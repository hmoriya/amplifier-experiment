# Phase 2: Value Definition - 価値定義

企業の価値創造を定義し、**業種・セグメント特性に応じた**Value Stages (VS0-VS7) を確立します。

**重要**: Value Stage は「組織の活動」ではなく「顧客の状態」を定義します。

## 使用方法

```bash
/parasol:2-value              # インタラクティブ選択（業種→セグメント→VS）
/parasol:2-value industry     # 業種・セグメント設定
/parasol:2-value VS0          # VS0を直接指定
/parasol:2-value VS2          # VS2を直接指定
/parasol:2-value all          # 全VS概要を生成
/parasol:2-value exploration  # 3軸探索モード（価値軸/事業部軸/ハイブリッド軸）
```

## 目的

- **業種・セグメントを特定する** - 適切なVSパターンの選択
- **Value Proposition（価値提案）を明確化する** - 誰に、何の価値を、どのように提供するか
- **バリューステージ (VS0-VS7) を定義する** - 顧客の状態遷移として価値創造の全体像を定義
- **Phase 3 Capabilitiesへの入力を準備する** - 必要なCapabilitiesの特定

## 📚 価値方法論リファレンス

**重要**: 本フェーズは**バックキャスティングアプローチ**を採用しています。

詳細な方法論は [_value-methodology.md](./_value-methodology.md) を参照してください。

### 方法論の全体フロー

```
価値定義（最上位価値）
    ↓ 価値分解
VL1 → VL2 → VL3
    ↓ バックキャスティング（理想状態から逆算）
価値マイルストーン（MS5→MS4→MS3→MS2→MS1）
    ↓ ステージ変換
バリューステージ（VS0-VS7）
    ↓ CL分解と統合（Phase 3で実施）
プロジェクトマイルストーン = 価値マイルストーン × コアサブドメイン
```

### 主要概念

| 概念 | 説明 |
|------|------|
| **バックキャスティング** | 理想の未来状態（MS5）から現在に向かって逆算するアプローチ |
| **価値分解（VL1→VL2→VL3）** | 最上位価値を階層的に分解し、実現方法を明確化 |
| **価値マイルストーン（MS1-MS5）** | 時間軸に沿った価値実現の段階（3ヶ月〜18ヶ月） |
| **バリューステージ（VS0-VS7）** | 最終価値提供者にとっての「価値ある状態」の定義 |

### バリューストリームの設計原則

#### ⚠️ 最重要原則：1価値定義 = 1バリューストリーム

**バリューストリームは価値定義からバックキャスティングして作成する「1本の統合されたストリーム」です。**

```
価値定義（VL3）
    ↓ バックキャスティング
┌─────────────────────────────────────────────────────┐
│      1本のバリューストリーム（VS0 → VS1 → ... → VS7）    │
│                                                     │
│  ※複数の並列ストリームではない                      │
│  ※VS0-VS7は1本のストリーム内のステージ              │
└─────────────────────────────────────────────────────┘
```

#### パターン別バリューストリーム数

| パターン | 価値定義 | バリューストリーム数 |
|---------|---------|-------------------|
| **統合型（価値軸）** | 1つの統合価値 | **1本** |
| **セグメント型（事業部軸）** | 事業部別価値 | **事業部数と同じ本数**（例：飲料1本、酒類1本） |
| **ハイブリッド型** | 統合 + セグメント | **統合1本 + セグメント数** |

#### アンチパターン（避けるべき設計）

| ❌ 誤った設計 | ✅ 正しい設計 |
|-------------|-------------|
| 機能別に複数VStr作成（価値創造VStr、価値提供VStr等） | 1本のVStrにVS0-VS7として統合 |
| 製品別にVStr作成（天然水VStr、茶VStr等） | 事業部レベル（飲料VStr）で1本 |
| バリューステージ（VS0-VS7）を並列ストリーム扱い | VSはストリーム内のステージ（直列） |
| **シナジー駆動VStr統合**（「クロスセルのため」でVStr統合） | 事業間シナジーはSupporting/Generic Layerで実現 |
| **コングロマリット単一VStr**（全事業を1つのVStrで表現） | 事業部ごとに独立VStrを設計（事業部軸） |
| **「シナジーVStr」の定義**（事業部間連携をVStrとして定義） | シナジーはBC間関係（Context Map）で表現 |

#### 正しい設計フロー

バリューストリーム自体も、**最終価値提供者（End Value Recipient）に対して価値ある状態を定義**し、バックキャスティングで組み立てます：

1. **最終価値提供者を特定** - 誰に価値を届けるか
2. **理想の最終状態を定義** - その人にとっての最終的な価値ある状態
3. **逆算してステージを構築** - VS7 ← VS6 ← ... ← VS1 ← VS0

### 🎯 V5特有機能: 設計ストーリー出力

**重要**: Parasol V5では、価値分解・バックキャスティングの各段階で**設計ストーリー（なぜそう設計したか）**を出力します。

#### 設計ストーリーの目的

- **理解促進**: なぜこの構造にしたかを明確にし、チーム全員の理解を深める

- **学習資産**: プロジェクト固有の知見をナレッジとして蓄積

#### 出力タイミング

| フェーズ | 出力内容 |
|----------|----------|
| **価値分解（VL1→VL2→VL3）** | なぜこのVL2グループ分けにしたか、なぜこのVL3詳細にしたか |
| **MSバックキャスティング** | なぜMS5をこの状態に設定したか、なぜこの時間軸で逆算したか |
| **MS→VS変換** | なぜ組織視点をこのVSに変換したか、なぜこの粒度にしたか |

## 🏗️ ステージ・ケーパビリティ・セグメント構造

### 構造的原則

**重要**: ステージとケーパビリティは「階層」ではなく「マッピング」の関係です。

```
価値軸（横）: 認知 → 検討 → 購入 → 使用 → 継続 → 推奨
              ↑
              │ 直交（異なる次元）
              ↓
事業軸（縦）: 事業A / 事業B / 事業C / ...
```

**解決**: 顧客セグメントを接点層として設計

```
┌─────────────────────────────────────────────────────────────────────┐
│                  市場接点層（Layer 3）                                │
│              顧客価値ステージ（共通フレーム）                          │
│     認知 → 検討 → 購入 → 初回体験 → 継続 → 推奨                      │
└─────────────────────────────────────────────────────────────────────┘
                              ↓ マッピング
┌─────────────────────────────────────────────────────────────────────┐
│                  顧客セグメント層（接点層）                           │
│   ┌─────────┬─────────┬─────────┬─────────┬─────────┐              │
│   │ セグメントA│ セグメントB│ セグメントC│ セグメントD│ セグメントE│              │
│   │ （事業A）│ （事業B）│ （事業C）│ （事業D）│ （事業E）│              │
│   └─────────┴─────────┴─────────┴─────────┴─────────┘              │
└─────────────────────────────────────────────────────────────────────┘
                              ↓ 実現
┌─────────────────────────────────────────────────────────────────────┐
│                  ケーパビリティ層（Layer 1-2）                        │
│   L1: 共通ケーパビリティ（事業横断・DX優先）                          │
│   L2: ドメインケーパビリティ（セグメント固有）                        │
│   L3: 製品ケーパビリティ（ブランド固有）                              │
└─────────────────────────────────────────────────────────────────────┘
```

### 3層ケーパビリティモデル

| 層 | 定義 | 粒度 | DX優先度 | 例 |
|---|---|---|---|---|
| **L1: 共通** | 事業横断で再利用可能 | 大 | 最高 | 顧客データ基盤、EC基盤、品質管理 |
| **L2: ドメイン** | 顧客セグメント固有 | 中 | 高 | 育児支援、シニア栄養管理 |
| **L3: 製品** | ブランド固有 | 小 | 中 | 個別レシピ提案、フレーバー開発 |

### スイムレーン設計

各セグメントごとにバリューステージをスイムレーンとして整理：

```
┌─────────────────────────────────────────────────────────────────────┐
│ スイムレーン: [セグメント名]バリューステージ                          │
├─────────┬─────────┬─────────┬─────────┬─────────┬─────────┤
│ Stage1  │ Stage2  │ Stage3  │ Stage4  │ Stage5  │ Stage6  │
│ [状態]  │ [状態]  │ [状態]  │ [状態]  │ [状態]  │ [状態]  │
├─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│[活動]   │[活動]   │[活動]   │[活動]   │[活動]   │[活動]   │
└─────────┴─────────┴─────────┴─────────┴─────────┴─────────┘
         ↓           ↓           ↓           ↓           ↓
      [Cap L?]    [Cap L?]    [Cap L?]    [Cap L?]    [Cap L?]
```

### ステージ×ケーパビリティ マッピング

ケーパビリティは**複数のステージに「寄与」**する：

```
              認知    検討    購入    使用    継続    推奨
          ┌───────┬───────┬───────┬───────┬───────┬───────┐
顧客DB    │   ○   │   ○   │   ◎   │   ◎   │   ◎   │   ◎   │ L1
          ├───────┼───────┼───────┼───────┼───────┼───────┤
EC基盤    │       │   ○   │   ◎   │       │   ◎   │       │ L1
          ├───────┼───────┼───────┼───────┼───────┼───────┤
CRM基盤   │       │       │   ○   │   ◎   │   ◎   │   ◎   │ L1
          └───────┴───────┴───────┴───────┴───────┴───────┘
          ◎: 強い寄与  ○: 寄与あり
```

### DX進化単位

**推奨**: L1ケーパビリティを優先（最大レバレッジ）

1. **L1優先**: 事業横断で共有されるケーパビリティをDX対象に
2. **L2次点**: セグメント固有だが再利用可能な部分
3. **L3最後**: ブランド固有の差別化要素

## 🔀 4軸探索モード（exploration）

複数の分解軸を並行探索し、最適なバリューストリーム構造を発見するモードです。

### 使用方法

```bash
/parasol:2-value exploration              # worktree作成 + 4軸探索開始
/parasol:2-value exploration --no-worktree # worktreeなし（フォルダ構造のみ）
/parasol:2-value exploration status        # 4軸の進捗確認
/parasol:2-value exploration merge         # 結果を本体にマージ
```

### worktree自動作成

`exploration` 実行時、以下のGitコマンドを内部で実行してworktreeを作成します：

### 4つの探索軸

| # | 軸 | VStr数 | 特徴 | 適用シナリオ |
|---|-----|--------|------|-------------|
| 1 | **価値軸** | 1本 | 統合・一貫 | 強力なブランド統一が必要な場合 |
| 2 | **事業部軸** | N本 | 独立・最適化 | 事業間の独立性が高い場合、コングロマリット |
| 3 | **フュージョン型** | 1+N本 | 共通基盤＋事業固有 | **顧客体験が事業横断で連続する場合のみ** |
| 4 | **ケーパビリティ統合型** | CL1判定後 | 効率化重視 | 無駄なシステム排除が必要な場合 |

### 探索プロセス

```

    │
    ├── 価値軸: 1本の統合VStr設計
    ├── 事業部軸: 事業別VStr設計

### 軸3: フュージョン型

**目的**: 顧客体験が事業を跨いで連続する場合の統合設計

> ⚠️ **シナジー≠VStr原則**: シナジー（コスト削減・効率化・クロスセル）はVStr統合の理由にならない。
> VStr統合は「顧客状態遷移が事業を跨いで連続する場合」のみ正当化される。

**フュージョン型選択前のチェック**:
- [ ] 顧客が事業を跨いで状態遷移するか？（例: 酒類購入者が飲料も購入する自然な行動）
- [ ] その遷移は顧客の意思か、企業の都合か？

❌ 企業の都合（シナジー追求）→ 事業部軸を維持、シナジーはSupporting/Generic Layerで実現
✅ 顧客の自然な行動 → フュージョン型検討可

**構造**:
```
統合VStr（HD統括）
    │
    ├── 共通基盤・研究開発の共有
    ├── ブランド価値の相互強化
    └── サステナビリティ指標の統一
    │
    ▼ 価値・機能の提供
    │
┌───┴───┬───────┬───────┐
▼       ▼       ▼       ▼
事業A   事業B   事業C   ...
VStr    VStr    VStr
```

**適用場面**:
- R&D成果の横展開
- ブランド価値の共有
- サステナビリティ目標の統一

### 軸4: ケーパビリティ統合型

**目的**: ケーパビリティ分析に基づく無駄排除・効率化

**設計原則**:
- 統合可能なケーパビリティを融合（共通システム化）
- 個別必須なケーパビリティを抽出（事業別保持）
- 無駄なシステム重複を回避

**Phase 3（CL1）での検討項目**:

| # | 検討項目 | 判定基準 |
|---|---------|---------|
| 1 | 各CL1ドメインの統合可能性判定 | 全事業で共通化できるか？ |
| 2 | 個別保持の必要性判定 | 事業特性で差別化が必要か？ |
| 3 | 部分統合の可能性 | 一部共通・一部個別のパターン |
| 4 | 統合時のシナジー効果 | コスト削減・品質向上等 |
| 5 | 個別保持時の差別化価値 | 事業競争力への貢献 |

**判定カテゴリ**:
```
A: 統合（共通システム化）
B: 個別（事業別システム）
C: ハイブリッド（基盤は共通、実装は個別）
```

**適用場面**:
- IT基盤・DX基盤の統合
- バックオフィス機能の共通化
- 物流・SCMの最適化

### zen-architect の活用（ケーパビリティ統合判定）

CL1分析時にケーパビリティ統合型の構造を判定するためのサブエージェント活用：

```

   - 共通化のデメリット（柔軟性低下、調整コスト）

### フュージョン型 vs ケーパビリティ統合型

| 観点 | フュージョン型 | ケーパビリティ統合型 |
|------|--------------|-------------------|
| **主目的** | 価値の相互強化 | コスト最適化・無駄排除 |
| **出発点** | VStr構造ありき | ケーパビリティ分析 |
| **設計タイミング** | Phase 2で設計 | Phase 3（CL1）で判定 |
| **適用場面** | ブランド・R&D連携 | IT基盤・バックオフィス |
| **複雑性** | 中 | 高（詳細分析必要） |

**組み合わせも可能**: フュージョン型で価値連携を設計しつつ、ケーパビリティ統合型で基盤効率化を図る

## Value Proposition 発見質問

VS個別定義時に使用する発見質問：

### Business Context 質問
1. このVSが必要な事業上の理由は何ですか？
2. 現在の課題や問題点は何ですか？
3. 市場や競合の状況はどうですか？

### Stakeholder Value 質問
1. 顧客にとっての価値は何ですか？（具体的なメリット）
2. 従業員にとっての価値は何ですか？
3. 株主・投資家にとっての価値は何ですか？
4. 社会にとっての価値は何ですか？

### Business Metrics 質問
1. 成功を測る主要指標（KPI）は何ですか？
2. 現在のベースライン値はありますか？
3. 目標値と達成期限はありますか？

### Strategic Importance 質問
1. このVSは競争優位性にどう貢献しますか？
2. コア/サポート/ジェネリックのどれに分類されますか？

2. 現在その能力はどの程度成熟していますか？

## Value Stages (VS0-VS7) について

**重要**: Value Stage は「組織が何をするか」ではなく「顧客がどの状態にあるか」を定義します。

### 顧客状態としてのVS定義

| VS | 顧客状態 | 顧客視点の定義 | 対応するEnterprise Activity（EA）例 |
|----|----------|----------------|-------------------------------------|
| **VS0** | 潜在ニーズ状態 | 「まだ気づいていないが、この価値を必要としている」 | 市場調査、ニーズ発掘 |
| **VS1** | 認知到達状態 | 「この価値の存在を知る」 | マーケティング、ブランディング |
| **VS2** | 選好形成状態 | 「この価値を選びたいと思う」 | 製品開発、差別化戦略 |
| **VS3** | 意思決定状態 | 「この価値を獲得することを決める」 | 販売、提案活動 |
| **VS4** | 価値獲得状態 | 「この価値を手に入れる」 | デリバリー、オペレーション |
| **VS5** | 価値実感状態 | 「この価値を実感している」 | カスタマーサクセス、サポート |
| **VS6** | 価値継続状態 | 「この価値を継続的に享受している」 | リテンション、ロイヤルティ施策 |
| **VS7** | 価値共創状態 | 「この価値を他者と共有・推奨している」 | コミュニティ、アドボカシー |

### 設計原則

1. **主語は常に「顧客」** - 組織ではなく顧客の状態変化を記述
2. **バックキャスティング** - VS7（理想状態）からVS0に向かって逆算設計
3. **EA（Enterprise Activity）との分離** - 組織活動はVSを実現する手段として別途定義

これらは業種を問わず適用可能な汎用的な顧客状態遷移フレームワークです。
プロジェクトの特性に応じて、各VSの具体的な顧客状態を定義してください。
