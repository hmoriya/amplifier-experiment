# 価値方法論リファレンス

**Parasol V5 - バックキャスティングと価値分解の方法論**

---

## 概要

このドキュメントは、Parasol V5における価値定義からバリューストリーム設計、プロジェクトマイルストーン作成までの方法論を定義します。V4の体系的なバックキャスティングアプローチを継承しています。

---

## Value Spaceの3軸モデル（WHY / HOW / WHEN）

Parasol V5のValue Spaceは、3つの直交軸で構成されます。これらの軸が統合されて、顧客価値を体系的に設計・実現します。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     Value Spaceの3軸モデル                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                              WHY                                            │
│                         顧客価値レベル                                      │
│                       (Value Level: VL)                                     │
│                              │                                              │
│                              │ 「何を届けるか」                             │
│                              ▼                                              │
│                        ┌─────────┐                                          │
│                        │   VL1   │ 最上位価値                               │
│                        │   VL2   │ 価値グループ                             │
│                        │   VL3   │ 詳細価値                                 │
│                        └─────────┘                                          │
│                              │                                              │
│          ┌───────────────────┴───────────────────┐                          │
│          │                                       │                          │
│          ▼                                       ▼                          │
│    ┌───────────┐                           ┌───────────┐                    │
│    │    HOW    │                           │   WHEN    │                    │
│    │ ストリーム │                           │ ステージ  │                    │
│    │  (VStr)   │                           │ (VS0-VS7) │                    │
│    └───────────┘                           └───────────┘                    │
│    「どう流すか」                          「いつ届けるか」                  │
│    価値の流れ全体                          顧客ジャーニー上の状態            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3軸の定義

| 軸 | 概念 | 略称 | 説明 | 主語 |
|----|------|------|------|------|
| **WHY** | Value Level | VL1-VL3 | 顧客に**何を**届けるか（価値の階層分解） | 組織 |
| **HOW** | Value Stream | VStr | 価値が**どう**流れるか（全体フロー） | 組織 |
| **WHEN** | Value Stage | VS0-VS7 | **いつ**価値を届けるか（顧客状態遷移） | **顧客** |

### 3軸の関係

```
WHY（価値レベル）: 「期待を超えるおいしさで、心身の潤いを届ける」
        │
        ├─── HOW（ストリーム）: VS0→VS1→VS2→...→VS7 への価値流通プロセス全体
        │
        └─── WHEN（ステージ）:
                VS0: 未認知状態        「顧客がこの価値を知らない」
                VS1: 認知状態          「顧客がこの価値を認識した」
                VS2: 選好形成状態      「顧客がこの価値を選びたいと思う」
                VS3: 初回体験状態      「顧客が初めてこの価値を体験した」
                ...
                VS7: 推奨状態          「顧客が他者にこの価値を推奨する」
```

### なぜ3軸が重要か

| 軸 | 欠けると何が起こるか |
|----|---------------------|
| WHY | 目的不明のプロジェクト（何のためにやっているか分からない） |
| HOW | 価値が顧客に届かない（プロセスが設計されていない） |
| WHEN | ビジネス変更が複数ドメインに波及（時間軸で分割されていない） |

**重要**: WHENは従来の設計手法（DDD等）で見落とされがちな軸。Parasol V5はWHENを第一分割軸として採用し、ビジネス変更とシステム変更の一致を実現する。

## なぜこの方法論なのか？（設計ストーリー）

### 従来アプローチの問題

多くのDXプロジェクトは**「できること」から始めて**失敗します：

```
従来の失敗パターン:
「AIを導入したい」→ 技術選定 → PoC → 「で、何に使うの？」
「DXを推進したい」→ ツール導入 → 「業務が変わらない」
```

**問題の本質**: 手段（技術）から始めると、目的（価値）が曖昧なまま進み、投資対効果が見えない。

### バックキャスティングという解法

本方法論は**「実現したい価値」から始めて逆算**します：

```
本方法論:
「顧客にとって価値ある状態とは？」→ その状態を実現するには？ → 必要な能力は？ → 必要な技術は？
```

**なぜ効くのか**:
1. **投資判断が明確**: 価値に直結する投資か否かが判断できる
2. **優先順位が決まる**: 価値への貢献度で優先順位が決まる
3. **成功が測定可能**: 価値の実現度で成功を測定できる

### Vision/Mission/Core Valuesとパラソル価値（VL）の違い

**多くの組織が混同する2つの概念を明確に区別します。**

```
┌────────────────────────────────────────────────────────────────┐
│  企業価値体系              │  パラソル価値（VL）               │
│  (Vision/Mission/Values)  │  (Value Level)                   │
├────────────────────────────────────────────────────────────────┤
│  【HOWの規定】             │  【WHATの規定】                   │
│  どのように振る舞うか       │  何を顧客に届けるか               │
├────────────────────────────────────────────────────────────────┤
│  安定した錨（10年以上）     │  深化する理解（顧客理解と共に進化）│
├────────────────────────────────────────────────────────────────┤
│  組織のアイデンティティ     │  顧客への価値提案                 │
└────────────────────────────────────────────────────────────────┘
```

**企業価値体系（Vision/Mission/Core Values）— HOWの規定**

これらは「**組織としてどのように振る舞うか**」を規定する「安定した錨」です：
- **Vision**: 組織が目指す理想の姿（10-20年スパン）
- **Mission**: 社会における存在意義と役割
- **Core Values**: 判断と行動の基準となる価値観

**パラソル価値（VL1→VL2→VL3）— WHATの規定**

これらは「**顧客に何を届けるか**」を規定し、顧客理解の深化により**進化（深化）**します：
- **VL1**: 最上位の顧客価値（経営レベルの意思決定基準）
- **VL2**: 価値グループ（部門レベルの投資配分基準）
- **VL3**: 詳細価値（チームレベルの行動計画基準）

**重要な違い**:
- Vision/Missionが**変わる**と、それは組織の根本的な変革を意味する
- VLが**深化する**のは、顧客理解が成熟した証拠であり、健全な進化である

```
例: 飲料メーカーの場合

【安定した錨】
Vision: 「世界中の人々の潤いと健康に貢献する」
Mission: 「最高品質の飲料を通じて、日常に喜びを届ける」
Core Values: 「品質第一」「顧客尊重」「持続可能性」
→ 10年以上変わらない組織のアイデンティティ

【深化する理解】
VL1（当初）: 「おいしさで選ばれる」
VL1（深化後）: 「期待を超えるおいしさで、心身の潤いを届ける」
→ 顧客理解が深まり、提供価値の解像度が上がった
```

### 価値分解（VL1→VL2→VL3）の意味

**なぜ階層化するのか？**

```
VL1「期待を超えるおいしさ」← 抽象的すぎて行動に落ちない
    ↓ 分解
VL2「製品イノベーション価値」← 方向性は見えるが具体性不足
    ↓ 分解
VL3「発酵技術による味の革新」← 具体的な行動と投資判断が可能
```

**各レベルの役割**:
- **VL1（最上位価値）**: 経営レベルの意思決定基準
- **VL2（価値グループ）**: 部門レベルの投資配分基準
- **VL3（詳細価値）**: チームレベルの行動計画基準

### 価値の深化（Value Deepening）

**VLは「修正」されるのではなく「深化」します。**

```
【深化と修正の違い】

修正（Correction）:
  VL1（旧）: 「安い製品を提供」
  VL1（新）: 「高品質な製品を提供」
  → 方向性の転換、過去の否定

深化（Deepening）:
  VL1（当初）: 「おいしさで選ばれる」
  VL1（深化）: 「期待を超えるおいしさで、心身の潤いを届ける」
  → 同じ方向で解像度が上がる、過去の発展
```

**深化が起こる要因**:

1. **顧客理解の成熟**: 顧客との対話、フィードバックから本質的ニーズが見える
2. **市場での検証**: 仮説が検証され、より精緻な理解に到達
3. **実践からの学び**: 価値提供の経験から、より深い洞察を得る
4. **競争環境の変化**: 競合との差別化ポイントがより明確に

**深化のサイン**:

| サイン | 例 | 意味 |
|--------|-----|------|
| 解像度の向上 | 「便利」→「時間を節約し、大切な人と過ごせる」 | 抽象から具体へ |
| 感情価値の追加 | 「機能的価値」→「感情的価値も含む」 | 機能から体験へ |
| 本質への接近 | 「製品の良さ」→「顧客の人生への貢献」 | 手段から目的へ |

**深化を促進する活動**:

- 顧客インタビュー、エスノグラフィ調査
- NPS・顧客満足度調査の深掘り分析
- 成功事例・失敗事例のパターン分析
- 競合との差別化ポイントの再評価
- 年次レビューでのVL振り返り

**注意**: 深化は**健全な成長**であり、頻繁なVL変更とは異なります。年次レビューでの検討が適切であり、四半期ごとに変えるものではありません。

### バックキャスティング（VMS5→VMS1）の意味

**なぜ未来から逆算するのか？**

```
フォアキャスティング（現在→未来）:
「今できることは何か？」→ 少しずつ改善 → どこに着くか分からない

バックキャスティング（未来→現在）:
「18ヶ月後の理想は？」→ そこに到達するには？ → 今何をすべきか？
```

**逆算の効果**:
- **ゴールが明確**: 「何を達成したら成功か」が最初に定義される
- **ギャップが見える**: 理想と現実の差が明確になる
- **必要条件が導出**: 「〜が必要」という論理的な導出が可能

### VMS→VS変換の意味

**なぜ価値マイルストーン（VMS）をバリューステージ（VS）に変換するのか？**

```
VMS（組織視点）: 「我々が達成すべき状態」
    ↓ 視点変換
VS（顧客視点）: 「顧客にとって価値ある状態」
```

**変換の意図**:
- **VMSは組織の進捗指標**: 「研究基盤が整った」「サービスが展開された」
- **VSは顧客の価値指標**: 「新製品を選べるようになった」「便利に取引できる」

**なぜ両方必要か**:
- VMSだけだと「組織の自己満足」になりがち
- VSだけだと「実行計画」が立てられない
- 両方を対応づけることで「顧客価値を軸にした実行計画」が可能

## Value Stage（VS）の本質的定義

### 用語の明確化: Value Stage vs Value Stream

| 用語 | 略称 | 定義 | 例 |
|------|------|------|-----|
| **Value Stage** | VS | 顧客の「状態」（個々のステージ） | VS0, VS1, VS2...VS7 |
| **Value Stream** | VStr | 価値の「流れ」（VS0→VS7の遷移プロセス全体） | 全体のフロー |

```
Value Stream（VStr）= VS0からVS7への価値の流れ全体
┌──────────────────────────────────────────────────────────────────┐
│  VS0  →  VS1  →  VS2  →  VS3  →  VS4  →  VS5  →  VS6  →  VS7   │
│  ↑       ↑       ↑       ↑       ↑       ↑       ↑       ↑      │
│  └───────┴───────┴───────┴───────┴───────┴───────┴───────┘      │
│         各々が Value Stage（VS）= 顧客状態                        │
└──────────────────────────────────────────────────────────────────┘
```

**使い分け**:
- 「VS2を設計する」= 選好形成**ステージ**の詳細を設計
- 「VStrを設計する」= 顧客状態遷移の**流れ全体**を設計

### 顧客状態としてのValue Stage

**定義**: Value Stage（価値ステージ）とは、顧客が価値を認識・獲得・享受する過程における「**状態**」の定義です。組織の「活動」ではなく、**顧客にとっての「状態変化」**を記述します。

```
【核心的な視点転換】

誤った理解（組織活動視点）:
  「VS2: 製品開発」 → 主語は組織、「我々が開発する」

正しい理解（顧客状態視点）:
  「VS2: 選好形成状態」 → 主語は顧客、「顧客がこの価値を選びたいと思う」
```

| 観点 | 誤った理解 | 正しい理解 |
|------|-----------|-----------|
| **主語** | 組織 | **顧客** |
| **記述対象** | 活動（何をするか） | **状態（どうなるか）** |
| **視点方向** | 組織→顧客（フォワード） | **顧客←理想（バックキャスト）** |
| **問い** | 「我々は何をするか」 | **「顧客は何を得るか」** |

### Value Stage と Capability の関係

パラソルでは、顧客状態（VS）から組織能力（Capability）を導出します。

**概念の整理**:

| 概念 | 定義 | 主語 | Phase |
|------|------|------|-------|
| **Value Stage (VS)** | 顧客の状態 | 顧客 | Phase 2 |
| **Activity Area (AA)** | 活動領域識別（Core/Supporting/Generic**傾向**） | 組織 | Phase 3 CL1 |
| **Capability (Cap)** | 能力の単位（**正式分類**・投資判断・サービス境界） | 組織 | Phase 3 CL2 |
| **Operation (Op)** | 具体的な業務活動 | 組織 | Phase 3 CL3 |

```
【構造的関係】

Value Stage（顧客状態）
    ↓ 「この状態遷移を実現するために必要な能力は？」
Capability Domain（能力ドメイン）
    ↓ 「この能力領域をどう組織するか？」
Capability（ケーパビリティ）
    ↓ 「この能力を発現する具体的な活動は？」
Operation（オペレーション）= 組織の具体的活動
```

**重要**: 「組織活動（Enterprise Activity）」は独立した概念ではなく、**Capabilityを分解するとOperationとして導出**されます。

### DDDとParasolの対応関係

ParasolのCapability階層はDDD（ドメイン駆動設計）の概念と対応しています。

#### DDDの問題空間と解決空間

```
┌───────────────────────────────────────────────────────┐
│ 問題空間（Problem Space）                             │
│                                                       │
│  [問題領域] ──分割──→ [サブドメイン群]                │
│   Domain    （発見）   Core / Supporting / Generic   │
└───────────────────────────────────────────────────────┘
                          │
                          │ 分析・設計判断
                          ↓
┌───────────────────────────────────────────────────────┐
│ 解決空間（Solution Space）                            │
│                                                       │
│  [サブドメイン群] ──対応づけ──→ [Bounded Context群]   │
│                    （設計）      Context Map          │
│                                                       │
│  [BC群] ──実装──→ [サービス/モジュール]               │
└───────────────────────────────────────────────────────┘
```

#### Parasol V5のマッピング

```
┌───────────────────────────────────────────────────────┐
│ 問題空間（Problem Space）                   Phase 3   │
│                                                       │
│  [CL1 活動領域] ──分割──→ [CL2 ケイパビリティ]        │
│   傾向的分類    （発見）   Core/Supporting/Generic    │
│                            （正式分類・投資判断）     │
│                                  │                    │
│                                  ▼                    │
│                           [CL3 業務オペレーション]    │
│                            （詳細化・網羅性確保）     │
└───────────────────────────────────────────────────────┘
                          │
                          │ 分析・設計判断
                          ↓
┌───────────────────────────────────────────────────────┐
│ 解決空間（Solution Space）                Phase 4-5   │
│                                                       │
│  [CL3] ──対応づけ──→ [Bounded Context群]              │
│         （BC境界確定）  Context Map                   │
│                                                       │
│  [BC群] ──実装設計──→ [API / Domain Model / DB]       │
└───────────────────────────────────────────────────────┘
```

#### 対応表

| Parasol | Phase | DDD概念 | 性質 |
|---------|-------|---------|------|
| **CL1** 活動領域 | 3 | **Problem Space** | 傾向的分類（参考情報） |
| **CL2** ケイパビリティ | 3 | **Subdomain** | 正式分類（Core/Supporting/Generic） |
| **CL3** 業務オペレーション | 3 | **Domain Model詳細** | 問題領域の詳細化 |
| **BC境界** | 4 | **Bounded Context** | 解決領域への移行 |

#### 重要な違い

| 観点 | DDD | Parasol |
|------|-----|---------|
| 分類タイミング | Subdomain識別時に即座にCore/Supporting/Generic | CL1は傾向的、CL2で正式分類 |
| 分類基準 | 技術者視点（複雑さ、差別化） | ビジネス視点（顧客接点、投資判断） |
| BCへの流れ | Subdomain → BC（直接） | CL2 → CL3 → BC（業務詳細化を経由） |

```
【例：VS2（選好形成状態）の場合】

VS2「顧客がこの価値を選びたいと思う」
    ↓
CD: ブランド構築（Core）、製品開発（Core）
    ↓
Cap: brand-building, product-development
    ↓
Op: ブランドストーリー策定、製品差別化設計、品質設計...
    ↑
    これが「組織活動」= Capabilityから導出される
```

### 領域別工程フロー（ドメイン分解からサービス定義まで）

**重要**: 各活動領域（ドメイン）は以下の工程を順次辿り、最終的にサービスとして定義されます。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│ 領域別工程フロー（1つのドメインが辿る完全な流れ）                                │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  Phase 3a: CL1 活動領域識別                                                    │
│  ┌─────────────────────────────────────────────────────────────┐               │
│  │ 問題空間（Problem Space）開始                                │               │
│  │                                                             │               │
│  │ • 傾向的分類（Core傾向 / Supporting傾向 / Generic傾向）      │               │
│  │ • 投資方向性の初期判断（参考情報）                           │               │
│  │ • 例: 「基盤技術研究」→ Core傾向                            │               │
│  └─────────────────────────────────────────────────────────────┘               │
│                            │                                                   │
│                            ▼                                                   │
│  Phase 3b: CL2 ケイパビリティ設計 ★正式分類                                    │
│  ┌─────────────────────────────────────────────────────────────┐               │
│  │ サブドメイン分解                                             │               │
│  │                                                             │               │
│  │ • 正式分類（Core / Supporting / Generic）決定               │               │
│  │ • 投資判断の根拠明示                                         │               │
│  │ • サービス境界候補の識別                                     │               │
│  │ • 例: fermentation-research（Core）、ingredient-research（Core）│             │
│  └─────────────────────────────────────────────────────────────┘               │
│                            │                                                   │
│                            ▼                                                   │

### 内部活動（社内向けアクティビティ）の位置づけ

**課題**: バリューストリーム（VStr）を顧客中心で設計すると、顧客に直接関係しない内部活動（IT運用、HR、経理、内部ツール、セキュリティ等）が見えなくなる。

**解決**: Capability Domain分類（Core/Supporting/Generic）が内部活動を捕捉します。

    ┌───────────────────────┼───────────────────────┐
    │                       │                       │
    ▼                       ▼                       ▼
┌─────────┐           ┌─────────┐           ┌─────────┐
│  Core   │           │Supporting│           │ Generic │
│  CD     │           │  CD     │           │   CD    │
├─────────┤           ├─────────┤           ├─────────┤
│特定のVS │           │VStr全体 │           │組織全体 │
│に紐付く │           │を支える │           │の基盤  │
├─────────┤           ├─────────┤           ├─────────┤

```

**Capability Domain別の特性**:

| 分類 | 顧客接点 | VS紐付け | 例 | 設計時の扱い |
|------|---------|---------|-----|------------|
| **Core** | 直接 | 特定VSに紐付く | 製品開発、販売、顧客サービス | VS遷移から導出 |
| **Supporting** | 間接 | VStr全体を支援 | データ基盤、物流、分析 | Core要件から導出 |
| **Generic** | なし | 組織全体の基盤 | HR、経理、IT運用、セキュリティ | 組織要件から導出 |

**設計順序**:

```
1. VStr設計（顧客状態遷移）
    ↓
2. Core CD導出（どの能力がVS遷移を実現するか？）
    ↓
3. Supporting CD導出（Coreを支えるために何が必要か？）
    ↓
4. Generic CD確認（組織運営に必要な基盤は？）
```

**注意**: SupportingとGenericは「顧客状態」からは直接見えませんが、**Capabilityを実装する段階で必ず必要になる**ため、Phase 3のCL1分類で明示的に洗い出します。

### VS-Swimlane モデル：時間文脈としてのValue Stage

#### 従来のスイムレーン問題と解決

従来のスイムレーンは「アクター/責任者」を表現しますが、VSは「顧客状態」であり意味論的に不一致でした。

**解決**: VSを「時間文脈を持つオペレーションフェーズ」として再解釈します。

```
【VS-Swimlane モデル】

VSは単なる「顧客状態」ではなく「時間文脈」を提供する

┌─────────────────────────────────────────────────────────────────────────┐
│                          時間軸（Time）→                                │
├──────────┬──────────┬──────────┬──────────┬──────────┬─────────────────┤
│   VS1    │   VS2    │   VS3    │   VS4    │   VS5    │      VS6        │
│ 認知形成 │ 選好形成 │ 意思決定 │ 価値獲得 │ 価値実感 │    価値継続     │
├──────────┼──────────┼──────────┼──────────┼──────────┼─────────────────┤
│          │  Core:   │  Core:   │  Core:   │          │                 │
│          │ 製品開発 │ セールス │ 出荷管理 │          │                 │
│          │    │     │    ↑     │    ↑     │          │                 │
│          │    └─────┼────┘     │    │     │          │                 │
│          │       参照（依存）   │ 参照    │          │                 │
├──────────┼──────────┼──────────┼──────────┼──────────┼─────────────────┤
│Supporting│Supporting│Supporting│Supporting│Supporting│   Supporting    │
│ データ   │ データ   │ データ   │ データ   │ データ   │    データ       │
│ 基盤v1  │ 基盤v1  │ 基盤v2  │ 基盤v2  │ 基盤v3  │    基盤v3      │
│   └──────┴────┬─────┴────┬─────┴────┬─────┴────┬─────┘                │
│               進化       進化       進化       進化                     │
└─────────────────────────────────────────────────────────────────────────┘
```

#### VS-Swimlane の3つの特性

| 特性 | 説明 | 例 |
|------|------|-----|
| **時間性** | VSは時間文脈を提供し、オペレーションは時間に沿って配置される | VS2のオペレーションはVS4より先に実行 |
| **文脈最適化** | 同じ抽象Capability（例：データ基盤）でも、VS文脈で具体Operationが異なる | VS2向けデータ基盤 vs VS5向けデータ基盤 |
| **参照可能性** | 先行VSで確立したCapabilityを後続VSが参照・再利用可能 | VS2の製品マスタをVS4の出荷管理が参照 |

#### 概念の階層関係

```
【Capability vs Operation @ VS の関係】

Capability（抽象能力）
    │
    │ 時間非依存の「できること」
    │ 例：データ基盤、製品開発、顧客サービス
    │
    └──→ Operation @ VS（具体活動）
              │
              │ 時間文脈依存の「このフェーズですること」
              │ 例：VS2向けデータ基盤運用、VS5向け問合せ対応
              │
              └──→ 先行VSへの参照
                        │
                        │ 「VS2で確立した製品マスタを参照」
                        │ 「VS3で構築した顧客DBを利用」
```

#### VS-Swimlane の設計メリット

1. **チーム編成の最適化**
   - VSフェーズごとにチームの責任範囲を明確化
   - 時間軸に沿った引き継ぎポイントの特定

2. **デプロイ単位の分離**
   - VSフェーズごとに独立したデプロイ可能
   - 先行VSの安定性を後続VSが前提にできる

3. **段階的な能力構築**
   - 先行VSで基盤を確立し、後続VSで参照
   - 同じCapabilityの「進化」を時間軸で表現

#### VS-Swimlane と Capability Domain の統合

```
【統合モデル】

              VS1      VS2      VS3      VS4      VS5      VS6
              │        │        │        │        │        │
Core     ────┼────────●────────●────────●────────┼────────┼────
              │     製品開発  セールス  出荷管理  │        │
              │        │   ↑    │   ↑    │        │        │
              │        └───┼────┘   │    │        │        │
              │         参照      参照    │        │        │
              │                          │        │        │
Support  ────●────────●────────●────────●────────●────────●────
              │   データ基盤（各VSで最適化されたOperation）  │
              │   v1 ──→ v1 ──→ v2 ──→ v2 ──→ v3 ──→ v3  │
              │                                            │
Generic  ────●────────●────────●────────●────────●────────●────
              │        IT基盤・セキュリティ（全VS横断）      │
```

- **Core**: 特定VSに紐付き、VS間で参照関係を持つ
- **Supporting**: 全VSを支援しつつ、VS文脈で最適化（進化）
- **Generic**: 全VS横断で一貫した基盤を提供

#### 航海図がないとどうなるか

- ❌ 部門ごとに別々の地図を持って航海する
- ❌ 「良い仕事をした」が価値に繋がらない
- ❌ 手戻り・重複・抜け漏れが発生する
- ❌ 全体最適ではなく部分最適に陥る

> **「航海図なき航海は、ただの漂流である」**

### なぜ顧客状態視点が重要か

**バックキャスティングの本質**:
- 「顧客にとっての理想状態」から逆算して設計する
- 組織活動はその状態を実現する「手段」として導出される
- 手段から始めると目的が曖昧になる（従来のDX失敗パターン）

```
パラソルの設計順序:
1. 顧客にとっての理想状態（VS7）を定義
2. その状態に至るまでの顧客の状態変遷（VS6→VS5→...→VS0）を設計
3. 各状態を実現するために必要な組織能力（Capability）を導出
4. Capabilityを実装するサービス・システムを設計
```

## Phase 1: 価値宣言と価値分解

### 1.1 最上位価値の定義（VL1）

まず組織が提供する最上位の価値を宣言します。

```yaml
価値宣言:
  ビジョン: [組織が目指す理想の姿]

  提供価値:
    顧客への価値: [顧客が得る最終的な価値]
    社会への価値: [社会的インパクト]
    組織への価値: [組織が得る価値]

  価値の独自性:
    差別化要因: [他社にない独自価値]
    競争優位性: [持続可能な優位性]
    価値の持続性: [長期的な価値提供力]
```

### 1.2 価値分解（VL1→VL2→VL3）

最上位価値を階層的に分解します。

```yaml
VL1: 最上位価値
  定義: [組織全体の提供価値]
  測定指標: [成功の測定方法]

VL2: 価値グループ（3-5個）
  VL2-1: [価値グループ1]
    定義: [詳細な定義]
    VL1への貢献: [どのようにVL1に貢献するか]

  VL2-2: [価値グループ2]
    定義: [詳細な定義]
    VL1への貢献: [どのようにVL1に貢献するか]

  VL2-3: [価値グループ3]
    定義: [詳細な定義]
    VL1への貢献: [どのようにVL1に貢献するか]

VL3: 詳細価値（各VL2に3-5個）
  VL3-1-1: [詳細価値]
    定義: [具体的な価値]
    実現方法: [どのように実現するか]
    必要な活動: [必要な企業活動]
```

**分解のポイント**:
- VL2は3-5個が適切（多すぎると焦点がぼやける）
- VL3は実現可能な粒度まで分解
- 各レベルでVL1への貢献を明確化

## Phase 2: バックキャスティングによる価値マイルストーン

### 2.1 バックキャスティングとは

**未来の理想状態から現在に向かって逆算**するアプローチです。

```
従来のアプローチ（フォアキャスティング）:
現在 → 少し先 → もう少し先 → 未来（どこに着くか分からない）

バックキャスティング:
理想の未来（VMS5）← 逆算 ← 逆算 ← 逆算 ← 現在（VMS1）
```

### 2.2 価値マイルストーンの設計

**理想状態（VMS5）から現在（VMS1）へ逆算**します。

```yaml
VMS5: 顧客が期待を超える価値を日常的に享受している状態（18ヶ月後）
  状態: [理想の実現状態]
  成功基準: [測定可能な基準]
  価値指標: [具体的な数値目標]
  # ここから逆算を開始

VMS4: 顧客が全面的に価値を実感している状態（12ヶ月後）
  状態: [最適化された状態]
  成功基準: [測定可能な基準]
  VMS5への移行: [VMS5に到達するために必要な進化]

VMS3: 顧客が主要価値を日常的に体験できる状態（9ヶ月後）
  状態: [成熟した状態]
  成功基準: [測定可能な基準]
  VMS4への移行: [VMS4に到達するために必要な改善]

VMS2: 顧客が価値を認識し選択できる状態（6ヶ月後）
  状態: [展開された状態]
  成功基準: [測定可能な基準]
  VMS3への移行: [VMS3に到達するために必要な拡大]

VMS1: 顧客が最初の価値を体験できる状態（3ヶ月後）
  状態: [基礎が確立した状態]
  成功基準: [測定可能な基準]
  VMS2への移行: [VMS2に到達するために必要な発展]
```

**設計順序**（重要）:
1. VMS5（理想状態）を先に定義
2. VMS4を「VMS5に到達するには何が必要か」で定義
3. VMS3を「VMS4に到達するには何が必要か」で定義
4. VMS2を「VMS3に到達するには何が必要か」で定義
5. VMS1を「VMS2に到達するには何が必要か」で定義

### 2.3 VMS-OKR連携による戦略的目標管理

**VMSとOKRを統合して、価値実現を定量的に測定・管理します。**

```
価値トレーサビリティチェーン:

VL1（最上位価値）
  ├─ VL2 → VL3（詳細価値）
  │           └─ VMS（価値マイルストーン）
  │                 └─ OKR（測定層）
  │                       ├─ Objective: VMSの達成状態
  │                       └─ Key Results: 定量的な達成指標
  ↓
VS（バリューストリーム）→ CL1 → CL2 → CL3 → 実装
```

**OKRの位置づけ**:
- **Objective（目標）**: VMSで定義した「顧客価値状態」を目標として設定
- **Key Results（主要な結果）**: その状態を測定する定量的指標

**VMS-OKR対応例**:

```yaml
VMS2: 顧客が価値を認識し選択できる状態

OKR:
  Objective: "顧客が当社製品を「選びたい」と感じている"

  Key Results:
    - KR1: NPS（推奨度）が+30以上
    - KR2: 認知率が対象セグメントで60%以上
    - KR3: 購入意向率が認知者の40%以上
    - KR4: 競合比較で「選びたい」が1位

  測定サイクル: 四半期
  責任者: マーケティング本部長
```

**なぜVMS-OKR連携が重要か**:

1. **価値の測定可能化**: 抽象的な価値を具体的な数値で追跡
2. **組織横断の整合**: 各部門のOKRがVMSに紐づき、価値実現に向けて整合
3. **継続的な改善**: 四半期ごとのレビューで価値実現の進捗を確認
4. **投資判断の根拠**: KR達成状況に基づく合理的な資源配分

### 2.4 4段階戦略レビューサイクル

**価値の実現と深化を継続的に管理するためのレビュー体制です。**

| サイクル | 対象 | 頻度 | 主な活動 |
|---------|------|------|----------|
| **四半期レビュー** | KR進捗・VMS達成 | 3ヶ月 | OKR達成状況確認、戦術調整 |
| **年次レビュー** | VL2/VL3深化・OKR再設定 | 1年 | 価値理解の更新、年間目標設定 |
| **中期レビュー** | VL1・VS構造見直し | 3-5年 | Value Stream再構築検討 |
| **長期レビュー** | Vision/Mission対応 | 5-10年 | 組織価値体系との再整合 |

```
【レビューサイクルの関係】

長期レビュー（5-10年）─────────────────────────────────────────┐
  │ Vision/Missionとの整合性確認                               │
  ↓                                                           │
中期レビュー（3-5年）───────────────────────────────┐          │
  │ VL1・VS構造の妥当性検証                         │          │
  ↓                                                │          │
年次レビュー（1年）───────────────────┐            │          │
  │ VL2/VL3の深化、OKR再設定           │            │          │
  ↓                                   │            │          │
四半期レビュー─┬─┬─┬─┬────────────────┴────────────┴──────────┘
              Q1 Q2 Q3 Q4
              │  │  │  │
              KR進捗確認・戦術調整
```

**レビューのポイント**:

- **四半期**: 「KRは達成できているか？」「戦術の修正は必要か？」
- **年次**: 「VLの理解は深まったか？」「来年のOKRは何か？」
- **中期**: 「顧客の状態遷移（VS）は適切か？」「新しいVSが必要か？」
- **長期**: 「Vision/Missionは時代に合っているか？」「根本的な再定義が必要か？」

## Phase 3: バリューストリームとバリューステージ

### 3.1 バリューストリームのバックキャスティング

#### ⚠️ 最重要原則：1価値定義 = 1バリューストリーム

**バリューストリームは「1本の統合されたストリーム」として設計します。**

```
【正しい設計】
価値定義（VL3）
    ↓ バックキャスティング
┌─────────────────────────────────────────────────────┐
│   1本のバリューストリーム                            │
│   VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7    │
│                                                     │
│   ※VS0-VS7はストリーム内の「ステージ」（直列）      │
│   ※複数の並列ストリームではない                    │
└─────────────────────────────────────────────────────┘

【誤った設計（アンチパターン）】
×  VStr-1: 価値創造ストリーム
×  VStr-2: 価値提供ストリーム  ← 機能別に分割するのは誤り
×  VStr-3: 価値持続ストリーム
```

#### パターン別バリューストリーム数

| パターン | 価値定義 | VStr数 | 例 |
|---------|---------|--------|-----|
| 統合型 | 1つの統合価値 | 1本 | 全社共通VStr |
| セグメント型 | 事業部別価値 | 事業部数 | 飲料VStr、酒類VStr、その他VStr |
| ハイブリッド型 | 統合 + セグメント | 1 + N | 統合VStr + 事業部別VStr |

バリューストリーム自体も、最終価値提供者に対して**価値ある状態を定義**してバックキャスティングで組み立てます。

```
最終価値提供者（End Value Recipient）
    ↓ 「この人にとって価値ある状態とは？」
価値ある最終状態の定義
    ↓ バックキャスティング
VS7 ← VS6 ← VS5 ← VS4 ← VS3 ← VS2 ← VS1 ← VS0
```

### 3.2 VMS→VS変換

価値マイルストーンをバリューステージに変換します。

**重要**: VSは「顧客の状態」として定義します。主語は常に顧客です。

```yaml
# 標準型 VS0-VS7（顧客状態ステージ）

VS0: 潜在ニーズ状態
  顧客状態: 「まだ気づいていないが、この価値を必要としている」
  対応VMS: 全VMSの前提
  組織活動（参考）: 戦略策定、市場理解
  分類: Generic

VS1: 認知到達状態
  顧客状態: 「この価値の存在を知った」
  対応VMS: VMS1前半
  組織活動（参考）: 認知形成、情報発信
  分類: Supporting

VS2: 選好形成状態
  顧客状態: 「この価値を選びたいと思う」
  対応VMS: VMS1後半-VMS2前半
  組織活動（参考）: 製品開発、差別化
  分類: Core

VS3: 意思決定状態
  顧客状態: 「この価値を得ることを決めた」
  対応VMS: VMS2後半-VMS3前半
  組織活動（参考）: マーケティング、販売促進
  分類: Core

VS4: 価値獲得状態
  顧客状態: 「価値を手に入れた」
  対応VMS: VMS3後半-VMS4前半
  組織活動（参考）: 流通、配送、提供
  分類: Supporting

VS5: 価値実感状態
  顧客状態: 「価値を体験し、満足している」
  対応VMS: VMS4後半
  組織活動（参考）: 品質保証、サポート
  分類: Supporting

VS6: 価値継続状態
  顧客状態: 「この価値が日常の一部になった」
  対応VMS: VMS5前半
  組織活動（参考）: 関係維持、リテンション
  分類: Generic

VS7: 価値共創状態
  顧客状態: 「価値を広め、新たな価値を共に創る」
  対応VMS: VMS5後半
  組織活動（参考）: コミュニティ、イノベーション
  分類: Generic
```

#### 顧客状態の記述ルール

| ルール | 説明 | 例 |
|--------|------|-----|
| **主語は顧客** | 組織ではなく顧客を主語にする | ○「顧客が知る」 ×「我々が伝える」 |
| **状態を記述** | 活動ではなく状態を記述 | ○「満足している」 ×「サポートする」 |
| **価値視点** | 機能ではなく価値で記述 | ○「潤いを得る」 ×「水を飲む」 |

### 3.3 ステージ設計のポイント

各ステージは**最終価値提供者にとっての「価値ある状態」**として定義します：

1. **誰に対して？** → 最終価値提供者（End Value Recipient）
2. **何が価値？** → その人にとっての価値ある状態
3. **どう測定？** → 状態を示す成功基準

```yaml
# 例: 製品開発Value Streamの場合
最終価値提供者: 消費者
価値ある状態の定義:
  VS7: 消費者が期待を超える製品を継続的に受け取れる状態
  VS6: 消費者のニーズに完全に応える製品が提供される状態
  VS5: 安定した品質の製品が提供される状態
  VS4: 多様な製品ラインナップから選択できる状態
  VS3: 初めての製品体験で価値を感じられる状態
  VS2: 製品の価値を理解できる状態
  VS1: 製品の存在を知る状態
  VS0: 製品にアクセスできる基盤がある状態
```

## Phase 4: プロジェクトマイルストーンの作成

### 4.1 統合の公式

**プロジェクトマイルストーン = 価値マイルストーン × コアサブドメイン**

```
価値マイルストーン（VMS1-VMS5）
        ×
コアサブドメイン（CL1→CL2→CL3で特定）
        ↓
プロジェクトマイルストーン（実行計画）
```

### 4.2 作成手順

**Step 1: コアサブドメインの特定**

Phase 3（ケーパビリティ分解）で特定したCoreドメインから：

```yaml
コアサブドメイン例:
  - fermentation-research（発酵研究）
  - ingredient-research（素材研究）
  - premium-beer-development（プレミアムビール開発）
```

**Step 2: 価値マイルストーンとの交差**

各価値マイルストーン（VMS1-VMS5）に対して、コアサブドメインごとの具体的な達成目標を定義：

```yaml
PM1: プロジェクトマイルストーン1（3ヶ月）
  対応VMS: VMS1（顧客が最初の価値を体験できる状態）

  fermentation-research:
    達成目標: 酵母データベースの基盤構築
    成果物: 酵母株100種の登録完了

  ingredient-research:
    達成目標: 素材評価フレームワークの確立
    成果物: 評価基準書の策定完了

  premium-beer-development:
    達成目標: 製品開発プロセスの標準化
    成果物: 開発ガイドライン策定完了

PM2: プロジェクトマイルストーン2（6ヶ月）
  対応VMS: VMS2（顧客が価値を認識し選択できる状態）

  fermentation-research:
    達成目標: 発酵最適化アルゴリズムの実装
    成果物: 最適化ツールv1.0リリース

  # ... 以下同様
```

**Step 3: 依存関係の整理**

```yaml
依存関係マトリクス:
  PM1:
    fermentation-research → ingredient-research: 酵母特性データ提供
    ingredient-research → premium-beer-development: 素材選定基準提供

  PM2:
    前提: PM1の全目標達成
    fermentation-research → premium-beer-development: 発酵レシピ提供
```

### 4.3 プロジェクトマイルストーンテンプレート

```yaml
プロジェクトマイルストーン:
  PM{N}:
    期間: {開始日} - {終了日}
    対応価値マイルストーン: VMS{N}

    コアサブドメイン別目標:
      {subdomain-1}:
        達成目標: [具体的な目標]
        成果物: [具体的な成果物]
        担当チーム: [チーム名]
        リソース: [必要リソース]

      {subdomain-2}:
        達成目標: [具体的な目標]
        成果物: [具体的な成果物]
        担当チーム: [チーム名]
        リソース: [必要リソース]

    横断的目標:
      - [全サブドメイン共通の目標]

    依存関係:
      - [サブドメイン間の依存]

    リスク:
      - [想定されるリスクと対策]

    成功基準:
      - [マイルストーン達成の判定基準]
```
