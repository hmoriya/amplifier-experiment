# 価値方法論リファレンス

**Parasol V5 - バックキャスティングと価値分解の方法論**

---

## 概要

このドキュメントは、Parasol V5における価値定義からバリューストリーム設計、プロジェクトマイルストーン作成までの方法論を定義します。V4の体系的なバックキャスティングアプローチを継承しています。

---

## なぜこの方法論なのか？（設計ストーリー）

### 従来アプローチの問題

多くのDXプロジェクトは**「できること」から始めて**失敗します：

```
従来の失敗パターン:
「AIを導入したい」→ 技術選定 → PoC → 「で、何に使うの？」
「DXを推進したい」→ ツール導入 → 「業務が変わらない」
```

**問題の本質**: 手段（技術）から始めると、目的（価値）が曖昧なまま進み、投資対効果が見えない。

### バックキャスティングという解法

本方法論は**「実現したい価値」から始めて逆算**します：

```
本方法論:
「顧客にとって価値ある状態とは？」→ その状態を実現するには？ → 必要な能力は？ → 必要な技術は？
```

**なぜ効くのか**:
1. **投資判断が明確**: 価値に直結する投資か否かが判断できる
2. **優先順位が決まる**: 価値への貢献度で優先順位が決まる
3. **成功が測定可能**: 価値の実現度で成功を測定できる

### 価値分解（VL1→VL2→VL3）の意味

**なぜ階層化するのか？**

```
VL1「期待を超えるおいしさ」← 抽象的すぎて行動に落ちない
    ↓ 分解
VL2「製品イノベーション価値」← 方向性は見えるが具体性不足
    ↓ 分解
VL3「発酵技術による味の革新」← 具体的な行動と投資判断が可能
```

**各レベルの役割**:
- **VL1（最上位価値）**: 経営レベルの意思決定基準
- **VL2（価値グループ）**: 部門レベルの投資配分基準
- **VL3（詳細価値）**: チームレベルの行動計画基準

### バックキャスティング（MS5→MS1）の意味

**なぜ未来から逆算するのか？**

```
フォアキャスティング（現在→未来）:
「今できることは何か？」→ 少しずつ改善 → どこに着くか分からない

バックキャスティング（未来→現在）:
「18ヶ月後の理想は？」→ そこに到達するには？ → 今何をすべきか？
```

**逆算の効果**:
- **ゴールが明確**: 「何を達成したら成功か」が最初に定義される
- **ギャップが見える**: 理想と現実の差が明確になる
- **必要条件が導出**: 「〜が必要」という論理的な導出が可能

### MS→VS変換の意味

**なぜマイルストーン（MS）をバリューステージ（VS）に変換するのか？**

```
MS（組織視点）: 「我々が達成すべき状態」
    ↓ 視点変換
VS（顧客視点）: 「顧客にとって価値ある状態」
```

**変換の意図**:
- **MSは組織の進捗指標**: 「研究基盤が整った」「サービスが展開された」
- **VSは顧客の価値指標**: 「新製品を選べるようになった」「便利に取引できる」

**なぜ両方必要か**:
- MSだけだと「組織の自己満足」になりがち
- VSだけだと「実行計画」が立てられない
- 両方を対応づけることで「顧客価値を軸にした実行計画」が可能

---

## 方法論の全体像

```
価値定義（最上位価値）
    ↓ 価値分解
VL1 → VL2 → VL3
    ↓ バックキャスティング
価値マイルストーン（MS5→MS4→MS3→MS2→MS1）
    ↓ ステージ変換
バリューステージ（VS0-VS7）
    ↓ CL分解と統合
プロジェクトマイルストーン = 価値マイルストーン × コアサブドメイン
```

## Phase 1: 価値宣言と価値分解

### 1.1 最上位価値の定義（VL1）

まず組織が提供する最上位の価値を宣言します。

```yaml
価値宣言:
  ビジョン: [組織が目指す理想の姿]

  提供価値:
    顧客への価値: [顧客が得る最終的な価値]
    社会への価値: [社会的インパクト]
    組織への価値: [組織が得る価値]

  価値の独自性:
    差別化要因: [他社にない独自価値]
    競争優位性: [持続可能な優位性]
    価値の持続性: [長期的な価値提供力]
```

### 1.2 価値分解（VL1→VL2→VL3）

最上位価値を階層的に分解します。

```yaml
VL1: 最上位価値
  定義: [組織全体の提供価値]
  測定指標: [成功の測定方法]

VL2: 価値グループ（3-5個）
  VL2-1: [価値グループ1]
    定義: [詳細な定義]
    VL1への貢献: [どのようにVL1に貢献するか]

  VL2-2: [価値グループ2]
    定義: [詳細な定義]
    VL1への貢献: [どのようにVL1に貢献するか]

  VL2-3: [価値グループ3]
    定義: [詳細な定義]
    VL1への貢献: [どのようにVL1に貢献するか]

VL3: 詳細価値（各VL2に3-5個）
  VL3-1-1: [詳細価値]
    定義: [具体的な価値]
    実現方法: [どのように実現するか]
    必要な活動: [必要な企業活動]
```

**分解のポイント**:
- VL2は3-5個が適切（多すぎると焦点がぼやける）
- VL3は実現可能な粒度まで分解
- 各レベルでVL1への貢献を明確化

## Phase 2: バックキャスティングによる価値マイルストーン

### 2.1 バックキャスティングとは

**未来の理想状態から現在に向かって逆算**するアプローチです。

```
従来のアプローチ（フォアキャスティング）:
現在 → 少し先 → もう少し先 → 未来（どこに着くか分からない）

バックキャスティング:
理想の未来（MS5）← 逆算 ← 逆算 ← 逆算 ← 現在（MS1）
```

### 2.2 価値マイルストーンの設計

**理想状態（MS5）から現在（MS1）へ逆算**します。

```yaml
MS5: 価値の完全実現（18ヶ月後）
  状態: [理想の実現状態]
  成功基準: [測定可能な基準]
  価値指標: [具体的な数値目標]
  # ここから逆算を開始

MS4: 価値の最適化（12ヶ月後）
  状態: [最適化された状態]
  成功基準: [測定可能な基準]
  MS5への移行: [MS5に到達するために必要な進化]

MS3: 価値の成熟（9ヶ月後）
  状態: [成熟した状態]
  成功基準: [測定可能な基準]
  MS4への移行: [MS4に到達するために必要な改善]

MS2: 価値の展開（6ヶ月後）
  状態: [展開された状態]
  成功基準: [測定可能な基準]
  MS3への移行: [MS3に到達するために必要な拡大]

MS1: 価値の基礎確立（3ヶ月後）
  状態: [基礎が確立した状態]
  成功基準: [測定可能な基準]
  MS2への移行: [MS2に到達するために必要な発展]
```

**設計順序**（重要）:
1. MS5（理想状態）を先に定義
2. MS4を「MS5に到達するには何が必要か」で定義
3. MS3を「MS4に到達するには何が必要か」で定義
4. MS2を「MS3に到達するには何が必要か」で定義
5. MS1を「MS2に到達するには何が必要か」で定義

## Phase 3: バリューストリームとバリューステージ

### 3.1 バリューストリームのバックキャスティング

バリューストリーム自体も、最終価値提供者に対して**価値ある状態を定義**してバックキャスティングで組み立てます。

```
最終価値提供者（End Value Recipient）
    ↓ 「この人にとって価値ある状態とは？」
価値ある最終状態の定義
    ↓ バックキャスティング
VS7 ← VS6 ← VS5 ← VS4 ← VS3 ← VS2 ← VS1 ← VS0
```

### 3.2 MS→VS変換

価値マイルストーンをバリューステージに変換します。

```yaml
VS0: 基盤ステージ（汎用・横断的）
  対応MS: 全MSの基盤
  定義: すべてのステージで共通利用される基盤価値
  状態: 基盤サービスが利用可能な状態

VS1: 認知形成ステージ
  対応MS: MS1前半
  定義: 価値の存在と可能性が認知された状態
  状態: [最終価値提供者にとっての価値ある状態]

VS2: 理解促進ステージ
  対応MS: MS1後半-MS2前半
  定義: 価値の活用方法が理解された状態
  状態: [最終価値提供者にとっての価値ある状態]

VS3: 初期実践ステージ
  対応MS: MS2後半-MS3前半
  定義: 実際に価値を体験した状態
  状態: [最終価値提供者にとっての価値ある状態]

VS4: 展開拡大ステージ
  対応MS: MS3後半-MS4前半
  定義: 価値が組織内で広く活用される状態
  状態: [最終価値提供者にとっての価値ある状態]

VS5: 成熟運用ステージ
  対応MS: MS4後半
  定義: 価値が安定的に提供される状態
  状態: [最終価値提供者にとっての価値ある状態]

VS6: 価値最大化ステージ
  対応MS: MS5前半
  定義: 期待された価値が完全に実現された状態
  状態: [最終価値提供者にとっての価値ある状態]

VS7: 革新創造ステージ
  対応MS: MS5後半
  定義: 新たな価値を継続的に創造する状態
  状態: [最終価値提供者にとっての価値ある状態]
```

### 3.3 ステージ設計のポイント

各ステージは**最終価値提供者にとっての「価値ある状態」**として定義します：

1. **誰に対して？** → 最終価値提供者（End Value Recipient）
2. **何が価値？** → その人にとっての価値ある状態
3. **どう測定？** → 状態を示す成功基準

```yaml
# 例: 製品開発Value Streamの場合
最終価値提供者: 消費者
価値ある状態の定義:
  VS7: 消費者が期待を超える製品を継続的に受け取れる状態
  VS6: 消費者のニーズに完全に応える製品が提供される状態
  VS5: 安定した品質の製品が提供される状態
  VS4: 多様な製品ラインナップから選択できる状態
  VS3: 初めての製品体験で価値を感じられる状態
  VS2: 製品の価値を理解できる状態
  VS1: 製品の存在を知る状態
  VS0: 製品にアクセスできる基盤がある状態
```

## Phase 4: プロジェクトマイルストーンの作成

### 4.1 統合の公式

**プロジェクトマイルストーン = 価値マイルストーン × コアサブドメイン**

```
価値マイルストーン（MS1-MS5）
        ×
コアサブドメイン（CL1→CL2→CL3で特定）
        ↓
プロジェクトマイルストーン（実行計画）
```

### 4.2 作成手順

**Step 1: コアサブドメインの特定**

Phase 3（ケーパビリティ分解）で特定したCoreドメインから：

```yaml
コアサブドメイン例:
  - fermentation-research（発酵研究）
  - ingredient-research（素材研究）
  - premium-beer-development（プレミアムビール開発）
```

**Step 2: 価値マイルストーンとの交差**

各価値マイルストーン（MS1-MS5）に対して、コアサブドメインごとの具体的な達成目標を定義：

```yaml
PM1: プロジェクトマイルストーン1（3ヶ月）
  対応MS: MS1（価値の基礎確立）

  fermentation-research:
    達成目標: 酵母データベースの基盤構築
    成果物: 酵母株100種の登録完了

  ingredient-research:
    達成目標: 素材評価フレームワークの確立
    成果物: 評価基準書の策定完了

  premium-beer-development:
    達成目標: 製品開発プロセスの標準化
    成果物: 開発ガイドライン策定完了

PM2: プロジェクトマイルストーン2（6ヶ月）
  対応MS: MS2（価値の展開）

  fermentation-research:
    達成目標: 発酵最適化アルゴリズムの実装
    成果物: 最適化ツールv1.0リリース

  # ... 以下同様
```

**Step 3: 依存関係の整理**

```yaml
依存関係マトリクス:
  PM1:
    fermentation-research → ingredient-research: 酵母特性データ提供
    ingredient-research → premium-beer-development: 素材選定基準提供

  PM2:
    前提: PM1の全目標達成
    fermentation-research → premium-beer-development: 発酵レシピ提供
```

### 4.3 プロジェクトマイルストーンテンプレート

```yaml
プロジェクトマイルストーン:
  PM{N}:
    期間: {開始日} - {終了日}
    対応価値マイルストーン: MS{N}

    コアサブドメイン別目標:
      {subdomain-1}:
        達成目標: [具体的な目標]
        成果物: [具体的な成果物]
        担当チーム: [チーム名]
        リソース: [必要リソース]

      {subdomain-2}:
        達成目標: [具体的な目標]
        成果物: [具体的な成果物]
        担当チーム: [チーム名]
        リソース: [必要リソース]

    横断的目標:
      - [全サブドメイン共通の目標]

    依存関係:
      - [サブドメイン間の依存]

    リスク:
      - [想定されるリスクと対策]

    成功基準:
      - [マイルストーン達成の判定基準]
```

## 方法論の適用フロー

### V5コマンドとの対応

| フェーズ | V5コマンド | 本方法論の対応セクション |
|---------|-----------|------------------------|
| Phase 2 | `/parasol:2-value` | Phase 1（価値宣言・分解） + Phase 2（バックキャスティング） |
| Phase 3 | `/parasol:3-capabilities` | CL1→CL2→CL3分解、コアサブドメイン特定 |
| Phase 4 | `/parasol:4-architecture` | Phase 3（バリューステージ設計） |
| 実行計画 | （プロジェクト管理） | Phase 4（プロジェクトマイルストーン） |

### 実行チェックリスト

**価値定義フェーズ**
- [ ] VL1（最上位価値）が明確に定義されているか
- [ ] VL2（価値グループ）が3-5個に分解されているか
- [ ] VL3（詳細価値）が実現可能な粒度か
- [ ] 各VLでVL1への貢献が明確か

**バックキャスティングフェーズ**
- [ ] MS5（理想状態）から設計を開始したか
- [ ] 各MSが測定可能な成功基準を持つか
- [ ] MS間の移行条件が明確か
- [ ] 時間軸が現実的か

**バリューストリームフェーズ**
- [ ] 最終価値提供者が明確か
- [ ] 各VSが「価値ある状態」として定義されているか
- [ ] MS→VS変換が適切か
- [ ] VS間の価値フローが明確か

**プロジェクトマイルストーンフェーズ**
- [ ] コアサブドメインが特定されているか
- [ ] 各PMに全コアサブドメインの目標があるか
- [ ] 依存関係が整理されているか
- [ ] リソースとスケジュールが現実的か
