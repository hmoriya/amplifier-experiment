# V5.6 検討：ステージ定義とVS0-VS7マッピング

> **ステータス**: V5.6検討中
> **作成日**: 2026-01-13
> **更新日**: 2026-01-14
> **目的**: 5W1Hフレームワークのステージ概念とV5標準VS0-VS7の関係を整理
> **重要原則**: Out→In設計に基づき、顧客セグメント別に単一バリューストリームを設計

## 1. 概念の整理：2種類の「ステージ」

V5.6検討において、「ステージ」という用語に2つの異なる概念が存在することを明確化する必要がある。

### 1.1 概念の定義

| 概念 | V5での名称 | 5W1Hでの名称 | 定義 | 主語 |
|------|-----------|-------------|------|------|
| **顧客状態** | Value Stage (VS0-VS7) | - | 顧客が価値を認識・獲得・享受する過程の状態 | **顧客** |
| **運用段階** | - | ステージ (WHEN) | バリューストリーム内の価値蓄積区切り | **組織** |

### 1.2 2つの概念の関係（Out→In原則）

**重要**: V5.6では**Out→In設計原則**に基づき、顧客セグメントごとに**単一のバリューストリーム**を設計する。組織機能別（製造、マーケティング等）に複数のVStrを持つことは**In→Out思考**であり、V5の設計思想に反する。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    顧客状態（Value Stage: VS）                       │
│                         顧客視点・状態遷移                           │
│                                                                     │
│  VS0      VS1      VS2      VS3      VS4      VS5      VS6      VS7 │
│  未認知 → 認知 → 選好形成 → 初回体験 → 継続利用 → 定着 → 発展 → 推奨│
└─────────────────────────────────────────────────────────────────────┘
                              │
                              │ 顧客セグメントごとに1本
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│              単一バリューストリーム（Value Stream）                  │
│                    顧客が体験する価値の流れ                          │
│                                                                     │
│  顧客体験VStr: 認知→検討→購入→利用→定着→推奨                       │
│                                                                     │
│  ※組織機能（製造、物流等）はVStr内の運用段階として内包              │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              │ VStr内に複数
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│                  運用段階（Operational Stage）                       │
│                    組織視点・業務フロー                              │
│                                                                     │
│  VS3支援OS: [店頭誘導]→[購入処理]→[製品提供]→[体験フォロー]         │
│                          ↑                                          │
│                    （製造・物流は内部プロセスとして内包）            │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.3 Out→In vs In→Out の対比

| 観点 | Out→In（V5.6推奨） | In→Out（非推奨） |
|------|-------------------|-----------------|
| VStr数 | 顧客セグメント別に1本 | 組織機能別に複数 |
| 設計視点 | 顧客が体験する流れ | 組織が実行する活動 |
| 主語 | 「顧客が〜する」 | 「組織が〜する」 |
| 例 | 「顧客体験VStr」 | 「製造VStr」「物流VStr」 |

## 2. V5標準 Value Stage (VS0-VS7) 定義

V5では、顧客の状態遷移を8段階で標準化している。

### 2.1 VS0-VS7の定義

| VS | 名称 | 顧客状態の定義 | 顧客の問い |
|----|------|---------------|-----------|
| **VS0** | 未認知 | 顧客がこの価値を知らない | - |
| **VS1** | 認知 | 顧客がこの価値を認識した | 「これは何？」 |
| **VS2** | 選好形成 | 顧客がこの価値を選びたいと思う | 「これがいいかも」 |
| **VS3** | 初回体験 | 顧客が初めてこの価値を体験した | 「試してみよう」 |
| **VS4** | 継続利用 | 顧客がこの価値を繰り返し利用する | 「また使おう」 |
| **VS5** | 定着 | 顧客の生活にこの価値が定着した | 「これがないと困る」 |
| **VS6** | 発展 | 顧客が関連する追加価値を求める | 「もっと知りたい」 |
| **VS7** | 推奨 | 顧客が他者にこの価値を推奨する | 「これいいよ」 |

### 2.2 VS設計の原則（V5標準）

1. **主語は常に顧客**: 「組織が〜する」ではなく「顧客が〜になる」
2. **状態として定義**: 活動ではなく状態変化を記述
3. **測定可能性**: 各VSに到達したかを測定可能にする
4. **価値視点**: 組織活動ではなく顧客が得る価値で定義

## 3. 5W1H 運用段階（Operational Stage）定義

5W1Hフレームワークでは、WHEN要素としてバリューストリーム内の運用段階を定義する。

### 3.1 運用段階の特徴

| 特徴 | 説明 |
|------|------|
| **価値蓄積** | 各段階で価値が蓄積される区切り |
| **判断ポイント** | 意思決定や承認が発生する |
| **接点** | ステークホルダーとの接点がある |
| **境界** | ケイパビリティの適用範囲を決める |

### 3.2 良い運用段階 vs 悪い運用段階

**良い運用段階**:
- 価値が蓄積される区切り
- 判断・意思決定ポイント
- ステークホルダーとの接点

**悪い運用段階**:
- 単なるタスク（「システム入力」「書類作成」）
- 担当者の切り替え（組織都合）

### 3.3 バリューストリーム種類別の運用段階例

| バリューストリーム種類 | 運用段階例 |
|---------------------|-----------|
| **価値創造系** | 企画→設計→検証→承認→展開 |
| **価値提供系** | 受注→引当→出荷→配送→納品 |
| **価値維持系** | 監視→検知→対応→復旧→改善 |
| **支援系** | 要求→評価→調達→導入→運用 |

## 4. 2つの概念の統合：V5.6提案

### 4.1 階層的統合モデル（Out→In原則適用）

**核心原則**: 顧客セグメントごとに**単一のバリューストリーム**を持つ。

```
┌──────────────────────────────────────────────────────────────────────┐
│ Level 1: Value Stage (VS0-VS7) - 顧客状態                           │
│                                                                      │
│  顧客の価値認識ジャーニー全体を8段階で標準化                         │
│  VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7                      │
└─────────────────────────────────────┬────────────────────────────────┘
                                      │
                                      │ 顧客セグメント別に1本
                                      ▼
┌──────────────────────────────────────────────────────────────────────┐
│ Level 2: Value Stream (VStr) - 顧客体験の流れ                       │
│                                                                      │
│  顧客が体験する価値の流れ全体（VS0-VS7を一貫して支援）              │
│  ※組織機能別に分割しない                                           │
└─────────────────────────────────────┬────────────────────────────────┘
                                      │
                                      │ VSステージごとに1:N
                                      ▼
┌──────────────────────────────────────────────────────────────────────┐
│ Level 3: Operational Stage (OS) - 運用段階                          │
│                                                                      │
│  各VSステージ内の価値蓄積区切り                                     │
│  例: VS3(初回体験)内 = [店頭誘導]→[購入処理]→[製品提供]→[体験支援] │
└──────────────────────────────────────────────────────────────────────┘
```

### 4.2 用語の標準化提案

| V5現行 | V5.6提案 | 英語表記 | 定義 |
|--------|---------|----------|------|
| Value Stage | **Value Stage (VS)** | Value Stage | 顧客状態（VS0-VS7） |
| - | **Value Stream (VStr)** | Value Stream | 顧客セグメント別の価値体験の流れ（1本） |
| - | **Operational Stage (OS)** | Operational Stage | 運用段階（VSステージ内） |

### 4.3 単一VStr設計例（飲料メーカー・B2C消費者セグメント）

**重要**: 以下は**単一のVStr**内でのVSステージ別運用段階の設計例。

```
【B2C消費者向け VStr】（単一のバリューストリーム）

VS0-VS1(認知)
  └── OS: [広告企画]→[制作]→[配信]→[認知測定]

VS1-VS2(選好形成)
  └── OS: [情報提供]→[体験機会創出]→[比較情報]→[選好醸成]

VS2-VS3(初回体験)
  └── OS: [店頭誘導]→[購入処理]→[製品提供]→[初回体験支援]
              ※製造・物流は「製品提供」の内部プロセス

VS3-VS4(継続利用)
  └── OS: [関係構築]→[再購入促進]→[習慣化支援]→[継続確認]

VS4-VS5(定着)
  └── OS: [価値深化]→[ロイヤリティ醸成]→[離脱防止]

VS5-VS6(発展)
  └── OS: [ニーズ発見]→[関連提案]→[クロスセル]

VS6-VS7(推奨)
  └── OS: [体験共有支援]→[コミュニティ運営]→[推奨促進]
```

### 4.4 複数セグメントの場合

顧客セグメントが異なる場合のみ、別のVStrを持つことが許容される。

```
【セグメント別VStr例】

セグメント1: B2C一般消費者
  └── B2C消費者VStr（上記の例）

セグメント2: B2B法人顧客（飲食店等）
  └── B2B法人VStr
      VS0-VS1: [法人向け広告]→[営業アプローチ]
      VS2-VS3: [商談]→[契約]→[導入支援]
      VS3-VS4: [定期配送]→[在庫管理支援]→[追加提案]
      ...
```

### 4.5 セグメント判定基準

どの場合に別VStrとすべきか、明確な基準を設ける。

**別VStrとすべき条件**:
1. 顧客体験の流れが**根本的に異なる**
   - 例: B2B（商談→契約→導入）vs B2C（認知→購入→利用）
2. 提供価値の**本質が異なる**
   - 例: 法人向け「コスト削減」vs 個人向け「利便性」
3. VSの各ステージの活動が**50%以上異なる**

**同一VStrとすべき条件**:
1. 顧客体験の流れが**類似**
2. 提供価値の本質が**同一**
3. 差異は**パラメータ**で吸収可能
   - 例: 「大企業」と「中小企業」は同一VStr、規模パラメータで調整

## 5. VS と運用段階の設計原則

### 5.1 VS設計の原則

1. **顧客起点**: 常に顧客の状態変化として定義
2. **測定可能**: 各VSへの到達を測定できる指標を定義
3. **標準8段階**: VS0-VS7の標準を基本とし、必要に応じて細分化

### 5.2 運用段階設計の原則

1. **価値蓄積単位**: 各段階で価値が蓄積される
2. **ケイパビリティ境界**: 1ケイパビリティが複数段階を跨がない
3. **判断ポイント**: 意思決定や承認を明確にする
4. **測定ポイント**: 進捗や品質を測定できる

### 5.3 両者の関係設計原則

```
設計順序:
1. VS設計（顧客状態の定義）
   ↓
2. VStr設計（どのVS遷移を支援するか）
   ↓
3. OS設計（VStr内の運用段階）
   ↓
4. Capability設計（各OSに必要な能力）
```

## 8. ZIGZAGプロセス：分解→統合

### 8.1 VStr設計のZIGZAGフロー

VL1レベルのバリューストリームを設計するには、**分解→統合**のZIGZAGプロセスが必要。

```
【ZIGZAGプロセス】

Step 1: VL1仮説（トップダウン）
  └── 企業全体の価値提供の大枠を仮説として設定

Step 2: セグメント分解
  └── 顧客セグメントを識別（B2C/B2B、地域、製品カテゴリ等）

Step 3: セグメント別VStr設計（ボトムダウン）
  └── 各セグメントごとに単一VStrを詳細設計

Step 4: 統合（ボトムアップ）
  └── セグメント別VStrから共通パターンを抽出
  └── VL1レベルの統合VStrを導出

Step 5: VL1 VStr確定
  └── 統合VStrと各セグメントVStrの関係を明確化
```

### 8.2 分解→統合の意義

```
【なぜ統合が必要か】

具体（セグメント）から抽象（企業全体）を導出することで：
  1. 各セグメントの実態に基づいた設計が可能
  2. 机上の空論ではなく、実運用に即した統合
  3. ボトムアップで成長可能な構造
  4. 後からセグメントを追加しても統合可能
```

## 9. スケーラビリティ：1製品からホールディングスまで

### 9.1 スケールレベル

V5.6のVStr設計は、1製品から始めてホールディングスレベルまでボトムアップで成長可能。

```
【スケールレベル】

Level 1: 単一製品/サービス
  └── 単一VStr（最もシンプル）
  └── 例: 「ビール製品A」のVStr

Level 2: 複数製品（同一ブランド）
  └── 同一VStrに複数製品が出現
  └── 例: 「ビールブランドX」の製品A, B, CがVStr内に

Level 3: 複数ブランド/事業部
  └── ブランド/事業部ごとにVStr
  └── 共通Platform VS（VS0経営支援、VS7改善等）
  └── 例: ビール事業VStr、清涼飲料事業VStr

Level 4: ホールディングス
  └── グループ共通VS + 事業会社別VStr
  └── ポートフォリオ統合ビュー
  └── 例: グループ統合VStr + 各事業会社VStr
```

### 9.2 ボトムアップ成長パス

```
【成長パス例】

1製品から開始:
  └── 製品A VStr設計

製品追加:
  └── 製品B追加 → 同一VStr内に出現

ブランド分離:
  └── ブランドX VStr（製品A, B）
  └── ブランドY VStr（製品C）

事業部統合:
  └── 事業部統合VStr（ブランドX, Y）

ホールディングス化:
  └── グループ統合VStr
      ├── 事業会社A VStr
      └── 事業会社B VStr
```

## 10. 統合VStrと製品マトリクス

### 10.1 統合VStrでの複数製品出現

企業・ホールディングスレベルでは、**同一VSに複数製品/サービスが出現**する。これにより：
- ポートフォリオ分析
- シナジー発見
- クロスセル機会の特定

が可能になる。

### 10.2 製品マトリクス表現（飲料メーカー例）

```
【統合VStr × 製品マトリクス】

VS           VS2(選好)      VS3(決定)      VS4(獲得)      VS5(実感)
─────────────────────────────────────────────────────────────────────
ビール       ✅ 味試飲      ✅ 購入        ✅ 配送        ✅ 飲用体験
             ※発酵技術     ※酒類規制     ※温度管理     ※食事ペアリング

清涼飲料     ✅ 試飲        ✅ 購入        ✅ 配送        ✅ リフレッシュ
             ※健康訴求     ※価格競争     ※自販機       ※季節性

健康食品     ✅ 情報収集    ✅ 定期購入    ✅ D2C配送     ✅ 効果実感
             ※エビデンス   ※サブスク     ※パーソナル   ※継続性
─────────────────────────────────────────────────────────────────────

【シナジー発見ポイント】
  VS2: 発酵技術 → 健康食品への転用可能
  VS4: 配送インフラ → 共有可能
  VS5: 顧客データ → クロスセル機会
```

### 10.3 統合VStrの活用シーン

| シーン | 活用方法 |
|--------|---------|
| **ポートフォリオ分析** | 各製品がどのVSで強み/弱みを持つか可視化 |
| **シナジー発見** | 同一VSで共有可能なケイパビリティを特定 |
| **クロスセル設計** | VS5→VS6遷移で関連製品を提案 |
| **投資判断** | どのVS/製品に投資すべきか判断 |
| **M&A評価** | 買収候補のVStrとの統合可能性評価 |

## 11. BIZBOKとの差異と正当化

### 11.1 意図的な差異

| 観点 | BIZBOK | V5.6 | 差異の理由 |
|------|--------|------|-----------|
| VStr数 | 可変・多対多 | 顧客セグメント別1本 | Out→In原則の徹底 |
| 同一顧客への複数VStr | 許容 | 不可 | DX/マイクロサービス設計への直線的トレーサビリティ |
| ケイパビリティの重複 | 許容（推奨） | VS単位で独立 | サービス境界の明確化 |

### 11.2 差異の正当性

V5の設計思想は「DX/マイクロサービス設計への直線的トレーサビリティ」を重視。

```
BIZBOK思想: 組織全体の能力ポートフォリオ管理
  → 同一Capabilityが複数ステージに出現可能
  → EA（エンタープライズアーキテクチャ）視点

V5.6思想: サービス境界の明確化
  → VS→CL1→CL2→BC→Serviceの直線的パス
  → DX/マイクロサービス視点
```

**結論**: V5.6はDX最適化のため意図的に異なるアプローチを採用。BIZBOKとの差異は設計思想の違いであり、どちらが正しいというものではない。

## 13. 両面市場モデルへの適用

### 13.1 両面市場モデルとは

両面市場（Two-Sided Market）は、2つの異なる顧客グループが相互に価値を提供し合うビジネスモデル。人材派遣、不動産仲介、マッチングプラットフォーム等が該当する。

V5.6では、**顧客側面ごとに別VStr**を設計し、**交点を共有ケイパビリティ**として扱う。

### 13.2 両面モデルの基本構造

```
┌─────────────────────────────────────────────────────────────────┐
│  側面A VStr（例：求職者）                                        │
│                                                                 │
│  顧客：サービスを求める個人/組織                                 │
│  価値：ニーズの充足                                             │
│                                                                 │
│  VS0→VS7: 独自の顧客体験ジャーニー                              │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ マッチング（価値の交点）
                              │ ← 共有ケイパビリティ
                              │
┌─────────────────────────────────────────────────────────────────┐
│  側面B VStr（例：企業）                                          │
│                                                                 │
│  顧客：サービスを提供する個人/組織                               │
│  価値：ニーズの充足                                             │
│                                                                 │
│  VS0→VS7: 独自の顧客体験ジャーニー                              │
└─────────────────────────────────────────────────────────────────┘
```

### 13.3 人材派遣会社の適用例

#### 13.3.1 なぜ2本のVStrか

| 判定基準 | 求職者 | 企業 | 判定 |
|---------|-------|------|------|
| 顧客体験の流れ | 登録→面談→就業 | 要件定義→選考→契約 | **根本的に異なる** |
| 提供価値の本質 | キャリア実現 | 人材確保 | **異なる** |
| VSステージの活動 | 大半が異なる | 大半が異なる | **50%以上異なる** |

→ **別VStrとすべき条件を満たす**

## 14. 結論と次のステップ

### 14.1 結論

1. **2種類のステージ概念を明確に区別**: VS (顧客状態) と OS (運用段階)
2. **Out→In原則の徹底**: 顧客セグメントごとに単一VStr
3. **ZIGZAGプロセス**: 分解→統合で具体から抽象を導出
4. **スケーラブル設計**: 1製品からホールディングスまでボトムアップ成長可能
5. **統合VStrの活用**: 製品マトリクスでポートフォリオ可視化

### 14.2 V5.6 VStr設計原則（まとめ）

```yaml
V5.6_VStr設計原則:
  1_顧客視点で1本:
    - メインVStrは顧客セグメントごとに1本
    - 組織機能別に分けない（Out→In）

  2_セグメント別は許容:
    - 顧客が異なれば別VStr
    - 判定基準: 体験の流れ、提供価値の本質

  3_分解→統合のZIGZAG:
    - VL1仮説 → セグメント分解 → 統合 → VL1確定
    - 具体から抽象を導出

  4_スケーラブル:
    - 1製品から始めてボトムアップで成長
    - 個社→ブランド→事業部→ホールディングス

  5_統合VStrでの複数製品:
    - 同一VSに複数製品/サービスが出現
    - ポートフォリオ分析、シナジー発見に活用
```
