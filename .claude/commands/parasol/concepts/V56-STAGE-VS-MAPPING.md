## 1. 概念の整理：2種類の「ステージ」

V5.6検討において、「ステージ」という用語に2つの異なる概念が存在することを明確化する必要がある。

### 1.1 概念の定義

| 概念 | V5での名称 | 5W1Hでの名称 | 定義 | 主語 |
|------|-----------|-------------|------|------|
| **顧客状態** | Value Stage (VS0-VS7) | - | 顧客が価値を認識・獲得・享受する過程の状態 | **顧客** |
| **運用段階** | - | ステージ (WHEN) | バリューストリーム内の価値蓄積区切り | **組織** |

### 1.2 2つの概念の関係

```
┌─────────────────────────────────────────────────────────────────────┐
│                    顧客状態（Value Stage: VS）                       │
│                         顧客視点・状態遷移                           │
│                                                                     │
│  VS0      VS1      VS2      VS3      VS4      VS5      VS6      VS7 │
│  未認知 → 認知 → 選好形成 → 初回体験 → 継続利用 → 定着 → 発展 → 推奨│
│    │       │        │        │         │        │       │       │  │
└────┼───────┼────────┼────────┼─────────┼────────┼───────┼───────┼──┘
     │       │        │        │         │        │       │       │
     ▼       ▼        ▼        ▼         ▼        ▼       ▼       ▼
┌─────────────────────────────────────────────────────────────────────┐
│                  運用段階（Operational Stage）                       │
│                    組織視点・業務フロー                              │
│                                                                     │
│  バリューストリーム1: 製品供給                                       │
│  [受注]→[引当]→[出荷]→[配送]→[納品]→[請求]                         │
│                                                                     │
│  バリューストリーム2: 製品製造                                       │
│  [計画]→[調達]→[調合]→[充填]→[検査]→[保管]                         │
│                                                                     │
│  バリューストリーム3: 顧客獲得                                       │
│  [認知]→[興味]→[検討]→[購入]→[評価]→[リピート]                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. V5標準 Value Stage (VS0-VS7) 定義

V5では、顧客の状態遷移を8段階で標準化している。

### 2.1 VS0-VS7の定義

| VS | 名称 | 顧客状態の定義 | 顧客の問い |
|----|------|---------------|-----------|
| **VS0** | 未認知 | 顧客がこの価値を知らない | - |
| **VS1** | 認知 | 顧客がこの価値を認識した | 「これは何？」 |
| **VS2** | 選好形成 | 顧客がこの価値を選びたいと思う | 「これがいいかも」 |
| **VS3** | 初回体験 | 顧客が初めてこの価値を体験した | 「試してみよう」 |
| **VS4** | 継続利用 | 顧客がこの価値を繰り返し利用する | 「また使おう」 |
| **VS5** | 定着 | 顧客の生活にこの価値が定着した | 「これがないと困る」 |
| **VS6** | 発展 | 顧客が関連する追加価値を求める | 「もっと知りたい」 |
| **VS7** | 推奨 | 顧客が他者にこの価値を推奨する | 「これいいよ」 |

### 2.2 VS設計の原則（V5標準）

1. **主語は常に顧客**: 「組織が〜する」ではなく「顧客が〜になる」
2. **状態として定義**: 活動ではなく状態変化を記述
3. **測定可能性**: 各VSに到達したかを測定可能にする
4. **価値視点**: 組織活動ではなく顧客が得る価値で定義

## 3. 5W1H 運用段階（Operational Stage）定義

5W1Hフレームワークでは、WHEN要素としてバリューストリーム内の運用段階を定義する。

### 3.1 運用段階の特徴

| 特徴 | 説明 |
|------|------|
| **価値蓄積** | 各段階で価値が蓄積される区切り |
| **判断ポイント** | 意思決定や承認が発生する |
| **接点** | ステークホルダーとの接点がある |
| **境界** | ケイパビリティの適用範囲を決める |

### 3.2 良い運用段階 vs 悪い運用段階

**良い運用段階**:
- 価値が蓄積される区切り
- 判断・意思決定ポイント
- ステークホルダーとの接点

**悪い運用段階**:
- 単なるタスク（「システム入力」「書類作成」）
- 担当者の切り替え（組織都合）

### 3.3 バリューストリーム種類別の運用段階例

| バリューストリーム種類 | 運用段階例 |
|---------------------|-----------|
| **価値創造系** | 企画→設計→検証→承認→展開 |
| **価値提供系** | 受注→引当→出荷→配送→納品 |
| **価値維持系** | 監視→検知→対応→復旧→改善 |
| **支援系** | 要求→評価→調達→導入→運用 |

## 4. 2つの概念の統合：V5.6提案

### 4.1 階層的統合モデル

```
┌──────────────────────────────────────────────────────────────────────┐
│ Level 1: Value Stage (VS0-VS7) - 顧客状態                           │
│                                                                      │
│  顧客の価値認識ジャーニー全体を8段階で標準化                         │
│  VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7                      │
└─────────────────────────────────────┬────────────────────────────────┘
                                      │
                                      │ 1:N 関係
                                      ▼
┌──────────────────────────────────────────────────────────────────────┐
│ Level 2: Value Stream (VStr) - 価値の流れ                           │
│                                                                      │
│  各VSの状態遷移を実現するための価値提供フロー                        │
│  例: VS3→VS4遷移を支援するVStr = 製品供給ストリーム                 │
└─────────────────────────────────────┬────────────────────────────────┘
                                      │
                                      │ 1:N 関係
                                      ▼
┌──────────────────────────────────────────────────────────────────────┐
│ Level 3: Operational Stage (OS) - 運用段階                          │
│                                                                      │
│  バリューストリーム内の価値蓄積区切り                               │
│  例: 製品供給VStr = [受注]→[引当]→[出荷]→[配送]→[納品]             │
└──────────────────────────────────────────────────────────────────────┘
```

### 4.2 用語の標準化提案

| V5現行 | V5.6提案 | 英語表記 | 定義 |
|--------|---------|----------|------|
| Value Stage | **Value Stage (VS)** | Value Stage | 顧客状態（VS0-VS7） |
| - | **Value Stream (VStr)** | Value Stream | 価値の流れ全体 |
| - | **Operational Stage (OS)** | Operational Stage | 運用段階（VStr内） |

### 4.3 VS × VStr マッピング例（飲料メーカー）

```
Value Stage      主要 Value Stream           運用段階例
───────────────────────────────────────────────────────────────────
VS0→VS1          ブランド認知 VStr          [企画]→[制作]→[配信]→[測定]
(未認知→認知)

VS1→VS2          購買検討支援 VStr          [情報提供]→[体験機会]→[比較支援]
(認知→選好)

VS2→VS3          初回購買 VStr              [店頭誘導]→[購入]→[体験]→[評価取得]
(選好→初回体験)

VS3→VS4          リピート促進 VStr          [関係構築]→[再購入促進]→[習慣化支援]
(初回→継続)

VS4→VS5          顧客定着 VStr              [価値深化]→[ロイヤリティ醸成]→[離脱防止]
(継続→定着)

VS5→VS6          クロスセル VStr            [ニーズ発見]→[提案]→[関連購入]
(定着→発展)

VS6→VS7          アドボカシー VStr          [体験共有支援]→[コミュニティ]→[推奨促進]
(発展→推奨)
```

## 5. VS と運用段階の設計原則

### 5.1 VS設計の原則

1. **顧客起点**: 常に顧客の状態変化として定義
2. **測定可能**: 各VSへの到達を測定できる指標を定義
3. **標準8段階**: VS0-VS7の標準を基本とし、必要に応じて細分化

### 5.2 運用段階設計の原則

1. **価値蓄積単位**: 各段階で価値が蓄積される
2. **ケイパビリティ境界**: 1ケイパビリティが複数段階を跨がない
3. **判断ポイント**: 意思決定や承認を明確にする
4. **測定ポイント**: 進捗や品質を測定できる

### 5.3 両者の関係設計原則

```
設計順序:
1. VS設計（顧客状態の定義）
   ↓
2. VStr設計（どのVS遷移を支援するか）
   ↓
3. OS設計（VStr内の運用段階）
   ↓
4. Capability設計（各OSに必要な能力）
```

## 6. 5W1H文書のステージ → V5マッピング

### 6.1 製品供給バリューストリームのマッピング

```
【5W1H文書】
[受注] → [引当] → [出荷] → [配送] → [納品] → [請求・回収]

【V5.6マッピング】
VS4（継続利用）を支援する「製品供給 VStr」の運用段階として位置づけ:

VS4支援

    ├── OS2: 在庫引当
    │     └── Cap: 在庫管理
    ├── OS3: 出荷準備
    │     └── Cap: 出荷管理
    ├── OS4: 配送
    │     └── Cap: 配送管理
    ├── OS5: 納品
    │     └── Cap: 納品管理
    └── OS6: 請求・回収
          └── Cap: 債権管理
```

## 7. V5.6 拡張提案：Operational Stage の正式導入

### 7.1 現行V5との差分

| 項目 | V5現行 | V5.6提案 |
|------|--------|---------|
| VS定義 | VS0-VS7（顧客状態） | 変更なし |
| VStr定義 | 価値の流れ | 変更なし |
| **運用段階** | **明示的定義なし** | **OS (Operational Stage) として正式導入** |
| ステージ-Cap関係 | 暗黙的 | 明示的マッピング |

### 7.2 導入効果

1. **概念の明確化**: 顧客状態(VS)と運用段階(OS)の混同を防止
2. **設計の精緻化**: OS-Capability の1:1対応が明確に
3. **BIZBOKとの整合**: BIZBOK用語との対応が取りやすい
4. **DDDとの接続**: OSがBounded Context境界の参考になる

### 7.3 Phase構成への影響

```
Phase 2: 価値定義
  └── VS0-VS7の顧客状態設計 ← 変更なし

Phase 3: ケイパビリティ分解
  ├── CL1: Activity Area識別
  ├── CL2: Capability定義
  └── CL3: Sub-capability定義（Operation含む）
      └── **OS (Operational Stage) との対応を明示** ← 追加

Phase 4: アプリケーション設計
  └── OSとBCの対応関係を活用 ← 明確化
```

### 8.1 結論
