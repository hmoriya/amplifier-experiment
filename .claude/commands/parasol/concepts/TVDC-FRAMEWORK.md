# TVDC Framework: Three-Dimensional Value-Driven Classification

**バージョン**: V1.0
**作成日**: 2026-01-07

---

## エグゼクティブサマリー

TVDC（Three-Dimensional Value-Driven Classification）Frameworkは、Parasol V5におけるケイパビリティ分類の新しい4分類体系です。従来のDDDの3分類（Core/Supporting/Generic）に**VCI（Value-Critical Infrastructure）**を追加し、価値駆動設計の理念に基づいた分類を実現します。

---

## 1. 背景：なぜ4分類が必要か

### DDDの3分類の限界

従来のDDD（Domain-Driven Design）では、ケイパビリティを以下の3分類で整理していました：

| 分類 | 定義 | 投資優先度 |
|------|------|-----------|
| **Core** | 競争優位の源泉 | 最高 |
| **Supporting** | ビジネス支援機能 | 中 |
| **Generic** | 汎用的な基盤機能 | 低（SaaS活用推奨） |

### 「配送問題」の発見

Parasol V5のValue Stream設計において、以下の矛盾が発見されました：

```
例：ECサイトにおける「配送」能力

DDDの判定:
- 競争優位の源泉か？ → No（他社も同じことをしている）
- 差別化要因か？ → No（標準的なサービス）
→ 結論: Generic（投資優先度：低）

しかし実際は:
- 配送が失敗すると顧客価値は完全に毀損
- 「商品を届ける」という価値の根幹
- 品質低下は致命的なビジネス影響
→ 「投資優先度：低」は明らかに不適切
```

### 3軸の視点

この問題を分析すると、「Core」には3つの異なる視点があることが判明しました：

```
軸1: バリューストリーム全体から見てCore
     「顧客への価値提供に本質的に必要か？」

軸2: ステージ内での状態遷移に対してCore
     「このステージの価値実現に必須か？」

軸3: 全体戦略・差別化を意識してCore
     「競争優位の源泉となるか？」
```

---

## 2. TVDC Frameworkの構造

### 2次元 + 属性モデル

討議の結果、3軸は「直交」ではなく「階層」であることが判明しました：

```
┌─────────────────────────────────────────────────┐
│  次元A: 価値必然性軸（Value Necessity）         │
│  ├─ 軸1: VSTR全体視点（マクロ粒度）            │
│  └─ 軸2: ステージ内視点（ミクロ粒度）          │
│       → 同一次元の異なる解像度                  │
│                                                 │
│  次元B: 差別化軸（Differentiation）            │
│  └─ 軸3: 競争優位性視点                        │
│       → 次元Aと直交する独立軸                   │
│                                                 │
│  属性: ステージ焦点（Stage Focus）             │
│  └─ どのステージで特にCoreか                    │
└─────────────────────────────────────────────────┘
```

### 4分類体系

| 分類 | 定義 | 価値必然性 | 差別化 | 投資優先度 |
|------|------|-----------|--------|-----------|
| **Core** | 価値必然性★★★ かつ 差別化★★★ | 極めて高い | 高い | 最高 |
| **VCI** | 価値必然性★★★ かつ 差別化★以下 | 極めて高い | 低い | 高（品質重視） |
| **Supporting** | 価値必然性★★ かつ 差別化★★以下 | 中程度 | 中〜低 | 中 |
| **Generic** | 価値必然性★ かつ 差別化★ | 低い | 低い | 低（SaaS活用） |

## 3. VCI（Value-Critical Infrastructure）

### 定義

**VCI = 価値実現に本質的だが差別化にならない能力**

```
VCIの特徴:
- 顧客価値の実現に「絶対に必要」
- しかし「それだけでは選ばれない」
- 失敗すると価値が完全に毀損
- 品質・信頼性が最重要
```

### VCIの具体例

| 能力 | 価値必然性 | 差別化 | 従来DDD | TVDC |
|------|-----------|--------|---------|------|
| 配送 | ★★★ | ★ | Generic | **VCI** |
| 決済 | ★★★ | ★ | Generic | **VCI** |
| 認証 | ★★★ | ★ | Generic | **VCI** |
| 在庫引当 | ★★★ | ★ | Supporting | **VCI** |
| 通知配信 | ★★ | ★ | Generic | Generic |
| 会計処理 | ★ | ★ | Generic | Generic |

### VCIの投資戦略

```
VCIへの投資方針:
┌─────────────────────────────────────────────────┐
│ ❌ 「差別化にならないから外注」               │
│ ✅ 「失敗は致命的なので品質に投資」           │
│                                                 │
│ 具体的なアクション:                            │
│ - 信頼性の高いSaaSを選定                       │
│ - 冗長化・フェイルオーバー設計                 │
│ - SLA監視の徹底                                │
│ - 障害時の代替手段確保                         │
└─────────────────────────────────────────────────┘
```

## 4. 分類判定フロー

### 標準判定フロー

```
                    ┌─────────────────┐
                    │ ケイパビリティ │
                    └────────┬────────┘
                             ▼
        ┌────────────────────────────────────────┐
        │ 価値必然性チェック                      │
        │ Q1: VSTR全体で価値状態遷移に必須か？   │
        │ Q2: ステージ内で状態変化の要か？        │
        └────────────────────┬───────────────────┘
                             ▼
              ┌──────────────┴──────────────┐
              │                             │
         高必然性                      低必然性
         (Q1またはQ2がYes)            (両方No)
              │                             │
              ▼                             ▼
    ┌─────────────────┐           ┌─────────────────┐
    │ 差別化チェック   │           │    Generic      │
    │ 競争優位源泉か？ │           │ 外部サービス    │
    │ 他社が真似困難か？│           │ 活用推奨        │
    └────────┬────────┘           └─────────────────┘
             ▼
    ┌────────┴────────┐
    │                 │
  高差別化         低差別化
  (両方Yes)        (どちらかNo)
    │                 │
    ▼                 ▼
┌───────┐        ┌───────┐
│ Core  │        │  VCI  │
│ 最高  │        │ 品質  │
│ 投資  │        │ 重視  │
└───────┘        └───────┘
```

### 詳細判定マトリクス

```
           │ 差別化：高        │ 差別化：低
───────────┼───────────────────┼─────────────────
価値必然性 │                   │
    高     │      Core         │      VCI
           │ 「競争の武器」    │「失敗は致命的」
───────────┼───────────────────┼─────────────────
価値必然性 │                   │
    中     │    Supporting     │   Supporting
           │「重要な支援機能」 │「効率化対象」
───────────┼───────────────────┼─────────────────
価値必然性 │    (稀なケース)   │
    低     │    Supporting     │    Generic
           │                   │「外部活用」
───────────┴───────────────────┴─────────────────
```

### 判定質問リスト

**価値必然性の判定**:

| 質問 | Yes → 高必然性 | No → 次の質問へ |
|------|----------------|-----------------|
| この能力がないと顧客は価値を得られないか？ | ★★★ | - |
| この能力の品質低下は顧客体験を直接損なうか？ | ★★★ | - |
| 価値ストリームの状態遷移に必須か？ | ★★〜★★★ | - |
| 他の能力の前提条件となっているか？ | ★★ | ★ |

**差別化の判定**:

| 質問 | Yes → 高差別化 | No → 低差別化 |
|------|----------------|---------------|
| この能力で競合と差別化できるか？ | ★★★ | - |
| 他社が真似するのに時間がかかるか？ | ★★★ | - |
| 独自のノウハウや技術が必要か？ | ★★〜★★★ | - |
| 顧客がこの能力を理由に選んでくれるか？ | ★★★ | - |

## 5. ステージ依存パターン

### 同一能力のステージ別分類

能力によっては、Value Stageごとに分類が変わる場合があります：

```
例：「商品説明」能力

VS1（認知形成）: Core（ブランド認知に直結）
VS2（理解促進）: Core（購買決定に直結）
VS3（初期実践）: Supporting（使用方法の補助）
VS4（継続活用）: Generic（参照頻度低下）
```

### ステージ依存Coreの記法

```yaml
capability:
  name: product-description
  classification:
    default: Supporting
    stage_overrides:
      VS1: Core
      VS2: Core
  notes: "VS1-VS2では顧客の意思決定に直結するためCore"
```

### パターン分類

| パターン | 説明 | 例 |
|---------|------|-----|
| **通貫Core** | 全ステージでCore | 独自アルゴリズム |
| **ステージ依存Core** | 特定ステージでCore | 商品説明（VS1-2） |
| **戦略的Core昇格** | 戦略判断でCoreに昇格 | 配送（翌日配送特化時） |
| **VCI** | 全ステージで価値必然だが非差別化 | 決済、認証 |

## 6. 投資配分ガイドライン

### 分類別投資配分

```
推奨投資配分（目安）:

┌────────────────────────────────────────────────┐
│ Core:       40-50%                             │
│ ├─ 独自技術開発                                │
│ ├─ 専門人材確保                                │
│ └─ 継続的改善                                  │
├────────────────────────────────────────────────┤
│ VCI:        20-25%                             │
│ ├─ 信頼性投資（冗長化、監視）                  │
│ ├─ SLA保証                                    │
│ └─ 障害対応体制                                │
├────────────────────────────────────────────────┤
│ Supporting: 15-25%                             │
│ ├─ 効率化投資                                  │
│ ├─ 部分的SaaS活用                             │
│ └─ 標準化推進                                  │
├────────────────────────────────────────────────┤
│ Generic:    10-15%                             │
│ ├─ SaaS利用料                                 │
│ ├─ 統合コスト                                  │
│ └─ 最小限のカスタマイズ                        │
└────────────────────────────────────────────────┘
```

### VCIの特別な投資原則

```
VCI投資の原則:

1. 「安さ」より「信頼性」
   - 最安値のSaaSより、実績あるサービスを選択
   - SLAを契約条件に含める

2. 「効率」より「冗長性」
   - シングルポイント障害を排除
   - フェイルオーバー機構を用意

3. 「機能」より「可用性」
   - 99.9%以上の稼働率を目標
   - 障害発生時の影響を最小化

4. 「開発」より「監視」
   - 自前開発は避ける
   - 監視・アラート体制を充実
```

## 7. DDD戦略的設計との対応

### 従来DDDとの比較

| DDD概念 | TVDC対応 | 変更点 |
|---------|---------|--------|
| Core Domain | Core | 定義を「差別化 + 価値必然性」に明確化 |
| Supporting Subdomain | Supporting | 変更なし |
| Generic Subdomain | Generic または VCI | VCIの分離 |

### Context Mapへの影響

```
VCIのContext Map統合パターン:

┌─────────────┐
│ Core BC     │
└──────┬──────┘
       │ Anticorruption Layer
       ▼
┌─────────────┐
│ VCI BC      │ ← 外部サービスのACL
│ (配送)      │
└─────────────┘

推奨パターン:
- VCI → Core: Anticorruption Layer
- VCI → VCI: Published Language
- Core → VCI: Customer-Supplier (VCIがSupplier)
```

## 8. Phase 2との連動：価値分解からの価値必然性継承

### 連動モデル

**重要**: 価値必然性はPhase 2の価値分解（VL3）で評価し、Phase 3 CL2に継承します。

```
Phase 2: 価値分解
┌─────────────────────────────────────────────────┐
│ VL1: 最上位価値                                  │
│   └─ VL2: 価値グループ                          │
│        └─ VL3: 価値要素                          │
│              └─ 【価値必然性スコア】★★★          │ ← ここで評価
│              └─ 【VCI候補フラグ】true/false      │ ← ここで識別
└─────────────────────────────────────────────────┘
                    │
                    │ 継承（再評価不要）
                    ▼
Phase 3 CL2: ケイパビリティ分類
┌─────────────────────────────────────────────────┐
│ ケイパビリティ                                   │
│   ├─ 価値必然性: VL3から継承 ★★★                │ ← 自動継承
│   └─ 差別化: CL2で評価 ★★★                      │ ← ここで評価
│                    ↓                             │
│              TVDC分類決定                        │
└─────────────────────────────────────────────────┘
```

### VL3での価値必然性評価

Phase 2の価値要素（VL3）定義時に以下を評価：

```yaml
VL3-X-Y:
  name: "{価値要素名}"
  parent: VL2-X

  # 価値必然性評価（Phase 2で実施）
  value_necessity:
    score: "★★★"  # ★/★★/★★★

    evaluation:
      # Q1: この価値要素がないと顧客価値は？
      value_impact: "完全毀損"  # 完全毀損/部分低下/影響小

      # Q2: VSTR状態遷移に必須か？
      state_transition_required: true
      related_stages: ["VS3", "VS4"]  # 必須となるステージ

      # Q3: 失敗時の影響は？
      failure_impact: "致命的"  # 致命的/重大/軽微

    rationale: |
      - この能力がないと製品を届けられない
      - VS3→VS4の状態遷移に絶対必須
      - 品質低下は顧客価値を直接毀損

    # VCI候補識別
    vci_candidate: true  # 価値必然だが差別化しにくい場合
    vci_reason: "失敗は致命的だが、他社も同等サービスを提供"

  # Phase 3への引継ぎ情報
  phase3_handoff:
    target_capability: "delivery-fulfillment"
    differentiation_hints: |
      - 翌日配送で差別化可能か検討
      - 配送追跡UXで差別化可能か検討
```

### VSTRからのVCI候補自動識別

価値状態遷移（VSTR）マップから、状態遷移に必須な能力を自動識別：

```
VSTR状態遷移マップ
┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐
│ VS2 │───→│ VS3 │───→│ VS4 │───→│ VS5 │
│理解 │    │購入 │    │獲得 │    │利用 │
└─────┘    └─────┘    └─────┘    └─────┘
                │         │
           【遷移必須】 【遷移必須】
                │         │
        ┌───────┴─────┐   │
        │ 決済処理     │←──┘
        │ 在庫引当     │
        │ 配送手配     │
        └─────────────┘
              ↓
        VCI候補として自動識別
        （価値必然性★★★、差別化は要CL2評価）
```

### 連動判定フロー

```
Phase 2: VL3定義時
┌─────────────────────────────────────────┐
│ 価値必然性評価（3つの質問）              │
│                                         │
│ Q1: この価値要素がないと顧客価値は？     │
│     「完全毀損」→ ★★★                   │
│     「部分低下」→ ★★                    │
│     「影響小」 → ★                      │
│                                         │
│ Q2: VSTR状態遷移に必須か？               │
│     Yes → +★（必然性を上方修正）         │
│                                         │
│ Q3: 失敗時の影響は？                     │
│     「致命的」→ VCI候補フラグON          │
└─────────────────────────────────────────┘
                    │
                    ▼
Phase 3 CL2: 差別化評価のみ
┌─────────────────────────────────────────┐
│ 価値必然性: Phase 2から継承（評価済み）  │
│                                         │
│ Q: 差別化要素か？（CL2で評価）          │
│     「競争優位の源泉」→ ★★★             │
│     「重要だが標準的」→ ★★              │
│     「汎用的」      → ★                │
│                                         │
│ 【自動判定】                            │
│ 必然性★★★ + 差別化★★★ → Core           │
│ 必然性★★★ + 差別化★   → VCI            │
│ 必然性★★  + 任意     → Supporting      │
│ 必然性★   + 差別化★  → Generic         │
└─────────────────────────────────────────┘
```

### 連動のメリット

| 観点 | 連動前 | 連動後 |
|------|--------|--------|
| **トレーサビリティ** | 価値→分類の根拠が曖昧 | VL3から明確に継承 |
| **評価の一貫性** | Phase 2と3で重複評価 | 価値必然性はPhase 2のみ |
| **VCI識別精度** | CL2で新規判断 | Phase 2でVCI候補特定済み |
| **設計根拠** | 分類理由が不明確 | 価値から導出された根拠 |
| **効率性** | CL2で全項目評価 | CL2は差別化のみ評価 |

## 9. 実践ガイド

### Phase 3での適用

CL2（ケイパビリティ設計）で以下を実施：

1. **価値必然性評価**: 各ケイパビリティについて質問リストで評価
2. **差別化評価**: 競争優位性の観点で評価
3. **4分類決定**: マトリクスに基づいて分類
4. **ステージ依存チェック**: 必要に応じてステージ別分類を設定
5. **投資配分設計**: 分類に基づいて投資優先度を設定

### テンプレート

```yaml
# CL2ケイパビリティ分類（TVDC版）

capability:
  name: "{capability-name}"

  # TVDC分類
  classification:
    type: "{Core | VCI | Supporting | Generic}"

    value_necessity:
      score: "{★ | ★★ | ★★★}"
      rationale: "{価値必然性の根拠}"

    differentiation:
      score: "{★ | ★★ | ★★★}"
      rationale: "{差別化の根拠}"

    stage_dependency:
      is_stage_dependent: "{true | false}"
      overrides:
        VS1: "{classification if different}"
        VS2: "{classification if different}"

  # 投資方針
  investment:
    priority: "{最高 | 高 | 中 | 低}"
    strategy: "{投資戦略の説明}"
    budget_ratio: "{推奨予算比率}"
```

## 9. まとめ

### TVDC Frameworkの価値

1. **価値駆動設計との整合**: DDDの「差別化」重視から、Parasol V5の「価値」重視へ
2. **VCIの明示化**: 「失敗は致命的だが差別化にならない」能力の適切な扱い
3. **投資判断の精度向上**: 4分類による、より適切な投資配分
4. **ステージ依存の考慮**: 同一能力のステージ別分類への対応

### 適用の原則

```
TVDC適用の3原則:

1. 価値起点: 「顧客価値への貢献」を最優先で評価
2. 品質重視: VCIは「安さ」より「信頼性」
3. 文脈依存: ステージや事業特性による分類変動を許容
```
