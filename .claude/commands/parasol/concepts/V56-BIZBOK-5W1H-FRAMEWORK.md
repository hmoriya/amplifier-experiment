# Parasol V5.6 検討：BIZBOKベース5W1Hビジネスアーキテクチャフレームワーク

> **ステータス**: V5.6検討中
> **作成日**: 2026-01-13
> **目的**: BIZBOKの概念をV5に統合し、5W1Hによる構造化を検討

---

## 概要

Parasol V5は、価値起点でビジネスアーキテクチャからシステム実装までを一貫して設計するためのフレームワークである。BIZBOKの概念を取り入れつつ、DDDとの接続を重視した構造を持つ。

---

## 1. 5W1Hによる構造化

### 基本構造

| 要素 | 問い | 説明 |
|------|------|------|
| **Why** | なぜ | 価値（企業理念・存在意義） |
| **What** | 何を | 問題領域（外向き：価値構成要素、内向き：ケイパビリティ） |
| **How** | どうやって | バリューストリーム（価値を届ける流れ） |
| **When** | いつ | ステージ（バリューストリームの各段階） |
| **Who** | 誰が | 組織（実行主体）※ステージを跨ぐことがある |
| **Where** | どこで | チャネル・接点 |

### 設計原則

- **ケイパビリティはステージを跨がない**（設計原則として守る）
- **組織はステージを跨ぐことがある**（現実の制約として受け入れる）
- **製品はパラメータとして流れる**（バリューストリームは共通、製品で条件が変わる）

## 2. WHY：価値

### 定義

顧客およびステークホルダーに提供する価値の意味・目的。企業理念・存在意義に相当する。

### WHATへの分解軸

WHYからWHATへの分解には複数の方向性がある。

#### 軸1：ステークホルダー軸（誰への価値か）

```
WHY: 企業の存在意義
│
├── 顧客価値（Customer Value）
│     └── 製品、サービス、体験、関係、情報
│
├── 株主価値（Shareholder Value）
│     └── 収益、成長、配当、企業価値
│
├── 従業員価値（Employee Value）
│     └── 報酬、成長機会、働きがい、安全
│
├── パートナー価値（Partner Value）
│     └── 取引機会、成長支援、関係性
│
└── 社会価値（Social Value）
      └── 環境、地域、雇用、文化
```

#### 軸2：時間軸（いつの価値か）

```
WHY: 企業の存在意義
│
├── 現在価値（Current Value）
│     └── 今、提供している価値
│
├── 将来価値（Future Value）
│     └── これから提供する価値
│
└── 持続価値（Sustainable Value）
      └── 長期的に維持すべき価値
```

#### 軸3：価値の性質軸（どんな価値か）

```
WHY: 企業の存在意義
│
├── 機能的価値（Functional Value）
│     └── 実用性、品質、性能、価格
│
├── 情緒的価値（Emotional Value）
│     └── 体験、感動、安心、喜び
│
└── 社会的価値（Social Value）
      └── 所属、ステータス、共感、貢献
```

## 3. WHAT：問題領域

### WHATの二面性

WHATには外向きと内向きの2つの側面がある。

```
WHAT = 問題領域
│
├── 価値構成要素（外向き）
│     「顧客に何を提供するか」
│
└── ケイパビリティ（内向き）
      「自社に何ができるか」
```

### 3.1 価値構成要素（外向きWHAT）

顧客に提供する価値を構成する要素。

| 要素 | 説明 | 例（飲料メーカー） |
|------|------|-------------------|
| **製品価値** | 製品そのものの価値 | 本格的な味、品質、バリエーション |
| **サービス価値** | 利便性・アクセス性 | 入手しやすさ、価格、安定供給 |
| **体験価値** | 購入・使用時の体験 | 購入の楽しさ、飲む瞬間、香り |
| **関係価値** | ブランドとの関係性 | 愛着、習慣、共感 |
| **情報価値** | 提供される情報 | 新商品情報、キャンペーン |

### 3.2 ケイパビリティ（内向きWHAT）

組織が持つ能力。「何ができるか」を表す。

#### ケイパビリティの特徴

- **What視点**：何ができるか（Howではない）
- **安定性**：数十年変わらない
- **技術非依存**：実現手段から独立
- **組織横断**：複数組織が1つのケイパビリティを担うことがある

#### ケイパビリティの階層構造（飲料メーカー例）

```
Level 1              Level 2                   Level 3
──────────────────────────────────────────────────────────────
製品開発             ├── 市場調査              ├── 消費者調査
                    ├── 商品企画              ├── コンセプト開発
                    └── 製品設計              └── レシピ開発

製造                 ├── 調達                  ├── 原料調達
                    ├── 生産                  ├── 調合
                    │                         ├── 充填
                    └── 品質管理              └── 工程検査

販売                 ├── チャネル管理          ├── コンビニ営業
                    ├── 受注管理              ├── 受注処理
                    └── 販売分析              └── POS分析

ブランド管理         ├── ブランド戦略
                    └── マーケティング
```

### 3.3 価値構成要素 × ケイパビリティ 対応分析

外向きWHATと内向きWHATの対応関係がギャップ分析の基盤となる。

```
価値構成要素（外向き）         ケイパビリティ（内向き）
─────────────────────────────────────────────────────────────

製品価値                 ←→   製品系ケイパビリティ
├── 本格的な味                 ├── レシピ開発
├── 品質・安全                 ├── 品質管理
└── バリエーション             └── 製造

サービス価値             ←→   供給系ケイパビリティ
├── 入手しやすさ               ├── チャネル管理
├── 価格の手頃さ               ├── 在庫管理
└── 安定供給                   └── 配送管理

体験価値                 ←→   体験設計系ケイパビリティ
├── 購入時の楽しさ             ├── 店頭体験設計
└── 飲む瞬間                   └── パッケージ設計

関係価値                 ←→   関係構築系ケイパビリティ
├── ブランド愛着               ├── ブランド管理
└── 習慣・安心感               └── 顧客管理

情報価値                 ←→   情報提供系ケイパビリティ
├── 新商品情報                 ├── マーケティング
└── キャンペーン               └── 販促管理
```

### 3.4 時間軸別ケイパビリティ

価値の時間軸によって必要なケイパビリティが異なる。

| 価値の時間軸 | 必要なケイパビリティ |
|-------------|---------------------|
| **現在価値** | 製造、供給、販売、ブランド管理 |
| **将来価値** | R&D、イノベーション、新規事業開発、デジタル |
| **持続価値** | ブランド管理、環境管理、人材開発、水源保全 |

## 4. HOW：バリューストリーム

### 定義

ステークホルダーに価値を提供するための一連の活動（ステージ）の流れ。

### バリューストリームの特徴

- エンドツーエンドの価値創出フローを可視化
- トリガー（開始点）から価値提供（終了点）まで
- プロセスより抽象度が高い
- 企業全体で10〜30本程度

### バリューストリームの種類

| 種類 | 説明 | 例 |
|------|------|-----|
| **価値創造系** | 新しい価値を生み出す | 製品開発、ブランド育成 |
| **価値提供系** | 価値を届ける | 製品製造、製品供給、顧客獲得 |
| **価値維持系** | 価値を維持する | 顧客維持、品質保証 |
| **支援系** | 上記を支援する | 人材確保・育成、設備管理 |

### バリューストリーム例：製品供給

```
トリガー: 顧客からの注文

[受注] → [引当] → [出荷] → [配送] → [納品] → [請求・回収]
                                                    ↓
                                           価値: 製品の届け + 代金

ステージ        内容                    主要アウトプット
─────────────────────────────────────────────────────────────
受注            注文を受け付け           受注データ
引当            在庫を確保               引当済み在庫
出荷            倉庫から出庫             出荷済み製品
配送            物流で輸送               輸送中製品
納品            顧客に届ける             納品完了
請求・回収      代金を回収               売上・入金
```

### バリューストリームの設計原則

#### 製品別バリューストリームは作らない（アンチパターン）

```
✗ 悪い例:
├── 住宅ローン提供バリューストリーム
├── マイカーローン提供バリューストリーム
└── 教育ローン提供バリューストリーム

○ 良い例:
└── 融資提供バリューストリーム（共通）
      └── 製品はパラメータとして流れる
```

#### 分ける判断基準

- **同じバリューストリームにする**：ステージの流れが同じ、必要なケイパビリティが同じ
- **別のバリューストリームにする**：ステージの流れが根本的に異なる、ステークホルダーへの価値が異なる

## 5. WHEN：ステージ

### 定義

バリューストリームを構成する各段階。価値が蓄積される区切り。

### ステージの設計原則

#### 良いステージ

- 価値が蓄積される区切り
- 判断・意思決定ポイント
- ステークホルダーとの接点

#### 悪いステージ

- 単なるタスク（「システム入力」「書類作成」）
- 担当者の切り替え（組織都合）

### ステージ × ケイパビリティ マッピング

各ステージに必要なケイパビリティをマッピングする。**ケイパビリティはステージを跨がない**。

```
製品製造バリューストリーム:

[計画] → [調達] → [調合] → [充填] → [検査] → [保管]
   │       │        │        │        │        │
   ▼       ▼        ▼        ▼        ▼        ▼
 生産計画  原料調達  調合管理  充填管理  品質検査  在庫管理
 需要予測  サプライヤー管理          ライン管理
```

## 6. 製品・ブランドの位置づけ

### 製品の定義（BIZBOK）

> 顧客のニーズを満たすために提供される、商品とサービスの組み合わせによる総合的な体験。

### 製品の位置づけ

製品は独立したドメインではなく、**価値（WHY）を構成する要素（WHAT）の一つ**。

```
WHY: 顧客価値
│
└── WHAT: 価値構成要素
      ├── 製品（Product）      ← ここ
      ├── サービス（Service）
      ├── 体験（Experience）
      ├── 関係（Relationship）
      └── 情報（Information）
```

### 飲料メーカーの製品階層

```
ブランド（マーケティング単位）
    │
    │  1:N
    ▼
製品/SKU（販売・製造単位）
    │
    │  N:1
    ▼
カテゴリ（製造技術単位）

例:
ブランド「ボス」
    ├── ボス贅沢微糖 185g缶  ─┬─→ カテゴリ「コーヒー」
    ├── ボス無糖ブラック     ─┘
    └── ボス フルーツオレ    ──→ カテゴリ「乳飲料」
```

### バリューストリームとの関係

| バリューストリーム | 関係する製品階層 |
|-------------------|-----------------|
| ブランド育成 | ブランド |
| 製品開発 | ブランド × SKU |
| 製品製造 | カテゴリ × SKU |
| 販売・流通 | SKU |

## 7. 組織の位置づけ（WHO）

### 組織とケイパビリティの関係

- 1つのケイパビリティを複数組織が担うことがある
- 1つの組織が複数のケイパビリティを持つ

### ステージとの関係

組織はステージを跨ぐことがある（現実の制約として受け入れる）。

```
バリューストリーム × 組織 マトリクス:

           [受注] [引当] [出荷] [配送] [納品]
─────────────────────────────────────────────
営業部       ●      ●                    ●
物流部              ●      ●      ●
配送業者                          ●      ●

→ 引当ステージに2組織が関与 = 引継ぎコスト発生
```

## 8. 在庫管理の構造

### 在庫ケイパビリティの分解

```
在庫管理（Level 1）
│
├── 原料在庫管理（Level 2）
│     ├── 原料受入
│     ├── 原料保管
│     └── 原料払出
│
├── 中間在庫管理（Level 2）← 液体特有
│     ├── タンク管理
│     ├── 調合液管理
│     └── バッチ追跡
│
├── 製品在庫管理（Level 2）
│     ├── 製品入庫
│     ├── 製品保管
│     ├── 在庫引当
│     └── 製品出庫
│
└── 在庫計画（Level 2）
      ├── 需要予測
      ├── 適正在庫設定
      └── 在庫分析
```

### 在庫の流れ（飲料メーカー）

```
原料倉庫        タンク          製品倉庫         得意先
   │              │                │              │
   ▼              ▼                ▼              ▼
┌──────┐      ┌──────┐         ┌──────┐      ┌──────┐
│原料  │  →   │調合液│    →    │製品  │  →   │販売  │
│在庫  │      │在庫  │         │在庫  │      │      │
└──────┘      └──────┘         └──────┘      └──────┘
                                    │
                                    ▼
                               輸送在庫（配送中）
```

### 配送と在庫の関係

- **輸送在庫**は「在庫の一形態」（在庫管理ケイパビリティの責務）
- **配送**は「在庫を移動させる活動」（配送管理ケイパビリティの責務）
- 両者はイベントで連携（出荷→輸送在庫発生、到着→輸送在庫消込）

## 9. BOM（部品表）の構造

### BOMの定義

製品を構成する部品・材料のリスト。

### 飲料のBOM構造

```
SKU: ボス贅沢微糖 185g缶
│
├── 液体: ボスコーヒー液     185 ml
│         │
│         └── レシピ（子BOM）
│               ├── コーヒー抽出液    60%
│               ├── 砂糖             3%
│               ├── 乳成分           5%
│               ├── 香料             0.1%
│               └── 水              31.9%
│
├── 容器: アルミ缶185g       1 個
├── 蓋:   プルタブ蓋         1 個
└── 包装: 24本入りトレー      1/24 個
```

### 飲料特有の課題

1. **同一液体 → 複数SKU**：1つの液体から複数の製品が生まれる
2. **液体の配合比率**：比率（%）で管理、バッチサイズで実数量が変わる
3. **副産物・ロス**：歩留まり率の管理が必要

## 11. 設計プロセス

### 価値からのバックキャスティング

```
Step 1: WHY（価値）を定義
        企業理念、ステークホルダーへの価値

Step 2: WHAT（価値構成要素）に分解
        製品、サービス、体験、関係、情報
        × ステークホルダー軸 × 時間軸 × 性質軸

Step 3: WHAT（ケイパビリティ）を導出
        各価値構成要素を実現するために必要な能力

Step 4: ギャップ分析
        価値構成要素 × ケイパビリティ のマトリクス
        現状評価と投資優先度決定

Step 5: HOW（バリューストリーム）を設計
        価値構成要素をケイパビリティで届ける流れ

Step 6: WHEN（ステージ）に分解
        各バリューストリームをステージに分解
        ステージ × ケイパビリティ マッピング

Step 7: 境界づけられたコンテキストへ
        ケイパビリティを実現するドメインを設計

Step 8: ユースケース → アグリゲート
        DDDによる実装設計
```

## 付録A: 飲料メーカー適用例

### 価値の多次元分解

```

## 付録B: 用語集

| 用語 | 定義 |
|------|------|
| **価値（Value）** | 顧客およびステークホルダーに提供する意味・目的 |
| **価値構成要素** | 価値をブレークダウンした具体的な提供物（製品、サービス、体験、関係、情報） |
| **ケイパビリティ** | 組織が「何ができるか」を表す能力の抽象的な定義 |
| **バリューストリーム** | ステークホルダーに価値を提供するための一連の活動の流れ |
| **ステージ** | バリューストリームを構成する各段階 |
| **境界づけられたコンテキスト** | DDDにおける、特定のドメインモデルが適用される範囲 |
| **BOM（部品表）** | 製品を構成する部品・材料のリスト |
| **輸送在庫** | 出荷済み・未着荷の在庫（In-Transit Inventory） |

## 付録C: V5適用分析（2026-01-13）

### C.1 既存V5との整合性評価

| 文書の概念 | V5での対応 | 整合性 | 備考 |
|-----------|----------|--------|------|
| WHY（価値） | L1価値レイヤー | ✅ 整合 | V5のVision→Mission→Strategy→VS→VLと対応 |
| WHAT（ケイパビリティ） | CL1/CL2/CL3階層 | ⚠️ 調整必要 | V5は3階層+BC、文書は3レベル |
| HOW（バリューストリーム） | Value Stream | ✅ 整合 | 同一概念 |
| WHEN（ステージ） | VS0-VS7 | ⚠️ 調整必要 | V5は8標準ステージ定義済み |
| WHO（組織） | 横軸として扱う | ✅ 整合 | V5でも組織はクロスカッティング |
| WHERE（チャネル） | V5では明示なし | 📝 拡張候補 | 有用な視点 |

### C.2 V5との主要な差異

#### C.2.1 バリューステージの標準化

**文書の記述**：ステージを汎用的に定義

**V5の定義**：8標準ステージ
```
VS0: Strategy（戦略立案・方針策定）
VS1: Discovery/Insights（市場調査・顧客理解）
VS2: Product Development（製品開発・イノベーション）★Core
VS3: Branding/Marketing（ブランディング・販促）★Core
VS4: Distribution/Supply Chain（流通・サプライチェーン）
VS5: Customer Support/Quality Assurance（顧客対応・品質保証）
VS6: Data/DX（データ活用・デジタル化）
VS7: Operations/Improvement（運用・改善）
```

**推奨**：飲料メーカー例のバリューストリーム「製品製造」「製品供給」は、VS2/VS4にマッピングすべき

#### C.2.2 TVDC分類の欠落

文書ではケイパビリティの分類基準が明示されていない。V5.5ではTVDC（Three-Dimensional Value-Driven Classification）を採用：

```
          高  ┌────────────────────────────────────┐
              │           │                        │
         価   │  Supporting│      Core             │
         値   │  （効率化）  │   （競争優位）         │
         必   │           │                        │
         然   ├───────────┼────────────────────────┤
         性   │           │                        │
              │  Generic  │      VCI               │
              │（標準化）  │ （価値維持基盤）        │
         低   │           │                        │
              └────────────────────────────────────┘
                   低      差別化度              高
```

#### C.2.3 価値構成要素（外向きWHAT）の位置づけ

**文書の独自貢献**：
```
価値構成要素（外向きWHAT）
├── 製品価値（Product Value）
├── サービス価値（Service Value）
├── 体験価値（Experience Value）
├── 関係価値（Relationship Value）
└── 情報価値（Information Value）
```

これはV5で明示的に定義されていない概念で、**Value Proposition Canvas**的な視点を提供。V5.6への拡張候補として有用。

### C.3 V5適用のための修正提案

#### C.3.1 ステージ→VS標準へのマッピング

**製品製造バリューストリーム → VS2（Product Development）配下**
```
VS2: Product Development
│
├── VL2-1: 計画（生産計画策定）
├── VL2-2: 調達（原料調達完了）
├── VL2-3: 調合（製品液調合完了）
├── VL2-4: 充填（製品充填完了）
├── VL2-5: 検査（品質検査合格）
└── VL2-6: 保管（製品在庫化）

※VL = Value Level（価値水準）：各ステージでの価値蓄積
```

**製品供給バリューストリーム → VS4（Distribution/Supply Chain）配下**

VS4: Distribution/Supply Chain
│
├── VL4-1: 受注（注文受付完了）
├── VL4-2: 引当（在庫確保完了）
├── VL4-3: 出荷（出庫完了）
├── VL4-4: 配送（輸送中→到着）
├── VL4-5: 納品（顧客受取完了）

#### C.3.2 ケイパビリティ階層のV5対応

```

            │
            └── BC: 境界づけられたコンテキスト（CL2分類を継承）
```

**飲料メーカー例のV5対応**：
```
CL1: 製造（Activity Area）
│

### C.5 結論

この文書は、Parasol V5の概念を独自の視点で整理した有用なリファレンスであり、以下の点で価値がある：

**強み**：
1. **5W1Hによる明快な構造化** - V5の複雑な概念をアクセスしやすく整理
2. **外向きWHAT/内向きWHATの区別** - 価値と能力の関係を明確化
3. **飲料メーカー固有のドメイン知識** - BOM、中間在庫など業界特有の概念を詳細化
4. **設計原則の明示** - 「ケイパビリティはステージを跨がない」等

**V5適合のための調整必要事項**：
1. バリューステージをVS0-VS7標準にマッピング
2. ケイパビリティにTVDC分類を適用
3. CL1→CL2→CL3→BCの階層名称をV5に統一
4. 価値トレーサビリティ検証の組み込み

**V5.6拡張への貢献候補**：
- 価値構成要素フレームワーク（Value Proposition分解）
- 価値構成要素×ケイパビリティ マトリクス
