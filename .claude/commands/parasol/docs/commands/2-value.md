# Phase 2: Value Definition & Strategic Capabilities - 価値定義・戦略ケイパビリティ

企業の価値創造を定義し、**業種・セグメント特性に応じた**Value Stages (VS0-VS7) を確立します。さらに、**V4統合により戦略ケイパビリティ層（L1 Strategic Capabilities）**を構築し、価値実現の基盤を設計します。

**重要**: 
- Value Stage は「組織の活動」ではなく「顧客の状態」を定義します
- **戦略ケイパビリティは価値ストリームを実現可能にする組織能力を定義します**

## 使用方法

```bash
# Phase 2.1: Value Definition（従来）
/parasol:2-value              # インタラクティブ選択（業種→セグメント→VS）
/parasol:2-value industry     # 業種・セグメント設定
/parasol:2-value VS0          # VS0を直接指定
/parasol:2-value VS2          # VS2を直接指定
/parasol:2-value all          # 全VS概要を生成
/parasol:2-value exploration  # 6軸探索モード（capability/business-unit/value/platform/fusion/multi-tier-vstr）
/parasol:2-value es-ps        # ES/PS VStr（従業員・パートナー向け）設計

# Phase 2.2: Strategic Capabilities（V4統合）
/parasol:2-strategic          # L1戦略ケイパビリティ設計
/parasol:2-strategic map      # VL1→L1→VS価値マッピング
/parasol:2-strategic maturity # 戦略ケイパビリティ成熟度評価
/parasol:2-strategic vms2      # VMS2マイルストーン設定

# Phase 2.3: Value Component Analysis（V5.6）
/parasol:2-vc                 # Value Component分析
/parasol:2-vc decompose       # 3軸価値分解（ステークホルダー/時間/性質）
/parasol:2-vc vs-matrix       # VS-VC優先度マトリクス生成
/parasol:2-vc portfolio       # Value Portfolio（投資配分）設計
```

## 目的

**Phase 2.1: Value Definition（従来）**
- **Phase 1からの入力を活用する** - コングロマリットスコア・アクターインベントリを基に軸パターンを自動推奨
- **業種・セグメントを特定する** - 適切なVSパターンの選択
- **Value Proposition（価値提案）を明確化する** - 誰に、何の価値を、どのように提供するか
- **バリューステージ (VS0-VS7) を定義する** - 顧客の状態遷移として価値創造の全体像を定義
- **多段VStr（ES/PS）の必要性を判定する** - 従業員・パートナー向けVStrの設計判断

**Phase 2.2: Strategic Capabilities（V4統合）**
- **価値ヒエラルキーを戦略ケイパビリティに変換する** - VL1→VL2→VL3からL1戦略能力を設計
- **価値ストリーム実現のための組織能力を定義する** - VS実現に必要な戦略的能力
- **価値成熟度評価システムを構築する** - 各戦略能力の現在・目標・ギャップを測定
- **VMS2マイルストーンを設定する** - 戦略能力構築の成功基準と測定指標
- **Phase 3 Capabilitiesへの価値トレーサビリティを確立する** - L1→CL1→CL2→CL3の価値追跡可能性

**Phase 2.3: Value Component Analysis（V5.6）**
- **VL3をValue Componentで分析する** - 各詳細価値の5VC構成（製品/サービス/体験/関係/情報）を定義
- **3軸価値分解を実行する** - ステークホルダー軸/時間軸/性質軸による多角的価値分析
- **VS-VC優先度マトリクスを作成する** - 各Value Stageで重要なVCを特定
- **Value Portfolioを設計する** - VC別・TVDC別の投資配分を決定
- **Phase 3への価値引継ぎを準備する** - VC-Capability対応の基盤情報

## 🎯 設計哲学の適用

### 🔬 VL: 情報公理適用

> 📚 [zigzag-foundations.md](../philosophy/zigzag-foundations.md#axiomatic-designの2つの公理)

**なぜVLに情報公理か**:

価値構造（VL階層）が複雑だと、後続フェーズ全体が肥大化します。

```
VL階層が深い → CL3が爆発的に増加 → BC数も爆発 → 保守不能なシステム
```

**VLチェック（情報公理）**:
- [ ] VL階層 ≤ 3（VL1→VL2→VL3で十分）
- [ ] 重複VLなし（同じ価値を複数箇所で定義しない）

**具体例**:
```
❌ 情報公理違反:
  VL1: 健康的な生活を支援
    └─ VL2: 食事管理
        └─ VL3: 栄養計算
            └─ VL4: カロリー計算      ← 4階層目は過剰！
                └─ VL5: 三大栄養素計算  ← 5階層目は完全に過剰！

✅ 情報公理遵守:
  VL1: 健康的な生活を支援
    └─ VL2: 食事管理
        └─ VL3: 栄養バランス可視化（カロリー・三大栄養素含む）
                 ↑ 3階層で必要十分！
```

**違反した場合の影響**:
| 指標 | 情報公理違反 | 情報公理遵守 |
|------|-------------|-------------|
| CL3数 | 50+ → 管理不能 | 15-25 → 管理可能 |
| BC数 | 30+ → チーム分割困難 | 5-10 → 適切なチーム編成 |
| 開発期間 | 2-3年 | 6-12ヶ月 |

### 🧠 VS: 認知負荷管理

> ⚠️ **注意**: VSは「設計パラメータ」ではなく「顧客状態の記述」です。
> 情報公理ではなく、認知科学の原則（Miller's Law: 7±2）を適用します。

**なぜVSに認知負荷管理か**:

VSは顧客ジャーニーの時間軸（WHEN）を表します。主語は「組織」ではなく「顧客」です。

| 項目 | VL（Value Level） | VS（Value Stage） |
|------|------------------|-------------------|
| **本質** | 組織が設計する価値構造 | 顧客状態の遷移 |
| **主語** | 組織 | 顧客 |
| **適用原則** | 情報公理 | 認知負荷管理（7±2） |

**VSチェック（認知負荷管理）**:
- [ ] VS数 5-8（Miller's Law: 人間が一度に把握できる7±2の範囲）
- [ ] 各VSが明確な顧客状態変化を表現
- [ ] ステークホルダーが直感的に理解できる粒度

**違反した場合の影響**:
| 指標 | VS過多（10+） | VS適正（5-8） |
|------|--------------|--------------|
| 理解 | ステークホルダー間で認識齟齬 | 共通理解が容易 |
| 設計 | CL分解が複雑化 | 適切な粒度でCL分解 |
| 運用 | 顧客ジャーニー追跡困難 | 明確なステージ管理 |

### 価値の実証要件 - 想像から観察へ

> **重要**: すべての価値主張は**測定可能な現実**に基づく必要があります

#### 必須の価値検証要件

**現在の状態（ベースライン）**:
- 現在の顧客行動パターンの観察データ
- 実際に測定されている指標とその値
- 顧客の声（実際のフィードバック・苦情・要望）
- 現在の収益構造と顧客ライフサイクル

**目標の状態（測定可能な改善）**:
- 具体的な数値目標（売上、満足度、効率性等）
- 成功の測定方法とタイミング
- 失敗の早期警告指標
- ROI/KPIの定義と測定計画

**価値実現の検証方法**:
- 実験可能な仮説設定
- 段階的な価値検証計画
- ステークホルダーの合意確認
- リスクと制約の現実的な評価

#### ❌ 避けるべき「想像の価値」パターン

- 競合他社の成功例をそのまま仮定
- 顧客調査なしの価値主張
- 測定不可能な抽象的な価値
- 現在の問題を把握しない改善計画
- 実現可能性を検証しない理想状態

### 方法論の全体フロー（価値先行アプローチ）

**重要**: 価値マイルストーン（MS）は「プロセス完了」ではなく「**顧客が得ている価値状態**」として定義します。

```
STEP 1: 価値分解（VLだけを先に完成）
┌────────────────────────────────────────────┐
│  VL1（最上位価値）                          │
│    ↓                                       │
│  VL2（価値グループ）                        │
│    ↓                                       │
│  VL3（詳細価値）← 実行可能な粒度まで分解    │
└────────────────────────────────────────────┘
              ↓ VL分解が完了してから

STEP 2: 価値マイルストーン導出（バックキャスト）
┌─────────────────────────────────────────────┐
│  VMS5「顧客が理想の価値を享受している」← 起点 │
│    ↑                                        │
│  VMS4「顧客が全面的に価値を実感している」     │
│    ↑                                        │
│  VMS3「顧客が主要価値を体験できる」           │
│    ↑                                        │
│  VMS2「顧客が価値を認識し選択できる」         │
│    ↑                                        │
│  VMS1「顧客が最初の価値を体験できる」         │
│                                             │
│  ※主語は常に「顧客」、内容は「状態」         │
└─────────────────────────────────────────────┘
              ↓ VMSが確定してから

STEP 3: バリューストリーム設計（VS0-VS7）
┌────────────────────────────────────────────┐
│  VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7 │
│  潜在  認知  選好  決定  獲得  実感  継続  共創  │
└────────────────────────────────────────────┘
              ↓ VSが確定してから

STEP 4: VL×VSマッピング
┌────────────────────────────────────────────┐
│  どの価値（VL3）がどのステージ（VS）で       │
│  実現されるかを可視化                        │
└────────────────────────────────────────────┘
              ↓

STEP 5: 優先順位決定
┌─────────────────────────────────────────────┐
│  次のVMS達成に必要なVS×VLの組み合わせを特定  │
│  → 実行計画に反映                            │
└─────────────────────────────────────────────┘
```

### 主要概念

| 概念 | 説明 |
|------|------|
| **価値分解（VL1→VL2→VL3）** | 最上位価値を階層的に分解し、実行可能な粒度まで詳細化 |
| **バックキャスティング** | 理想の未来状態（VMS5）から現在に向かって逆算するアプローチ |
| **価値マイルストーン（VMS1-VMS5）** | **顧客が得ている価値状態**（❌プロセス完了ではない） |
| **バリューステージ（VS0-VS7）** | 顧客の状態遷移を定義する価値の流れ |
| **VL×VSマッピング** | どの価値がどのステージで実現されるかの可視化 |
| **優先順位決定** | VMS達成に必要なVS×VLの組み合わせを特定 |

### Value Spaceの3軸モデル（WHY / HOW / WHEN）

Value Spaceは内部に3つの軸を持ちます：

| 軸 | 概念 | 略称 | 説明 | 主語 |
|----|------|------|------|------|
| **WHY** | Value Level | VL1-VL3 | 顧客に**何を**届けるか | 組織 |
| **HOW** | Value Stream | VStr | 価値が**どう**流れるか | 組織 |
| **WHEN** | Value Stage | VS0-VS7 | **いつ**価値を届けるか（顧客状態遷移） | **顧客** |

```
WHY = 顧客価値レベル（VL1→VL2→VL3）
      「何を顧客に届けるか」
                │
    ┌──────────┴──────────┐
    ▼                     ▼
HOW = バリューストリーム    WHEN = バリューステージ
      (VStr)                    (VS0-VS7)
      「どう価値を流すか」       「いつ価値を届けるか」
      全体フロー                 顧客ジャーニー時間軸
```

**重要な区別**:
- **WHY・HOW**: 組織視点（価値を「届ける」側）
- **WHEN**: 顧客視点（価値を「受け取る」側の状態変化）

> 📚 詳細は [_value-methodology.md](./_value-methodology.md#value-spaceの3軸モデルwhy--how--when) を参照

### ⚠️ VMSの正しい定義

| 観点 | ❌ 誤（プロセス完了） | ✅ 正（顧客価値状態） |
|------|---------------------|---------------------|
| **主語** | 組織（我々が〜した） | **顧客**（顧客が〜を得ている） |
| **記述** | 活動の完了 | **状態の達成** |
| **例** | 「システム開発完了」 | 「顧客が24時間注文可能」 |
| **測定** | タスク完了率 | **顧客価値の実現度** |

### バリューストリームの設計原則

#### ⚠️ 最重要原則：1価値定義 = 1バリューストリーム

**バリューストリームは価値定義からバックキャスティングして作成する「1本の統合されたストリーム」です。**

```
価値定義（VL3）
    ↓ バックキャスティング
┌─────────────────────────────────────────────────────┐
│      1本のバリューストリーム（VS0 → VS1 → ... → VS7）    │
│                                                     │
│  ※複数の並列ストリームではない                      │
│  ※VS0-VS7は1本のストリーム内のステージ              │
└─────────────────────────────────────────────────────┘
```

#### パターン別バリューストリーム数

| パターン | 価値定義 | バリューストリーム数 |
|---------|---------|-------------------|
| **統合型（価値軸）** | 1つの統合価値 | **1本** |
| **セグメント型（事業部軸）** | 事業部別価値 | **事業部数と同じ本数**（例：飲料1本、酒類1本） |
| **ハイブリッド型** | 統合 + セグメント | **統合1本 + セグメント数** |

#### アンチパターン（避けるべき設計）

| ❌ 誤った設計 | ✅ 正しい設計 |
|-------------|-------------|
| 機能別に複数VStr作成（価値創造VStr、価値提供VStr等） | 1本のVStrにVS0-VS7として統合 |
| 製品別にVStr作成（天然水VStr、茶VStr等） | 事業部レベル（飲料VStr）で1本 |
| バリューステージ（VS0-VS7）を並列ストリーム扱い | VSはストリーム内のステージ（直列） |
| **シナジー駆動VStr統合**（「クロスセルのため」でVStr統合） | 事業間シナジーはSupporting/Generic Layerで実現 |
| **コングロマリット単一VStr**（全事業を1つのVStrで表現） | 事業部ごとに独立VStrを設計（事業部軸） |
| **「シナジーVStr」の定義**（事業部間連携をVStrとして定義） | シナジーはBC間関係（Context Map）で表現 |

#### 正しい設計フロー

バリューストリーム自体も、**最終価値提供者（End Value Recipient）に対して価値ある状態を定義**し、バックキャスティングで組み立てます：

1. **最終価値提供者を特定** - 誰に価値を届けるか
2. **理想の最終状態を定義** - その人にとっての最終的な価値ある状態
3. **逆算してステージを構築** - VS7 ← VS6 ← ... ← VS1 ← VS0

### 🎯 V5特有機能: 設計ストーリー出力

**重要**: Parasol V5では、価値分解・バックキャスティングの各段階で**設計ストーリー（なぜそう設計したか）**を出力します。

- **理解促進**: なぜこの構造にしたかを明確にし、チーム全員の理解を深める

#### 出力タイミング

| フェーズ | 出力内容 |

### 🎯 V5.1特有機能: VL3価値必然性評価（TVDC連動）

**重要**: Parasol V5.1では、VL3（価値要素）定義時に**価値必然性**を評価し、Phase 3のTVDC分類に継承します。

> 📚 詳細は [TVDC-FRAMEWORK.md](../../concepts/TVDC-FRAMEWORK.md#8-phase-2との連動価値分解からの価値必然性継承) を参照

#### なぜPhase 2で価値必然性を評価するか

```
従来（連動なし）:
Phase 2: 価値分解     Phase 3: TVDC分類
   VL3定義 ─────?─────→ 価値必然性★★★を再評価
                       差別化★★★を評価
                       → 評価が重複、根拠が曖昧

V5.1（連動あり）:
Phase 2: 価値分解     Phase 3: TVDC分類
   VL3定義            差別化★★★のみ評価
     └─ 価値必然性★★★ ─継承→ 価値必然性は継承
     └─ VCI候補フラグ  ─継承→ VCI候補として認識
                       → 効率的、根拠が明確
```

#### VL3価値必然性評価テンプレート

VL3定義時に以下を追加：

```yaml
VL3-X-Y:
  name: "{価値要素名}"
  description: "{価値要素の説明}"
  parent: VL2-X

  # === 価値必然性評価（TVDC連動） ===
  value_necessity:
    score: "★★★"  # ★（低）/ ★★（中）/ ★★★（高）

    evaluation:
      # Q1: この価値要素がないと顧客価値は？
      value_impact: "完全毀損"  # 完全毀損 / 部分低下 / 影響小
      value_impact_reason: "この能力がないと商品を届けられない"

      # Q2: VSTR状態遷移に必須か？
      state_transition_required: true
      related_stages: ["VS3", "VS4"]
      transition_reason: "VS3→VS4の購入完了に絶対必須"

      # Q3: 失敗時の影響は？
      failure_impact: "致命的"  # 致命的 / 重大 / 軽微
      failure_reason: "配送失敗は返金・信頼喪失に直結"

    # VCI候補識別
    vci_candidate: true  # 価値必然性★★★ かつ 差別化しにくい場合
    vci_reason: "失敗は致命的だが、他社も同等サービスを提供"

  # === Phase 3への引継ぎ情報 ===
  phase3_handoff:
    target_capability: "delivery-fulfillment"
    inherited_necessity: "★★★"  # CL2に継承される
    differentiation_hints: |
      - 翌日配送で差別化可能か検討
      - 配送追跡UXで差別化可能か検討
      - 時間指定精度で差別化可能か検討
```

## 🏗️ ステージ・ケーパビリティ・セグメント構造

### 構造的原則

**重要**: ステージとケーパビリティは「階層」ではなく「マッピング」の関係です。

```
価値軸（横）: 認知 → 検討 → 購入 → 使用 → 継続 → 推奨
              ↑
              │ 直交（異なる次元）
              ↓
事業軸（縦）: 事業A / 事業B / 事業C / ...
```

**解決**: 顧客セグメントを接点層として設計

```
┌─────────────────────────────────────────────────────────────────────┐
│                  市場接点層（Layer 3）                                │
│              顧客価値ステージ（共通フレーム）                          │
│     認知 → 検討 → 購入 → 初回体験 → 継続 → 推奨                      │
└─────────────────────────────────────────────────────────────────────┘
                              ↓ マッピング
┌─────────────────────────────────────────────────────────────────────┐
│                  顧客セグメント層（接点層）                           │
│   ┌─────────┬─────────┬─────────┬─────────┬─────────┐              │
│   │ セグメントA│ セグメントB│ セグメントC│ セグメントD│ セグメントE│              │
│   │ （事業A）│ （事業B）│ （事業C）│ （事業D）│ （事業E）│              │
│   └─────────┴─────────┴─────────┴─────────┴─────────┘              │
└─────────────────────────────────────────────────────────────────────┘
                              ↓ 実現
┌─────────────────────────────────────────────────────────────────────┐
│                  ケーパビリティ層（Layer 1-2）                        │
│   L1: 共通ケーパビリティ（事業横断・DX優先）                          │
│   L2: ドメインケーパビリティ（セグメント固有）                        │
│   L3: 製品ケーパビリティ（ブランド固有）                              │
└─────────────────────────────────────────────────────────────────────┘
```

### 3層ケーパビリティモデル

| 層 | 定義 | 粒度 | DX優先度 | 例 |
|---|---|---|---|---|
| **L1: 共通** | 事業横断で再利用可能 | 大 | 最高 | 顧客データ基盤、EC基盤、品質管理 |
| **L2: ドメイン** | 顧客セグメント固有 | 中 | 高 | 育児支援、シニア栄養管理 |
| **L3: 製品** | ブランド固有 | 小 | 中 | 個別レシピ提案、フレーバー開発 |

### スイムレーン設計

各セグメントごとにバリューステージをスイムレーンとして整理：

```
┌─────────────────────────────────────────────────────────────────────┐
│ スイムレーン: [セグメント名]バリューステージ                          │
├─────────┬─────────┬─────────┬─────────┬─────────┬─────────┤
│ Stage1  │ Stage2  │ Stage3  │ Stage4  │ Stage5  │ Stage6  │
│ [状態]  │ [状態]  │ [状態]  │ [状態]  │ [状態]  │ [状態]  │
├─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│[活動]   │[活動]   │[活動]   │[活動]   │[活動]   │[活動]   │
└─────────┴─────────┴─────────┴─────────┴─────────┴─────────┘
         ↓           ↓           ↓           ↓           ↓
      [Cap L?]    [Cap L?]    [Cap L?]    [Cap L?]    [Cap L?]
```

### ステージ×ケーパビリティ マッピング

ケーパビリティは**複数のステージに「寄与」**する：

```
              認知    検討    購入    使用    継続    推奨
          ┌───────┬───────┬───────┬───────┬───────┬───────┐
顧客DB    │   ○   │   ○   │   ◎   │   ◎   │   ◎   │   ◎   │ L1
          ├───────┼───────┼───────┼───────┼───────┼───────┤
EC基盤    │       │   ○   │   ◎   │       │   ◎   │       │ L1
          ├───────┼───────┼───────┼───────┼───────┼───────┤
CRM基盤   │       │       │   ○   │   ◎   │   ◎   │   ◎   │ L1
          └───────┴───────┴───────┴───────┴───────┴───────┘
          ◎: 強い寄与  ○: 寄与あり
```

### DX進化単位

**推奨**: L1ケーパビリティを優先（最大レバレッジ）

1. **L1優先**: 事業横断で共有されるケーパビリティをDX対象に
2. **L2次点**: セグメント固有だが再利用可能な部分
3. **L3最後**: ブランド固有の差別化要素

### fusion-axis（融合軸）

**目的**: 顧客体験が事業を跨いで連続する場合の統合設計

> ⚠️ **シナジー≠VStr原則**: シナジー（コスト削減・効率化・クロスセル）はVStr統合の理由にならない。
> VStr統合は「顧客状態遷移が事業を跨いで連続する場合」のみ正当化される。

**フュージョン型選択前のチェック**:
- [ ] 顧客が事業を跨いで状態遷移するか？（例: 酒類購入者が飲料も購入する自然な行動）
- [ ] その遷移は顧客の意思か、企業の都合か？

❌ 企業の都合（シナジー追求）→ 事業部軸を維持、シナジーはSupporting/Generic Layerで実現
✅ 顧客の自然な行動 → フュージョン型検討可

**構造**:
```
統合VStr（HD統括）
    │
    ├── 共通基盤・研究開発の共有
    ├── ブランド価値の相互強化
    └── サステナビリティ指標の統一
    │
    ▼ 価値・機能の提供
    │
┌───┴───┬───────┬───────┐
▼       ▼       ▼       ▼
事業A   事業B   事業C   ...
VStr    VStr    VStr
```

**適用場面**:
- R&D成果の横展開
- ブランド価値の共有
- サステナビリティ目標の統一

### capability-axis（ケイパビリティ軸）

**目的**: ケーパビリティ分析に基づく無駄排除・効率化

**設計原則**:
- 統合可能なケーパビリティを融合（共通システム化）
- 個別必須なケーパビリティを抽出（事業別保持）
- 無駄なシステム重複を回避

**Phase 3（CL1）での検討項目**:

| # | 検討項目 | 判定基準 |
|---|---------|---------|
| 1 | 各CL1ドメインの統合可能性判定 | 全事業で共通化できるか？ |
| 2 | 個別保持の必要性判定 | 事業特性で差別化が必要か？ |
| 3 | 部分統合の可能性 | 一部共通・一部個別のパターン |
| 4 | 統合時のシナジー効果 | コスト削減・品質向上等 |
| 5 | 個別保持時の差別化価値 | 事業競争力への貢献 |

**判定カテゴリ**:
```
A: 統合（共通システム化）
B: 個別（事業別システム）
C: ハイブリッド（基盤は共通、実装は個別）
```

**適用場面**:
- IT基盤・DX基盤の統合
- バックオフィス機能の共通化
- 物流・SCMの最適化

### zen-architect の活用（ケーパビリティ統合判定）

CL1分析時にケーパビリティ統合型の構造を判定するためのサブエージェント活用：

   - 共通化のデメリット（柔軟性低下、調整コスト）

   - 個別保持のデメリット（重複コスト）

### platform-axis（プラットフォーム軸）

**目的**: 需要者と供給者の両面市場を最適化するマーケットプレイス・エコシステム型ビジネス向け

**構造**:
```
┌──────────────────────────────────────────────────────────────────────┐
│  【需要者向け VStr】(Demand Side) - DS0〜DS7                          │
│  価値の受け手: 購入者/利用者（買い手）                                │
│  ニーズ潜在 → 認知 → 探索 → 選択 → 取引 → 利用 → 継続 → 推奨         │
└──────────────────────────────────────────────────────────────────────┘
                        ↕ マッチング
┌──────────────────────────────────────────────────────────────────────┐
│  【プラットフォーム基盤】Core Platform                                │
│  マッチング │ 決済 │ 信頼構築 │ サポート │ データ                     │
└──────────────────────────────────────────────────────────────────────┘
                        ↕ マッチング
┌──────────────────────────────────────────────────────────────────────┐
│  【供給者向け VStr】(Supply Side) - SS0〜SS7                          │
│  価値の受け手: 出品者/提供者（売り手）                                │
│  参入検討 → 認知 → 登録 → 出品 → 取引 → 収益 → 拡大 → パートナー化    │
└──────────────────────────────────────────────────────────────────────┘
```

**適用条件**:
- [ ] 需要者と供給者をマッチングするプラットフォーム事業
- [ ] ネットワーク効果（片側または双方向）が事業価値の源泉
- [ ] 取引手数料/サブスクリプションが主要収益モデル

**業界別適用例**:
| 業界 | 需要者 | 供給者 | 例 |
|------|--------|--------|-----|
| ECマーケットプレイス | 購入者 | 出品者 | メルカリ、Amazon Marketplace |
| シェアリングエコノミー | 利用者 | ホスト/ドライバー | Airbnb、Uber |
| 人材マッチング | 企業（採用側） | 求職者 | Wantedly、LinkedIn |

→ 詳細: `_patterns/_axes/platform-axis.md` 参照

## Value Stages (VS0-VS7) について

**重要**: Value Stage は「組織が何をするか」ではなく「顧客がどの状態にあるか」を定義します。

### 顧客状態としてのVS定義

| VS | 顧客状態 | 顧客視点の定義 | 対応するEnterprise Activity（EA）例 |
|----|----------|----------------|-------------------------------------|
| **VS0** | 潜在ニーズ状態 | 「まだ気づいていないが、この価値を必要としている」 | 市場調査、ニーズ発掘 |
| **VS1** | 認知到達状態 | 「この価値の存在を知る」 | マーケティング、ブランディング |
| **VS2** | 選好形成状態 | 「この価値を選びたいと思う」 | 製品開発、差別化戦略 |
| **VS3** | 意思決定状態 | 「この価値を獲得することを決める」 | 販売、提案活動 |
| **VS4** | 価値獲得状態 | 「この価値を手に入れる」 | デリバリー、オペレーション |
| **VS5** | 価値実感状態 | 「この価値を実感している」 | カスタマーサクセス、サポート |
| **VS6** | 価値継続状態 | 「この価値を継続的に享受している」 | リテンション、ロイヤルティ施策 |
| **VS7** | 価値共創状態 | 「この価値を他者と共有・推奨している」 | コミュニティ、アドボカシー |

### 設計原則

1. **主語は常に「顧客」** - 組織ではなく顧客の状態変化を記述
2. **バックキャスティング** - VS7（理想状態）からVS0に向かって逆算設計
3. **EA（Enterprise Activity）との分離** - 組織活動はVSを実現する手段として別途定義

これらは業種を問わず適用可能な汎用的な顧客状態遷移フレームワークです。
プロジェクトの特性に応じて、各VSの具体的な顧客状態を定義してください。

## 🔄 顧客価値と組織能力の統合ガバナンス

Phase 2では、顧客価値（VS）と組織能力（CL）の間で適切な統合・調整を行う必要があります。

### 統合ポイント

│                                                                                  │

### 原則: 顧客価値優先（Customer Value First）

> **組織能力は「顧客価値実現の手段」であり、目的ではない。**
> **判断に迷った場合は、常に「顧客が得る価値」を優先する。**

### 統合チェックリスト

各統合ポイントで以下を確認：

| # | チェック項目 | 確認タイミング |
|---|-------------|---------------|
| 1 | この組織能力は、顧客が得る価値に変換できるか？ | VMS→VS変換時 |
| 2 | 変換できない能力は、Supporting/Genericに分類されているか？ | ドメイン分類時 |
| 3 | Core Domainの能力は、顧客価値に直結しているか？ | CL1分類時 |
| 4 | 投資判断は、顧客価値への貢献度で優先順位がついているか？ | 投資配分時 |
| 5 | 顧客状態（VS）から逆算して必要能力を導出しているか？ | Capability導出時 |

### トレードオフ判断基準

顧客価値と組織効率が対立する場合の判断基準：

| 状況 | 判断 | 例 |
|------|------|-----|
| 組織効率↑ but 顧客価値→ | **実施**（価値を毀損しない限りOK） | バックオフィス自動化 |
| 組織効率↑ but 顧客価値↓ | **慎重検討**（顧客価値毀損を最小化） | コスト削減による品質低下 |
| 組織効率↓ but 顧客価値↑ | **優先実施**（顧客価値が最優先） | 手厚いサポート体制構築 |

### 統合ガバナンスの適用例

```
例: 「コールセンター統合」の判断

❌ 組織視点のみ:
   「コスト削減のため全事業部のコールセンターを統合」
   → VMS実現のみを追求

✅ 顧客価値視点を統合:
   「顧客は専門的な回答を求めている（顧客価値）」
   「専門性が薄れると顧客価値が低下」
   「一次対応は統合、専門対応は事業部別を維持」
   → 組織効率と顧客価値のバランス
```
