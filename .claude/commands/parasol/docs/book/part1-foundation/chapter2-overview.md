# 第2章　Parasol V5の全体像 ― ビジネス変革への8つの旅路

## はじめに：理想のシステムを建築する

理想の家を建てることを想像してみてください。

いきなり工事を始める人はいません。まず土地を調査し、家族のニーズを聞き、予算を確認します。そして最も重要なのが「設計」です。優れた設計なくして、快適な家は生まれません。

複雑なビジネスシステムの構築も、まさに「建築」と同じです。しっかりとした設計図なしに、いきなりコーディングを始めても、住みやすい（使いやすい）システムは作れません。

Parasol V5は、あなたが理想のシステムを「建築」するための、体系的な設計プロセスです。経験豊富な建築家のように、ビジネスの要求を美しく機能的なシステムへと設計する方法を、段階的に導きます。

前章で、なぜParasol V5が必要なのかを理解していただきました。変化し続ける要求、多様なステークホルダー、積み重なる技術負債—— これらの課題に対して、V5は新しいアプローチを提示します。

本章では、そのV5の全体像を俯瞰します。8つのフェーズ（登山でいうキャンプ地）を経て、どのようにビジネスの要求が動くシステムへと変換されていくのか。その道筋を、一緒に辿っていきましょう。

## 本章で学ぶこと

この章を読み終えた時、あなたは以下のことが理解できているはずです：

- **8フェーズプロセス**：Phase 0からPhase 7まで、各段階で何をするのか、なぜその順番なのか
- **価値駆動設計**：すべての設計判断が「価値」を軸に行われる仕組み
- **トレーサビリティ**：要求から実装まで、一本の糸でつながる追跡可能性
- **6軸探索**：見落としを防ぐ、多角的な分析アプローチ
- **品質ゲート**：各フェーズで品質を保証する仕組み

難しそうに聞こえるかもしれません。でも安心してください。一つずつ、具体例を交えながら説明していきます。

なぜ8つものフェーズが必要なのでしょうか？

それは、**複雑さを管理可能なサイズに分解する**ためです。一度に全てを考えようとすると、頭がパンクしてしまいます。でも、「今はPhase 2だから、価値の流れだけに集中すればいい」となれば、扱いやすくなります。

### 3つの大きな変換ポイント

8つのフェーズは、大きく3つのグループに分けられます：

#### 1. 調査・基本設計（Phase 0-2）
「どんな家（システム）を建てるか構想する」段階です。

- **Phase 0**: 施主（ステークホルダー）の要望を聞く
- **Phase 1**: 土地（既存環境）の条件を調査する  
- **Phase 2**: 家のコンセプト（価値）を設計する

優れた建築家は、いきなり図面を引きません。まず施主の生活スタイルを理解し、土地の特性を活かした設計を考えます。

#### 2. 詳細設計（Phase 3-5）
「設計を具体的な図面に落とし込む」段階です。

- **Phase 3**: 間取り（機能配置）を決める
- **Phase 4**: 構造（アーキテクチャ）を設計する
- **Phase 5**: 配管・配線（API・データフロー）を設計する

ここが設計の核心です。美しいコンセプトを、実際に建築可能な詳細図面へと具体化していきます。

#### 3. 建設・仕上げ（Phase 6-7）
「設計を形にし、住める家にする」段階です。

- **Phase 6**: 内装・設備（動的な機能）を整える
- **Phase 7**: インフラ（電気・水道・ガス）を整備する

設計図を元に、実際に使えるシステムを構築します。良い設計があれば、建設はスムーズに進みます。

### 価値中心アプローチ ― すべては価値から始まる

V5の最大の特徴は、**すべての判断が「価値」に基づく**ことです。

従来の開発では、こんなことがよく起きます：

「この機能、なんで作ったんだっけ？」
「誰が使うんだろう、これ？」
「要件定義書には書いてあったから...」

V5では、このような「迷子」は起きません。なぜなら、すべての機能、すべての設計判断が、明確な「価値」に紐づいているからです。

例えば：
- 「在庫確認API」は、
- 「在庫管理ケイパビリティ」の一部であり、
- 「注文処理バリューストリーム」を実現するために必要で、
- 「顧客の注文体験を向上させる」という価値を生む

この繋がりが、常に明確になっています。

## 価値トレーサビリティシステム ― 見える化の力

### なぜトレーサビリティが重要か

食品を買う時、産地や生産者が分かると安心しませんか？「このトマトは○○農園で、こういう方法で作られました」という情報があれば、品質への信頼が生まれます。

システム開発も同じです。

「このAPIは、どの要求から生まれたのか？」
「この設計変更は、どんな影響があるのか？」
「このバグは、どの価値に影響するのか？」

これらの質問に即座に答えられることが、品質の高いシステムを作る鍵になります。

### トレーサビリティの実現メカニズム

V5では、独自の識別子システムにより、要求から実装まで追跡可能にします：

```
価値ストリーム: VS3（注文処理）
        ↓
ケイパビリティ: CL3-2（在庫管理）
        ↓
アーキテクチャ: AR3-2-1（在庫サービス）
        ↓
API/データ: AD3-2-1-api（在庫確認API）
        ↓
イベント: EV3-2-1-evt（在庫更新イベント）
```

この番号体系により、「EV3-2-1-evt」を見れば、即座に「これはVS3（注文処理）に関連する在庫管理のイベントだ」と分かります。

### 実例：ECサイトの注文処理を追跡する

具体例で見てみましょう。あるECサイトで「在庫切れ商品の注文を防ぐ」という要求があったとします。

**従来の開発**：
- 要件定義書のどこかに記載
- 設計書のどこかで言及
- コードのコメントに痕跡
- でも、全体像は見えない...

**V5での開発**：
1. VS3（注文処理）の品質要求として記録
2. CL3-2（在庫管理）で在庫確認機能として具体化
3. AR3-2-1（在庫サービス）でリアルタイム確認の設計
4. AD3-2-1-api（在庫確認API）で実装仕様
5. EV3-2-1-evt（在庫更新イベント）で在庫変動を追跡

すべてが「VS3」という共通の起点から派生し、相互に関連付けられています。

仕様変更が必要になった時、影響範囲が一目瞭然です。「VS3を変更するなら、CL3-2、AR3-2-1、AD3-2-1-api、EV3-2-1-evtすべてを確認する必要がある」—— これが即座に分かるのです。

## 6軸探索システム ― 見落としを防ぐ多角的視点

### なぜ多角的分析が必要か

古典的な寓話に、複数の人が巨大な物体の異なる部分に触れて、それぞれ異なる解釈をする話があります。

一人は太い部分に触れて「柱のようだ」と言い、別の人は細長い部分に触れて「ロープのようだ」と言い、また別の人は平たい部分に触れて「壁のようだ」と言う。それぞれの観察は部分的には正しいのですが、誰も全体像を把握していません。

複雑なビジネスシステムも同じです。一つの視点だけでは、システムの本質を見落とす危険があります。

### 6つの探索軸の詳細

V5では、6つの異なる「視点」から分析を行います：

#### 1. capability-axis（ケイパビリティ軸）
**問い**：このビジネスは、どんな能力を持つべきか？

例：ECサイトなら「商品管理」「注文処理」「配送管理」「顧客管理」など

#### 2. business-unit-axis（ビジネスユニット軸）
**問い**：組織のどの部門が、何に責任を持つか？

例：「商品管理は商品部」「配送は物流部」「顧客対応はCS部」など

#### 3. value-axis（価値軸）
**問い**：どのような価値を、誰に、どう届けるか？

例：「顧客の購買時間を50%削減」「運営コストを30%削減」「顧客満足度95%達成」

#### 4. platform-axis（プラットフォーム軸）
**問い**：どのような共通基盤が、複数の価値創造を支えるか？

例：「統合顧客DB」「共通決済基盤」「マイクロサービス基盤」

#### 5. fusion-axis（フュージョン軸）
**問い**：異なる要素を組み合わせることで、どんな新しい価値が生まれるか？

例：「EC×物流×金融」「製造×IoT×AI」「医療×介護×保険」

#### 6. multi-tier-vstr-axis（多層価値ストリーム軸）
**問い**：価値はどのように階層的に流れ、積み上がるか？

例：「現場→部門→全社」「店舗→地域→本部」「個人→チーム→組織」

```
┌────────────────────────────────────────────────────────────┐
│                    6軸システム                              │
├──────────────────────┬─────────────────────────────────────┤
│ 軸                   │ 問い（分析の焦点）                   │
├──────────────────────┼─────────────────────────────────────┤
│ capability-axis      │ 何ができるか？（能力）               │
│ business-unit-axis   │ 誰が担うか？（組織）                 │
│ value-axis           │ どんな価値か？（成果）               │
│ platform-axis        │ 何が支えるか？（基盤）               │
│ fusion-axis          │ 組み合わせ効果は？（融合）           │
│ multi-tier-vstr-axis │ どう積み上がるか？（階層）           │
└──────────────────────┴─────────────────────────────────────┘
```

### 軸の組み合わせによる発見

本当の力は、これらの軸を組み合わせた時に発揮されます：

**capability × business-unit**
→ 「在庫管理機能を商品部と物流部が共有している。これは将来の拡張時にボトルネックになるかも」

**value × platform**
→ 「顧客満足度向上（価値）を実現するには、統合顧客DB（基盤）が必要。これにより個別最適化が可能になる」

**fusion × multi-tier**
→ 「店舗のPOSデータ×ECデータ×物流データを融合し、地域レベルで在庫最適化、全社レベルで調達最適化」

このように、複数の視点を組み合わせることで、単一視点では見えない洞察が得られます。

### 実践例：地方銀行の6軸分析

ある地方銀行が「デジタルバンキングサービス」を構築する例で見てみましょう：

**capability-axis での発見**：
- 口座管理、送金、ローン申請、投資相談の能力が必要

**business-unit-axis での発見**：
- リテール部門と法人部門で要求が大きく異なる
- IT部門のキャパシティが制約になっている

**value-axis での発見**：
- 口座開設時間を3日→30分に短縮（顧客価値）
- 運営コストを年間5億円削減（内部価値）
- コンプライアンス対応を自動化（リスク価値）

**platform-axis での発見**：
- 統合認証基盤が全サービスの土台
- APIゲートウェイで外部連携を一元化
- データレイクで分析基盤を構築

**fusion-axis での発見**：
- 口座×ローン×運用の統合提案
- オンライン×オフライン融合サービス
- 銀行×地域商店街×自治体の連携

**multi-tier-vstr-axis での発見**：
- 支店レベル：窓口業務のデジタル化
- 地域レベル：支店間の情報共有
- 全社レベル：統合的な顧客サービス

これらの分析により、「段階的にデジタル化を進めつつ、まずは口座開設プロセスから着手し、顧客情報の統合を並行して進める」という戦略が見えてきます。

## 各フェーズの詳細ガイド

それでは、8つのフェーズを一つずつ見ていきましょう。各フェーズで何をするのか、どんな成果物が生まれるのか、品質をどう保証するのかを説明します。

### Phase 0: 事前準備 ― 施主の夢を聞く

**建築で例えると**：初回の打ち合わせで、施主がどんな家に住みたいか、予算はどれくらいか、いつまでに建てたいかを確認する段階です。

**目的**：プロジェクトの土台を作り、基本情報を収集する

**主要活動**：
1. **企業情報の収集**
   - ミッション、ビジョン、バリュー
   - 主要な製品・サービス
   - 市場でのポジション

2. **初期スコープの定義**
   - 今回のプロジェクトで扱う範囲
   - 扱わない範囲の明確化
   - 大まかなタイムライン

3. **ステークホルダーの識別**
   - 誰が関わるか
   - それぞれの期待は何か
   - キーパーソンは誰か

**成果物**：
- `0-init-analysis.md`：企業の基本分析
- `parasol.yaml`：プロジェクト設定ファイル

**品質ゲート**：
- [ ] 企業のミッションが明文化されている
- [ ] 主要なステークホルダーがリストアップされている
- [ ] プロジェクトのスコープが合意されている
- [ ] 基本的な制約条件が識別されている

**実例**：
ある地方銀行のケースでは、Phase 0で以下を明確にしました：
- ミッション：「地域に根ざした金融サービスで、お客様の人生を支える」
- スコープ：「個人向けインターネットバンキングの全面刷新」
- 主要ステークホルダー：個人顧客、行員、規制当局、システム部門

### Phase 1: 敷地調査 ― 建築条件の把握

**建築で例えると**：土地の形状、法規制、周辺環境、地盤の強度などを詳しく調査する段階です。

**目的**：ビジネスの現状と制約を深く理解する

**主要活動**：
1. **ステークホルダー分析の深化**
   ```
   ステークホルダー | ニーズ | 影響力 | 関心度
   ------------------|---------|---------|--------
   個人顧客（若年層）| 使いやすいUI | 中 | 高
   個人顧客（高齢層）| 安心感 | 高 | 中
   規制当局 | コンプライアンス | 高 | 高
   ```

2. **制約条件の詳細化**
   - 技術的制約（レガシーシステムとの連携）
   - 規制制約（金融庁のガイドライン）
   - 組織的制約（IT人材の不足）
   - 時間的制約（競合の動向）

3. **ビジネスゴールの具体化**
   - SMART原則での目標設定
   - 成功指標の定義
   - 優先順位の明確化

**成果物**：
- `1-stakeholders.md`：詳細なステークホルダー分析
- `1-constraints.md`：制約条件カタログ
- `1-goals.md`：ビジネスゴールと成功指標

**品質ゲート**：
- [ ] すべての主要ステークホルダーのニーズが文書化されている
- [ ] 制約条件が優先度付けされている
- [ ] ゴールがSMART基準を満たしている
- [ ] ステークホルダー間の利害対立が識別されている

### Phase 2: コンセプト設計 ― 価値ある家の設計

**建築で例えると**：「光あふれる家」「家族が集まる家」など、どんな価値を持つ家にするかコンセプトを決める段階です。ここが最も創造的で重要な段階です。

**目的**：ビジネスがどのように価値を生み出し、届けているかを可視化する

**主要活動**：
1. **バリューストリームの識別**
   
   地方銀行の例：
   - VS1: 口座開設（見込み客→口座保有者）
   - VS2: 日常取引（入出金、振込、残高照会）
   - VS3: ローン申込（ニーズ発生→審査→実行）
   - VS4: 資産運用相談（資産形成ニーズ→商品提案→運用）

2. **価値提供プロセスの分析**
   
   VS1（口座開設）の詳細：
   ```
   申込受付 → 本人確認 → 審査 → 口座開設 → カード発行
     ↓          ↓         ↓        ↓         ↓
   10分      2-3日    即日    即日     7日
   ```
   
   ※ボトルネック（本人確認、カード発行）は上記フロー図で強調表示される箇所

3. **ボトルネックの発見**
   - 本人確認に2-3日かかる（郵送ベース）
   - これが顧客離脱の主要因

**成果物**：
- `2-value-streams.md`：バリューストリームカタログ
- `2-value-analysis-*.md`：各VSの詳細分析
- `2-pain-points.md`：改善ポイント一覧

**6軸探索の適用（Phase 2では価値発見に最適な3軸を選択）**：
- **value-axis**：どんな価値が、誰に、どれくらい届けられるか
- **capability-axis**：価値実現に必要な能力は何か
- **multi-tier-vstr-axis**：価値はどの階層で生まれ、派生するか

**品質ゲート**：
- [ ] 主要な価値の流れがすべて識別されている
- [ ] 各バリューストリームの開始と終了が明確
- [ ] 測定可能な価値指標が設定されている
- [ ] ボトルネックが特定され、優先順位付けされている

### Phase 3: 間取り設計 ― 必要な機能の配置

**建築で例えると**：リビング、キッチン、寝室など、必要な部屋（機能）を決め、最適な配置を設計する段階です。

**目的**：価値を実現するために必要なビジネス能力を体系的に整理する

**主要活動**：
1. **活動領域識別（CL1）- 傾向的分類（参考情報）**
   ```
   Core傾向（差別化要因）:
   - 地域密着型相談サービス
   - 中小企業向け融資判断

   Supporting傾向（必要だが差別化にならない）:
   - 口座管理
   - 決済処理

   Generic傾向（汎用的）:
   - 認証・認可
   - 帳票出力
   ```
   ※CL1の分類は参考情報。CL2で個別に正式分類される。

2. **ケイパビリティ設計（CL2）- 正式分類（投資判断の根拠）**

   口座管理ケイパビリティの分解と正式分類：
   - 口座開設（Supporting）
   - 口座維持（Supporting）
   - 口座解約（Generic）

3. **オペレーション定義（CL3）- 分類なし（網羅性重視）**

   口座開設の業務オペレーション：
   - **申込受付**：オンライン/窓口からの申込処理
   - **本人確認**：eKYCまたは郵送による確認
   - **口座有効化**：審査完了後の口座開設処理

> **注意**: Bounded Context（BC）の定義はPhase 3には含まれません。

## 品質ゲートシステム ― 確実な前進のために

### 段階的な品質保証

建築では、各工程で建築基準を満たしているか検査を受けます。基礎工事の後、骨組みの後、そして完成時。これらの検査なしに次の工程に進むことはできません。

V5の品質ゲートも同じ考え方です。各フェーズの終わりに「検査」を行い、次の工程に進む品質が確保されているかを確認します。

### 品質ゲートの実例

Phase 2からPhase 3に進む際の品質ゲートを見てみましょう：

**Phase 2 → Phase 3 移行チェックリスト**

**必須項目**（これらがすべて完了していること）：
- [ ] すべての主要価値ストリームが識別されている
- [ ] 各価値ストリームの開始点と終了点が明確
- [ ] ステークホルダーからの合意が得られている
- [ ] 測定可能な価値指標が設定されている

**推奨項目**（できれば完了していること）：
- [ ] 現状の問題点が定量化されている
- [ ] 改善後の目標値が設定されている
- [ ] 優先順位が明確になっている

**確認方法**：
1. チームでのレビューセッション
2. ステークホルダーへのプレゼンテーション
3. 品質保証チェックリストの確認

もし必須項目が満たされていない場合は、Phase 3に進まず、Phase 2の作業を継続します。これは「手戻り」ではなく、「確実な基礎固め」です。

### 継続的な改善

品質ゲートは固定的なものではありません。プロジェクトを重ねるごとに、より適切なチェック項目に改善していきます。

例えば、ある銀行では、最初のプロジェクトで「規制要件の確認」が漏れていたことが判明しました。それ以降、Phase 1の品質ゲートに「規制要件チェックリスト」を追加し、同じ問題が起きないようにしています。

## 実践的な適用例

### 小規模プロジェクトでの適用

「全部で8フェーズもあるなんて、小さなプロジェクトには重すぎる」—— そう思われるかもしれません。

安心してください。V5は柔軟です。

**スタートアップの新機能開発（2週間）の例**：

- **Phase 0-1**: 半日でまとめて実施
- **Phase 2**: 1日で主要な価値の流れだけ確認
- **Phase 3**: 1日で必要最小限のケイパビリティ定義
- **Phase 4-5**: 2日でアーキテクチャとAPI設計
- **Phase 6-7**: 実装と並行して段階的に詳細化

重要なのは、「すべてのフェーズを通過すること」です。時間は短縮できても、スキップしてはいけません。

### 大規模プロジェクトでの適用

逆に、大規模プロジェクトでは、複数チームが並行して作業します。

**全社システム刷新（2年）の例**：

```
チームA: 顧客管理領域
  Phase 0-1: 共通実施
  Phase 2-7: 顧客管理に特化して実施

チームB: 商品管理領域  
  Phase 0-1: 共通実施
  Phase 2-7: 商品管理に特化して実施

統合チーム:
  各チームのPhase 3完了後に統合ポイントを調整
  Phase 5でAPI仕様を統一
```

このように、領域ごとに並行して進めながら、重要なポイントで統合を図ります。

### アジャイルとの統合

「V5はウォーターフォールなの？」—— いいえ、違います。

V5は、アジャイルと完全に統合可能です：

**スクラムとの統合例**：
- Sprint 0: Phase 0-1を実施
- Sprint 1-2: Phase 2を実施し、最初の価値ストリームを選択
- Sprint 3-5: 選択したVSに対してPhase 3-5を実施
- Sprint 6-8: 実装しながらPhase 6-7を詳細化
- Sprint 9-: 次の価値ストリームに対して同様のサイクル

重要なのは、「段階的詳細化」の考え方です。全体を薄く広く進めるのではなく、価値の高い部分から深く確実に進めていきます。
