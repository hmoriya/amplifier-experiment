# 第6章　Phase 2：価値発見と価値設計 ― ビジネスの心臓部を見つける

## はじめに：価値とは何か？

ある経営会議での一コマです。

「我が社の価値は何ですか？」

コンサルタントの問いに、役員たちは口々に答えます。

「高品質な製品です」
「優れた技術力です」
「長年の信頼関係です」
「充実したサポート体制です」

コンサルタントは静かに首を振りました。

「それらは素晴らしい『特徴』です。でも『価値』ではありません。価値とは、顧客があなたに『お金を払う理由』です」

会議室に沈黙が流れました。

この章では、Parasol V5の心臓部である「価値」の発見と設計を学びます。技術やプロセスの前に、「なぜ顧客はお金を払うのか」「何が本当の競争力なのか」を明らかにすることが、成功への最短距離なのです。

## 価値の本質：特徴ではなく、顧客の変化

### 特徴と価値の決定的な違い

多くの企業が「特徴」を「価値」と混同しています。

```yaml
feature_vs_value:
  製造業の例:
    特徴: "精度±0.001mmの加工技術"
    価値: "不良品ゼロで生産ラインが止まらない"
    
    # 顧客が払う理由
    理由: "ライン停止による損失（1時間1000万円）を防げる"
    
  ITサービスの例:
    特徴: "99.99%の稼働率"
    価値: "ビジネスが24時間止まらない安心"
    
    # 顧客が払う理由
    理由: "機会損失と信用失墜のリスクがない"
    
  コンサルティングの例:
    特徴: "MBA保有者10名のチーム"
    価値: "3ヶ月で売上が20%向上する"
    
    # 顧客が払う理由
    理由: "投資対効果が明確で確実"
```

価値は常に「顧客の状態がどう変わるか」で定義されます。

### 価値の3層構造

価値は、表層から深層まで3つの層を持っています。

```yaml
value_layers:
  surface_value:  # 表層価値
    what: "見える成果"
    example: "発注処理が5分→1分に短縮"
    measurement: "時間、回数、量"
    
  business_value:  # ビジネス価値
    what: "経営への影響"
    example: "業務効率化で人件費30%削減"
    measurement: "売上、コスト、利益"
    
  strategic_value:  # 戦略的価値
    what: "競争優位性"
    example: "競合より3倍速い顧客対応で差別化"
    measurement: "市場シェア、顧客満足度、ブランド価値"
```

優れたシステムは、3層すべてで価値を生み出します。

## Phase 2の実践：価値ストリームマッピング

### ステップ1：現状の価値の流れを可視化する

まず、現在のビジネスで「価値がどう流れているか」を可視化します。

**事例：オンライン書店の価値ストリーム**

```yaml
current_value_stream:
  name: "書籍販売"
  
  trigger: "顧客が本を探し始める"
  end: "本を読み終えて満足する"
  
  steps:
    1_search:
      action: "欲しい本を探す"
      current_time: "平均15分"
      pain_points:
        - "検索精度が低い"
        - "レコメンドが的外れ"
        - "在庫状況が不明"
      value_loss: "50%が購入せずに離脱"
      
    2_purchase:
      action: "購入を決定する"
      current_time: "平均5分"
      pain_points:
        - "レビューが少ない"
        - "価格比較ができない"
        - "送料が分かりにくい"
      value_loss: "カートの30%が放棄"
      
    3_delivery:
      action: "商品を受け取る"
      current_time: "2-3日"
      pain_points:
        - "配送状況が不明"
        - "受取時間の制約"
        - "梱包が過剰"
      value_loss: "5%が再配達"
      
    4_reading:
      action: "読書を楽しむ"
      current_time: "1-2週間"
      pain_points:
        - "読後の共有場所がない"
        - "次の本の推薦がない"
        - "質問する相手がいない"
      value_loss: "次回購入率30%"
```

この可視化により、価値が失われている「出血点」が明確になります。

## 価値設計の極意：6軸システムの活用

### なぜ6軸が必要なのか

ビジネスの価値は多面的です。一つの軸だけでは、全体像を捉えきれません。

```yaml
six_axis_system:
  capability_axis:  # ケイパビリティ軸
    focus: "このビジネスは、どんな能力を持つべきか？"
    example: "24時間注文受付能力"
    
  business_unit_axis:  # ビジネスユニット軸
    focus: "組織のどの部門が、何に責任を持つか？"
    example: "法人営業部の効率化"
    
  value_axis:  # 価値軸
    focus: "どのような価値を、誰に、どう届けるか？"
    example: "受注処理時間75%削減"
    
  platform_axis:  # プラットフォーム軸
    focus: "どのような共通基盤が、複数の価値創造を支えるか？"
    example: "統合顧客データベース"
    
  fusion_axis:  # フュージョン軸
    focus: "異なる要素を組み合わせることで、どんな新しい価値が生まれるか？"
    example: "在庫×物流の最適化"
    
  multi_tier_vstr_axis:  # 多層価値ストリーム軸
    focus: "価値はどのように階層的に流れ、積み上がるか？"
    example: "現場→部門→全社への波及"
```

### 産業別の軸選択パターン

業界特性により、重視すべき軸の組み合わせが変わります。

```yaml
industry_patterns:
  manufacturing:  # 製造業
    primary_axes:
      - capability_axis: "生産能力の向上"
      - platform_axis: "IoT基盤の構築"
      - fusion_axis: "設計×製造×品質の統合"
    reason: "物理的な制約と統合が鍵"
    
  retail:  # 小売業
    primary_axes:
      - value_axis: "顧客体験の向上"
      - business_unit_axis: "店舗別最適化"
      - multi_tier_vstr_axis: "店舗→地域→本部"
    reason: "顧客接点と階層構造が重要"
    
  finance:  # 金融業
    primary_axes:
      - platform_axis: "リスク管理基盤"
      - capability_axis: "即時処理能力"
      - value_axis: "コンプライアンス価値"
    reason: "信頼性と処理能力が生命線"
```

### 価値マトリクスの作成

選択した3軸を使って、価値を立体的に分析します。

```yaml
value_matrix_example:
  # 小売業の例（3軸：価値×事業部×多層）
  
  店舗レベル:
    価値軸:
      - "レジ待ち時間削減"

      - "全社在庫の最適化"
```

## Claude Codeとの協働：価値の深堀り

### AIが見つける隠れた価値

人間は既存の枠組みに囚われがちですが、AIは新しい視点を提供します。

```
あなた：弊社の受注システムで、まだ発見できていない価値を探してください。

Claude Code：受注データと関連情報を分析します。

[データ分析実行]

隠れた価値の発見：

1. 予測価値
   現状：受注後に在庫確認
   発見：受注パターンから3日後の需要を85%の精度で予測可能
   価値：先行手配により納期を2日短縮、緊急配送コスト年間3000万円削減

2. 関係性価値  
   現状：顧客を個別に管理
   発見：発注パターンから企業間の関連性を検出
   価値：グループ提案により受注額30%向上の可能性

3. タイミング価値
   現状：顧客からの発注を待つ
   発見：過去データから「発注忘れ」パターンを特定
   価値：プロアクティブな提案で失注を年間5000万円防止

4. 異常検知価値
   現状：クレーム後に対応
   発見：通常と異なる発注パターンから問題を事前検知
   価値：顧客満足度の低下を未然に防止

実現のための次のステップ：
[具体的な実装提案]
```

## 価値設計のアンチパターン

### 1. 機能の羅列を価値と勘違いする

**誤り**：
```yaml
wrong_value_definition:
  システムの価値:
    - "100個の新機能"
    - "最新のAI搭載"
    - "マイクロサービス化"
    - "クラウドネイティブ"
```

**正しい定義**：
```yaml
correct_value_definition:
  顧客にとっての価値:
    - "商品選定時間が50%短縮"
    - "誤発注による損失がゼロに"
    - "24時間サポートで機会損失なし"
    - "導入3ヶ月で投資回収"
```

### 2. 内部の都合を価値にすり替える

**誤り**：
```yaml
internal_focused:
  "価値":
    - "部門間の連携強化"  # → で、顧客にどんないいことが？
    - "データの一元管理"  # → それで何が改善される？
    - "業務の標準化"      # → 顧客にとっての利点は？
```

**正しい視点**：
```yaml
customer_focused:
  真の価値:
    - "部門連携により即日回答"
    - "一元管理で履歴から最適提案"
    - "標準化により品質保証"
```

### 3. 測定できない価値に逃げる

**誤り**：
```yaml
unmeasurable_value:
  - "イノベーションの促進"  # どう測る？
  - "組織文化の改善"        # 具体的には？
  - "将来への投資"          # いつ回収？
```

**測定可能な形に**：
```yaml
measurable_value:
  - "新商品開発期間：6ヶ月→3ヶ月"
  - "従業員エンゲージメント：3.2→4.0"
  - "3年後の新規事業売上：50億円"
```

```yaml
value_prioritization:

      - "戦略的投資"
      - "段階的実施"
      
    low_impact_easy:  # 第3優先
      - "余力があれば"
      - "自動化で対応"
      
    low_impact_hard:  # 対象外
      - "見送り"
      - "代替案を検討"
```

## まとめ：価値なくして設計なし

Phase 2で学んだことを振り返りましょう。

### 価値発見の要諦

1. **価値は顧客の変化**
   - 機能や特徴ではない
   - 顧客の状態がどう変わるか

2. **価値は多面的**
   - 直接価値だけでなく間接価値も
   - 6軸システムで立体的に捉える

3. **価値は測定可能**
   - 定量化できない価値は管理できない
   - 投資対効果を明確に

### 価値設計の実践

1. **現状の可視化から始める**
   - 価値の流れを追う
   - 出血点を特定する

2. **理想との差分を設計**
   - あるべき姿を描く
   - 段階的な実現計画

3. **継続的な検証と改善**
   - 仮説は必ず検証
   - 柔軟に軌道修正

### 次章への橋渡し

価値が明確になったら、次は「その価値を実現するために必要な能力（ケイパビリティ）」を設計します。

Phase 3では、発見した価値を、具体的な「できること」に分解していきます。抽象的な価値を、実装可能な単位にまで具体化する—— これがケイパビリティ設計の真髄です。

価値という設計図ができました。次は、その設計図を実現する「部材」を準備する番です。
