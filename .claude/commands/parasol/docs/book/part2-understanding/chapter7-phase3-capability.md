# 第7章　Phase 3：ケイパビリティ分解 ― 価値を実現する能力を設計する

## はじめに：できることとやりたいこと

ある企業の経営戦略会議でのことです。

「我が社は、顧客への即日配送を実現します！」

社長の力強い宣言に、役員たちは拍手を送りました。素晴らしいビジョンです。顧客価値も明確です。

3ヶ月後、プロジェクトは暗礁に乗り上げていました。

「在庫情報がリアルタイムで分からない...」
「配送業者とのシステム連携ができていない...」
「そもそも注文を即座に処理する仕組みがない...」

価値（やりたいこと）は明確でした。でも、それを実現する能力（できること）の設計が欠けていたのです。

この章では、Phase 2で発見した「価値」を、実現可能な「能力」に分解する技術を学びます。夢を現実にする、最も重要なステップです。

## ケイパビリティとは何か

### 能力の本質的な理解

ケイパビリティ（能力）とは、「組織が価値を生み出すために必要な、具体的にできること」です。

**ケイパビリティとは**

- 価値を実現する具体的な能力
- 人・プロセス・技術の統合
- 測定可能な成果を生む仕組み

### CL1/CL2/CL3 ― 3段階のケイパビリティ構造

Parasol V5では、ケイパビリティを3段階で定義します。この構造はDDDの概念と連携し、価値から実装への橋渡しを担います。

**CL1（活動領域識別）― 傾向的分類**

| 観点 | 内容 |
|------|------|
| 目的 | バリューストリームにおける活動の傾向を把握 |
| 分類 | Core / Supporting / Generic |
| 性質 | **参考情報のみ**（CL2に継承されない） |
| 用途 | 全体像の理解、初期の議論材料 |

> **重要**: CL1の分類は「初期仮説」として参考にすぎません。この段階の分類に過度に依存しないでください。

**CL2（ケイパビリティ設計）― 正式分類**

| 観点 | 内容 |
|------|------|
| 目的 | 投資判断のための正式な分類 |
| 分類 | Core / Supporting / Generic（CL1とは独立に再評価） |
| 性質 | **投資判断の根拠** |
| 用途 | リソース配分、内製/外注判断、優先順位決定 |

```
CL2分類の判断基準：
━━━━━━━━━━━━━━━━━━━━━━
Core        : 競争優位の源泉、差別化要素
Supporting  : 必要だが差別化にはならない
Generic     : 汎用的、外部調達可能
━━━━━━━━━━━━━━━━━━━━━━
```

**CL3（オペレーション定義）― 網羅性重視**

| 観点 | 内容 |
|------|------|
| 目的 | 業務オペレーション（Business Operation）の定義 |
| 分類 | **分類なし**（網羅性を重視） |
| 性質 | ビジネス観点での業務プロセス定義 |
| 用途 | 業務フローの網羅的把握、Phase 4-5への入力 |

> **Phase 4-5との関係**: CL3で定義される業務オペレーションは、Phase 4-5でBounded Context（BC）の境界確定と実装設計の入力となります。Phase 3はビジネス観点（What）、Phase 4-5は実装観点（How）という役割分担です。

**CL1からCL3への流れ**

```
CL1（活動領域識別）
  │  傾向的分類（参考のみ）
  │  ※CL2に継承されない
  ▼
CL2（ケイパビリティ設計）
  │  正式分類（投資判断の根拠）
  │  ※独立して再評価
  ▼
CL3（オペレーション定義）
  │  分類なし（網羅性重視）
  │  = Business Operation（業務オペレーション）
  ▼
Phase 4-5（BC境界確定・実装設計）
  ▼
パラソルドメイン言語 → 実装
```

**なぜCL1の分類はCL2に継承されないのか？**

CL1は「森を見る」ための俯瞰です。CL2は「木を見る」ための詳細分析です。

例：ECサイトの場合
- CL1で「物流」をSupportingと仮分類
- CL2で詳細分析すると「独自の即日配送能力」が競争優位の源泉と判明
- CL2ではCoreに再分類

CL1の分類に縛られていたら、この重要な発見を見逃すところでした。だからこそ、CL2では白紙から再評価するのです。

**ケイパビリティではないもの**

- 単なる機能やツール
- 部門や組織
- 一時的なプロジェクト

**良いケイパビリティの例**

- 24時間以内に注文を配送する能力
- 顧客の購買履歴から最適な商品を推薦する能力
- 在庫を店舗間で融通する能力

**ケイパビリティではない例**

| 誤った表現 | 理由 |
|-----------|------|
| ECサイト | これはチャネル |
| 物流部門 | これは組織 |
| 在庫管理システム | これはツール |

### ケイパビリティの3要素

優れたケイパビリティは、3つの要素が統合されています。

**人（People）**

| 要素 | 説明 |
|-----|------|
| スキル | 必要なスキルと知識 |
| 役割 | 明確な役割と責任 |
| 文化 | 実行を支える文化 |

**プロセス（Process）**

| 要素 | 説明 |
|-----|------|
| ワークフロー | 価値を生む手順 |
| ルール | 意思決定の基準 |
| メトリクス | 成果の測定方法 |

**技術（Technology）**

| 要素 | 説明 |
|-----|------|
| システム | 実現するシステム |
| データ | 必要な情報 |
| 統合 | 連携の仕組み |

> **悪い例：技術だけに偏る**
>
> 「最新のAIを導入しました！」
> - でも使える人がいない
> - プロセスも未整備
> - 結果：宝の持ち腐れ

> **良い例：3要素の統合**
>
> 「AI需要予測能力」
> - データサイエンティストを採用（人）
> - 予測→発注の業務フロー確立（プロセス）
> - 機械学習基盤を構築（技術）

## Phase 3の実践：ケイパビリティマップの作成

### ステップ1：価値からケイパビリティを導出

Phase 2で定義した価値から、必要な能力を論理的に導き出します。

**事例：オンライン食品スーパーの場合**

**価値**：注文から2時間以内に配送

**必要なケイパビリティ**

| # | ケイパビリティ名 | 説明 | 目標 |
|---|----------------|------|------|
| 1 | 高速注文処理能力 | 注文を即座に処理し、配送準備を開始 | 受注から5分以内に処理完了 |
| 2 | リアルタイム在庫管理能力 | 全拠点の在庫を秒単位で把握 | 在庫情報の遅延1秒以内 |
| 3 | 動的配送ルート最適化能力 | 交通状況を考慮した最適ルート算出 | ルート計算30秒以内 |
| 4 | 高速ピッキング能力 | 注文商品を効率的に集める | 50商品を10分以内 |
| 5 | ラストマイル配送能力 | 顧客宅への確実な配送 | 指定時間の±15分 |

### ステップ2：ケイパビリティの依存関係分析

能力は独立して存在するのではなく、相互に依存しています。

**コアケイパビリティ**：2時間配送能力

**依存関係**

| 依存レベル | ケイパビリティ | 理由 |
|-----------|--------------|------|
| 必須（Critical） | リアルタイム在庫 | 在庫がなければ配送できない |
| 必須（Critical） | 注文処理 | 注文を受けられなければ始まらない |
| 重要（Important） | ルート最適化 | 効率が落ちるが配送は可能 |
| 重要（Important） | 顧客コミュニケーション | 不安を与えるが配送は可能 |
| あれば良い（Nice to have） | 需要予測 | 在庫効率が上がる |
| あれば良い（Nice to have） | 梱包自動化 | コストが下がる |


### ステップ3：ケイパビリティの成熟度評価

現在の能力レベルを正直に評価します。

**成熟度レベル定義**

| レベル | 名称 | 説明 |
|-------|-----|------|
| 0 | なし | 能力なし |
| 1 | 初期 | 属人的・場当たり的 |
| 2 | 発展中 | 部分的に仕組み化 |
| 3 | 定義済み | プロセス確立 |
| 4 | 管理 | 測定・管理 |
| 5 | 最適化 | 継続的改善 |

**現状評価の例**

| ケイパビリティ | 現在レベル | 目標とのギャップ | エビデンス |
|--------------|-----------|----------------|-----------|
| 注文処理 | 2 | 3 | 一部自動化されているが手作業多い、ベテランがいないと回らない、繁忙期にはミスが増える |
| 在庫管理 | 1 | 4（致命的） | Excelで日次更新、店舗ごとにやり方が違う、在庫切れが頻発 |
| 配送実行 | 3 | 2（比較的小さい） | 配送業者との連携確立、SLAも定義済み、追跡も可能 |

## ケイパビリティ設計の高度なテクニック

### コア・競争・基礎の3分類

すべてのケイパビリティが同じ重要度ではありません。

```
ケイパビリティ3分類
==================

┌─────────────────────────────────────────────────────────────┐
│ コアケイパビリティ（競争優位の源泉）          【最重要】   │
├─────────────────────────────────────────────────────────────┤
│  ・1時間配送                                                │
│  ・AI味覚マッチング                                         │
│  ・産地直送ネットワーク                                     │
└─────────────────────────────────────────────────────────────┘
            │
            ▼
┌─────────────────────────────────────────────────────────────┐
│ 競争的ケイパビリティ（業界標準レベル）        【標準】     │
├─────────────────────────────────────────────────────────────┤
│  ・クレジット決済                                           │
│  ・商品検索                                                 │
│  ・顧客サポート                                             │
└─────────────────────────────────────────────────────────────┘
            │
            ▼
┌─────────────────────────────────────────────────────────────┐
│ 基礎的ケイパビリティ（ビジネスの前提条件）    【基盤】     │
├─────────────────────────────────────────────────────────────┤
│  ・経理処理                                                 │
│  ・給与計算                                                 │
│  ・オフィスIT                                               │
└─────────────────────────────────────────────────────────────┘
```

**コアケイパビリティ（競争優位の源泉）**

- **定義**：競争優位の源泉
- **特徴**：他社が真似しにくい、顧客価値に直結、自社の強み
- **例**：1時間配送（業界最速）、AI味覚マッチング（特許技術）、産地直送ネットワーク（独占契約）
- **戦略**：徹底的に投資し、磨き上げる

**競争的ケイパビリティ（業界標準レベル）**

- **定義**：業界標準レベルが必要
- **特徴**：顧客が当然期待、劣ると致命的、差別化は困難
- **例**：クレジットカード決済、商品検索機能、カスタマーサポート
- **戦略**：効率的に業界水準を達成

**基礎的ケイパビリティ（ビジネスの前提条件）**

- **定義**：ビジネスの前提条件
- **特徴**：なければ営業できない、差別化要素ではない、アウトソース可能
- **例**：経理処理、給与計算、オフィスIT
- **戦略**：標準化・外部活用

### ケイパビリティのモジュール化設計

大きなケイパビリティを、組み合わせ可能な部品に分解します。

**大きなケイパビリティ**：パーソナライズドマーケティング能力

**モジュールへの分解**

| モジュール | 機能 | 入力 | 出力 | 再利用性 |
|-----------|------|-----|------|---------|
| データ収集 | 顧客データ収集 | Webログ、購買履歴、アンケート | 統合顧客プロファイル | 他の分析にも活用可能 |
| 分析エンジン | 行動パターン分析 | 統合顧客プロファイル | セグメント、予測スコア | 需要予測にも転用 |
| コンテンツ生成 | パーソナライズドコンテンツ生成 | セグメント、商品情報 | 最適化されたコンテンツ | メール、Web、アプリで共通 |
| 配信オーケストレーション | 最適タイミング配信 | コンテンツ、顧客状態 | 配信実行、結果記録 | 通知全般で利用 |

**モジュール化のメリット**

- 段階的な構築が可能
- 部分的な改善が容易
- 他用途への転用
- ベンダー選定の柔軟性

ケイパビリティは、ビジネスの成長と共に進化します。

## ケイパビリティ設計のアンチパターン

### 1. 手段の目的化

**誤り**：

> 「最新のマイクロサービスアーキテクチャを導入する能力」
> - なぜマイクロサービスが必要？
> - 価値は何？

**正しいアプローチ**：

> 「サービスを独立して素早くリリースする能力」
> - 価値：市場投入速度の向上
> - 手段：マイクロサービス（も選択肢の一つ）

### 2. 現状の制約に囚われる

**誤り**：

> 「うちには技術者がいないから...」
> 「予算がないから...」
> 「今のシステムでは無理だから...」
> → 最初から諦める

**正しい思考**：

1. 理想のケイパビリティを定義
2. 現状とのギャップを認識
3. 段階的な獲得計画を作成
4. 創造的な解決策を探る
   - パートナーシップ
   - 段階的な内製化
   - クラウドサービス活用

### 3. 総花的な能力羅列

**誤り**：

> - AI活用能力
> - ビッグデータ分析能力
> - ブロックチェーン能力
> - IoT能力
> - 量子コンピューティング能力
> → 焦点が定まらない

**選択と集中**：

- **コアフォーカス**：顧客理解に基づく最適提案能力
- **サポート**：
  - 購買データ分析能力
  - リアルタイムレコメンド能力
- **見送り**：その他の先端技術は当面見送り

## まとめ：能力なくして価値なし

Phase 3で学んだことを整理しましょう。

### ケイパビリティ設計の要点

1. **価値からの逆算**
   - 何のための能力か常に問う
   - 手段を目的化しない

2. **統合的な視点**
   - 人・プロセス・技術の三位一体
   - 単独では機能しない

3. **現実的な計画**
   - 現状を正直に評価
   - 段階的な獲得戦略

4. **選択と集中**
   - コアケイパビリティに投資
   - すべてを自前で持たない

### 次章への橋渡し

ケイパビリティが明確になったら、次はそれを支える「アーキテクチャ」の設計です。

Phase 4-7では、抽象的な能力を、具体的なシステム構造に落とし込んでいきます。いよいよ、建物の具体的な設計図を描く段階です。

価値（なぜ）とケイパビリティ（何を）が明確になりました。
次は、アーキテクチャ（どのように）を設計する番です。

基礎工事は完了。構造設計に入る準備はできましたか？

次章で、理想が現実になり始めます。
