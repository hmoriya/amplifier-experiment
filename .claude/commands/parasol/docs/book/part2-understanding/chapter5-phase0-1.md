# 第5章　Phase 0-1：プロジェクト基盤構築 ― 成功の80%はここで決まる

## はじめに：なぜ基礎工事に時間をかけるのか

ある建設現場での話です。

施主が現場監督に詰め寄ります。「もう2週間も経つのに、まだ穴を掘ってるだけじゃないか！いつになったら建物が建つんだ？」

現場監督は落ち着いて答えます。「30階建てのビルを建てるには、しっかりした基礎が必要です。地上に見える部分より、地下の基礎工事の方が重要なんです」

1年後、立派なビルが完成しました。隣の土地では、基礎を急いで建てた10階建てのビルが、すでに傾き始めていました。

ソフトウェア開発も同じです。

Phase 0-1は、Parasol V5の「基礎工事」。地味で、時間がかかり、目に見える成果も少ない。でも、ここでの丁寧な作業が、プロジェクト全体の成否を決めるのです。

## Phase 0：初期化 ― 正しい問いから始める

### なぜPhase 0が必要なのか

「要件定義から始めましょう」—— 多くのプロジェクトはこう始まります。でも、ちょっと待ってください。

要件を定義する前に、もっと根本的な問いに答える必要があります：

- そもそも、なぜこのプロジェクトが必要なのか？
- 誰のために、何を解決しようとしているのか？
- 成功とは何か？どう測るのか？

これらの問いに明確に答えられないまま進むプロジェクトは、高確率で迷走します。

### ステップ1：プロジェクトの「生い立ち」を理解する

すべてのプロジェクトには「生い立ち」があります。

**プロジェクトの生い立ち**

> **発端となった出来事**
> - 何が起きたか：競合他社が革新的なサービスをリリース
> - 時期：2025年3月
> - 影響：3ヶ月で市場シェア10%を奪われた
>
> **最初の反応**
> - 誰が：CEO緊急会議
> - 決定：我々も対抗サービスを作る
> - 期限：6ヶ月以内にリリース
>
> **本当の問題**
> - 表面的：競合に追いつく
> - より深く：顧客ニーズの変化に対応できていない
> - 根本原因：既存システムが硬直化している

表面的な「競合対抗」から、本質的な「システムの硬直性」まで掘り下げることで、本当に解決すべき問題が見えてきます。

### ステップ2：ステークホルダーマッピング

「誰が関わるのか」を明確にすることは、想像以上に重要です。

**ステークホルダーマップ**

| 分類 | 関係者 | 関心事 |
|------|--------|--------|
| **推進者** | CEO | 市場での競争力回復 |
| | CTO | 技術的負債の解消 |
| **主要利用者** | 営業部門 | 顧客への提案を素早く |
| | カスタマーサポート | 問い合わせに即座に対応 |
| **副次利用者** | 経理部門 | 請求処理の自動化 |
| | 人事部門 | 社員情報の一元管理 |
| **影響者** | 情報システム部 | 既存システムとの整合性 |
| | 法務部 | コンプライアンス遵守 |
| | 労働組合 | 業務負荷の増加防止 |
| **隠れた関係者** | 既存ベンダー | 仕事を奪われる恐れ |
| | 地方拠点 | 本社主導への不満 |

特に「隠れたステークホルダー」の存在は重要です。彼らの懸念を早期に把握し、対処することで、後々の抵抗を防げます。

## Phase 1：Context理解 ― 企業の本質を掴む

### ビジネスコンテキスト：何で稼いでいるのか

システムを設計する前に、そのシステムが支える「ビジネス」を理解する必要があります。

**事例：中堅商社の場合**

**ビジネスコンテキスト例：グローバルトレード株式会社**

> **ビジネスモデル**
> - 事業内容：工業用部品の輸入販売
> - 価値提案：小ロット・短納期・技術サポート

**収益構造**

| 区分 | 比率 | 内容 |
|------|------|------|
| 商品販売 | 70% | 部品販売 |
| サービス | 20% | 技術サポート・コンサル |
| リカーリング | 10% | 保守契約 |

**利益ドライバー**
- プラス要因
  - 独占輸入契約商品（粗利率40%）
  - 技術サポート（粗利率80%）
- マイナス要因
  - 在庫コスト（売上の5%）
  - 為替リスク（利益の±10%）

**競争優位性**
- 独自ポジション：日本で唯一の正規代理店（主力3製品）
- 顧客関係：技術担当の常駐（大手顧客5社）
- ノウハウ：20年の業界知識とネットワーク

このビジネス理解があって初めて、「何を優先すべきか」が見えてきます。この例では、独占商品の在庫管理と、技術サポートの効率化が最重要となります。

### 組織構造：誰がどう動いているのか

組織図を見るだけでは不十分です。実際の「情報とお金の流れ」を理解する必要があります。

**組織ダイナミクス**

**公式組織構造**
- 営業本部
  - 第一営業部：大手顧客担当
  - 第二営業部：中小顧客担当
  - 技術営業部：技術サポート

**非公式ネットワーク（キーパーソン）**

| 人物 | 影響力 |
|------|--------|
| 山田部長 | 大手顧客の意思決定に影響大 |
| 技術の鈴木 | 実質的な技術判断の要 |
| 経理の佐藤 | システム関連の予算承認 |

**意思決定フロー**
- 日常業務：現場で即断
- システム変更：部長会議で協議
- 戦略的判断：役員会決裁

**各部門のペインポイント**

| 部門 | 課題 |
|------|------|
| 営業 | 在庫確認に時間がかかる |
| 技術 | 過去の対応履歴が散在 |
| 経理 | 手作業の請求書処理 |
| 物流 | 出荷指示の二重入力 |

この理解により、システム設計時に「誰の、どんな痛みを解決するか」が明確になります。

### 既存システムランドスケープ：現実を直視する

理想を語る前に、現実を正確に把握します。

**システムランドスケープ**

**基幹システム一覧**

| システム | 名称 | 経過年数 | 技術 | 状態 | データ品質 |
|---------|------|---------|------|------|-----------|
| 基幹システム | TradeCore | 15年 | COBOL/Oracle | 安定稼働だが改修困難 | ★★★★☆ |
| 在庫管理 | Excel/Access | バラバラ | — | 属人化が激しい | ★★☆☆☆ |

**連携ポイント**
- ファイル連携
  - 日次バッチ（CSV）：基幹→経理
  - 週次レポート（Excel）：営業→役員
- 手動連携
  - 在庫データ：Excel→基幹（手入力）
  - 顧客情報：名刺→Excel→基幹

**技術的負債**

| 深刻度 | 項目 |
|--------|------|
| 🔴 Critical | バックアップが手動、パスワードが共通 |
| 🟠 High | エラー時の手動リカバリ、データ重複（顧客マスタ） |
| 🟡 Medium | UIが古い、レポートが遅い |

この現実を踏まえた上で、段階的な改善計画を立てます。一気にすべてを変えようとすると、必ず失敗します。

### 文化と風土：見えない力を理解する

最も重要で、最も見落とされがちなのが「組織文化」です。

**文化評価**

**意思決定スタイル**
- 方式：ボトムアップ（稟議重視）
- 速度：慎重（平均2週間）
- リスク許容度：保守的

**変革への準備度**

| 過去の経験 | 結果 | 時期 |
|-----------|------|------|
| POSシステム導入 | 成功 | 5年前 |
| CRM導入 | 失敗（利用率10%） | 3年前 |

*学んだ教訓：*
- 現場を巻き込まないと失敗する
- 一気に変えると抵抗が大きい
- 教育には時間をかける

**コミュニケーションスタイル**
- 公式：会議と議事録
- 非公式：喫煙所とランチ
- 効果的な方法：部長への根回し

**価値観**
- 明示的：顧客第一、品質重視
- 暗黙的：
  - 長期的な関係性重視
  - 失敗を恐れる
  - 前例踏襲

この文化に逆らうシステムは、どんなに技術的に優れていても受け入れられません。文化に沿った変革のアプローチが必要です。

### ステークホルダー分析の深化

Claude Codeは、表面的な情報から深い洞察を導き出します。

```

   - 営業本部長を早期に味方につける
```

## Phase 0-1の実践的テクニック

### 1. ドキュメントの「生き物化」

従来の要件定義書は、作った瞬間から陳腐化が始まります。V5では、ドキュメントを「生きた状態」に保ちます。

**生きたドキュメントの運用**

**フォーマット**
- 推奨形式：YAML/Markdown in Git
- 利点：
  - バージョン管理可能
  - 差分が明確
  - レビュー可能
  - Claude Codeが解析可能

**更新トリガー**
- ステークホルダーMTG後
- 重要な決定時
- 前提条件の変化時

**検証方法**

| 種別 | 項目 |
|------|------|
| 自動検証 | YAMLフォーマットチェック、相互参照の整合性、必須項目の確認 |
| 手動検証 | 月次レビュー会、ステークホルダー確認 |

### 2. 「小さな成功」の積み重ね

Phase 0-1は成果が見えにくいフェーズです。だからこそ、小さな成功を可視化します。

## まとめ：基礎があってこその高層ビル

Phase 0-1は、Parasol V5の中で最も地味なフェーズかもしれません。でも、最も重要なフェーズでもあります。

ここでしっかりとした基礎を作ることで：
- プロジェクトの方向性が明確になる
- ステークホルダーの期待値が揃う
- 現実的な計画が立てられる
- リスクに備えられる
- チームの士気が上がる

逆に、ここを疎かにすると：
- 後になって「それは聞いていない」が頻発
- 現実と乖離した設計
- 予期せぬ抵抗による停滞
- 手戻りの連続

次章では、この強固な基礎の上に、いよいよ「価値」という建物を建て始めます。Phase 2の価値分析・価値設計で、ビジネスの本質的な価値を発見し、それをシステムとして実現する道筋を描いていきます。
