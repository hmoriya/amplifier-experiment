# BizBOK vs V5：設計思想の根本的違い

**バージョン**: 1.0
**最終更新**: 2025-01-08
**目的**: BizBOKとV5の設計アプローチの根本的な違いを明確化する

---

## 核心的違い

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│  BizBOK: Capability-Centric（能力中心）                                     │
│          「組織は能力の集合体である」                                        │
│          → Inside-Out（内から外へ）                                         │
│                                                                              │
│  V5:     Value Stream-Centric（価値中心）                                   │
│          「組織は価値提供の仕組みである」                                    │
│          → Outside-In（外から内へ）                                         │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

---

## 設計の起点の違い

### BizBOK：Capability-Centric（能力中心）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  BizBOKの設計フロー                                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│      「我々は何ができるか？」からスタート                                   │
│                    │                                                        │
│                    ▼                                                        │
│      ┌─────────────────────────┐                                           │
│      │    Capability Map       │  ← 最初にこれを作る                       │
│      │    （組織の能力棚卸し）  │                                           │
│      └───────────┬─────────────┘                                           │
│                  │                                                          │
│                  ▼                                                          │
│      「この能力でどう価値を届けるか？」                                     │
│                  │                                                          │
│                  ▼                                                          │
│      ┌─────────────────────────┐                                           │
│      │    Value Stream Map     │  ← 後からこれを作る                       │
│      └───────────┬─────────────┘                                           │
│                  │                                                          │
│                  ▼                                                          │
│      ┌─────────────────────────┐                                           │
│      │    Cross-Mapping        │  ← 別途作成して関連づける                 │
│      └─────────────────────────┘                                           │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### V5：Value Stream-Centric（価値中心）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  V5の設計フロー                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│      「どんな価値を届けたいか？」からスタート                               │
│                    │                                                        │
│                    ▼                                                        │
│      ┌─────────────────────────┐                                           │
│      │    Value Stream (VS)    │  ← 最初にこれを定義（Phase 2）            │
│      │    （価値の流れ）        │                                           │
│      └───────────┬─────────────┘                                           │
│                  │                                                          │
│                  ▼                                                          │
│      「この価値を実現するには何が必要か？」                                 │
│                  │                                                          │
│                  ▼                                                          │
│      ┌─────────────────────────┐                                           │
│      │    CL1/CL2/CL3          │  ← VSから導出する（Phase 3）              │
│      │    （スイムレーン内）    │     Cross-Mappingは構造に内包            │
│      └───────────┬─────────────┘                                           │
│                  │                                                          │
│                  ▼                                                          │
│      ┌─────────────────────────┐                                           │
│      │    BC / サービス設計    │  ← 直接実装に結びつく（Phase 4-6）        │
│      └─────────────────────────┘                                           │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

---

## 問いの順序の違い

| 順序 | BizBOK（能力中心） | V5（価値中心） |
|------|-------------------|----------------|
| **1st** | 何ができるか？（Capability） | 誰にどんな価値を届けるか？（VS） |
| **2nd** | どう価値に結びつけるか？（Cross-Mapping） | その価値を実現するには何が必要か？（CL） |
| **3rd** | どこに投資すべきか？（Portfolio Management） | どう実装するか？（BC/サービス） |

## 組織観の違い

### BizBOK：能力のポートフォリオとして組織を捉える

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  BizBOKの組織観                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│      ┌─────────────────────────────────────────────────────────────────┐   │
│      │                    組織 = 能力の集合体                           │   │
│      │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐   │   │
│      │  │Cap A  │ │Cap B  │ │Cap C  │ │Cap D  │ │Cap E  │ │Cap F  │   │   │
│      │  └───────┘ └───────┘ └───────┘ └───────┘ └───────┘ └───────┘   │   │
│      │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐   │   │
│      │  │Cap G  │ │Cap H  │ │Cap I  │ │Cap J  │ │Cap K  │ │Cap L  │   │   │
│      │  └───────┘ └───────┘ └───────┘ └───────┘ └───────┘ └───────┘   │   │
│      └─────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│      → 能力を棚卸しし、管理し、価値に結びつける                            │
│      → 組織変更に依存しない安定した視点を提供                              │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### V5：価値の流れとして組織を捉える

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  V5の組織観                                                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│      組織 = 価値提供の仕組み                                                │
│                                                                              │
│      VS0 ══════════════════════════════════════════════▶ 価値A             │
│           Stage1 → Stage2 → Stage3 → Stage4                                 │
│                                                                              │
│      VS1 ══════════════════════════════════════════════▶ 価値B             │
│           Stage1 → Stage2 → Stage3                                          │
│                                                                              │
│      VS2 ══════════════════════════════════════════════▶ 価値C             │
│           Stage1 → Stage2 → Stage3 → Stage4 → Stage5                        │
│                                                                              │
│      → 価値を定義し、必要な能力を導出する                                  │
│      → 価値提供の最適化がそのまま組織設計になる                            │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────────────────┐

│                              │                                              │

│                    │   「何ができるか」  │                                   │
│                    └─────────┬─────────┘                                   │
│                              │                                              │
│                              ▼                                              │
│                    ┌─────────────────────┐                                 │
│                    │   Value Stream      │                                  │
│                    │   「どう届けるか」   │                                  │
│                    └─────────┬─────────┘                                   │
│                              │                                              │

```

## 各アプローチの強みと弱み

### BizBOK（Capability-Centric）

| 区分 | 内容 |
|------|------|
| **強み** | |
| ✅ | 組織全体の能力を俯瞰できる |
| ✅ | 組織変更に依存しない安定した視点 |
| ✅ | 能力の重複・ギャップを発見しやすい |
| ✅ | 投資ポートフォリオ管理に適する |
| ✅ | M&A時の能力統合分析に有効 |
| **弱み** | |
| ⚠️ | 「能力はあるが価値に結びついていない」状態が起きうる |
| ⚠️ | Cross-Mappingを別途作らないと価値との関係が不明確 |
| ⚠️ | DX/サービス設計への直結性が弱い |
| ⚠️ | 能力ありきで価値を後付けするリスク |

### V5（Value Stream-Centric）

| 区分 | 内容 |
|------|------|
| **強み** | |
| ✅ | 価値から逆算するので「無駄な能力」が生まれにくい |
| ✅ | Cross-Mappingが構造に内包される |
| ✅ | DX/マイクロサービス設計に直結 |
| ✅ | 「なぜこの能力が必要か」が常に明確 |
| ✅ | アジャイル/リーン思考と親和性が高い |
| **弱み** | |
| ⚠️ | 組織横断の能力俯瞰がやや弱い |
| ⚠️ | VS間で類似CLが重複する可能性 |
| ⚠️ | 能力ポートフォリオ管理には追加の視点が必要 |
| ⚠️ | 既存組織の能力棚卸しには別途作業が必要 |

## 思想的背景

| 観点 | BizBOK | V5 |
|------|--------|-----|
| **時代背景** | エンタープライズアーキテクチャ（2000年代〜） | DX/クラウドネイティブ（2010年代〜） |
| **影響を受けた思想** | ポーターの価値連鎖、RBV（資源ベース理論） | リーンスタートアップ、DDD |
| **主な関心** | 組織能力の最適化・ガバナンス | 価値提供の迅速化・サービス設計 |
| **変革アプローチ** | 能力ギャップ分析 → 投資計画 | 価値ストリーム最適化 → サービス実装 |
| **典型的ユースケース** | 大企業の能力管理、M&A統合 | 新規サービス設計、DX推進 |

## 適用場面の違い

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  どちらのアプローチが適するか                                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【BizBOKが適する場面】                                                     │
│                                                                              │
│  ・既存の大企業で組織能力を棚卸ししたい                                     │
│  ・M&Aで2つの組織の能力を統合分析したい                                    │
│  ・投資ポートフォリオを能力ベースで管理したい                               │
│  ・組織変更に依存しない安定した視点が必要                                   │
│  ・「我々は何ができるか」を明確にしたい                                     │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【V5が適する場面】                                                         │
│                                                                              │
│  ・新規サービス/プロダクトを設計したい                                      │
│  ・DXで既存業務をデジタル化したい                                           │
│  ・マイクロサービスアーキテクチャを設計したい                               │
│  ・「顧客にどんな価値を届けるか」から考えたい                               │
│  ・アジャイル/リーンな開発プロセスと連携したい                              │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 補完的活用

両アプローチは排他的ではなく、補完的に活用できる：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  補完的活用パターン                                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  パターン1: V5で設計 → BizBOKで俯瞰                                        │
│                                                                              │
│      V5でVS起点の設計を行った後、                                          │
│      BizBOK的視点で組織横断の能力俯瞰を追加する                            │
│      → 価値駆動 + 能力管理 の両立                                          │
│                                                                              │
│  パターン2: BizBOKで棚卸し → V5で再設計                                    │
│                                                                              │
│      BizBOKで既存能力を棚卸しした後、                                      │
│      V5で価値起点の再設計を行う                                            │
│      → 現状理解 + 将来設計 の両立                                          │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 結論

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│  BizBOKとV5は「どちらが正しいか」ではなく                                   │
│  「どちらの視点で設計するか」の違い                                         │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  BizBOK: 「我々は何ができるか」→「どう価値に結びつけるか」          │   │
│  │          能力ポートフォリオ管理に強み                                │   │
│  │          Inside-Out（内から外へ）                                    │   │
│  │                                                                      │   │
│  │  V5:     「どんな価値を届けるか」→「何が必要か」                     │   │
│  │          DX/サービス設計に強み                                       │   │
│  │          Outside-In（外から内へ）                                    │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  目的と状況に応じて、適切なアプローチを選択する                             │
│  または両者を補完的に活用することで、より包括的な視点を得る                 │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```
