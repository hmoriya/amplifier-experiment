## 前提

ビジネスプロセスコミュニティは、エンタープライズのオペレーティングモデルの形式化により深く関与するようになり、情報モデリングやビジネスケイパビリティなど、多くのビジネスアーキテクチャドメインをツールセットに含めるよう、その範囲を急速に拡大しています。

ビジネスモデルの急速な変化に伴い、オペレーションおよびサポートプロセスは、作業を達成し管理するためのより良い方法を提供するために適応する必要があります。この現実は、ビジネスアーキテクチャコミュニティ全般、特にBusiness Architecture Guild®（Guild）にとって、ビジネスプロセス管理コミュニティとの協働と関与を受け入れる時宜を得た機会を提供しています。

ビジネスアーキテクチャの原則に忠実でありながら、いくつかの重要なビジネスプロセス管理（BPM）の基礎的視点を収容できれば、Guildはビジネスアーキテクチャの影響力と重要な貢献を高めることができます。

これらの原則の主な側面は、**ビジネスプロセスの定義と扱い**、そしてBIZBOK® Guideで定義されるバリューストリームとケイパビリティの概念との関係です。これは、Lean、SAFe、その他の専門コミュニティで使用される類似用語の表現とは異なります。

## 価値の実現

ビジネスアーキテクチャコミュニティは、**顧客およびその他のステークホルダーバリューストリームのレンズを通じて価値創造を記述**します。バリューストリームのステージは一連のバリューアイテムを生み出し、バリューストリームステージのすべてのバリューアイテムが集約されて、トリガーするステークホルダーのためのバリュープロポジションを実現します。

同様に、現代のビジネスプロセス管理実務者は、同じステークホルダーに提供する価値創造を表現する必要があります。これは、高レベル（高度に抽象化された）、エンドツーエンド（ステークホルダーからステークホルダーへ）のビジネスプロセスモデルを使用し、価値のある出力を特定することで行われます。

しかし、BPM実務者が生成しなければならない追加の出力があります。これらは他のプロセスに必要とされますが、規制報告書など、顧客にとってのバリューアイテムではない場合があります。

**両コミュニティは価値創造レベルでのモデルを必要としています。** しかし、ビジネスプロセスコミュニティは詳細な活動のセットも必要とし、ビジネスアーキテクチャコミュニティはビジネスケイパビリティのクロスマッピングを必要とする可能性が最も高いです。

## プロセスとバリューストリームの違い

より低い抽象度のビジネスプロセスモデルは、しばしばビジネスオペレーションに固有の依存関係を反映します。一方、バリューストリームは論理的な順序（例：左から右）で提示されることが多いですが、その順序は必ずしもバリューアイテムの作成に関連するビジネスオペレーションの実際のシーケンスを反映しているわけではありません。

| 特性 | バリューストリーム | ビジネスプロセス |
|------|------------------|-----------------|
| **焦点** | ステークホルダー価値の蓄積 | オペレーションの詳細 |
| **抽象度** | 高レベル | 低〜高レベル |
| **順序** | 論理的順序（必ずしも実際の順序ではない） | 依存関係に基づく順序 |
| **トリガー** | ビジネスイベント | ビジネスイベント |
| **クロスマッピング対象** | ケイパビリティ | 活動の詳細 |

### 製造業 vs サービス業

- **製造業の場合：** バリューストリームステージのシーケンスは、製造オブジェクトを作成するオペレーションプロセスを通常反映する
- **サービス業（医療など）の場合：** オペレーションプロセスパスは、一時停止、放棄、反復を収容する必要がある。ただし、患者は依然として診断および治療プロトコルから一般的に増加する価値を期待する

## ビジネスアーキテクチャとビジネスプロセスの視点の整合

歴史的に、ビジネスアーキテクチャとビジネスプロセスのコミュニティは、バリューストリームとビジネスプロセスは異なるものだと考えていました。

**今日では、両者は実際には同じ価値の創造に関する非常に近い視点を表しており、両方の視点が各タイプの実務者にとって有用であることが明らかです。** それぞれがオペレーション活動の異なる側面や視点を明らかにします。

### 各視点の特徴

| ビジネスアーキテクチャ バリューストリームの視点 | 現代のビジネスプロセス管理の視点 |
|------------------------------------------|--------------------------------|
| 高レベルのステークホルダー価値蓄積に関連する価値ベースの視点 | オペレーション活動の依存的性質の完全なエンドツーエンドビュー |
| イネーブリングケイパビリティへのクロスマッピング | 製品またはサービス受領者への価値の生産 |
| ビジネスイベントによってトリガー | ビジネスイベントによってトリガー |

## 2つの重要な関係

図4で注目すべき2つの重要な関係があります：

### 1. プロセス活動/サブプロセス → バリューストリームステージ

**プロセス活動またはサブプロセスがバリューストリームステージを実装します。**

- **ビジネスアーキテクチャの視点：** イネーブリングケイパビリティもバリューストリームステージにクロスマッピングされる
- **プロセスの視点：** バリューストリームステージを活動のコレクションとして見る
- **ビジネスアーキテクチャの視点：** バリューストリームステージをケイパビリティのコレクションとして見る

**両方のビューが重要です。**

### 2. プロセス活動 → ケイパビリティインスタンス

**プロセス活動とケイパビリティインスタンスの間の関係。** インスタンスは活動の実際の実装です。

## 結論

ビジネスアーキテクチャのバリューストリームと関連するケイパビリティは、組織構造、場所、製品バリアント、特定の手順、または特定のビジネスやビジネスチェーンを区別するその他のオペレーション特性に関係なく、あるタイプのビジネス（例：任意のホテル）を運営するために必要なケイパビリティに関する**価値ベースの視点**を提供します。

対照的に、価値ベースのビジネスプロセスとその活動およびフローへの分解は、（潜在的に一般的な）ビジネスの活動を通じた商品、情報、および成果の達成のフローに関する**価値ベースの視点**を提供します。

**本ホワイトペーパーの著者は、視点の二重性を調和させることが、2つのコミュニティが共存するだけでなく、すべてのモデルとビジネス設計を同期させ、それらから導き出されるビジネスソリューションを強化するために協力することで卓越するために不可欠であると信じています。**

## Parasol V5への示唆

この文書はParasol V5の設計に以下の重要な示唆を与えています：

### 1. バリューストリーム vs プロセスの明確な区別

| 概念 | Parasol V5での扱い |
|------|-------------------|
| **バリューストリーム** | Phase 2: VS0-VS7（顧客状態ベース） |
| **ビジネスプロセス** | Phase 3 → CL3活動 → ユースケース |

### 2. 二重視点の統合

BizBOKが示す「両方のビューが重要」という原則をParasol V5で実現：

```
VS0-VS7（価値の蓄積視点）
    │
    ├─→ CL1-CL3（ケイパビリティマッピング）
    │
    └─→ ユースケース（プロセス/活動の詳細）
```

### 3. スイムレーン方式の根拠

このホワイトペーパーの図3「共有された価値創造構造」は、Parasol V5のスイムレーン方式の理論的根拠を補強：

- **各ステージ** に **ケイパビリティ** と **プロセス** がマッピングされる
- Parasol V5はこれをCL3レベルで一体化して表現

### 4. 製造業 vs サービス業の違い

- **製造業**: バリューストリームステージ ≈ オペレーションプロセス順序
- **サービス業**: 論理的な価値蓄積順序（実際の処理順序とは異なる場合あり）

**Parasol V5**: 顧客状態遷移として抽象化することで、両方に対応可能
