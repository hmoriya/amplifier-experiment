## 1. 概要

組織は、管理すべき多数の脅威、改善すべき弱点、活用すべき強み、追求すべき機会に直面しています。これらの課題に対処する戦略を実行しようとする際、共通の障害に遭遇します：**共有された目標を追求するために、複数のビジネスユニットと管理レベル全体で協調して行動できないこと**です。

この課題に対処するには、チーム、プロジェクト、ビジネスユニット、パートナー全体にスケールする明確に定義された目標に基づいた、協調されたクロスビジネス戦略を定義し実行することが含まれます。

### 本ホワイトペーパーの目的

本ホワイトペーパーは、以下を目指す組織のためのガイドを提供します：
- 複雑な戦略を実行するための基盤としてビジネスアーキテクチャを形式化しスケールする
- 根本原因分析を実行する
- ビジネス課題に対応する
- より短い時間と少ないコストでより効果的なソリューションを提供する

### メタモデルとは

> **メタモデル**は「モデルのクラスの抽象構文」として定義されます。メタモデルの概念に馴染みのない読者は、モデルを記述するモデル、つまり抽象的な概念を（特定のコンテキスト内で）具体的なものに変換するために使用されるモデルと考えることができます。

ビジネスアーキテクチャは、ケイパビリティ、エンドツーエンドの価値提供、情報、組織構造のホリスティックで多次元的なビジネスビュー、およびこれらのビジネスビューと戦略、製品、ポリシー、イニシアチブ、ステークホルダー、メトリクスとの関係を表します。

### 4つのコアドメイン

| ドメイン | 説明 |
|---------|------|
| **Value Stream** | ステークホルダー価値提供のエンドツーエンド視点 |
| **Capability** | ビジネスが行うこと - 基本的な構成要素 |
| **Information** | ビジネス用語とセマンティクスの表現 |
| **Organization** | 組織構造とビジネスユニット |

### 6つの拡張ドメイン

| ドメイン | 説明 |
|---------|------|
| **Strategy** | 目標、目的、行動方針 |
| **Stakeholder** | 価値を受け取る/トリガーする個人/組織 |
| **Initiative** | 戦略の実行、プログラム/プロジェクト |
| **Product** | 顧客のニーズを満たす商品とサービスの組み合わせ |
| **Policy** | 内部/外部のポリシーと規制 |
| **Metrics** | パフォーマンス測定と評価 |

### 4.1 ビジネスアーキテクチャ ナレッジベース

ビジネスアーキテクチャの完全な価値を実現するには、ビジネスアーキテクチャをキャプチャ、管理、活用するためのフレームワークを展開する必要があります。

### 4.2 ビジネスアーキテクチャ ブループリント

**クロスマッピング**は、2つ以上のビジネスアーキテクチャドメイン間の関連を示すブループリントです。これは複数のビジネスアーキテクチャドメインを互いに、または関連するビジネスおよびIT分野に関連付ける技術です。

### 4.3 ビジネスアーキテクチャ シナリオ

最も重要なクロスマッピングブループリントの1つは、**バリューストリーム/ケイパビリティ クロスマッピング**です。このブループリントは、バリューストリームの各ステージで価値提供を可能にする特定のケイパビリティを特定します。

#### 5.1.1 バリューストリーム クロスマッピング

| クロスマッピング | 用途 |
|----------------|------|
| Value Stage ↔ Capability | どのケイパビリティがバリューアイテムの提供を可能にするかを強調 |

| Stakeholder ↔ Value Stream ↔ Value Proposition | どのステークホルダーが価値提案を望むかを強調 |
| Stakeholder ↔ Value Stage | バリューアイテムの提供に貢献するステークホルダーを特定 |

### 5.2 Capability（ケイパビリティ）

ケイパビリティはビジネスの基本的な構成要素を表します。これらの構成要素は、無限の範囲のビジネス目標を達成するために、さまざまな方法で使用、改善、再配置、活用できます。

#### ケイパビリティの3つの視点

| 視点 | 説明 |
|------|------|
| **Capability Instance** | 特定のビジネスユニットまたは他の状況的コンテキストに存在する、または存在することが想定されるケイパビリティの特定の実現 |
| **Capability Behavior** | 特定の状況またはインスタンスでケイパビリティが行動する方法 |
| **Outcome** | イベントまたはアクション、または一連のイベントとアクションの結果である最終製品 |

### 5.4 Organization（組織）

組織は、ビジネスアーキテクチャにおけるもう1つのコアビジネスコンセプトであり、ビジネスの「誰が」と「何を」のビューを提供します。

#### 組織マップの利点

- 課題分析
- ビジネス計画
- ソリューション展開
- 変革計画

### 5.5 Stakeholder（ステークホルダー）

ビジネスアーキテクチャの主要な目的は、ビジネス内のすべてのステークホルダーを明確に特定し表現することです。

#### ステークホルダーの分類

| 分類 | 説明 |
|------|------|
| **Triggering Stakeholder** | バリューストリームのバリュープロポジションを望むステークホルダー |
| **Participating Stakeholder** | バリューアイテムの達成に参加するステークホルダー |
| **External** | 外部のステークホルダー（顧客、パートナーなど） |
| **Internal** | 内部のステークホルダー（従業員など） |

---

### 5.7 Initiative（イニシアチブ）

イニシアチブは戦略の実行を表します。イニシアチブは、組織が戦略の目的を達成するために行った選択です。

---

### 5.8 Policy（ポリシー）

ポリシーは、特に銀行や公共セクターなどの高度に規制された分野で、ビジネスを行う上で重要な役割を果たすことができます。

| ポリシータイプ | 説明 |
|--------------|------|
| **Internal** | 企業の内部組織構造によって設定・維持される |
| **External** | 規制、業界慣行、商業契約など、実施し遵守しなければならない外部からの命令 |

---

### 5.9 Product（製品）

顧客の視点から見ると、製品は顧客のニーズを満たすために提供される商品とサービスの組み合わせによって提供される全体的な体験です。

## 6. 学際的な整合

学際的な整合は、ナレッジベースを拡張して非ビジネスアーキテクチャの視点を含めます。

### 6.1 要件の整合

ビジネス要件は、ビジネスが顧客やその他のステークホルダーへの価値を向上させるために何を必要とするか、または望むかを記述するためのコア表現です。

### 6.3 ビジネスプロセスの整合

ビジネスプロセスは、バリューストリーム、バリューストリームステージ、ケイパビリティ、および対応するアウトカムを通じてビジネスアーキテクチャと複数の関連を持ちます。

## 7. まとめ

ビジネスアーキテクチャは、ビジネスモデルに記述された実世界のものの抽象化です。すべての参加者がそのビジネスの同じメンタルモデルを理解し共有することを確保するには、合理化され明確に定義された語彙、価値提供モデル、および関連する視点が必要です。

### 主要なポイント

1. **メタモデル**はドメイン、基礎となるドメイン関係、およびビジネスのモデルを定義するための使用ルールの包括的なセットを定義
2. **ナレッジベース**はビジネスドメインと関連を容易にアクセス可能な形で保存しながら、モデル化されたドメインと関係の整合性を維持
3. **4つのコアドメイン**（Value Stream, Capability, Information, Organization）と**6つの拡張ドメイン**が連携して顧客価値を提供
4. **クロスマッピング**により、組織の弱点のターゲティング、機会の活用、脅威への対処、組織の強みの最大化が可能

### Industry Reference Models

Guildは、ビジネスアーキテクチャのベースラインを確立するための取り組みを加速するために、業界リファレンスモデルを提供しています：
- 金融サービス
- ヘルスケア
- 保険
- 政府
- 製造
- 電気通信
- 輸送
- 共通/汎用ビジネスアーキテクチャベースライン

### 2. メタモデルの重要な関係

BizBOKメタモデルで定義される以下の関係をParasol V5でも継承：

- **Value Stage → enabled by → Capability**
- **Capability → achieves → Outcome**
- **Outcome → assigned to → Value Item**
- **Stakeholder → triggers → Value Stream**

### 3. Capability Instance/Behaviorの概念

BizBOKの「Capability Instance」「Capability Behavior」概念は、Parasol V5のCL3活動設計に反映：

| BizBOK概念 | Parasol V5での扱い |
|-----------|-------------------|
| **Capability** | CL1-CL2（抽象的な能力） |
| **Capability Instance** | CL3（具体的な活動） |
| **Capability Behavior** | ユースケース記述 |

### 4. カスタマージャーニーとの整合

図31で示される「Value Stream → Customer Journey」の関係は、Parasol V5のVS0-VS7設計で実現：

```
Parasol V5:
  VS0-VS7（顧客状態遷移） ≈ Customer Journey Stage
  各VSでのValue Item ≈ Touchpoint での価値交換
```
