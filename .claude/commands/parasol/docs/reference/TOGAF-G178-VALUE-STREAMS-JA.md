## 1 はじめに

### 1.1 「価値」とは何か？

「価値（value）」という言葉は、古フランス語の*valoir*（「価値がある」の意）に由来します。それは、何かが持つに値するとされる評価であり、何かの重要性、価値、有用性を指します。ビジネスアーキテクチャの文脈では、価値を何かの物質的または金銭的価値として定義する比較的狭い会計・財務的観点ではなく、**有用性、利点、便益、または望ましさという最も一般的な意味**で考えることが重要です。

ビジネス界における非金銭的な価値の例には、要求された製品やサービスの正常な提供、クライアントの問題のタイムリーな解決、より良いビジネス決定を行うための最新情報へのアクセスなどがあります。

**価値は、組織が行うすべてのことの根本です。組織が存在する主な理由は、一つ以上のステークホルダーに価値を提供することです。** それは、ビジネスがどのように価値を創造、提供、獲得するかの根拠を説明するビジネスモデルの基盤です。

ビジネスアーキテクトまたはエンタープライズアーキテクトは、企業が特定のステークホルダーに対して価値を達成するさまざまな方法をモデル化、測定、分析できる必要があります。これには、価値の創造、獲得、提供を、ビジネスケイパビリティの効果的な適用によって可能になる価値生産活動の個別のステージに分解する能力が含まれます（TOGAF®シリーズガイド：ビジネスケイパビリティを参照）。

### 1.2 価値分析のアプローチ

過去、ビジネス価値をモデル化、測定、分析するためにいくつかのアプローチが使用されてきました。よく知られている（しかしよく誤解される）3つの技法には、バリューチェーン、バリューネットワーク、リーンバリューストリームがあります。各アプローチには異なる目的と焦点領域があります：

| 技法 | 主な焦点 |
|------|---------|
| バリューチェーン | 経済的価値の観点 |
| バリューネットワーク | 価値の創造・提供に関わる参加者の特定 |
| リーンバリューストリーム | ビジネスプロセスの最適化（主に製造業の文脈） |
| ビジネスアーキテクチャ バリューストリーム | 顧客/ステークホルダーの視点からのエンド・ツー・エンドの価値観点 |

**ビジネスアーキテクチャのバリューストリームのみが、顧客（またはステークホルダー）の視点からのエンド・ツー・エンドの価値観点を作成するように設計されており**、財務、組織、または運用モデルよりも、組織のビジネスモデルの実現により密接に整合しています。

### 1.3 ビジネスアーキテクチャにおけるバリューストリーム

ビジネスアーキテクチャで使用される価値分析のアプローチは、James Martinの著書『The Great Transition』に由来します。バリューストリームは、**顧客、ステークホルダー、またはエンドユーザーに対して全体的な結果を生み出す、価値付加活動のエンド・ツー・エンドの集まり**として描かれます。モデリング用語では、これらの価値付加活動は**バリューストリームステージ**として表現され、各ステージが次のステージへと段階的にステークホルダー価値を生み出し、付加していきます。

**バリューストリーム例：小売商品の取得**

```
┌─────────────┬─────────────┬─────────────┬─────────────┬─────────────┐
│  チャネル   │   商品の    │   選択の    │   支払い    │   商品の    │
│  の広告    │   陳列     │   支援     │   処理     │   配送     │
└─────────────┴─────────────┴─────────────┴─────────────┴─────────────┘
```

**主な特徴：**
- バリューストリームは**外部トリガー**の場合がある（例：商品を取得する小売顧客）
- バリューストリームは**内部トリガー**の場合がある（例：新規採用を求めるマネージャー）
- バリューストリームは**企業レベル**または**事業部門レベル**で定義される場合がある

バリューストリームの重要な原則は、**価値は常にステークホルダー（顧客、エンドユーザー、または作業によって生み出された製品・サービス・成果物の受領者）の視点から定義される**ということです。

### 1.4 バリューストリームと他のビジネスアーキテクチャ概念との関係

ビジネスアーキテクチャの実践の一環として、私たちは関心事を分離します：

| 関心事 | 要素 |
|-------|------|
| 何をするか | ビジネスケイパビリティ |
| 誰がするか | 組織 |
| どのように価値を達成するか | バリューストリーム |
| どのように実装/実行されるか | プロセス |
| どのような情報が必要か | 情報 |
| どのようなシステムが使用されるか | システム |

**バリューストリームモデリングの関係：**

```
                    ┌───────────────┐
                    │ステークホルダー│
                    └───────┬───────┘
                            │ トリガー / 受領
                            ▼
┌─────────┐  生成    ┌─────────────┐  構成要素   ┌─────────────┐
│  価値   │◄────────│バリューストリーム│───────────►│バリューステージ│
└─────────┘          └─────────────┘             └──────┬──────┘
                                                       │ 有効化
                                                       ▼
                     ┌─────────────┐              ┌────────────┐
                     │  プロセス   │◄─────────────│ケイパビリティ│
                     └─────────────┘ 運用化       └────────────┘
```

バリューストリームの完全なセットは、組織がケイパビリティを編成してステークホルダー価値を創造するさまざまな方法を描写します。バリューストリームは、組織のビジネスモデル（特に価値提案）に直接リンクしています。

**重要な違い：**
- **バリューストリーム**は外部から内部への視点を取り、ステークホルダーに関連して価値がどのように創造され移動するかを示す
- **ビジネスプロセス**は内向きの運用レベルの焦点を持ち、作業がどのように行われるかに関心がある

### 1.5 バリューストリームとバリューストリームマッピングの利点

バリューストリームをマッピングすることは、ビジネス全体のスナップショットを素早く簡単に取得する方法です。なぜなら、それらのバリューストリームは、価値提供の観点からビジネスが実行する必要のあるすべての作業を表しているからです。

**利点には以下が含まれます：**
- ビジネスリーダーが戦略計画の影響を想定し優先順位付けし、ステークホルダーエンゲージメントを管理し、新しいビジネスソリューションを展開するのを支援
- ビジネス分析と計画のための価値ベースの顧客中心のコンテキストを提供
- より効果的なビジネス要件分析、ケース管理、ソリューション設計のためのフレームワークを提供
- 外部から内部へのデザインアプローチを通じて、より効果的なビジネスモデルと運用モデルにつながる

## 2 バリューストリームの記述、分解、マッピング

### 2.1 バリューストリームの記述

バリューストリームは4つの標準要素を使用して定義されます：

| 要素 | 説明 |
|------|------|
| **名前** | ステークホルダーの視点から明確に理解できる必要がある。能動態（動詞-名詞構文）を使用。例：「小売商品の取得」「従業員の採用」 |
| **説明** | 活動の範囲を明確にする短く正確な説明 |
| **ステークホルダー** | バリューストリームを開始またはトリガーする人または役割 |
| **価値** | 正常完了時に期待される価値（ステークホルダー用語で） |

**例：**

| 要素 | 内容 |
|------|------|
| 名前 | 小売商品の取得 |
| 説明 | 希望する小売商品を探し、選択し、入手する活動 |
| ステークホルダー | 商品を購入したい小売買い物客 |
| 価値 | 顧客が希望する商品を見つけ、タイムリーに入手できる |

### 2.2 バリューストリームの分解

ステークホルダー価値が単一のステップの結果として生み出されることはまれです。価値は、一連の順次および/または並行のアクション、つまり**バリューストリームステージ**を通じて達成され、各ステージが次のステージへと段階的にステークホルダー価値を生み出し、付加していきます。

**各バリューストリームステージは以下で構成されます：**

| 要素 | 説明 |
|------|------|
| **名前** | このステージで達成される内容を特定する2〜3語 |
| **説明** | 実行される目的と活動を説明する数文 |
| **ステークホルダー** | 測定可能な価値を受け取るか、価値の創造/提供に貢献するアクター |
| **入口基準** | ステージをトリガーする開始条件または状態変化 |
| **出口基準** | 完了を示す終了状態条件（次のステージの入口基準になる） |
| **価値項目** | ステージによって創造または提供される増分価値 |

**重要：** バリューストリームは必ずしも継続的に流れるわけではありません。各ステージの後、望ましい価値が創造または提供されていない場合は停止する機会があります。

### 2.3 バリューストリームステージへのケイパビリティのマッピング

エンド・ツー・エンドのバリューストリームを定義した後、以下によって各バリューストリームステージを有効にするために必要なビジネスケイパビリティを特定します：
1. ビジネスケイパビリティマップをレビューする
2. 関連するビジネスケイパビリティを各バリューストリームステージにリンク（クロスマッピング）する

**目的：**
- ステークホルダー価値の提供に重要なケイパビリティを特定する
- 十分な品質基準を必要とするケイパビリティを特定する
- コアバリューストリームに貢献しないケイパビリティを特定する（排除候補の可能性）

## 3 バリューストリーム作成のアプローチ

よくある質問：**バリューストリームマップとビジネスケイパビリティモデル、どちらを先に作成すべきか？**

**回答：** どちらでも構いません。なぜなら、一方が他方なしに長く存在すべきではないからです。バリューストリームは、*なぜ*ビジネスケイパビリティが必要かというステークホルダーコンテキストを提供し、ケイパビリティはバリューストリームが成功するために*何が*必要かを示します。

実際には、バリューストリームを先にドラフトする方が速く簡単なことが多いです。なぜなら、より早く結果を示し、ビジネスケイパビリティモデルの開発に情報を提供できるからです。

### 3.1 指導原則

1. **明確に定義されたトリガーするステークホルダーが必要** – 例：カスタマーサービスエージェントとの議論を開始する顧客

2. **外部（通常は顧客ベース）のバリューストリームから始める** – これらは、内部に焦点を当てたバリューストリームの可能な範囲を枠組み化するのに役立つ

3. **バリューストリームはケイパビリティではなく、ケイパビリティの構成要素でもない** – それらは価値を生産/実現するための活動のシーケンスを説明する。ビジネスがそれらの活動を実行できるかどうかは、そのビジネスケイパビリティの機能である

4. **簡潔に保つ** – バリューストリームの作成には、運用レベルの詳細に下がる必要はない。それはビジネスプロセス設計の領域である

## 4 バリューストリームマッピングのシナリオ

### 4.1 基本例

**バリューストリーム定義：**

| 要素 | 内容 |
|------|------|
| 名前 | 小売商品の取得 |
| 説明 | 希望する小売商品を探し、選択し、入手する活動 |
| ステークホルダー | 商品を購入したい小売買い物客 |
| 価値 | 顧客が希望する商品を見つけ、タイムリーに入手できる |

**バリューストリームステージ：**

| ステージ | 説明 | 参加ステークホルダー | 入口基準 | 出口基準 | 価値項目 |
|---------|------|---------------------|---------|---------|---------|
| チャネルの広告 | 顧客に会社の製品を認知させる | 店舗/ウェブサイトオーナー、マーケティング | 顧客が製品を検索 | 顧客がチャネルを選択 | 顧客が利用できる小売チャネル |
| 商品の陳列 | 物理的またはデジタル検索可能な形式で製品を提示 | 店舗従業員、在庫管理者、ウェブデザイナー | 顧客がチャネルを選択 | 顧客が製品を閲覧 | 顧客に提供される製品オプション |
| 選択の支援 | 顧客のニーズに最適な製品のフィルタリングと評価を可能にする | 店舗従業員、ウェブデザイナー、小売買い物客 | 顧客が製品を閲覧 | 顧客が製品を選択 | 希望の製品が見つかった |
| 支払い処理 | 顧客から支払いを受け取り処理する | レジ係、金融機関、小売買い物客 | 顧客が製品を選択 | 料金支払い完了 | 配送コミットメント |
| 商品の配送 | 製品を顧客の手に届ける | 倉庫、輸送、小売買い物客 | 料金支払い完了 | 製品配送完了 | ユーザーが製品を所有 |

### 4.2 バリューストリームとビジネスケイパビリティのマッピング

**例：従業員採用バリューストリーム**

| 要素 | 内容 |
|------|------|
| 名前 | 従業員の採用 |
| 説明 | 適切な資格を持つ従業員を特定し雇用する活動 |
| ステークホルダー | 特定のポジションを埋めようとしている採用マネージャー |
| 価値 | 職務に適合し、迅速に採用され、生産的に働く新しい従業員 |

**バリューストリームステージ：**

```
┌─────────────┬─────────────┬─────────────┬─────────────┬─────────────┐
│  ポジション │  ポジション │   応募の    │   候補者    │   従業員    │
│  の定義    │  の周知    │   評価     │  の面接    │ のオンボーディング │
└─────────────┴─────────────┴─────────────┴─────────────┴─────────────┘
```

| ステージ | 説明 | 参加ステークホルダー | 入口基準 | 出口基準 | 価値項目 |
|---------|------|---------------------|---------|---------|---------|
| ポジションの定義 | 人員ニーズの決定、スキル/資格の特定、文書化 | 採用マネージャー、HR採用リード | 人員変更が特定された | 採用ニーズが特定された | 検索の時間と費用の節約 |
| ポジションの周知 | 広告、ポータルや採用イベントへの求人掲載 | リクルーター | 採用ニーズが特定された | ポジションが周知された | 資格のある候補者を見つける高い可能性 |

### 4.3 ヒートマッピングシナリオ

**シナリオ：** ある架空の企業が、市場需要の増加に対応して採用を行ってきました。承諾率が低下傾向にあり、ソーシャルメディアでは新入社員の扱いが悪いことが示唆されています。

**プロセス：**
1. 採用ディレクターがクロスファンクショナルチーム（プログラム管理、財務、資産管理、ビジネスアーキテクト）を編成
2. チームが自社向けに従業員採用バリューストリームを洗練
3. ビジネスケイパビリティをバリューストリームステージにマッピング
4. ギャップを示す初期ヒートマップを作成（赤 = 重大なギャップ、黄 = 若干の調整が必要）
5. ギャップ領域に関するより詳細な情報を収集
6. 再集合して根本原因を特定

**主な発見：** 退職したIT従業員がオンボーディングシーケンスを手動で追跡していました。誰もその役割を引き継がなかったため、新入社員は完全な生産性に達するまでより長い遅延を経験しました。

**ヒートマップの凡例：**
- 赤：必要な状態と現在の状態の間に重大なギャップ
- 黄：若干の調整が必要
- 緑：適切なパフォーマンス

**解決策：**
1. 最初のバリューストリームステージにポリシー普及を主要ケイパビリティとして追加
2. 従業員オンボーディングをカバーする下位バリューストリームの必要性を認識
3. 会社の採用を改善するためのバリューストリームオーナーの必要性を特定

## 5 結論

バリューストリームの使用は、主要なステークホルダーの各グループに対してどのような価値を創造し提供する必要があるかを理解するために設計された重要なビジネスアーキテクチャ技法です。

**重要なポイント：**
- バリューストリームをステージに分解することで、価値がどのように創造/提供されるかをより明確に理解できる
- 重要な価値付加活動を支援するために必要なビジネスケイパビリティを特定する
- バリューストリームはビジネスケイパビリティの対となる概念
- **ケイパビリティ**はビジネスが*何を*するかを扱う
- **バリューストリーム**はビジネスが*どのように*ステークホルダーに価値を提供するかに焦点を当てる

バリューストリームとビジネスケイパビリティは共に、以下のための強力な分析装置を形成します：
- ビジネスの構成部分を分解する
- 内部の仕組みを理解する
- ミッションと活動の整合性を評価する
- 改善機会を特定する

バリューストリームマッピングは、ビジネス全体のスナップショットを取得する強力な方法を提供し、ビジネスリーダーがステークホルダーのために価値を創造、獲得、提供する組織の有効性を評価するのに役立ちます。

## 付録A: 代替価値分析技法の比較

### A.1 バリューチェーン

バリューチェーンは、Michael Porterの『Competitive Advantage（競争優位）』で導入されました：

> 「バリューチェーンは、コストの行動と（競争上の）差別化の既存の潜在的源泉を理解するために、企業を戦略的に関連する活動に分解します。」

**Porterのバリューチェーン構造：**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    支援活動                                         │
├─────────────────────────────────────────────────────────────────────┤
│ 企業インフラ                                                        │
│ 人的資源管理                                                        │
│ 技術開発                                                           │
│ 調達                                                               │
├─────────────────────────────────────────────────────────────────────┤
│                    主活動                                          │
├───────────┬───────────┬───────────┬───────────┬───────────┬────────┤
│ 購買物流   │  製造    │ 出荷物流  │マーケティング│ サービス  │ マージン │
│           │          │          │  & 販売   │          │        │
└───────────┴───────────┴───────────┴───────────┴───────────┴────────┘
```

**バリューストリームとの主な違い：**
- バリューチェーンは活動コストとマージン（経済的価値）に焦点を当てる
- ビジネスがケイパビリティを組み合わせて成果を生み出す方法をモデル化する構造がない
- 経済的価値生産のマクロレベルのビュー
- バリューストリームは、ケイパビリティによって有効化されるコア活動のシーケンスへの分解を可能にする

### A.2 バリューネットワーク

バリューネットワークは、ビジネス内およびビジネス間の社会的・技術的リソースを説明します。

**特徴：**
- ノードは人または役割を表す
- 接続はインタラクション/成果物を表す
- 成果物は有形（製品、現金）または無形（知識）の場合がある

**主な目的：** ビジネス内およびビジネス間の関係と価値交換領域の複雑なウェブを理解する。

### A.3 リーンバリューストリーム

リーンバリューストリームは、リーンシンキング運動（トヨタの「モノと情報の流れ図」）から生まれました。

**コア前提：** プロセスから無駄を特定して排除し、エンドユーザーに価値を創造または増加させる活動のみを保持する。

**定義：** 商品またはサービスを設計、生産、顧客に提供するために必要な活動のシーケンスの図式的表現。

**典型的な指標：**
- 処理時間
- 応答時間
- 完成率と正確率

**ビジネスアーキテクチャのバリューストリームとの主な違い：**
- 主な目的：情報/材料のフローを文書化、分析、改善する
- より広いアーキテクチャ目的のために設計されていない
- 重要な価値創造ステージへの分解に適していない
- 有効化ビジネスケイパビリティへのクロスマッピングのために設計されていない
