# TOGAF/BizBOK Value Streams vs Parasol V5 比較

**Parasol V5の理論的基盤と独自拡張の整理**

**最終更新**: 2025-01-08
**検証元**: TOGAF Series Guide: Value Streams (G178)

---

## 概要

Parasol V5は、TOGAF Series Guide（Business Architecture）およびBizBOK（Business Architecture Body of Knowledge）のバリューストリーム概念を基盤としつつ、独自の拡張を加えています。

本ドキュメントでは、引用元との共通点と相違点を明確にし、理論的基盤と独自拡張を区別します。

---

## 1. 価値（Value）の定義

### TOGAF/BizBOKの定義

> 「価値（Value）」とは、会計上の金銭的価値に限定されません。これは「何かが持つに値するとされる評価」、すなわち「重要性、価値、有用性」を指しています。

> ビジネスコンテキストにおいては、「要求された製品やサービスが成功裏に提供されること」や「より良い意思決定のために最新情報にアクセスできること」といった、金銭以外の有用性、利点、便益または望ましさという最も一般的な意味で価値を考えることが重要です。

> **組織は、一つ以上のステークホルダー（利害関係者）に価値を提供するために存在しており、価値は組織のビジネスモデルの基盤と言えます。**

### Parasol V5での採用

| 観点 | TOGAF/BizBOK | Parasol V5 | 状態 |
|------|--------------|------------|------|
| 金銭以外の価値 | 有用性、利点、便益 | VL1-VL3で非金銭的価値も定義 | **採用** |
| ステークホルダー視点 | 受け手の認識が重要 | 顧客視点で価値を定義 | **採用** |
| ビジネスモデルの基盤 | 組織存在の理由 | 設計の出発点として位置づけ | **採用** |

## 2. バリューストリーム（Value Stream）の定義

### TOGAF/BizBOKの定義

> ビジネスアーキテクチャで用いるバリューストリームは、James Martinの著作に由来し、「顧客、ステークホルダー、またはエンドユーザーに対して全体的な結果を生み出す、価値付加活動のエンド・ツー・エンドの集まり」として定義されます。
>
> "An end-to-end collection of value-adding activities that create an overall result for a customer, stakeholder, or end-user."

> モデル上では、これらの価値付加活動は「バリューストリームステージ」として表現され、各ステージが次のステージへと段階的なステークホルダー価値を生み出し、価値を追加していきます。

> **バリューストリームの最も重要な原則は、「価値は、常にステークホルダー（顧客、エンドユーザー、受領者）の視点から定義される」という点です。**

### Parasol V5での採用と拡張

| 観点 | TOGAF/BizBOK | Parasol V5 | 状態 |
|------|--------------|------------|------|
| 定義 | James Martinの定義 | エンド・ツー・エンドの価値提供 | **採用** |
| 視点 | ステークホルダー視点 | 顧客状態の変化として設計 | **採用** |
| ステージ構造 | 活動ベース・可変数 | 顧客状態ベース・固定8段階 | **拡張**（後述） |

## 3. バリューストリームステージの比較（重要な相違点）

### TOGAFのバリューストリームステージ

TOGAFでは、バリューストリームステージは**活動ベース**で定義され、バリューストリームごとに**可変数**のステージを設計します。

**TOGAFステージの定義要素:**

| 要素 | 説明 |
|------|------|
| **Name** | 2〜3語でこのステージで達成される内容を特定 |
| **Description** | 目的と実行される活動の説明 |
| **Stakeholders** | 価値を受け取る、または創造に貢献するアクター |
| **Entrance Criteria** | ステージをトリガーする開始条件 |
| **Exit Criteria** | 完了を示す終了条件（次ステージの入口基準） |
| **Value Item** | このステージで創造または提供される増分価値 |

**TOGAFの例: 小売商品取得（5ステージ）**

```
┌─────────────┬─────────────┬─────────────┬─────────────┬─────────────┐
│  チャネル   │   商品の    │   選択の    │   支払い    │   商品の    │
│  の広告    │   陳列     │   支援     │   処理     │   配送     │
└─────────────┴─────────────┴─────────────┴─────────────┴─────────────┘
```

### Parasol V5のバリューストリームステージ（VS0-VS7）

Parasol V5では、バリューストリームステージを**顧客状態ベース**で定義し、**固定8段階**の汎用モデルとして実装しています。

```
VS0: 潜在状態（Latent）      - 価値を知らない
VS1: 認知状態（Aware）       - 価値の存在を認識
VS2: 選好形成状態（Prefer）  - 価値を選びたい
VS3: 決定状態（Decide）      - 購入/契約を決定
VS4: 獲得状態（Acquire）     - 価値を手に入れた
VS5: 実感状態（Experience）  - 価値を日常的に実感
VS6: 継続状態（Continue）    - 継続的に価値を享受
VS7: 推奨状態（Advocate）    - 他者に価値を推奨
```

### 相違点の整理

| 観点 | TOGAF | Parasol V5 | 相違の意図 |
|------|-------|------------|-----------|
| **ステージの性質** | 活動（Activity）ベース | 状態（State）ベース | 顧客体験ジャーニーとの統合 |
| **ステージ数** | 可変（バリューストリームごとに設計） | 固定8段階 | 汎用モデルによる比較可能性 |
| **名前の形式** | 動詞+名詞（例: 商品を陳列） | 状態名（例: 認知状態） | 顧客視点の一貫性 |
| **入口/出口基準** | 明示的に定義 | 状態遷移として暗黙化 | 設計の簡素化 |

**注**: Parasol V5のVS0-VS7は、TOGAFの「ステージ」概念を**拡張・再解釈**したものであり、完全な採用ではありません。

## 4. 類似概念との違い

### TOGAF/BizBOKの説明

> バリューストリームという言葉は、従来も業務プロセス効率を改善するための技法としても使われており、これはビジネスアーキテクチャで用いている意味とは異なります。

| 概念 | 由来 | 焦点 | BAバリューストリームとの違い |
|------|------|------|---------------------------|
| **リーンバリューストリーム** | トヨタ（モノと情報の流れ図） | プロセス最適化、ムダ排除 | 製造業のプロセス改善が目的、ケイパビリティへのマッピングなし |
| **バリューチェーン** | マイケル・ポーター | 企業活動中心、経済的価値（金銭）創出 | 顧客視点ではなく企業視点、コストとマージンに焦点 |
| **バリューネットワーク** | Verna Allee | 価値交換・関係性の記述 | リソース間の関係性が焦点、エンド・ツー・エンドではない |

### Parasol V5での位置づけ

| 概念 | Parasol V5での扱い |
|------|-------------------|
| **バリューストリーム（BA）** | **採用** - VS0-VS7として実装（顧客状態モデルに拡張） |
| **リーンバリューストリーム** | **区別** - 採用せず（目的が異なる） |
| **バリューチェーン** | **区別** - 企業視点ではなく顧客視点を採用 |
| **バリューネットワーク** | 明示的に言及なし |

## 5. ケイパビリティとの関係

### TOGAF/BizBOKの説明

> ビジネスアーキテクチャでは、私たちは、
> - 何をするのか（ビジネスケイパビリティ）
> - 誰がそれをするのか（組織）
> - どのように価値が達成されるのか（バリューストリーム）
> - どのように運用レベルで実行されるのか（プロセス）
> - 必要とされる情報
> - 使用されるシステム
>
> それぞれの関心ごとを切り離し、理解しようとします。

> **アーキテクチャの重要な考え方は、関心ごとを分離し、そしてそれらの関係性を理解することで、全体を理解しようとする試みです。**

### TOGAF/BizBOKのケイパビリティマッピング

> バリューストリームを定義した後、各バリューストリームステージを有効にするために必要なビジネスケイパビリティを特定します。
>
> 目的:
> - ステークホルダー価値の提供に重要なケイパビリティを特定
> - 十分な品質基準を必要とするケイパビリティを特定
> - コアバリューストリームに貢献しないケイパビリティを特定（排除候補）

### 【重要】TOGAFのケイパビリティ重複許容（G178原文より）

**注: TOGAFの表現形式**

TOGAF G178では、ケイパビリティマッピングを**UMLや正式な図面ではなく、テキスト・テーブル形式**で表現しています。これは「運用詳細に入らない」という指導原則に沿った簡潔な表現です。

| 形式 | 用途 | 特徴 |
|------|------|------|
| **テーブル** | ケイパビリティマッピング | 列=ステージ、セル=関連Capability（重複許容） |
| **ASCIIボックス図** | ステージの流れ | 左→右の時間軸を表現 |
| **概念関係図** | 要素間の関係 | UMLではない簡易図 |

---

TOGAF G178の「Recruit Employee」マッピング例では、**同一ケイパビリティが複数ステージに重複して出現することを許容**しています:

```
| Define Position | Communicate Position | Assess Responses | Interview Candidates | Onboard Employee |
|-----------------|---------------------|------------------|---------------------|------------------|
| Finance Mgmt.   | Asset Mgmt.         | Agreement Mgmt.  | HR Mgmt.            | HR Mgmt.         |
| HR Mgmt.        | Asset Allocation    | Security Mgmt.   | Agreement Structuring| Benefits Mgmt.   |
| Program Mgmt.   | Position Advertising| HR Mgmt.         | HR Mgmt.            | Employee Background|
| Employee Supply | ...                 | Competency Mgmt. | Employee Supply     | Onboard Tracking |
```

**注目点**: `HR Mgmt.` が **4つのステージ**（Define Position, Assess Responses, Interview Candidates, Onboard Employee）に重複して出現。

**TOGAFの特徴:**
- **多対多マッピング**: Capability ↔ Stage は多対多の関係
- **同一ケイパビリティの横断**: 一つのケイパビリティが複数ステージにまたがることを許容
- **時間軸を超越**: ケイパビリティは時間的分布を持たない概念として扱う

### Parasol V5のスイムレーン方式との対比

| 観点 | TOGAF | Parasol V5 |
|------|-------|------------|
| **マッピング関係** | 多対多（Capability ↔ Stage） | 一対多（Stage → CL3活動） |
| **同一Capability** | **複数ステージに重複出現を許容** | ステージごとに活動として時間分解 |
| **分析視点** | 「どのCapabilityが必要か」 | 「このステージで何をするか」 |
| **時間軸** | Capabilityは時間を超越 | **スイムレーンで時間分布を明示** |

**DX/サービス設計への影響:**

```
【TOGAF方式】同一ケイパビリティがステージ横断

Stage1   Stage2   Stage3   Stage4   Stage5
  │        │        │        │        │
  ├───── HR Mgmt. ─────────────────────┤  ← 同一Capabilityが横断
  │        │        │        │        │

  → サービス境界が曖昧になりやすい
  → 「HR Service」は全ステージを担当？分割？

【Parasol V5方式】ステージごとに時間分解

Stage1   Stage2   Stage3   Stage4   Stage5
  │        │        │        │        │
  ├─求人定義─┼─応募受付─┼─選考支援─┼─入社処理─┤
  │        │        │        │        │

  → サービス境界 = ステージ境界の候補
  → マイクロサービス設計との親和性
```

**Parasol V5がスイムレーン方式を採用した理由:**
1. **サービス境界の明確化**: ステージ = Bounded Context の候補
2. **責任の分離**: 同じ「人事」でもステージごとに異なる責任を明示
3. **イベント駆動設計**: ステージ遷移 = ドメインイベント
4. **DX親和性**: マイクロサービス/API設計への直接的なマッピング

### Parasol V5での採用と拡張

| 関心ごと | TOGAF/BizBOK | Parasol V5 | 状態 |
|----------|--------------|------------|------|
| **WHAT**（何をするか） | ビジネスケイパビリティ | CL1-CL3（3階層） | **採用・階層化** |
| **WHO**（誰がするか） | 組織 | Phase 1 コンテキスト | **採用** |
| **HOW**（価値達成） | バリューストリーム | VS0-VS7 | **採用・拡張** |
| **HOW**（実行） | プロセス | CL3→ユースケース | **採用・連携** |
| **WITH**（情報・システム） | 情報・システム | Phase 5 ソフトウェア設計 | **採用・拡張** |

### ケイパビリティ分類の比較

| 分類 | TOGAF/BizBOK | Parasol V5 |
|------|--------------|------------|
| **Core** | 差別化の源泉 | **採用** - 差別化度★★★、価値必然性★★★ |
| **Supporting** | Coreを支援 | **採用** - 差別化度★★、価値必然性★★ |
| **Generic** | 業界標準 | **採用** - 差別化度★、価値必然性★ |
| **VCI** | （なし） | **独自追加** - 差別化度★、価値必然性★★★ |

## 6. バリューストリーム作成の指導原則

### TOGAFの指導原則

TOGAF G178では、バリューストリーム作成の4つの指導原則を定めています:

1. **明確に定義されたトリガーするステークホルダーが必要**
   - 例: カスタマーサービスエージェントとの議論を開始する顧客

2. **外部（顧客ベース）のバリューストリームから始める**
   - 外部VSが内部VSの範囲を枠組み化する

3. **バリューストリームはケイパビリティではない**
   - VSは価値を生産/実現するための活動のシーケンスを説明
   - その活動を実行できるかはケイパビリティの機能

4. **簡潔に保つ**
   - 運用レベルの詳細は不要（それはプロセス設計の領域）

### Parasol V5での採用

| 原則 | TOGAF | Parasol V5 | 状態 |
|------|-------|------------|------|
| トリガーするステークホルダー | 必須 | 顧客を起点に設計 | **採用** |
| 外部から内部へ | 顧客VSから開始 | 顧客状態遷移から開始 | **採用** |
| VSとケイパビリティの区別 | 明確に区別 | Phase 2(VS) と Phase 3(CL) で分離 | **採用** |
| 簡潔さ | 運用詳細は別 | CL3/Phase 5で詳細化 | **採用** |

## 7. Parasol V5 独自の拡張

### 7.1 価値分解（VL1→VL2→VL3）

TOGAF/BizBOKでは価値の階層的分解は明示されていません。

```
VL1: 最上位価値（経営判断基準）
  └─ VL2: 価値グループ（部門投資基準）
       └─ VL3: 詳細価値（チーム行動基準）★価値必然性評価
```

**独自性の理由**: 価値を実行可能な粒度まで分解し、各レベルで異なる意思決定者に対応

### 7.2 価値必然性評価

VL3レベルで「この価値要素が欠けた場合の影響」を評価し、TVDC分類に継承。

```
価値必然性: ★★★（致命的）/ ★★（重要）/ ★（あると良い）
```

**独自性の理由**: TOGAFには価値要素の重要度評価の概念がない

### 7.3 VMS（価値マイルストーン）バックキャスティング

理想状態（VMS5）から現在（VMS1）へ逆算する手法。

```
VMS5: 理想状態 ← ★ここから始める
  ↑
VMS4: 価値実感状態
  ↑
VMS3: 主要価値体験状態
  ↑
VMS2: 価値認識状態
  ↑
VMS1: 初期価値体験状態
```

**独自性の理由**: TOGAFはフォワード設計、Parasolはバックキャスティングを採用

### 7.4 TVDC 4分類（VCI追加）

従来の3分類に「VCI（Value-Critical Infrastructure）」を追加。

```
                  差別化度
               高 ←────→ 低
            ┌──────────┬──────────┐
価値    高  │   Core   │   VCI    │ ← 独自追加
必然性      ├──────────┼──────────┤
        低  │Supporting│  Generic │
            └──────────┴──────────┘
```

**独自性の理由**: 「差別化しないが不可欠」なケイパビリティの投資判断を明確化

### 7.5 固定8段階の顧客状態モデル（VS0-VS7）

TOGAFの可変ステージを、固定8段階の顧客状態モデルとして再定義。

**独自性の理由**:
- 顧客体験ジャーニーとの統合
- 異なるプロジェクト間での比較可能性
- 設計の標準化

### 7.6 2つのバックキャスティング

| 軸 | 問い | アウトプット |
|----|------|-------------|
| **VL→VMS（深さ）** | 価値の本質は何か | 達成すべき状態の定義 |
| **VS→Stage（流れ）** | 価値をどう届けるか | 体験の旅路の設計 |

### 7.7 Phase 1-7 一貫フロー

ビジネス価値からソフトウェア実装までの一貫したトレーサビリティ。

```
Phase 1: コンテキスト理解
    ↓
Phase 2: 価値定義・バリューストリームデザイン
    ↓ VL3価値必然性継承
Phase 3: ケイパビリティ分解（TVDC 4分類）
    ↓ TVDC分類継承
Phase 4: アプリケーションアーキテクチャ設計
    ↓
Phase 5: ソフトウェア設計
    ↓
Phase 6-7: 実装・運用
```

**独自性の理由**: TOGAFはビジネスアーキテクチャに焦点、Parasolはソフトウェア実装まで一貫
