# Parasol V5 ビジネスアーキテクチャ対応

**バージョン**: 1.0
**最終更新**: 2025-01-08
**目的**: V5がBizBOKの定義するビジネスアーキテクチャとして成立することを示す

---

## 評価結論

> **Parasol V5は、BizBOKが定義するビジネスアーキテクチャとして成立する**
>
> 形式はBizBOKと異なるが、4コアドメインを実質的にカバーし、
> 最重要ブループリント（Cross-Mapping）も実現している。

## 各ドメインの詳細対応

### 1. Value Stream ドメイン

**BizBOKの定義**:
> ステークホルダーに価値を提供するために必要な一連の活動のエンド・ツー・エンド視点

**V5での実現**:

| 要素 | V5での対応 |
|------|-----------|
| **価値の流れ** | VS0-VS7で企業活動を価値軸で分解 |
| **ステージ** | 各VSを価値提供ステージに分解 |
| **ステークホルダー** | VS定義時にトリガー/達成価値として明記 |
| **E2E視点** | Phase 2で全体俯瞰を先に設計 |

**V5の特徴**:
- 価値を「起点」として設計を開始（VS-First Design）
- 8本のVSで企業活動を網羅的にカバー

---

### 2. Capability ドメイン

**BizBOKの定義**:
> ビジネスが行うこと - ビジネスアーキテクチャの基本的な構成要素

**V5での実現**:

| BizBOK | V5 | 説明 |
|--------|-----|------|
| L1 Capability | CL1 Activity Area | 活動領域としての傾向的分類 |
| L2 Capability | CL2 Capability | TVDC正式分類を持つケイパビリティ |
| L3 Capability | CL3 Business Operation | BC実装単位の業務オペレーション |

**V5の特徴**:
- BizBOKと異なり、各レベルが**異なる概念**を表す
- CL1の分類はCL2に継承されない（独立した分類体系）
- TVDC 4分類（Core/VCI/Supporting/Generic）はCL2でのみ適用

**意味論の違い**:

```
BizBOK:  L1 → L2 → L3 （すべて「Capability」の詳細化）
         同じ概念の異なる粒度

V5:      CL1 → CL2 → CL3 （異なる概念の階層）
         Activity Area → Capability → Business Operation
```

---

### 3. Information ドメイン

**BizBOKの定義**:
> ビジネス用語とセマンティクスの表現

**V5での実現**:

| 要素 | V5での対応 |
|------|-----------|
| **ビジネス用語** | BCごとのドメインモデルで定義 |
| **セマンティクス** | エンティティ関係として表現 |
| **情報の流れ** | BC間のコンテキストマッピングで表現 |

**V5の特徴**:
- DDD（ドメイン駆動設計）のドメインモデルとして実現
- 各BCが独自のユビキタス言語を持つ
- Phase 5でソフトウェア設計と一体化

---

### 4. Organization ドメイン

**BizBOKの定義**:
> 組織構造とビジネスユニット

**V5での実現**:

| 要素 | V5での対応 |
|------|-----------|
| **組織構造** | CL2/CL3に組織オペレーションとして統合 |
| **ビジネスユニット** | CL2の責任主体として表現 |
| **役割** | CL3の業務オペレーション担当として表現 |

**V5の特徴**:
- 独立ドメインではなく、Capability階層に**統合**
- 組織がステージをまたぐケースも表現可能
- 「誰がやるか」がCL2/CL3の属性として明示

**ステージ横断組織の例**:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  VS: 顧客獲得バリューストリーム                                             │
├────────────────┬────────────────┬────────────────┬────────────────┬────────┤
│   認知獲得     │   興味喚起     │   検討支援     │   契約締結     │  ...   │
├────────────────┴────────────────┴────────────────┴────────────────┴────────┤
│                                                                              │
│   マーケティング部門 ─────────────────┐                                     │
│   （ステージ1-3を横断）                │                                     │
│                                        │                                     │
│                         営業部門 ──────┴─────────────────                   │
│                         （ステージ2-4を横断）                                │
│                                                                              │
│   → 組織がVSの複数ステージに関与することを表現可能                         │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Cross-Mapping の実現

### BizBOKの要件

> **「最も重要なクロスマッピングブループリントの1つは、バリューストリーム/ケイパビリティ クロスマッピングです。」**

### V5での実現：スイムレーン方式

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  BizBOK Cross-Mapping vs V5 スイムレーン                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【BizBOK方式】多対多マッピング                                             │
│                                                                              │
│      Value Stream A    Value Stream B    Value Stream C                     │
│           │                 │                 │                             │
│           ▼                 ▼                 ▼                             │
│      ┌─────────────────────────────────────────────┐                       │
│      │           Capability Map                    │                        │
│      │  ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐     │                        │
│      │  │Cap│ │Cap│ │Cap│ │Cap│ │Cap│ │Cap│     │                        │
│      │  │ A │ │ B │ │ C │ │ D │ │ E │ │ F │     │                        │
│      │  └───┘ └───┘ └───┘ └───┘ └───┘ └───┘     │                        │
│      └─────────────────────────────────────────────┘                       │
│      ★ 同一Capが複数VSに出現（多対多）                                     │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【V5方式】スイムレーン統合                                                 │
│                                                                              │
│      VS-A ──┬──────┬──────┬──────┬──────┐                                  │
│             │CL1-a │CL1-b │CL1-c │CL1-d │                                  │
│             │CL2.. │CL2.. │CL2.. │CL2.. │                                  │
│             │CL3.. │CL3.. │CL3.. │CL3.. │                                  │
│             └──────┴──────┴──────┴──────┘                                  │
│                                                                              │
│      VS-B ──┬──────┬──────┬──────┐                                         │
│             │CL1-x │CL1-y │CL1-z │                                         │
│             │CL2.. │CL2.. │CL2.. │                                         │
│             │CL3.. │CL3.. │CL3.. │                                         │
│             └──────┴──────┴──────┘                                         │
│                                                                              │
│      ★ 各VSが独自のCL階層を持つ（スイムレーン）                            │
│      ★ Cross-MappingがVS構造に内包される                                   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 両方式の比較

| 観点 | BizBOK（多対多） | V5（スイムレーン） |
|------|------------------|-------------------|
| **Capabilityの再利用** | 明示的（同一Capが複数VSに出現） | 暗黙的（類似CLが複数VSに存在しうる） |
| **Cross-Mapping** | 別途ブループリントとして作成 | VS構造に内包 |
| **設計目的** | 組織横断の能力ポートフォリオ管理 | DX/マイクロサービス設計への直結 |
| **表現の粒度** | Capability単位 | VS/ステージ単位 |

## V5がビジネスアーキテクチャとして成立する根拠

### 1. 4コアドメインの完全カバー

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  チェックリスト                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  ✅ Value Stream    - VS0-VS7で価値の流れを定義                             │
│  ✅ Capability      - CL1/CL2/CL3で能力を階層化                             │
│  ✅ Information     - BCごとのドメインモデルで情報を定義                    │
│  ✅ Organization    - CL2/CL3に組織オペレーションを統合                     │
│                                                                              │
│  → 4ドメインすべてをカバー                                                  │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. Cross-Mappingの実質的実現

BizBOKが「最重要」とするCross-Mappingが、スイムレーン方式で実現されている：
- 「どのケイパビリティがどのステージで価値を生むか」が可視化される
- VS/ステージ/CL の関係が明確

### 3. ステークホルダー価値の明確化

BizBOKの批判：
> 「ステークホルダー価値がどのように達成されるかのビジョンがない場合、ビジネスケイパビリティの展開は困難になる」

V5の対応：
- VS定義時にトリガー（誰が）と達成価値（何を得る）を明記
- Phase 2で価値の流れを最初に設計

### 4. 組織横断の表現能力

- 組織がステージをまたぐケースを表現可能
- CL2/CL3で組織の責任範囲を明示

## BizBOKとの構造的違い

### 違いは「形式」であり「本質」ではない

| 観点 | BizBOK | V5 | 本質的等価性 |
|------|--------|-----|-------------|
| **4ドメイン** | 独立した4つ | VS中心に統合 | ✅ 同じ情報をカバー |
| **Cross-Mapping** | 別途作成 | VS構造に内包 | ✅ 同じ関係を表現 |
| **Organization** | 独立ドメイン | CL統合 | ✅ 同じ組織情報を保持 |
| **設計順序** | 任意 | VS-First | ⚡ V5がより明確 |

### V5の追加価値

BizBOKにない特徴：
1. **TVDC分類**: Core/VCI/Supporting/Genericの4分類
2. **DX直結設計**: マイクロサービス/BC設計への直線的トレーサビリティ
3. **VS-First**: 価値起点の設計強制
