# Parasol V5 と BizBOK/TOGAF の比較分析

**バージョン**: 1.0
**最終更新**: 2025-01-08
**目的**: V5の設計思想を明確化し、改善可能な点を特定する

## 1. 基本概念の比較

### バリューストリームの定義

| 観点 | BizBOK/TOGAF | Parasol V5 |
|------|--------------|------------|
| **定義** | "An end-to-end collection of activities that creates a result for a customer" | 「顧客に対して全体的な結果を生み出す、価値付加活動のエンド・ツー・エンドの集まり」（James Martinに準拠） |
| **起源** | James Martin（1995）引用 | James Martin（1995）引用 |
| **整合性** | ✅ 同一定義 | ✅ 同一定義 |

### バリューストリームステージの構造

| 観点 | BizBOK/TOGAF | Parasol V5 |
|------|--------------|------------|
| **ステージ数** | **可変**（組織・目的により定義） | **標準8ステージ**（VS0-VS7、カスタマイズ可能） |
| **ステージ定義基準** | **活動ベース**（何をするか） | **顧客状態ベース**（顧客がどんな状態か） |
| **ステージ例** | "Identify Customer", "Assess Needs", "Deliver Solution" | VS0:潜在, VS1:認知, VS2:選好, VS3:決定, VS4:獲得, VS5:実感, VS6:継続, VS7:推奨 |
| **境界条件** | Entrance/Exit Criteria定義 | 顧客状態の遷移トリガー |

```
【BizBOK: 活動ベースの可変ステージ】
┌─────────────────────────────────────────────────────────┐
│ Identify → Assess → Propose → Negotiate → Deliver → ??? │
│   顧客      ニーズ   提案      交渉       配送         │
│ (個別に定義、数も内容も組織依存)                        │
└─────────────────────────────────────────────────────────┘

【V5: 顧客状態ベースの標準ステージ】
┌─────────────────────────────────────────────────────────┐
│ VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7          │
│ 潜在  認知  選好  決定  獲得  実感  継続  推奨          │
│ (標準8ステージ候補、業種に応じてカスタマイズ可能)       │
└─────────────────────────────────────────────────────────┘
```

## 2. L1/L2/L3の分解構造：根本的な違い

### BizBOK: Capability L1/L2/L3（同一概念の階層分解）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  BizBOK Capability Decomposition                                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【すべてが「Capability」】                                                  │
│                                                                              │
│  Capability L1（大分類）                                                    │
│  └─ 例: "人材管理"                                                          │
│      │                                                                       │
│      ├─→ Capability L2（中分類）                                            │
│      │   └─ 例: "採用管理", "育成管理", "評価管理"                          │
│      │       │                                                               │
│      │       └─→ Capability L3（詳細）                                      │
│      │           └─ 例: "求人作成", "応募者評価", "面接調整"                │
│      │                                                                       │
│      └─→ 全レベルで同じ「能力」の粒度違いとして扱う                        │
│                                                                              │

```

### Parasol V5: CL1/CL2/CL3（異なる概念の階層）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  Parasol V5 CL Hierarchy                                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【3つの異なる概念】                                                        │
│                                                                              │
│  CL1: Activity Area（活動領域）                                            │
│  └─ 例: "VS2 商品開発の活動領域"                                           │
│      │  └─ 傾向的分類（参考情報）: Core傾向 / Supporting傾向 / Generic傾向 │
│      │                                                                       │
│      ├─→ CL2: Capability（ケイパビリティ）                                 │
│      │   └─ 例: "発酵研究", "官能評価", "試作生産"                         │
│      │       │  └─ ★正式分類（TVDC）: Core / VCI / Supporting / Generic   │
│      │       │  └─ ★CL1の分類はCL2に継承されない                          │
│      │       │                                                               │
│      │       └─→ CL3: Sub-capability + BO（詳細能力+業務オペレーション）    │
│      │           └─ 例: "酵母株スクリーニング", "発酵条件最適化"           │
│      │               └─ 分類なし（網羅性が目的）                           │
│      │               └─ 複数アクターの協調作業                             │
│      │                                                                       │
│      └─→ 各レベルで異なる概念・異なる役割                                  │
│                                                                              │
│  ★ 特徴1: CL1/CL2/CL3は異なる概念（Activity Area / Capability / Operation）│
│  ★ 特徴2: 各VSが独自のCL構造を持つ（スイムレーン）                        │
│  ★ 特徴3: 分類はCL2のみ正式適用                                           │
│  ★ 特徴4: Value Streamに紐づいた階層構造                                   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

| 側面 | BizBOK | Parasol V5 |

| **L2の概念** | Capability（中分類） | Capability（ケイパビリティ） |

| **継承ルール** | 明示なし | CL1→CL2: 継承しない |

### なぜこの違いが重要か

```yaml
BizBOKの設計思想:
  目的: "組織全体でCapabilityを共有・再利用"
  利点:
    - 同一Capabilityが複数VSで参照される
    - 組織横断的な投資計画が容易
    - 重複投資の回避
  課題:
    - サービス境界が曖昧になりやすい
    - CapabilityとVSの責任分界が複雑

V5の設計思想:
  目的: "VS起点でサービス設計への直線的トレーサビリティ"
  利点:
    - VS→CL1→CL2→CL3→BC→Service の明確な流れ
    - CL2単位でのTVDC分類・投資判断
    - サービス境界の明確化
  課題:
    - 組織横断的なCapability再利用の可視化が弱い
    - 同名Capabilityが複数VSに存在する可能性
```

### DDD/マイクロサービス設計への影響

| 設計段階 | BizBOK | Parasol V5 |
|----------|--------|------------|
| **サービス境界** | Capabilityから導出（複数VS対応の可能性） | CL2から導出（VS単位で明確） |
| **Subdomain分類** | L1/L2/L3どこでも可能 | CL2のみ（TVDC 4分類） |
| **Bounded Context** | Capability Mapから抽出 | CL2→BCへ直接マッピング |
| **Context Map** | Capability間関係から導出 | VS間・BC間関係として定義 |

## 3. 設計思想の根本的違い

| 観点 | BizBOK/TOGAF | Parasol V5 |
|------|--------------|------------|
| **ケイパビリティの性質** | **再利用可能な組織能力**（どのステージでも使える汎用部品） | **各VSに紐づいた能力**（VSの文脈で定義） |
| **分析アプローチ** | ケイパビリティ→ステージへのマッピング（後付け） | VS→CL1→CL2→CL3の階層分解（前提から） |
| **重複の扱い** | **許容・推奨**（同一Capが複数ステージに） | **VS単位で独立**（スイムレーン的） |
| **サービス設計への影響** | 1つのCapが複数サービスから参照される可能性 | VSベースでサービス境界を決定しやすい |

## 4. V5の設計意図とメリット

### V5が標準8ステージ・スイムレーンを採用した理由

```yaml
v5_design_rationale:
  standard_8_stages:
    reason: "顧客ジャーニーの標準化による分析効率化"
    note: "標準テンプレートとして8ステージを提供、業種に応じてカスタマイズ可能"
    benefit:
      - "どの業界・組織でも共通のフレームワークで分析開始可能"
      - "ステージ定義の出発点を提供（ゼロから定義する必要なし）"
      - "VMS（価値マイルストーン）との対応が明確"

  swimlane_approach:
    reason: "サービス境界設計へのトレーサビリティ確保"
    benefit:
      - "CL2→Service→BCの直線的な設計フロー"
      - "ケイパビリティの責任所在が明確"
      - "DX（デジタル変革）でのサービス分割が容易"
```

### BizBOKの多対多がDXで困難な理由（V5の問題意識）

```
【BizBOKアプローチでのサービス設計の課題】

Cap-1「決済処理」が複数ステージで使われる場合:

Stage A ─────┐
             │
Stage B ─────┼───► Cap-1「決済処理」───► ???どのサービスに属する???
             │
Stage C ─────┘

問題:
1. 決済処理をどのサービスに配置するか不明確
2. 複数サービスから同一Capを参照→密結合のリスク
3. TVDC分類（Core/VCI等）がステージ横断で適用困難
```

```
【V5アプローチでのサービス設計】

VS3「決定」─────► CL2-3-1「決済処理」─────► BC-3「注文・決済BC」
                  TVDC: VCI
                  責任: VS3の文脈で完結

VS4「獲得」─────► CL2-4-1「配送管理」─────► BC-4「配送BC」
                  TVDC: Supporting
                  責任: VS4の文脈で完結

メリット:
1. 各CL2のサービス帰属が明確
2. TVDC分類がCL2単位で適用可能
3. BC間の境界が明確
```

## 5. V5の改善可能な点

### 5.1 BizBOKから取り入れ可能な概念

#### (A) Entrance/Exit Criteria（ステージ境界条件）

**BizBOKの定義**:
- **Entrance Criteria**: ステージをトリガーする開始条件
- **Exit Criteria**: 完了を示す終了条件

**V5への適用案**:

```yaml
# 現状のV5
vs3_decide:
  name: "決定状態"
  description: "顧客は購入/契約を決定した"
  # 境界条件が暗黙的

# 改善案: Entrance/Exit Criteriaを明示
vs3_decide:
  name: "決定状態"
  description: "顧客は購入/契約を決定した"
  entrance_criteria:
    - "顧客が商品/サービスを選択済み"
    - "価格・条件に同意"
    - "購入意思を表明"
  exit_criteria:
    - "決済が完了"
    - "契約が締結"
    - "注文が確定"
```

**期待効果**:
- VS間の境界が客観的に判定可能
- テスト設計時の受入基準として活用可能
- チーム間でのステージ認識の統一

#### (B) 5次元品質指標（Quality Metrics）

**BizBOKの定義**:
1. Quality and Correctness - 品質と正確性
2. Efficiency and Timeliness - 効率性と適時性
3. Consistency and Standardization - 一貫性と標準化
4. Availability to Stakeholder Community - ステークホルダーへの可用性
5. Performance against Expectations - 期待に対するパフォーマンス

**V5への適用案**:

```yaml
# 各VSステージに品質指標を追加
vs4_acquire:
  name: "獲得状態"
  quality_metrics:
    quality_correctness:
      description: "注文内容と配送品の一致率"
      target: "> 99.5%"
      measurement: "配送クレーム率"

    efficiency_timeliness:
      description: "注文から配送までのリードタイム"
      target: "< 48時間"
      measurement: "平均配送日数"

    consistency_standardization:
      description: "配送プロセスの標準化度"
      target: "全拠点で同一SOP適用"
      measurement: "SOP準拠率"

    availability:
      description: "配送状況の可視性"
      target: "リアルタイム追跡可能"
      measurement: "追跡システム稼働率"

    performance_expectation:
      description: "顧客満足度"
      target: "> 4.5/5.0"
      measurement: "配送後アンケートスコア"
```

**期待効果**:
- VSステージの「健全性」を定量評価可能
- 改善ポイントの特定が容易
- ヒートマップでの可視化（BizBOK推奨手法）

#### (C) Capability評価軸の拡張

**BizBOKの定義**:
- Criticality: 1-5スケール（重要度）
- Quality: 0-5スケール（現在の成熟度）

**V5との比較**:

```
【現状V5: 2軸の組み合わせによる4分類】
差別化度: ★（低）〜 ★★★（高）
価値必然性: ★（低）〜 ★★★（高）
→ Core / VCI / Supporting / Generic

【BizBOK: 数値スケールによる評価】
Criticality: 1〜5（重要度）
Quality: 0〜5（成熟度）
→ ヒートマップで可視化
```

**V5への適用案**:

```yaml
# 現状のTVDCに加えて、数値評価を追加
cl2_payment_processing:
  name: "決済処理"
  tvdc_classification: "VCI"  # 既存の4分類

  # 追加: 数値評価
  criticality: 5  # 1-5: 障害時の影響度
  quality: 3      # 0-5: 現在の成熟度

  # ギャップ分析
  gap_analysis:
    current_state: 3
    target_state: 5
    priority: "高"
    investment_needed: "信頼性向上投資"
```

**期待効果**:
- TVDC分類と数値評価の両立
- 投資優先度のより精緻な判断
- ヒートマップによる経営レポート

#### (D) Outcome（成果）の明示的定義

**BizBOKの定義**:
- Capability → Outcome（achieves）関係
- ケイパビリティが達成する具体的成果を定義

**V5への適用案**:

```yaml
# 現状V5: CL2定義
cl2_fermentation_research:
  name: "発酵研究"
  tvdc: "Core"
  operations:
    - "酵母株スクリーニング"
    - "発酵条件最適化"

# 改善案: Outcomeを明示
cl2_fermentation_research:
  name: "発酵研究"
  tvdc: "Core"

  # 追加: 成果定義
  outcomes:
    - id: "OUT-1"
      name: "新規酵母株の発見"
      measurement: "年間発見株数"
      target: "> 10株/年"

    - id: "OUT-2"
      name: "発酵効率の向上"
      measurement: "収率改善率"
      target: "> 5%/年"

  operations:
    - "酵母株スクリーニング"
    - "発酵条件最適化"
```

**期待効果**:
- CL2の価値貢献が定量化可能
- VL3（詳細価値）とOutcomeの紐付け
- 投資対効果の測定基盤

## 6. V5の現状維持すべき点

### 6.1 標準8ステージ（VS0-VS7）

**維持理由**:
- 顧客ジャーニーの**標準テンプレート**として出発点を提供
- 業界横断での比較可能性（共通フレームワーク）
- VMS（価値マイルストーン）との明確な対応
- **業種特性に応じたカスタマイズを許容**

**BizBOKの完全可変ステージを採用しない理由**:
- ゼロからのステージ定義は属人化リスクが高い
- 標準テンプレートがあることで分析開始が容易
- ただしV5も各VSの具体的定義はカスタマイズ可能

### 6.2 スイムレーン型ケイパビリティ分析

**維持理由**:
- サービス境界設計への直線的トレーサビリティ
- TVDC分類のCL2単位適用
- DXでのサービス分割の明確化

**BizBOKの多対多を採用しない理由**:
- サービス設計時の責任所在が曖昧
- 密結合のリスク
- 実装フェーズでの複雑性増大

### 6.3 TVDC 4分類

**維持理由**:
- Core/VCI/Supporting/Genericの投資方針が明確
- 差別化度×価値必然性の2軸評価が直感的
- Build/Buy/Partnerの判断基準として有効

## 8. 結論

### 💡 核心的洞察：設計思想の根本的違い

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│  BizBOKが「組織全体の能力ポートフォリオ管理」を重視するのに対し、           │
│  Parasol V5は「DX/マイクロサービス設計への直線的トレーサビリティ」を重視    │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  【BizBOK】                        【Parasol V5】                            │
│  ・組織横断的なCapability管理       ・VS起点のサービス設計                   │
│  ・能力の再利用・共有を最適化       ・実装境界の明確化を最適化               │
│  ・EA（エンタープライズ）視点       ・DX/マイクロサービス視点                 │
│  ・L1/L2/L3は同一概念の粒度分解     ・CL1/CL2/CL3は異なる概念の階層           │
│                                                                              │
│  → どちらが正しいではなく、目的に応じた選択                                 │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

この違いは、**両フレームワークの適用領域の違い**に起因します：

| 適用領域 | 推奨アプローチ |
|----------|----------------|
| **全社EA構築・M&A統合** | BizBOK（組織横断の能力管理） |
| **新規サービス開発・DX** | Parasol V5（サービス設計への直結） |
| **大企業の能力ポートフォリオ** | BizBOK |
| **スタートアップ・アジャイル開発** | Parasol V5 |

---

### V5の強み（維持）

1. **標準8ステージ**: テンプレートによる分析効率と比較可能性（カスタマイズ可能）
2. **スイムレーン型**: サービス設計へのトレーサビリティ
3. **TVDC 4分類**: 投資判断の明確な基準

### V5の改善機会（BizBOKから学ぶ）

1. **Entrance/Exit Criteria**: ステージ境界の客観化
2. **5次元品質指標**: VSステージの健全性評価
3. **Outcome明示**: CL2の価値貢献の定量化
4. **数値評価（Criticality/Quality）**: より精緻な投資判断

### 設計思想の違いの受容

BizBOKの「ケイパビリティ重複許容」はV5の設計思想と相容れない部分があるが、これは**意図的な設計判断**である。V5はDX（デジタル変革）を見据えたサービス設計への直線的なパスを優先しており、この点でBizBOKとは異なるアプローチを取っている。
