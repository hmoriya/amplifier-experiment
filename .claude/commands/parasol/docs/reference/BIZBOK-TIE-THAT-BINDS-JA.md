**出典**: Business Architecture Guild® - TSG, Inc.

## 概要

最近発表された「Business Capability Architecture Is the Tie that Binds All」という記事では、ビジネス戦略、エンタープライズ変革、プロジェクトポートフォリオの優先順位付けを結びつけるためにビジネスケイパビリティをどのように使用するかが議論されました。戦略、エンタープライズ変革、ポートフォリオ管理がビジネスアーキテクチャを使用してより効果的に管理されることには同意し、ケイパビリティがビジネスアーキテクチャのコンポーネントであることにも同意します。しかし、この記事の「ビジネスケイパビリティアーキテクチャ」という概念は不完全であると考えます。本稿では、この概念がなぜ不完全なのか、そしてバリューマッピングを通じてどのように拡張できるかを議論します。

---

## ビジネスアーキテクチャのコア

ビジネスアーキテクチャのコアは以下の4つのビジネスドメインで構成されます：

1. **ビジネスケイパビリティ（Business Capability）**
2. **組織（Organization）**
3. **情報（Information）**
4. **バリューストリーム（Value Streams）**

これら4つのビジネスドメインにより、大規模組織の複雑なビジネス視点をエンタープライズ規模で表現することが可能になります。このエンタープライズ視点は、エンタープライズ戦略を実行可能な結果に変換し、ポートフォリオ管理、変更管理、その他の優先ビジネス目標を実現するために不可欠です。

戦略、製品、ステークホルダー、イニシアチブ、および関連するビジネス視点は、これらの概念を4つのコアビジネスドメインにマッピングおよび整合させることで、ビジネスアーキテクチャに組み込まれます。これらのビューを集合的に活用することで、経営陣は課題を評価し、解決オプションを評価し、変更の影響を予測し、その他の戦略および詳細計画を実行することが可能になります。

---

## ビジネスケイパビリティアーキテクチャ（BCA）の限界

「Tie that Binds」の記事では、ビジネスケイパビリティアーキテクチャ（BCA）という概念が紹介されています。この記事はBCAをビジネス戦略とビジネスイニシアチブを解釈し実現するためのフレームワークとして位置づけています。

**しかし、私たちは価値提供の概念をバリューストリームとバリューネットワークを使用してより詳しく説明すべきだと考えます。** これらの価値指向の視点は、ケイパビリティがステークホルダー価値を提供するためにどのように使用されるかを明確にするために、ケイパビリティにリンクされます。

この価値指向の視点を無視すると、ビジネスケイパビリティを活用する能力は大幅に最適化されなくなります。これにより、組織がケイパビリティマッピングに投資しても、その投資からビジネス価値を引き出すのに苦労するシナリオが生まれます。

---

## バリューマッピングの追加

バリューマッピングの視点を追加することで、複数のケイパビリティがステークホルダー価値の提供にどのように使用されるかを容易に示すことができます。これは、戦略、変更管理、ポートフォリオ管理の影響を評価する上で不可欠な視点です。

### ステークホルダーの定義

> 「特定の成果を通じて価値を達成することに既得権益を持つ内部または外部の個人または組織」

### BIZBOK™の引用

> 「ビジネスケイパビリティだけでは、ビジネスが短期的および長期的な問題や課題に完全に対処できるようにするには不十分です。ステークホルダーのエンゲージメントの明確なビューや、ステークホルダー価値がどのように達成されるかのビジョンがない場合、ビジネスケイパビリティの展開は困難になる可能性があります。」

---

## クロスマッピングの重要性

図1は、Account Structuringなどのケイパビリティが「Acquire Loan」バリューストリームでどのように使用されるかを示しています。

バリューストリームは、ローンを要求する人によってトリガーされます。これらの同じケイパビリティの多くは、顧客がローンを返済したり、ローンを再構築したりする可能性のある他の多くのバリューストリームでも使用されます。これらのバリューストリームをイネーブリングケイパビリティに結びつけた集合的な視点は、ビジネスアーキテクチャの本質的な側面を構成します。

### ケイパビリティ単体の限界

ケイパビリティ自体はネットワークを表しません。ネットワークは、ケイパビリティが互いに、組織に、そしてバリューストリームにどのように関連するかを示す必要があります。ケイパビリティを六角形として、ネットワークを平面のテセレーションとして表現する概念は視覚的に魅力的ですが、ケイパビリティ間の関係の数が可変である（常に6つではない）こと、およびすべての隣接関係を視覚的に機能させることが困難であることによって挫折します。

私たちは、複数のビューを可能にするより従来型の図式的アプローチを好みます。これにより、適切なビュー選択によって、ビジネスアーキテクトは手元の問題に焦点を当てることができます。

## シナリオ例：ローンのライフサイクル

以下のシナリオを考えてみましょう：

1. **顧客がローンを取得** → Acquire Loan バリューストリーム
2. **顧客が支払いを滞納** → Default Loan バリューストリームがトリガー
3. **顧客がローン再構築を開始** → Restructure Loan バリューストリームでデフォルトを防止

これらのバリューストリーム全体でのアカウント構築やその他のケイパビリティの使用の一貫性は、非常に異なる価値コンテキストシナリオでそれらがどのように使用されるかを理解するために重要です。

**これらのシナリオがない場合、ケイパビリティをどのように改善、展開、または活用するかの決定は欠如し、ステークホルダー価値提供に投資を集中する能力も失われます。**

## モデルの関係性

| 関係 | 説明 |
|------|------|
| **ケイパビリティ → バリューストリーム** | ケイパビリティがバリューストリームと価値提供を可能にする |
| **バリューストリーム → 戦略計画** | 戦略計画、ステークホルダーエンゲージメント、製品計画のコンテキストを提供 |
| **バリューストリーム → ビジネスプロセス** | バリューストリームステージ/プロセス関係によりビジネスプロセスガバナンスのフレームワークを提供 |
| **ケイパビリティ → 情報** | ケイパビリティは情報、ビジネスユニット、製品と直接関係 |
| **イニシアチブ → ケイパビリティ/バリューストリーム** | イニシアチブ投資はケイパビリティとバリューストリームの組み合わせに焦点を当て、プロジェクト投資とステークホルダー価値提供への直接的なつながりを確保 |

---

## ケイパビリティ間のコミュニケーション

ケイパビリティの主要なネットワークは、バリューストリームによって課される関係と、バリューストリーム内のケイパビリティ間の価値のコミュニケーションが通常情報のコミュニケーションを伴うか、または情報のコミュニケーションによって実現されるという概念によって定義されます。

バリューストリームのコンテキストにおけるケイパビリティ間の情報コミュニケーションのパターンは、ケイパビリティがバリューストリームのステークホルダー価値を生み出すためにどのように協力するかを計画する上で非常に重要です。バリューストリームがITによってサポートされる場合、これらのパターンを理解することは不可欠です。

---

## 結論

ビジネスケイパビリティとビジネスケイパビリティのネットワークの概念は、戦略の策定と実行、およびビジネスの運用とリソース配分の監視において基本的なものです。

これらのケイパビリティのネットワークに加えて、価値分析、組織構造、戦略目標、情報フローを構築することで、アナリストはビジネスエグゼクティブの戦略的意思決定を支援することができます。

## Parasol V5への示唆

この文書はParasol V5の設計に以下の重要な示唆を与えています：

### 1. 4コアドメインの採用

BizBOKの4コアドメイン（Capability, Organization, Information, Value Streams）をParasol V5のPhase構造で実現：

| BizBOKドメイン | Parasol V5 Phase |
|----------------|------------------|
| Value Streams | Phase 2（VS0-VS7） |
| Capability | Phase 3（CL1-CL3） |
| Organization | Phase 1（コンテキスト） |
| Information | Phase 5（ソフトウェア設計） |

### 2. クロスマッピングの重要性

> 「ケイパビリティだけでは不十分」というBizBOKの主張は、Parasol V5の**VSスイムレーン方式**の理論的根拠を補強します。

### 3. BizBOKとの相違点

| BizBOK | Parasol V5 |
|--------|------------|
| 同一ケイパビリティの複数ステージ横断を許容 | ステージごとに活動として時間分解（スイムレーン） |
| 多対多マッピング | 一対多（Stage → CL3活動） |

**Parasol V5の選択理由**: DX/マイクロサービス設計において、サービス境界を明確にするため。
