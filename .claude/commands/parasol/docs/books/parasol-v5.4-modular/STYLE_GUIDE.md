# Parasol V5.4書籍スタイルガイド

## 目標比率とトーン
- **コード30% : 概念70%** を厳守
- **ストーリー優先**: 各章は実世界のストーリーで開始
- **段階的複雑性**: 簡単な物語 → 抽象概念 → 具体例 → 応用

## 読者層別アプローチ
### エグゼクティブ読者
- **位置**: 各章の最初の2-3セクション
- **内容**: ビジネス価値、ROI、決定要因
- **トーン**: 戦略的、簡潔、数値重視

### アーキテクト読者  
- **位置**: 章の中核セクション
- **内容**: 設計原則、トレードオフ、パターン
- **トーン**: 構造的、論理的、比較分析

### 開発者読者
- **位置**: 章末と付録
- **内容**: 実装詳細、コード例、技術仕様
- **トーン**: 実践的、具体的、手順明確

## 章構成テンプレート

### 1. フック（500-800語）
**目的**: 読者の関心を引く実世界のストーリー
**形式**: 
```
## [具体的なシナリオタイトル]
実際の企業/人物の課題を物語形式で紹介
```

### 2. 問題の本質（300-500語）
**目的**: ストーリーから抽象的な問題を抽出
**形式**:
```
## なぜこの問題が重要なのか
- ビジネスインパクト (エグゼクティブ向け)
- 技術的課題 (アーキテクト向け)
- 実装の困難さ (開発者向け)
```

### 3. 核心概念（800-1200語）
**目的**: 理論的フレームワークの説明
**形式**:
```
## [章の核心概念]
### 理論的背景
### 基本原則
### 他手法との違い
```

### 4. 実世界例（600-1000語）  
**目的**: 概念を具体的な例で説明
**形式**:
```
## 実践例：[具体的なタイトル]
段階的な適用プロセスを示す
コード例は最小限（2-3ブロック以内）
```

### 5. 実践ガイダンス（400-600語）
**目的**: 読者が実際に適用できる指針
**形式**:
```
## いつ・どのように使うべきか
### 適用タイミング
### 成功の条件
### よくある落とし穴
```

### 6. 関連技術との統合（300-500語）
**目的**: 既存手法との関係性を明確化
**形式**:
```
## 他の手法との組み合わせ
- Agile/Scrum
- マイクロサービス  
- DDD (Domain-Driven Design)
```

## コードブロック使用指針

### 制限事項
- **1章あたり最大3ブロック**
- **1ブロック最大15行**
- **説明なしのコードは禁止**

### 配置場所
- **概念説明後**: 理解を助ける最小例のみ
- **詳細実装**: 必ず付録に分離
- **設定例**: YAML/JSON設定は簡略化

### 例外規則
- **図表**: PlantUML/Mermaidは行数制限対象外  
- **設定ファイル**: 5行以内であれば本文可
- **擬似コード**: アルゴリズム説明時は簡略化して掲載

## ビジュアル要素

### 図表の優先順位
1. **概念図**: PlantUML/Mermaidで関係性を表現
2. **フローチャート**: プロセスの可視化
3. **比較表**: Before/After、優劣比較
4. **メトリクス**: 数値、グラフ、チャート

### 図表タイトル
```
図X-Y: [簡潔な説明] (章番号-図番号)
表X-Y: [簡潔な説明] (章番号-表番号)
```
