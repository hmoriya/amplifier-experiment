## 詳細構成

### Section 1: フック (500-800語)
**ストーリー候補**: ING銀行のアジャイル変革（2015年〜）
**選定理由**: 
- 伝統的銀行が明確なコンテキスト定義により劇的変革
- 「銀行らしくない銀行」への転換成功
- 組織全体のコンテキスト共有が鍵だった事例
**導入する緊張感**: 「なぜ同じアジャイル手法でも成功と失敗が分かれるのか？」

### Section 2: 問題の本質 (300-500語)
**抽象化する課題**: アセスメント結果と実行計画の間の断絶
**3層の関心事**:
- ビジネス課題: 変革の方向性が組織内で一致しない
- アーキテクチャ課題: 技術選択が組織文脈と乖離
- 実装課題: チームごとに異なる理解での実装

### Section 3: 核心概念 (800-1200語)
**中心となる理論/フレームワーク**: Parasolコンテキスト確立フレームワーク（PCF）
**導入順序**:
1. コンテキストの4要素（境界、制約、原則、関係性）
2. 変革ドメインの特定と優先順位付け
3. ステークホルダーマップとアライメント手法

**ビジュアル表現**: 
- 図7-1: コンテキストマップテンプレート
- 図7-2: ステークホルダー影響度マトリックス

### Section 4: 実世界例 (600-1000語)
**選定した例**: 製造業のDX推進でのコンテキスト確立
**段階的展開**:
1. 初期状態：各部門バラバラのデジタル化施策
2. 課題認識：全社コンテキストの不在が原因
3. 解決適用：PCFによる統一コンテキスト定義
4. 結果評価：部門横断プロジェクトの成功率向上

**コードブロック計画**:
- Block 1 (8行): コンテキスト定義の構造化フォーマット
- Block 2 (12行): 変革ドメイン優先度スコアリング
- Block 3 (10行): ステークホルダー合意形成トラッキング

### Section 5: 実践ガイダンス (400-600語)
**適用タイミング**: 
- アセスメント完了直後
- 変革チーム編成時
- 四半期ごとの見直しタイミング

**成功条件**: 
- 全ステークホルダーの参画
- 定期的な更新とコミュニケーション
- 測定可能な成功指標の設定

**よくある失敗**: 
- IT部門だけでコンテキスト定義
- 抽象的すぎる原則設定
- 一度作って更新しない

**チェックリスト**: 
- [ ] 変革ドメイン明確化
- [ ] ステークホルダー特定
- [ ] 制約条件の文書化
- [ ] 成功指標の定義

### Section 6: 技術統合 (300-500語)
**既存手法との関係**:
- DDD: 境界づけられたコンテキストとの整合
- SAFe: ポートフォリオレベルの文脈設定
- バランススコアカード: 戦略マップとの連携

**次章への流れ**: 確立したコンテキストを産業特性に応じてカスタマイズする方法を探る
