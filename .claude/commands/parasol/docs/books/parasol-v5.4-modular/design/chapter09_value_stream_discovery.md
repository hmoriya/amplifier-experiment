## 詳細構成

### Section 1: フック (500-800語)
**ストーリー候補**: Amazon倉庫の1冊の本の旅 - 注文から配送まで
**選定理由**: 誰もが経験する価値の流れを具体的に追跡できる
**導入する緊張感**: なぜ同じ本でも配送時間が異なるのか？

### Section 2: 問題の本質 (300-500語)
**抽象化する課題**: 組織内の価値創造プロセスの不可視性
**3層の関心事**:
- ビジネス課題: 投資対効果が測定できない
- アーキテクチャ課題: システム間の価値伝達が不明瞭
- 実装課題: 各コンポーネントの貢献度が不明

### Section 3: 核心概念 (800-1200語)
**中心となる理論/フレームワーク**: 価値ストリームマッピング（VSM）
**導入順序**:
1. 価値の定義 - 顧客視点での価値とは
2. ストリームの識別 - 始点から終点までの流れ
3. 測定と可視化 - リードタイム、処理時間、待機時間

**ビジュアル表現**: 
- 図9-1: 典型的な価値ストリームマップ（現状 vs 理想）
- 図9-2: 価値の蓄積と減衰のパターン

### Section 4: 実世界例 (600-1000語)
**選定した例**: 金融機関のローン審査プロセス改革
**段階的展開**:
1. 初期状態: 平均30日の審査期間
2. 課題認識: 待機時間が全体の80%
3. 解決適用: 並列処理とデジタル化
4. 結果評価: 5日に短縮、顧客満足度40%向上

**コードブロック計画**:
- Block 1 (5行): 価値ステージの定義構造
- Block 2 (10行): リードタイム計算の簡易実装
- Block 3 (8行): 価値損失の検出パターン

### Section 5: 実践ガイダンス (400-600語)
**適用タイミング**: 新規事業立ち上げ、既存プロセス改革時
**成功条件**: 経営層の支援、現場の協力、測定可能な指標
**よくある失敗**: 局所最適化、測定疲れ、改善の形骸化
**チェックリスト**: 
- 顧客価値は明確か
- End-to-Endで見ているか
- 測定は自動化されているか

### Section 6: 技術統合 (300-500語)
**既存手法との関係**:
- Agile/Scrum: スプリント内での価値提供サイクル
- マイクロサービス: サービス間の価値伝達契約
- DDD: ドメイン境界と価値ストリームの整合

**次章への流れ**: 発見した価値ストリームを詳細なステージに分解
