# 第2章 3つのスペース - WHY・WHAT・HOW

## 2.1 はじめに

ソフトウェア開発において、私たちはしばしば「何を作るか」と「どう作るか」を混同してしまいます。要件定義の段階で実装方法に言及したり、技術的な制約から機能を決めたりすることは珍しくありません。しかし、このような混同は、本来解決すべき問題を見失わせ、ユーザーにとって価値のないソフトウェアを生み出す原因となります。

Parasol V5.4の中核をなす概念が「3つのスペース」です。これは、ソフトウェア開発のプロセスを「価値スペース（WHY）」「問題スペース（WHAT）」「ソリューションスペース（HOW）」の3つの明確に分離された領域として捉え、それぞれを独立して探索・定義することで、より本質的な価値創造を実現するアプローチです。

本章では、この3つのスペースの概念を詳しく解説し、なぜこの分離が重要なのか、そしてどのように各スペースを探索し、それらを統合していくのかを実践的な例を交えながら説明します。

## 2.2 3つのスペースの概要

### 2.2.1 価値スペース（Value Space）- WHY

価値スペースは、「なぜそれが必要なのか」を探求する領域です。ここでは、ステークホルダーが真に求めている価値や、ビジネスが達成したい目標を明らかにします。

**価値スペースで扱う要素：**
- ビジネスゴール
- ユーザーが得たい成果（アウトカム）
- 組織のミッション・ビジョン
- 市場での競争優位性
- 投資対効果（ROI）

価値スペースでは、具体的な機能や実装方法には言及しません。代わりに、「顧客満足度を20％向上させる」「業務効率を30％改善する」といった、測定可能な成果に焦点を当てます。

### 2.2.2 問題スペース（Problem Space）- WHAT

問題スペースは、「何を解決すべきか」を定義する領域です。価値スペースで明らかになった目標を達成するために、どのような課題や機会が存在するのかを探索します。

**問題スペースで扱う要素：**
- ユーザーのペイン（痛み）
- 現状のワークフローの非効率性
- ユーザーのニーズと要望
- 制約と前提条件
- 解決すべき課題の優先順位

重要なのは、問題スペースでは「解決策」ではなく「問題」に焦点を当てることです。「検索機能が必要」ではなく、「ユーザーは必要な情報を見つけられない」という形で問題を定義します。

### 2.2.3 ソリューションスペース（Solution Space）- HOW

ソリューションスペースは、「どのように解決するか」を設計・実装する領域です。問題スペースで定義された課題に対して、具体的な解決策を検討し、実装します。

**ソリューションスペースで扱う要素：**
- 機能設計
- ユーザーインターフェース
- システムアーキテクチャ
- 技術選定
- 実装詳細

ソリューションスペースでは、複数の解決策を検討し、価値スペースで定義された目標に最も効果的にアプローチできる方法を選択します。

## 2.3 なぜ3つのスペースを分離するのか

### 2.3.1 早すぎる解決策の危険性

多くの開発プロジェクトで見られる失敗パターンは、問題を十分に理解する前に解決策に飛びつくことです。「AIを使った新機能を追加しよう」「モバイルアプリを開発しよう」といった提案は、それ自体は悪くないかもしれませんが、解決すべき問題が明確でなければ、単なる技術の無駄遣いに終わる可能性があります。

**例：オンラインショッピングサイトの改善**

❌ 悪い例：
「競合他社がAIレコメンデーション機能を導入したので、我々も同じ機能を実装しよう」

✅ 良い例：
1. **価値スペース**: 「リピート購入率を30％向上させる」
2. **問題スペース**: 「ユーザーは以前購入した商品の再購入時に、商品を探すのに時間がかかっている」
3. **ソリューションスペース**: 「購入履歴ベースのクイック再注文機能」（AIレコメンデーションとは異なる解決策）

### 2.3.2 視点の多様性の確保

各スペースを分離することで、異なるステークホルダーの視点を適切に取り入れることができます。

- **価値スペース**: 経営層、事業責任者の視点
- **問題スペース**: エンドユーザー、現場担当者の視点  
- **ソリューションスペース**: 開発者、デザイナーの視点

この分離により、技術的に優れているが事業価値の低い機能や、事業価値は高いが実現不可能な要求といった落とし穴を避けることができます。

### 2.3.3 変更への適応性

スペースを分離することで、変更に対してより柔軟に対応できます。

- **価値が変わった場合**: 新しい価値に基づいて問題を再定義
- **問題の理解が深まった場合**: 既存の解決策を見直し、新しいアプローチを検討
- **技術的制約が変わった場合**: 同じ問題に対して別の解決策を探る

## 2.4 各スペースの探索方法

### 2.4.1 価値スペースの探索

価値スペースの探索では、以下のような手法を活用します：

**1. ステークホルダーインタビュー**
```
質問例：
- このプロジェクトが成功したとき、何が変わっていますか？
- 1年後、どのような成果が出ていれば満足ですか？
- なぜ今このプロジェクトが必要なのですか？
```

**2. OKR（Objectives and Key Results）の設定**
```
Objective: 顧客体験を劇的に改善する
Key Results:
- NPS（Net Promoter Score）を40から60に向上
- カスタマーサポートへの問い合わせを50％削減
- 顧客の平均滞在時間を2倍に増加
```

**3. ビジネスモデルキャンバス**
顧客セグメント、価値提案、収益の流れなどを可視化し、どこに価値創造の機会があるかを特定します。

### 2.4.2 問題スペースの探索

問題スペースの探索では、ユーザーの実際の行動と痛みを深く理解することが重要です：

**1. ユーザーリサーチ**
```
観察ポイント：
- ユーザーが現在どのようにタスクを遂行しているか
- どこで躓いたり、時間を浪費しているか
- 回避策として何をしているか
```

**2. ジャーニーマップ**
ユーザーの体験を時系列で可視化し、ペインポイントを特定します。

```
例：ECサイトでの購入ジャーニー
1. 商品検索 → 😊 簡単に見つかった
2. 商品詳細確認 → 😐 情報が不足
3. カート追加 → 😊 スムーズ
4. 送料確認 → 😠 予想より高い（ペインポイント）
5. 決済 → 😐 手順が多い
```

**3. 問題仮説キャンバス**
```
誰が：定期的にオンラインショッピングをする30-40代の主婦
いつ：平日の午前中
どこで：自宅のスマートフォンから
何を：日用品の購入時に
なぜ：送料を含めた最終価格が購入直前まで分からず
どのように：購入を躊躇したり、他のサイトと比較したりしている
```

### 2.4.3 ソリューションスペースの探索

ソリューションスペースでは、創造性と実現可能性のバランスを取りながら解決策を探索します：

**1. アイディエーション**
```
手法例：
- ブレインストーミング
- SCAMPER法（Substitute, Combine, Adapt, Modify, Put to another use, Eliminate, Reverse）
- デザインスプリント
```

**2. プロトタイピング**
```
段階的アプローチ：
1. ペーパープロトタイプ（1日）
2. クリック可能なモックアップ（3日）
3. 機能限定版MVP（2週間）
4. 本格実装（1-3ヶ月）
```

**3. 技術検証（PoC）**
```
検証項目：
- 技術的実現可能性
- パフォーマンス要件の達成
- 既存システムとの統合性
- 運用・保守の複雑さ
```

## 2.5 3つのスペースの相互作用

### 2.5.1 トレーサビリティの確保

3つのスペースは独立していますが、相互に関連しています。価値から問題、問題から解決策への明確なトレーサビリティを確保することが重要です。

```
価値：顧客満足度の向上（NPS 40→60）
  ↓
問題：注文後の配送状況が分からず不安になる
  ↓
解決策：リアルタイム配送追跡システムの実装
```

### 2.5.2 フィードバックループ

実装後の結果を価値スペースにフィードバックし、継続的な改善サイクルを回します。

```
1. 実装：配送追跡システムをリリース
2. 測定：NPSが45に向上（目標の60には未達）
3. 分析：追跡情報の更新頻度が低いという新たな問題を発見
4. 改善：更新頻度を1時間ごとから15分ごとに変更
5. 再測定：NPSが58に向上
```

### 2.5.3 スペース間の整合性チェック

定期的に3つのスペース間の整合性を確認します：

**チェックリスト：**
- [ ] すべての解決策は特定の問題に対応しているか？
- [ ] すべての問題は価値創造に貢献しているか？
- [ ] 価値の優先順位は問題と解決策に反映されているか？
- [ ] 新たに発見された問題は価値スペースと整合しているか？

## 2.6 実践例：社内ナレッジ共有システムの構築

ここでは、架空の企業における社内ナレッジ共有システムの構築を例に、3つのスペースがどのように機能するかを見ていきましょう。

### 2.6.1 プロジェクトの背景

中規模のIT企業（従業員300名）で、技術情報やプロジェクトの知見が属人化しており、同じ問題で何度も躓く状況が発生していました。

### 2.6.2 価値スペースの定義

**ステークホルダーインタビューの結果：**

経営層：「エンジニアの生産性を向上させたい」
マネージャー：「プロジェクトの立ち上げ時間を短縮したい」
エンジニア：「過去の知見を活用して、より創造的な仕事に時間を使いたい」

**定義された価値：**
1. エンジニアの問題解決時間を平均50％削減
2. プロジェクト立ち上げ期間を2週間から1週間に短縮
3. 技術的な再発問題を80％削減

### 2.6.3 問題スペースの探索

**ユーザーリサーチの発見：**

1. **情報の散在**: ドキュメントがSlack、Wiki、Google Drive、個人のPCなど様々な場所に散在
2. **検索性の欠如**: キーワードを知らないと情報にたどり着けない
3. **情報の陳腐化**: 更新されていない古い情報が多い
4. **共有の心理的障壁**: 「まだ完璧でない」情報を共有することへの抵抗感

**問題の優先順位付け：**
1. 情報の検索性（影響度：高、解決難易度：中）
2. 情報の散在（影響度：高、解決難易度：高）
3. 共有の心理的障壁（影響度：中、解決難易度：中）
4. 情報の陳腐化（影響度：中、解決難易度：低）

### 2.6.4 ソリューションスペースの設計

**アイディエーション結果：**

問題1（検索性）に対する解決策案：
- A. AIを活用した自然言語検索
- B. タグベースの分類システム
- C. 専任のキュレーターによる整理

問題2（散在）に対する解決策案：
- A. 統合プラットフォームの構築
- B. 既存ツールをAPIで連携
- C. 定期的な情報集約プロセス

**選択された解決策：**

1. **統合検索プラットフォーム**: 既存の各種ツールをAPIで連携し、統一された検索インターフェースを提供
2. **段階的共有機能**: 「下書き」「レビュー中」「公開」のステータス管理で心理的障壁を下げる
3. **自動陳腐化アラート**: 6ヶ月更新されていない情報に自動的にフラグを立てる

### 2.6.5 実装とフィードバック

**フェーズ1（3ヶ月後）:**
- 基本的な統合検索機能をリリース
- 利用率：30％のエンジニアが週1回以上利用
- 問題解決時間：平均20％削減

**発見された新たな問題:**
- 検索結果の関連性が低い
- モバイルからのアクセスが困難

**フェーズ2（6ヶ月後）:**
- 検索アルゴリズムの改善
- モバイル対応
- 利用率：70％のエンジニアが週1回以上利用
- 問題解決時間：平均45％削減

## 2.7 3つのスペースを活用する際の注意点

### 2.7.1 完璧主義の罠

各スペースを完璧に定義してから次に進もうとすると、分析麻痺に陥る可能性があります。実践では、ある程度の仮説を持って進み、新たな発見があれば柔軟に見直すことが重要です。

### 2.7.2 スペースの混同

特に経験豊富な開発者ほど、問題を聞いた瞬間に解決策を思いつく傾向があります。しかし、「その解決策が本当に問題を解決するのか」「その問題が本当に価値を生むのか」を常に問い直す必要があります。

### 2.7.3 ステークホルダーとのコミュニケーション

異なるステークホルダーは異なるスペースに関心を持ちます：
- 経営層：主に価値スペース
- ユーザー：主に問題スペース
- 開発チーム：主にソリューションスペース

コミュニケーションの際は、相手の関心スペースに応じて情報を整理し、必要に応じて他のスペースとの関連を説明することが効果的です。

## 2.8 デジタル時代における3つのスペースの進化

### 2.8.1 AIとの協働

生成AIの登場により、特にソリューションスペースの探索が大きく変わりつつあります。AIは膨大な解決策のパターンを提案できますが、それが適切な問題を解決し、真の価値を生むかどうかの判断は、依然として人間の役割です。

### 2.8.2 リアルタイムフィードバック

デジタルプロダクトでは、ユーザーの行動データをリアルタイムで収集できます。これにより、価値の実現度を継続的に測定し、問題の仮説を素早く検証し、解決策を迅速に調整することが可能になっています。

### 2.8.3 プラットフォームエコシステム

現代のソフトウェアは単独で存在することは稀で、様々なプラットフォームやサービスと連携します。これにより：
- 価値スペース：エコシステム全体での価値創造を考える必要がある
- 問題スペース：自社だけでなく、パートナーやサードパーティの課題も考慮する
- ソリューションスペース：APIやプラグインなど、拡張可能な設計が重要になる

## 2.9 まとめ

3つのスペース（価値・問題・ソリューション）の明確な分離は、Parasol V5.4の基本的な考え方です。この分離により：

1. **本質的な価値創造**: 技術ドリブンではなく、価値ドリブンの開発が可能になります
2. **効果的な問題解決**: 表面的な症状ではなく、根本的な問題に取り組めます
3. **柔軟な適応**: 状況の変化に応じて、適切なレベルで対応を変更できます
4. **チームの協調**: 異なる視点を持つメンバーが、共通の枠組みで協力できます

次章では、これらのスペースをさらに詳細に分解し、それぞれのスペース内でどのようにシンプリシティを追求していくかを探っていきます。特に、各スペースで陥りがちな複雑性の罠と、それを回避するための具体的な手法について詳しく見ていきましょう。

重要なのは、3つのスペースは概念的なツールであり、それ自体が目的ではないということです。このフレームワークを通じて、より良いソフトウェアを、より効率的に、より確実に届けることが私たちの真の目標なのです。
