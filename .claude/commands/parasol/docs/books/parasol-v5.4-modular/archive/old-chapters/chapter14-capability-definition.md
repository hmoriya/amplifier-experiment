# Chapter 14: Capability Definition

## トヨタの「見えない強み」を可視化する挑戦

2019年、トヨタ自動車の若手エンジニアである田中浩二（仮名）は、ある重要なプロジェクトに配属された。彼のミッションは、トヨタ生産システム（TPS）の「強み」を新しい電気自動車製造ラインに適用することだった。しかし、経営陣から提示された要件は曖昧だった：「我々のカイゼン能力を活かして、競合他社を圧倒する生産性を実現せよ」。

田中は困惑した。「カイゼン能力」とは具体的に何を指すのか？それは個人のスキルなのか、組織のプロセスなのか、それとも文化なのか？彼が各部門にヒアリングに回ると、誰もが異なる答えを返した。生産部門は「現場の改善提案力」と言い、品質保証部門は「不良品の根本原因分析力」と答え、人事部門は「改善マインドを持つ人材育成力」と説明した。

この経験から田中は重要な洞察を得た。トヨタの真の競争優位性は、個々の技術やツールではなく、それらを統合し継続的に進化させる「組織能力（ケイパビリティ）」にあるのだ。しかし、その能力は暗黙知として組織に埋め込まれており、明確に定義も測定もされていなかった。

彼のチームは6ヶ月をかけて、トヨタの「カイゼン能力」を体系的に分解し、4つの構成要素（人材・プロセス・技術・情報）として再定義した。その結果、初めて「何に投資すれば能力が向上するか」が明確になり、新しい製造ラインへの適用が成功したのである。

## なぜケイパビリティ定義が重要なのか

多くの組織は「技術力」「営業力」「イノベーション力」といった曖昧な表現で自社の強みを語る。しかし、このような抽象的な表現では、以下の重要な意思決定ができない：

**ビジネスインパクト（エグゼクティブ向け）**
- どの能力に投資すべきか判断できない
- 競争優位性の源泉が特定できない
- M&A時の統合計画が立てられない

**技術的課題（アーキテクト向け）**
- 能力を支援するシステムアーキテクチャが設計できない
- 必要な技術投資の優先順位が決められない
- 能力の成熟度を客観的に評価できない

**実装の困難さ（開発者向け）**
- 改善目標が測定可能な形で定義されていない
- どのスキルやツールを習得すべきか不明確
- システムと業務プロセスの整合性が取れない

## ケイパビリティモデル：組織能力の構造化

### 理論的背景

ケイパビリティ（組織能力）とは、「組織が特定の成果を一貫して生み出すために必要な、人材・プロセス・技術・情報の統合された集合体」である。これは単なる機能や部門とは異なり、価値創造に直接貢献する能力単位として定義される。

### 基本原則

ケイパビリティモデルは以下の3つの原則に基づいて構築される：

**1. 4つの構成要素による定義**

```
ケイパビリティ = 人材 × プロセス × 技術 × 情報
```

これら4要素は相互に依存し、どれか1つが欠けても能力は発揮されない。例えば、優秀な人材（人）がいても、適切な手順（プロセス）とツール（技術）、必要なデータ（情報）がなければ、組織能力として機能しない。

**2. 3層階層構造**

- **CL1（Level 1）**: 戦略的ケイパビリティ（5-10個）
- **CL2（Level 2）**: 戦術的ケイパビリティ（20-50個）
- **CL3（Level 3）**: 運用的ケイパビリティ（100-300個）

この階層構造により、経営層は戦略的視点で、現場は運用的視点で、同じ能力を異なる粒度で管理できる。

**3. 成熟度モデル（5段階）**

1. **初期段階**: 能力が属人的で標準化されていない
2. **定義段階**: プロセスが文書化され始めている
3. **管理段階**: 測定可能で繰り返し可能
4. **最適化段階**: 継続的改善が組み込まれている
5. **革新段階**: 業界をリードする差別化能力

### 他手法との違い

従来の組織設計手法（機能分解、RACI、組織図）は「誰が何をするか」に焦点を当てるが、ケイパビリティモデルは「組織が何を成し遂げられるか」に焦点を当てる。これにより、部門の壁を越えた価値創造の視点が可能になる。

## 実践例：デジタルバンキングのケイパビリティ設計

ある地方銀行がデジタル変革を推進する際、従来の部門別アプローチではなく、ケイパビリティベースのアプローチを採用した事例を見てみよう。

**段階1：現状の機能組織評価**

銀行は最初、既存の組織図に基づいて能力を評価しようとした。しかし、「デジタルチャネル部」「IT部」「営業推進部」といった部門名では、実際に顧客に提供できる価値が不明確だった。

**段階2：ケイパビリティへの再編成**

ZIGZAGプロセスから得られた洞察を基に、顧客価値の観点から必要な能力を再定義した：

```
CL1: デジタル顧客体験
├── CL2: オムニチャネル統合
│   ├── CL3: モバイルバンキング運用
│   ├── CL3: Webバンキング運用
│   └── CL3: ATM/支店連携
├── CL2: リアルタイム取引処理
│   ├── CL3: 即時決済処理
│   └── CL3: リアルタイム残高更新
└── CL2: パーソナライズサービス
    ├── CL3: 顧客行動分析
    └── CL3: レコメンデーション生成
```

**段階3：成熟度評価**

各ケイパビリティの現状を5段階で評価し、目標レベルとのギャップを可視化した。例えば、「モバイルバンキング運用」は現状レベル2（定義段階）だが、競合他社に追いつくためにはレベル4（最適化段階）が必要と判明した。

**段階4：投資優先順位決定**

ビジネスインパクトと現状ギャップの分析により、以下の優先順位で投資を決定：

1. リアルタイム取引処理（顧客満足度への直接影響大）
2. モバイルバンキング運用（利用者数の急増に対応）
3. パーソナライズサービス（差別化要因として）

この体系的アプローチにより、銀行は2年間で顧客満足度を15%向上させ、デジタルチャネル利用率を40%増加させることに成功した。

## いつ・どのように使うべきか

### 適用タイミング

ケイパビリティ定義は以下のタイミングで特に有効である：

- **組織再編や統合時**: M&A、事業部統合での能力評価と統合計画
- **デジタル変革時**: 必要な新能力の特定と既存能力の進化計画
- **投資計画策定時**: 限られたリソースの最適配分決定
- **アーキテクチャ設計時**: ビジネスケイパビリティとITケイパビリティの整合

### 成功の条件

1. **経営層の関与**: 戦略的ケイパビリティ（CL1）の定義には経営層の参画が不可欠
2. **段階的アプローチ**: 全社一斉ではなく、重要領域から段階的に展開
3. **測定可能性の確保**: 各ケイパビリティに具体的なKPIを設定
4. **継続的な見直し**: 市場環境の変化に応じて定期的に再評価

### よくある落とし穴

- **機能組織との混同**: ケイパビリティ≠部門。複数部門で1つの能力を実現することが多い
- **過度の細分化**: CL3で300個を超えると管理が困難になる
- **技術偏重**: 4要素（人材・プロセス・技術・情報）のバランスを保つ
- **静的な管理**: ケイパビリティは進化するもの。定期的な見直しが必要

## 他の手法との組み合わせ

ケイパビリティモデルは、V5.4の他の要素と以下のように統合される：

**TOGAFビジネスアーキテクチャとの関係**
- TOGAFのビジネスケイパビリティマップと概念的に整合
- V5.4はより実践的な4要素分解と成熟度評価を提供
- Phase 0で組織アセスメントと連携して使用

**バリューストリームマッピングとの統合**
- バリューストリームは「価値の流れ」、ケイパビリティは「価値を生む能力」
- 各バリューステージで必要なケイパビリティを特定
- ケイパビリティギャップがボトルネックの原因となることが多い

**DDD（Domain-Driven Design）との連携**
- ケイパビリティはビジネス視点、Bounded Contextは技術視点
- 1つのケイパビリティが複数のBounded Contextにまたがることがある
- ケイパビリティ境界がContext境界設計の重要なインプットとなる

次章では、定義されたケイパビリティに対して、どのような制約が存在し、それらをどう管理すべきかを探求する。ケイパビリティは理想状態を示すが、現実には様々な制約により、その実現が制限される。制約管理は、理想と現実のギャップを体系的に扱うための重要な手法である。
