# Parasol V5 - 概要

## 🌂 価値の日傘（パラソル）

Parasol（パラソル）は、価値駆動設計（Value-Driven Design）を中核に据えたビジネス設計フレームワークです。

### パラソルのメタファー

```
       【価値の傘】
          ｜
    ╱────┼────╲
   ╱     ｜     ╲
  ╱      ｜      ╲
 ╱       ｜       ╲
```

**守るもの**：
- 🌧️ 想像の設計という「雨」から守る
- ☔ 根拠なき判断の「嵐」から守る
- 🌊 価値劣化の「洪水」から守る

**通すもの**：
- ☀️ 構造的必然性という「光」を通す
- 🌟 観測可能な価値という「輝き」を通す
- ✨ 段階的実現という「希望」を通す

詳細は [PARASOL-CONCEPT.md](./PARASOL-CONCEPT.md) を参照。

## 🏯 Parasol V5の革新性

### パラソルの構造

**🎯 中心軸：構造的必然性**
- 観測可能なデータに基づく判断
- 不可逆的な構造変化への適応
- 10年以上の継続的トレンド

**🏗️ 骨組み：九次元価値分析**
1. Impact（影響力）
2. Velocity（速度）
3. Messaging（伝達力）
4. Reach（到達力）
5. Sentiment（感情価値）
6. Metrics（測定可能性）
7. Balance（均衡）
8. Depth（深度）
9. Foundation（基盤）

**🎪 布地：価値トレーサビリティ**
- すべての判断を価値まで追跡
- フェーズ間の価値継承を保証
- 価値劣化を技術的に防止

**🌈 展開機構：MS1-MS5マイルストーン**
```
MS1: 軸を立てる（コンテキスト確立）
    ↓
MS2: 骨組みを広げる（戦略設計）
    ↓
MS3: 布地を張る（戦術設計）
    ↓
MS4: 補強する（運用設計）
    ↓
MS5: 完全展開（プロダクション基盤）
```

### 世界初の特徴

1. **想像の設計を技術的に不可能にする**
   - すべての設計判断に構造的必然性を要求
   - 観測可能なデータに基づかない判断を自動ブロック

2. **完全な価値トレーサビリティ**
   - すべての設計判断を価値まで追跡可能
   - フェーズ間の価値継承を自動検証

3. **段階的価値実現の保証**
   - MS1-MS5マイルストーンで価値実現を保証
   - 各段階で品質ゲートを自動実行

## 📚 7つのフェーズ

### Phase 1: Context - コンテキスト確立
- **目的**: プロジェクトの文脈を理解
- **成果物**: ステークホルダー分析、制約条件、VL1-VL3価値分解
- **マイルストーン**: MS1 - コンテキスト確立（1週間）

### Phase 2: Value - 価値定義
- **目的**: ビジネス価値の明確化
- **成果物**: バリューストリーム設計、戦略ケイパビリティ
- **マイルストーン**: MS2 - 戦略設計完了（3週間）

### Phase 3: Capabilities - ケイパビリティ分解
- **目的**: ビジネス能力の詳細化
- **成果物**: ドメイン分類、サブドメイン設計、BC定義
- **マイルストーン**: MS3 - 戦術設計完了（4週間）

### Phase 4: Application Design - アプリケーション設計
- **目的**: 技術的な境界定義
- **成果物**: サービス境界、API設計、データモデル
- **マイルストーン**: MS4 - 運用設計完了（6週間）

### Phase 5: Software Design - ソフトウェア設計
- **目的**: 実装詳細の設計
- **成果物**: ドメイン言語、DB設計、ビジネスオペレーション
- **進捗管理**: MS4達成への貢献

### Phase 6: Implementation - 実装
- **目的**: コードの生成と実装
- **成果物**: 実装ガイド、コード、テスト、ドキュメント
- **完了**: MS4マイルストーン完了

### Phase 7: Platform - プラットフォーム
- **目的**: 運用基盤の構築
- **成果物**: インフラ設計、CI/CD、監視システム
- **マイルストーン**: MS5 - プロダクション基盤完了（8週間）

## 🎯 Parasolの哲学

### 五本の支柱思想

1. **価値駆動** - すべては価値から始まる
2. **観測可能性** - 測定できないものは管理できない
3. **段階的実現** - 一歩ずつ確実に
4. **構造的思考** - 表層ではなく構造を見る
5. **継続的検証** - 常に価値を確認する

### 保守性と変更容易性

1. **シンプルで理解しやすいアーキテクチャ**
   - APIファーストの同期通信を基本
   - 複雑な分散トランザクションを避ける
   - 必要な場合のみバッチ処理を使用

2. **運用時の変更容易性**
   - サービスの停止・入れ替えが容易
   - ローカルでのテストが可能
   - デバッグとトラブルシューティングが簡単

3. **イベント駆動の落とし穴を回避**
   - キューシステムは停止できない → 修正が困難
   - トランザクション制御が複雑 → バグの温床
   - ローカルテストが困難 → リリースリスク増大
