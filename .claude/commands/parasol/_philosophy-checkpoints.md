# Parasol 哲学チェックポイント

**目的**: 各フェーズで設計哲学に沿った作業が行われているかを検証

---

## 🎯 共通チェックポイント（全フェーズ）

### 1. 想像の設計回避チェック

**✅ 合格基準**:
- [ ] すべての主張に客観的根拠がある
- [ ] 測定可能な指標が定義されている
- [ ] 現実の観察データに基づいている
- [ ] 仮定と事実が明確に区別されている

**❌ 危険信号**:
- 「おそらく〜だろう」「〜と思われる」
- 競合他社の事例をそのまま仮定
- 測定不可能な抽象的表現
- 現状分析なしの改善提案

### 2. ビジネス価値トレーサビリティ

**✅ 合格基準**:
- [ ] すべての要素が明確な価値ストリームに紐づく
- [ ] 技術的決定の根拠にビジネス価値が示されている
- [ ] ROI/KPIが具体的に定義されている
- [ ] ステークホルダーが価値を理解・合意している

**❌ 危険信号**:
- 「技術的に優れているから」が唯一の理由
- 価値への貢献が説明できない要素
- 抽象的・測定困難な価値主張
- ビジネス側の理解・合意が取れていない

### 3. トップダウン整合性チェック

**✅ 合格基準**:
- [ ] 上位フェーズの定義と整合している
- [ ] 全体最適の視点が保たれている
- [ ] 局所最適に陥っていない
- [ ] 統合設計が最初から考慮されている

**❌ 危険信号**:
- 前フェーズの定義と矛盾する内容
- 部分的な視点での最適化
- 統合が後付けになっている
- 個別チーム・部門の都合優先

---

## 📋 フェーズ別チェックポイント

### Phase 1: Context チェック

**実証性の検証**:
- [ ] 財務データが実際の数値に基づく
- [ ] 組織情報が公開情報で検証可能
- [ ] 市場分析が客観的データに基づく
- [ ] ステークホルダー情報が具体的

### Phase 2: Value Definition チェック

**価値の現実性**:
- [ ] 現在の状態が正確に把握されている
- [ ] 目標状態が測定可能で達成可能
- [ ] 価値実現の道筋が具体的
- [ ] リスクと制約が現実的に評価されている

### Phase 3: Capabilities チェック

**DDD的ボトムアップ回避**:
- [ ] 価値ストリームから分解されている
- [ ] ドメインエキスパートの意見が出発点になっていない
- [ ] 各ケイパビリティの価値貢献が明確
- [ ] 統合設計が最初から考慮されている

### Phase 4-7: Implementation チェック

**ビジネス連携維持**:
- [ ] 技術選択が価値実現を支援する
- [ ] 実装がビジネスオペレーションに適合
- [ ] 変更容易性が確保されている
- [ ] 保守性が実際の運用を考慮している

---

## 🚨 緊急停止条件

以下の場合は作業を停止し、前フェーズに戻る：

1. **根拠不明の主張**
   - 客観的証拠なしの設計決定
   - 「エキスパートが言った」が唯一の根拠

2. **価値との断絶**
   - ビジネス価値への貢献が説明できない
   - 技術的興味だけが動機

3. **整合性の破綻**
   - 上位フェーズとの矛盾
   - 全体設計からの逸脱

4. **想像の設計**
   - 現実と乖離した理想論
   - 検証不可能な仮定

## 💡 改善アクション

### 想像の設計が検出された場合

1. **事実収集**：客観的データの追加調査
2. **仮定明確化**：何が仮定で何が事実かを区別
3. **検証計画**：仮定を検証する方法の策定
4. **段階的実証**：小さな実験での仮説検証

### 価値トレーサビリティが失われた場合

1. **価値マッピング**：各要素の価値貢献を再確認
2. **ステークホルダー確認**：ビジネス側との合意確認
3. **優先度見直し**：価値の高い要素への集中
4. **不要要素除去**：価値貢献しない要素の削除

### トップダウン整合性が破綻した場合

1. **上位確認**：上位フェーズの定義を再確認
2. **全体視点**：局所最適に陥っていないかチェック
3. **統合設計**：全体最適の視点で再設計
4. **段階的修正**：影響範囲を考慮した修正実施

---

このチェックポイントは、Parasol哲学の実践を確実にし、「実在するビジネス価値のために、実在するシステムを作る」ことを保証します。
