# プラットフォーム軸パターン (Platform Axis)

## 概要

**プラットフォーム軸**は、需要者（Demand Side）と供給者（Supply Side）の双方に価値を提供するマーケットプレイス型・エコシステム型ビジネス向けのパターンです。両面市場の特性を持ち、ネットワーク効果が競争優位の源泉となる事業に適用します。

## 適用条件

| 条件 | 説明 |
|------|------|
| **推奨** | 需要者と供給者をマッチングするプラットフォーム事業 |
| **推奨** | ネットワーク効果（片側または双方向）が事業価値の源泉 |
| **推奨** | 取引手数料/サブスクリプションが主要収益モデル |
| **注意** | B2B2C構造との違いを明確に（仲介者 vs プラットフォーム） |
| **注意** | 片側のみのVStrでは不十分（両面設計が必須） |

## VStr 構成

| VStr | 価値の受け手 | 略称 | 主要KPI |
|------|-------------|------|---------|
| **需要者VStr** | 購入者/利用者 | DS | GMV、取引数、リピート率 |
| **供給者VStr** | 出品者/提供者 | SS | 出品数、供給者数、収益性 |
| **プラットフォーム基盤** | 両者 | - | マッチング精度、決済成功率 |

## 各VStrのステージ定義

### 需要者向け VStr（DS: Demand Stage）

| Stage | 状態 | 定義 |
|-------|------|------|
| DS0 | ニーズ潜在 | 「まだ探していないが、この価値を必要としている」 |
| DS1 | 認知到達 | 「このプラットフォームの存在を知る」 |
| DS2 | 探索開始 | 「このプラットフォームで探し始める」 |
| DS3 | 選択決定 | 「このプラットフォームで取引相手を選ぶ」 |
| DS4 | 取引完了 | 「このプラットフォームで取引を完了する」 |
| DS5 | 利用開始 | 「購入した価値を享受している」 |
| DS6 | 継続利用 | 「このプラットフォームを継続的に利用している」 |
| DS7 | 推奨共有 | 「このプラットフォームを他者に推奨している」 |

### 供給者向け VStr（SS: Supply Stage）

| Stage | 状態 | 定義 |
|-------|------|------|
| SS0 | 参入検討 | 「まだ出品を考えていないが、潜在的に機会がある」 |
| SS1 | 認知到達 | 「このプラットフォームの存在を知る」 |
| SS2 | 登録完了 | 「このプラットフォームに供給者として登録する」 |
| SS3 | 出品開始 | 「このプラットフォームで出品を開始する」 |
| SS4 | 取引成立 | 「このプラットフォームで取引が成立する」 |
| SS5 | 収益実感 | 「このプラットフォームから収益を得ている」 |
| SS6 | 拡大成長 | 「このプラットフォームでビジネスを拡大している」 |
| SS7 | パートナー化 | 「このプラットフォームの戦略的パートナーとなる」 |

## ネットワーク効果の設計

### 片側ネットワーク効果

```
需要者↑ → 需要者価値↑（同じ側が増えると価値上昇）
例: SNS - ユーザーが増えるとコンテンツが増える
```

### 双方向ネットワーク効果

```
需要者↑ → 供給者価値↑ → 供給者↑ → 需要者価値↑
例: Uber - 乗客が増えるとドライバーが増え、待ち時間が減る
```

### 設計ポイント

1. **鶏と卵問題の解決**: どちら側を先に獲得するか戦略的に決定
2. **クリティカルマス**: ネットワーク効果が発動する最小規模を特定
3. **マルチホーミング防止**: 一方または両方の側のロックイン戦略

## 業界別適用例

### ECマーケットプレイス（例：メルカリ、Amazon Marketplace）

```
需要者VStr: 購入ニーズ→認知→検索→購入決定→取引→受取→リピート→推奨
供給者VStr: 副業検討→認知→出品者登録→出品→販売成立→収益→ストア化→パワーセラー
```

### シェアリングエコノミー（例：Airbnb、Uber）

```
需要者VStr: 移動/宿泊ニーズ→認知→予約→利用→体験→リピート→推奨
供給者VStr: 収入機会検討→認知→ホスト/ドライバー登録→稼働→収益→拡大→スーパーホスト
```

### 人材マッチング（例：Wantedly、LinkedIn）

```
需要者VStr（企業）: 採用ニーズ→認知→掲載→応募受付→採用→定着→継続利用
供給者VStr（求職者）: 転職検討→認知→登録→応募→内定→入社→キャリア形成
```

## 多段VStrとの組み合わせ

プラットフォーム軸 + 多段VStrの組み合わせ：

```
┌─────────────────────────────────────────────────────────────┐
│                需要者VStr (DS0-DS7)                          │
│                供給者VStr (SS0-SS7)                          │
├─────────────────────────────────────────────────────────────┤
│                従業員VStr (ES0-ES7) ← プラットフォーム運営   │
│                パートナーVStr (PS0-PS7) ← 決済/物流パートナー│
└─────────────────────────────────────────────────────────────┘
```

## 他パターンとの違い

| パターン | 構造 | 適用場面 |
|---------|------|---------|
| **プラットフォーム軸** | DS + SS（需給両面） | マーケットプレイス |
| **両面市場型** | 統合VStr（セグメント別定義） | 相互依存が強い両面 |
| **事業部軸** | 独立VStr × N | 独立性が高い事業群 |

### 両面市場型との使い分け

| 特性 | 両面市場型 | プラットフォーム軸 |
|------|-----------|------------------|
| VStr構造 | 1本のVStr（Layer分離） | 2本の独立VStr |
| 設計フォーカス | 統合体験 | 各側の最適化 |
| 適用例 | 人材派遣（密接連携） | ECマーケットプレイス |
