# 多段VStrパターン (Multi-Tier Value Stream Axis)

## 概要

**多段VStrパターン**は、顧客向けValue Streamを主軸としつつ、従業員やパートナーなど他のステークホルダー向けのValue Streamを階層的に構成するアプローチです。従来の「Supporting Capability」を独立したValue Streamとして設計することで、各ステークホルダーへの価値提供を明示的に設計できます。

## 適用条件

| 条件 | 説明 |
|------|------|
| **推奨** | 従業員体験（EX）が経営課題の場合 |
| **推奨** | パートナーエコシステムが重要な場合 |
| **推奨** | 組織成熟度が高く、顧客VStrが確立済みの場合 |
| **推奨** | ESG/ステークホルダー資本主義への対応が必要な場合 |
| **注意** | スタートアップ・SMBでは過剰設計になりやすい |
| **注意** | まず顧客VStrを確立してから適用すべき |

## VStr 階層の定義

| VStr | Domain分類 | 価値の受け手 | 価値の内容 | 略称 |
|------|-----------|-------------|-----------|------|
| **顧客向け** | Core | 顧客 | 製品/サービス体験 | VS |
| **従業員向け** | Supporting | 従業員 | キャリア成長・報酬・意義 | ES |
| **パートナー向け** | Supporting | パートナー | 協業利益・市場アクセス | PS |
| **Generic** | Generic | 全ステークホルダー | 共通基盤サービス | - |

## 各VStrのステージ定義

### 顧客向け VStr（VS: Value Stage）

| Stage | 状態 | 定義 |
|-------|------|------|
| VS0 | 潜在ニーズ | 「まだ気づいていないが、この価値を必要としている」 |
| VS1 | 認知到達 | 「この価値の存在を知る」 |
| VS2 | 選好形成 | 「この価値を選びたいと思う」 |
| VS3 | 意思決定 | 「この価値を獲得することを決める」 |
| VS4 | 価値獲得 | 「この価値を手に入れる」 |
| VS5 | 価値実感 | 「この価値を実感している」 |
| VS6 | 価値継続 | 「この価値を継続的に享受している」 |
| VS7 | 価値共創 | 「この価値を他者と共有・推奨している」 |

### 従業員向け VStr（ES: Employee Stage）

| Stage | 状態 | 定義 |
|-------|------|------|
| ES0 | 就職潜在 | 「まだ転職を考えていないが、潜在的に機会を求めている」 |
| ES1 | 認知到達 | 「この組織の存在を知る」 |
| ES2 | 選考意向 | 「この組織で働きたいと思う」 |
| ES3 | 入社決定 | 「この組織に入ることを決める」 |
| ES4 | 活躍開始 | 「この組織で価値を発揮し始める」 |
| ES5 | 成長実感 | 「この組織で成長を実感している」 |
| ES6 | 定着継続 | 「この組織で継続的にキャリアを築いている」 |
| ES7 | 組織共創 | 「この組織の価値を他者に推奨し、組織文化を共に創っている」 |

### パートナー向け VStr（PS: Partner Stage）

| Stage | 状態 | 定義 |
|-------|------|------|
| PS0 | 協業潜在 | 「まだ協業を考えていないが、潜在的に機会がある」 |
| PS1 | 認知到達 | 「この組織との協業可能性を知る」 |
| PS2 | 提携意向 | 「この組織と協業したいと思う」 |
| PS3 | 契約決定 | 「この組織との協業を決める」 |
| PS4 | 協業開始 | 「この組織と協業を開始する」 |
| PS5 | 価値実感 | 「この協業から価値を実感している」 |
| PS6 | 拡大継続 | 「この協業を継続・拡大している」 |
| PS7 | 戦略共創 | 「この組織と戦略的パートナーとして共に成長している」 |

## 強み

- **ステークホルダー視点の設計**: 従業員体験（EX）・パートナー体験（PX）を明示的に設計可能
- **Capability境界の明確化**: 「誰の状態を遷移させるか」で責任境界を判断できる
- **投資判断の根拠**: 各VStrのボトルネック特定と投資優先順位付けが可能
- **サービス・プロフィット・チェーン整合**: 従業員満足→顧客満足→収益の因果を設計に反映

## 弱み

- **設計スコープの肥大化**: 単一VStrの3〜5倍の設計量
- **VStr間依存の複雑化**: 変更影響の把握が困難になる可能性
- **境界判断の難しさ**: 複数VStrに関わるCapabilityの所属判断が必要
- **組織成熟度要求**: 顧客VStrを確立していない組織には過剰

## VStr追加の判断基準

Supporting を独立 VStr に昇格させる条件：

| 基準 | 質問 |
|------|------|
| **価値の受け手** | 明確なステークホルダーが存在するか？ |
| **状態遷移** | 7ステップ以上のジャーニーが設計可能か？ |
| **KPI** | 独自の測定・改善サイクルが回せるか？ |
| **投資判断** | 経営レベルで投資対象になっているか？ |
| **チーム** | 独立したStream Teamで運用できるか？ |

## 段階的導入パターン（推奨）

```
Phase 1: 顧客VStr + Generic のみ（3-6ヶ月）
├── 顧客価値フローを確立
├── Generic基盤を最小限で構築
└── 「Supporting問題」が顕在化するまで待つ

Phase 2: + 従業員VStr（2-3ヶ月）
├── 条件: 人材課題が経営イシューになった時
├── 採用→オンボーディング→活躍のフロー設計
└── 顧客VStr への人材供給を最適化

Phase 3: + パートナーVStr（2-3ヶ月）
├── 条件: エコシステム拡大が戦略になった時
├── パートナー開拓→協業→拡大のフロー設計
└── 顧客VStr への能力補完を最適化
```

## 設計のポイント

1. **顧客VStrを主軸に据える**: 従業員/パートナーVStrは顧客価値実現の「イネーブラー」
2. **境界判断は「誰の状態遷移か」で決定**: 営業研修→従業員VStr、商談オーケストレーション→顧客VStr
3. **Generic は全VStr共通**: 認証・監査以外のShared Kernel拡大は避ける
4. **VStr間は契約（API/Event）で疎結合**: 同期依存チェーンを避ける
5. **一つのBCが複数VStrに属することは禁止**: 責任曖昧化を防止

## 業界別適用例

### SaaS企業

```
顧客VStr: 見込み客→トライアル→契約→活用→拡大→推奨
従業員VStr: 採用→オンボーディング→CS/Dev活躍→成長→定着
パートナーVStr: ISV認知→技術検証→統合→共同販売→エコシステム共創
```

### 製造業

```
顧客VStr: 認知→検討→発注→納品→稼働→保守→更新
従業員VStr: 採用→技能習得→製造現場活躍→熟練→伝承
パートナーVStr: サプライヤー選定→契約→納入→品質改善→戦略調達
```

### 人材サービス

```
顧客VStr（企業）: 人材ニーズ→依頼→人材受入→活用→継続発注
顧客VStr（求職者）: 転職検討→登録→マッチング→就業→キャリア形成
従業員VStr: 自社社員の採用→育成→活躍
パートナーVStr: 協業企業との提携
```
