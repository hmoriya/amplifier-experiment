# 事業部軸パターン (Business Unit Axis)

## 概要

**事業部軸**は、既存の事業部構造を尊重し、各事業部ごとに独立したバリューストリームを定義するアプローチです。現実の組織構造との整合性を重視します。

## 適用条件

| 条件 | 説明 |
|------|------|
| **推奨** | 事業部の独立性が高いコングロマリット |
| **推奨** | 段階的なDX推進を計画している場合 |
| **推奨** | 各事業部に強いオーナーシップがある場合 |
| **注意** | 縦割りを強化するリスクがある |

## 基本構造

```
┌─────────────────────────────────────────────────┐
│                  全社戦略層                      │
│              (共通VS0のみ)                       │
└─────────────────────────────────────────────────┘
        │           │           │
        ▼           ▼           ▼
   ┌────────┐  ┌────────┐  ┌────────┐
   │事業部A │  │事業部B │  │事業部C │
   │VS1-VS7 │  │VS1-VS7 │  │VS1-VS7 │
   │独自定義│  │独自定義│  │独自定義│
   └────────┘  └────────┘  └────────┘
```

## 強み

- **現実的**: 既存組織との整合性が高い
- **段階的適用**: 事業部単位で順次展開可能
- **オーナーシップ**: 各事業部が主体的に推進
- **抵抗が少ない**: 組織変革を最小化

## 弱み

- **縦割り強化**: サイロ化を助長するリスク
- **重複投資**: 事業部間で類似機能を個別開発
- **シナジー未発見**: 横断的な価値創造機会を逃す
- **全社最適困難**: 部分最適の集合になりがち

## CL2 = BC = チーム境界

事業部軸パターンにおいても、CL2レベルがBC（Bounded Context）となります：

```
┌─────────────────────────────────────────────────────────────────────┐
│  CL2 = BC = チーム境界（Conway's Law準拠）                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  事業部A                                                             │
│  └─ VS4: 販売・流通                                                 │
│     └─ CL1: サプライチェーン                                        │
│        │                                                            │
│        └─ CL2: 物流オーケストレーションチーム                        │
│           │   = BC: logistics-orchestration-bc                      │
│           │                                                         │
│           ├─ CL3: 配送計画オペレーション                            │
│           │   └─ Actor UC → View                                   │
│           │                                                         │
│           └─ CL3: 在庫補充オペレーション                            │
│               └─ Actor UC → View                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Phase 4連携

| Phase | 全社層 | 事業部層 |
|-------|--------|----------|
| Phase 3 | 共通CL2定義 | 事業部別CL2定義 |
| Phase 4 | 共通BC→サービス | 事業部BC→サービス |
| Phase 5 | 共通API/DB | 事業部API/DB |

## 設計のポイント

1. **共通レイヤーの明確化**
   - VS0（戦略）とVS6（データ基盤）は全社共通とする

2. **インターフェース定義**
   - 事業部間の連携ポイントを明示的に設計

3. **段階的統合計画**
   - 将来的な統合を見据えた設計指針を持つ

## 業種別の適用例

### 多角化企業（例：サントリー事業部型）
- 酒類事業部：VS1-VS7を酒類特化で定義
- 飲料事業部：VS1-VS7を清涼飲料特化で定義
- 健康食品事業部：VS1-VS7を健康食品特化で定義

### 持株会社
- 各子会社が独立したVS体系を持つ
- ホールディングスはVS0のみ統括

## リスク軽減策

| リスク | 軽減策 |
|--------|--------|
| 縦割り強化 | 定期的な横断会議体の設置 |
| 重複投資 | 共通基盤サービスの定義 |
| シナジー未発見 | 年次での横断レビュー実施 |

## コングロマリット設計原則

### 重要：シナジー ≠ Value Stream

**シナジーはVStrではない**。VStrの本質は「顧客の状態遷移」であり、シナジーは以下の3概念に分解される：

```
┌─────────────────────────────────────────────────────────────────┐
│                  「シナジー」の正しい位置づけ                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ❌ 誤: 「シナジーVStr」をVS0-VS7で定義                          │
│     → 顧客状態遷移として成立しない                               │
│                                                                  │
│  ✅ 正: 3つの概念に分解して配置                                  │
│                                                                  │
│  1️⃣ BC間の関係性（Context Map）                                  │
│     ・技術横展開 → Partnership関係                               │
│     ・顧客紹介 → Customer-Supplier関係                          │
│                                                                  │
│  2️⃣ Supporting Layer のケイパビリティ                           │
│     ・R&D基盤（技術共有サービス）                                │
│     ・品質保証基盤（ベストプラクティスDB）                        │
│     ・物流基盤（共同配送）                                       │
│                                                                  │
│  3️⃣ Generic Layer の共通基盤                                    │
│     ・統合調達（共同購買）                                       │
│     ・顧客データ基盤（LTV連携）                                  │
│     ・ブランド資産（信頼共有）                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### VStrの判定基準

「○○をVStrとして定義すべきか？」の判断：

| 基準 | Yes → VStr | No → 別概念 |
|------|-----------|-------------|
| 価値の受け手は明確か？ | 顧客、従業員、パートナー | 「事業部」「組織」 |
| 状態遷移が設計可能か？ | VS0→VS7のジャーニー | 関係性、連携パターン |
| 時間文脈があるか？ | 顧客ジャーニーの時間軸 | 構造的関係 |
| VS-Swimlaneに配置可能か？ | 列に配置できる | 配置不能 |

### 共通価値の扱い

コングロマリットの「共通価値」（例：「食と健康」「水と生きる」）は：

```
❌ 誤: 共通価値をVStrとして定義
   「食と健康VStr」VS0→VS7 ← 成立しない

✅ 正: 設計原則として全VStrに貫通
   ┌─────────────────────────────────────────────────┐
   │  設計原則: 「食と健康」                          │
   │  - 全VStrのVS2（選好形成）で健康価値を訴求       │
   │  - 全VStrのVS5（価値実感）で健康効果を実感させる │
   │  - KGI: 健康関連NPS、健康寿命貢献度              │
   └─────────────────────────────────────────────────┘
            │           │           │
            ▼           ▼           ▼
   ┌────────────┐ ┌────────────┐ ┌────────────┐
   │ 事業部A    │ │ 事業部B    │ │ 事業部C    │
   │ VStr       │ │ VStr       │ │ VStr       │
   │ VS0-VS7    │ │ VS0-VS7    │ │ VS0-VS7    │
   └────────────┘ └────────────┘ └────────────┘
```

## 多段VStrとの関係

事業部軸パターンでは、各事業部内で多段VStrを適用することが可能：

```
┌─────────────────────────────────────────────────┐
│                  全社戦略層                      │
│              (共通VS0のみ)                       │
└─────────────────────────────────────────────────┘
        │           │           │
        ▼           ▼           ▼
   ┌────────┐  ┌────────┐  ┌────────┐
   │事業部A │  │事業部B │  │事業部C │
   │┌──────┐│  │┌──────┐│  │        │
   ││顧客  ││  ││顧客  ││  │ 標準   │
   ││VStr  ││  ││VStr  ││  │ モデル │
   │├──────┤│  │├──────┤│  │        │
   ││従業員││  ││従業員││  │        │
   ││VStr  ││  ││VStr  ││  │        │
   │└──────┘│  │└──────┘│  │        │
   └────────┘  └────────┘  └────────┘
    多段VStr     多段VStr    単一VStr
```

**適用パターン**:
- 事業部ごとに成熟度が異なる場合、多段VStrを段階的に導入
- EX/PXが特に重要な事業部のみ先行して多段化
