# フュージョン型パターン (Fusion Axis)

## 概要

**フュージョン型**は、価値軸と事業部軸のハイブリッドアプローチです。共通バリューストリームで骨格を作り、事業部固有の拡張を許容する柔軟な構造を持ちます。

## 適用条件

| 条件 | 説明 |
|------|------|
| **推奨** | 事業間に共通性と独自性の両方がある場合 |
| **推奨** | 段階的な統合を目指す場合 |
| **推奨** | 現実的なバランスを重視する場合 |
| **注意** | 設計の複雑さが増す |

## 基本構造

```
┌─────────────────────────────────────────────────────────┐
│                    共通バリューストリーム                 │
│    VS0: 戦略  VS6: データ基盤  VS7: オペレーション       │
└─────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │ 事業部A  │    │ 事業部B  │    │ 事業部C  │
    │ VS1-VS5  │    │ VS1-VS5  │    │ VS1-VS5  │
    │ 固有拡張 │    │ 固有拡張 │    │ 固有拡張 │
    └──────────┘    └──────────┘    └──────────┘
```

## 層構造

| 層 | バリューストリーム | 責任 |
|----|-------------------|------|
| **共通層** | VS0, VS6, VS7 | 全社戦略・データ・オペレーション |
| **事業層** | VS1-VS5 | 事業固有の価値創造 |
| **連携層** | インターフェース | 事業間の連携実現 |

## 重要：共通層とシナジーの区別

### 共通層 ≠ シナジーVStr

フュージョン型の「共通層」は、しばしば「シナジー創出」と混同されるが、明確に区別が必要：

```
┌─────────────────────────────────────────────────────────────────┐
│                  共通層の正しい理解                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ❌ 誤解: 共通層 = 「シナジーVStr」                               │
│     VS0-VS7で「シナジーの状態遷移」を定義 ← 成立しない           │
│                                                                  │
│  ✅ 正解: 共通層 = Supporting/Generic Layer                      │
│     ・VS0（戦略）→ 設計原則として全VStrに貫通                    │
│     ・VS6（データ基盤）→ Generic Layer の共通基盤               │
│     ・VS7（オペレーション）→ Supporting Layer の支援機能        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### VStrの判定基準

| 基準 | Yes → VStr | No → Layer/Capability |
|------|-----------|----------------------|
| 価値の受け手は明確か？ | 顧客、従業員、パートナー | 「事業部」「組織」 |
| 状態遷移が設計可能か？ | VS0→VS7のジャーニー | 関係性、基盤機能 |
| 時間文脈があるか？ | 顧客ジャーニーの時間軸 | 構造的関係 |

### 連携層の正しい位置づけ

「連携層」は以下の3概念に分解される：

| 概念 | 配置場所 | 例 |
|------|----------|-----|
| BC間関係 | Context Map（Phase 4） | Partnership、Customer-Supplier |
| 支援機能 | Supporting Layer | R&D基盤、品質保証基盤 |
| 共通基盤 | Generic Layer | 統合調達、顧客データ基盤 |

## 強み

- **バランス**: 統合と独立のバランスを取れる
- **現実的**: 段階的な適用が可能
- **柔軟性**: 事業特性に応じた拡張が可能
- **シナジー**: 共通層で横断価値を実現

## 弱み

- **設計複雑**: 共通と固有の境界設計が難しい
- **ガバナンス**: 二重統制構造になりやすい
- **合意形成**: 共通化範囲の決定に時間がかかる

## VS-Swimlaneモデルとの統合

フュージョン型でも、各事業層でVS-Swimlaneモデルに基づく設計が必須です：

```

┃                  VS-Swimlane with Fusion Pattern                            ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃  【共通層: Generic Layer】                                                   ┃
┃  ┌─────────────────────────────────────────────────────────────────────┐    ┃
┃  │  VS0: 戦略  │  VS6: データ基盤  │  VS7: オペレーション              │    ┃

```

## CL2 = BC = チーム境界

フュージョン型においても、CL2レベルがBC（Bounded Context）となります：

```
┌─────────────────────────────────────────────────────────────────────┐
│  CL2 = BC = チーム境界（Conway's Law準拠）                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【共通層】                                                          │
│  └─ CL2: データ基盤チーム = BC: data-platform-bc                    │
│                                                                     │
│  【事業層A】                                                         │
│  └─ VS3: ブランディング                                             │
│     └─ CL2: 酒類マーケティングチーム = BC: liquor-marketing-bc      │
│        ├─ CL3: キャンペーン企画能力 → BO: 企画オペレーション        │
│        └─ CL3: 効果測定能力 → BO: 測定オペレーション                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Phase 4連携

| Phase | 共通層 | 事業層 |
|-------|--------|--------|
| Phase 3 | 共通CL2定義 | 事業別CL2定義 |
| Phase 4 | 共通BC→サービス | 事業BC→サービス |
| Phase 5 | 共通API/DB | 事業API/DB |

## 設計のポイント

1. **共通化の判断基準を明確に**
   ```
   共通化する条件:
   - 80%以上の事業部で同じ機能が必要
   - データの一元化が価値を生む
   - 規制対応など全社統一が必要

   固有とする条件:
   - 事業特性による差異が大きい
   - 競争優位の源泉となる
   - 顧客接点に直接関わる
   ```

2. **インターフェースの標準化**
   - 事業間連携のAPIやデータ形式を統一

3. **ガバナンス体制**
   - 共通層の意思決定プロセスを明確化

## 業種別の適用例

### 飲料・食品（例：サントリー フュージョン型）
- **共通層**:
  - VS0: 「水と生きる」統合戦略
  - VS6: 統合データ基盤
  - VS7: 品質保証・サステナビリティ
- **事業層**:
  - 酒類: VS2で発酵技術、VS3で成熟市場マーケ
  - 飲料: VS2で健康機能、VS4で自販機網
  - 健康食品: VS2で機能性研究、VS3でエビデンス訴求

## 導入ステップ

```
Phase 1: 共通層の定義
    ↓
Phase 2: 1事業部でパイロット
    ↓
Phase 3: 他事業部へ展開
    ↓
Phase 4: 連携層の強化
```

## 多段VStrとの関係

フュージョン型の「共通層」は、多段VStrパターンの「Generic Layer」と類似した役割を持つ：

| フュージョン型 | 多段VStrパターン | 役割 |
|--------------|----------------|------|
| 共通層（VS0, VS6, VS7） | Generic Layer | 全体を支える基盤 |
| 事業層（VS1-VS5） | 顧客VStr（VS0-VS7） | Core Domain |
| - | 従業員VStr / パートナーVStr | Supporting Domain |

### 組み合わせパターン

フュージョン型 + 多段VStrの組み合わせ：

```
┌─────────────────────────────────────────────────────────┐
│              共通バリューストリーム（Generic）            │
│    VS0: 戦略  VS6: データ基盤  VS7: オペレーション       │
└─────────────────────────────────────────────────────────┘
          │               │               │
    ┌─────┴─────┐   ┌─────┴─────┐   ┌─────┴─────┐
    │  事業部A   │   │  事業部B   │   │  事業部C   │
    │ 顧客VStr  │   │ 顧客VStr  │   │ 顧客VStr  │
    │ 従業員VStr│   │ 従業員VStr│   │ 従業員VStr│
    └───────────┘   └───────────┘   └───────────┘
```

**適用条件**: 事業部ごとの独立性を維持しつつ、各事業部内で多段VStrを展開する場合
