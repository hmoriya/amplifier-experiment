# 軸パターン別 価値貢献度ガバナンス

**バージョン**: V5.14
**作成日**: 2025-12-11

---

## 概要

パラソルは「価値定義に対して企業活動が同じベクトルになるようにバックキャスティングする」方法論です。軸パターンによって組織構造が異なるため、**価値への貢献度をどのようにコントロール・推進するか**もパターンごとに設計が必要です。

本ドキュメントは、各軸パターンにおける価値貢献度ガバナンスの設計指針を提供します。

---

## 共通原則: 価値貢献度の3原則

全軸パターンに共通する価値貢献度ガバナンスの原則：

### 原則1: 測定原則 - 価値は「顧客状態遷移」で測定する

```
❌ 誤った測定: 活動量を測る
   「研修を100回実施した」← 価値との因果が不明

✅ 正しい測定: 状態遷移を測る
   「VS5→VS6の遷移率が85%→92%に改善」← 価値実現を直接測定
```

### 原則2: トレーサビリティ原則 - 全活動がVL1に接続

```
Operation（活動）
    ↑ 「この活動はどのCapabilityを発現させるか？」
Capability（能力）
    ↑ 「この能力はどのVSを実現するか？」
Value Stage（価値ステージ）
    ↑ 「このVSはどのVL3に貢献するか？」
VL3 → VL2 → VL1

検証不能な活動 = 価値に貢献していない可能性
```

### 原則3: 連結原則 - インセンティブを価値で連結

```
全ての評価指標は最終的にVL1への貢献と連結

  部門評価 = 自部門KPI + 他部門支援 + VL1貢献
            (40%)      (40%)      (20%)
```

## 1. 価値軸パターンの価値貢献度ガバナンス

### 構造的特徴

```
┌─────────────────────────────────────────────────────────┐
│                    単一VStr                              │
│    VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7        │
│                                                         │
│    全社で1本のバリューストリームを共有                   │
└─────────────────────────────────────────────────────────┘
```

**特徴**: 全社が単一の価値創造フローを共有するため、**価値への整合性が最も高い**パターン。

### ガバナンス設計

| 観点 | 設計 |
|------|------|
| **価値定義** | VL1が全社共通、VL2-VL3も統一 |
| **測定ポイント** | 各VSの遷移率を全社で共有 |
| **課題** | 事業部固有の貢献が見えにくい |
| **対策** | VS内での事業部別ブレイクダウン |

### 価値貢献度フレームワーク

```
┌─────────────────────────────────────────────────────────────┐
│  【Tier 1: VL1貢献指標】                                     │
│  全社統一の価値指標（NPS、LTV、価値認知度）                  │
└─────────────────────────────────────────────────────────────┘
                    ↑ 集約
┌─────────────────────────────────────────────────────────────┐
│  【Tier 2: VS遷移指標】                                      │
│  ┌───────┬───────┬───────┬───────┬───────┬───────┬───────┐  │
│  │VS0→1  │VS1→2  │VS2→3  │VS3→4  │VS4→5  │VS5→6  │VS6→7  │  │
│  │遷移率 │遷移率 │遷移率 │遷移率 │遷移率 │遷移率 │遷移率 │  │
│  └───────┴───────┴───────┴───────┴───────┴───────┴───────┘  │
└─────────────────────────────────────────────────────────────┘
                    ↑ 分解
┌─────────────────────────────────────────────────────────────┐
│  【Tier 3: Capability貢献指標】                              │
│  各Capabilityが担当するVSの遷移率への寄与度                  │
│  ・R&D → VS2遷移率への寄与                                   │
│  ・マーケティング → VS1, VS3遷移率への寄与                  │
│  ・品質保証 → VS5遷移率への寄与                             │
└─────────────────────────────────────────────────────────────┘
```

### インセンティブ設計

```
各部門の評価 = 担当VS遷移率(50%) + 隣接VS支援(30%) + VL1達成(20%)

例: R&D部門
・担当VS遷移率: VS2（選好形成）の遷移率改善
・隣接VS支援: VS1（認知）への技術情報提供、VS3（決定）への差別化要因
・VL1達成: 全社の価値指標への貢献
```

### 推進メカニズム

| メカニズム | 内容 | 頻度 |
|-----------|------|------|
| **VS遷移レビュー** | 全VSの遷移率と責任部門の確認 | 月次 |
| **価値統合会議** | VS間のボトルネック特定と対策 | 四半期 |
| **VL1レビュー** | 全社価値指標の達成状況と投資判断 | 半期 |

## 2. 事業部軸パターンの価値貢献度ガバナンス

### 構造的特徴

```
┌─────────────────────────────────────────────────────────┐
│                 設計原則: VL1「共通価値」                 │
│     全事業部に貫通するKGI/設計指針（VStrではない）        │
└─────────────────────────────────────────────────────────┘
        │           │           │
        ▼           ▼           ▼
   ┌────────┐  ┌────────┐  ┌────────┐
   │事業部A │  │事業部B │  │事業部C │
   │VStr    │  │VStr    │  │VStr    │
   │VS0-VS7 │  │VS0-VS7 │  │VS0-VS7 │
   └────────┘  └────────┘  └────────┘
```

**特徴**: 各事業部が独立したVStrを持つため、**事業部間の価値整合性**が課題。

### ガバナンス設計

| 観点 | 設計 |
|------|------|
| **価値定義** | VL1は全社共通、VL2-VL3は事業部別に分解 |
| **測定ポイント** | 事業部別VS遷移率 + 全社VL1貢献度 |
| **課題** | 事業部最適が全社最適と乖離するリスク |
| **対策** | VL1貢献度を事業部評価に組み込む |

### 価値貢献度フレームワーク

```
┌─────────────────────────────────────────────────────────────┐
│  【Tier 1: VL1貢献指標】全社レベル                           │
│  共通価値への総合貢献度                                      │
│  ・全社NPS（加重平均）                                       │
│  ・共通価値認知度（ブランド調査）                           │
│  ・クロスセル率（事業間送客）                               │
└─────────────────────────────────────────────────────────────┘
                    ↑ 集約
┌─────────────────────────────────────────────────────────────┐
│  【Tier 2: 事業部別VL1貢献指標】                             │
│  ┌─────────────┬─────────────┬─────────────┐               │
│  │ 事業部A     │ 事業部B     │ 事業部C     │               │
│  │ VL1貢献度   │ VL1貢献度   │ VL1貢献度   │               │
│  │ ・共通価値  │ ・共通価値  │ ・共通価値  │               │
│  │  表現度    │  表現度    │  表現度    │               │
│  │ ・VStr遷移率│ ・VStr遷移率│ ・VStr遷移率│               │
│  └─────────────┴─────────────┴─────────────┘               │
└─────────────────────────────────────────────────────────────┘
                    ↑ 分解
┌─────────────────────────────────────────────────────────────┐
│  【Tier 3: 事業部内VS遷移指標】                              │
│  各事業部のVS0→VS7遷移率                                    │
│  （事業部ごとに顧客ジャーニーが異なる）                     │
└─────────────────────────────────────────────────────────────┘
```

### インセンティブ設計

```
事業部評価 = 事業KPI(50%) + VL1貢献度(30%) + シナジー創出(20%)

VL1貢献度の測定:
・共通価値の表現度（製品・サービスにおける体現）
・共通ブランドへの貢献（認知・信頼向上）
・他事業部との相互送客

シナジー創出の測定:
・Supporting Layer活用度
・技術/知見の横展開件数
・共同施策の実施・成果
```

### 推進メカニズム

| メカニズム | 内容 | 頻度 |
|-----------|------|------|
| **事業部レビュー** | 事業部内VS遷移率とVL1貢献度 | 月次 |
| **横断シナジー会議** | 事業間連携機会の発掘・推進 | 四半期 |
| **VL1整合レビュー** | 全事業部のVL1貢献度比較と調整 | 半期 |

### 縦割りリスクへの対策

```
┌─────────────────────────────────────────────────────────────┐
│  リスク: 事業部最適 ≠ 全社最適                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  対策1: VL1貢献度の評価ウェイト                              │
│  ・事業部評価の30%をVL1貢献度に連結                         │
│  ・VL1に貢献しない事業部単独施策の投資を抑制                │
│                                                             │
│  対策2: シナジー創出のインセンティブ化                      │
│  ・他事業部への貢献を評価に組み込む                         │
│  ・技術/知見の横展開を表彰・昇進要件に                     │
│                                                             │
│  対策3: 共通価値表現の標準化                                │
│  ・全事業部のVS2/VS3で共通価値を訴求                       │
│  ・ブランドガイドラインの策定・遵守                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. フュージョン型パターンの価値貢献度ガバナンス

### 構造的特徴

```
┌─────────────────────────────────────────────────────────┐
│                    共通層                                │
│    VS0: 戦略  VS6: データ基盤  VS7: オペレーション       │
└─────────────────────────────────────────────────────────┘
          │               │               │
    ┌─────┴─────┐   ┌─────┴─────┐   ┌─────┴─────┐
    │  事業層A   │   │  事業層B   │   │  事業層C   │
    │ VS1-VS5   │   │ VS1-VS5   │   │ VS1-VS5   │
    └───────────┘   └───────────┘   └───────────┘
```

**特徴**: 共通層と事業層が分離するため、**共通層の価値貢献が曖昧になるリスク**が最も高い。

### ガバナンス設計

| 観点 | 設計 |
|------|------|
| **価値定義** | VL1は共通層、VL2-VL3は事業層で分解 |
| **測定ポイント** | 共通層は「イネーブラー」として測定 |
| **課題** | 共通層が「お題目」になりやすい |
| **対策** | 共通層の貢献を事業層の成果で評価 |

### 価値貢献度フレームワーク

```
┌─────────────────────────────────────────────────────────────┐
│  【Tier 1: VL1貢献指標】全社レベル                           │
│  共通価値への総合貢献度                                      │
└─────────────────────────────────────────────────────────────┘
                    ↑ 集約
┌─────────────────────────────────────────────────────────────┐
│  【Tier 2: 共通層貢献指標】横断レベル                        │
│  ┌─────────────────┬─────────────────┬─────────────────┐   │
│  │ VS0: 戦略貢献   │ VS6: データ貢献 │ VS7: 品質貢献   │   │
│  │ ・戦略浸透率    │ ・データ活用率  │ ・品質達成率    │   │
│  │ ・価値整合度    │ ・分析精度向上  │ ・クレーム削減  │   │
│  │ ・投資効率     │ ・コスト削減額  │ ・安全スコア    │   │
│  └─────────────────┴─────────────────┴─────────────────┘   │
│                                                             │
│  ※ 共通層の評価 = 事業層の「価値創出能力」向上度           │
└─────────────────────────────────────────────────────────────┘
                    ↑ 支援
┌─────────────────────────────────────────────────────────────┐
│  【Tier 3: 事業層貢献指標】事業部レベル                      │
│  ┌─────────────────┬─────────────────┬─────────────────┐   │
│  │ 事業層A        │ 事業層B        │ 事業層C        │   │
│  │ ・VS遷移率     │ ・VS遷移率     │ ・VS遷移率     │   │
│  │ ・顧客LTV      │ ・顧客LTV      │ ・顧客LTV      │   │
│  │ ・市場シェア   │ ・市場シェア   │ ・市場シェア   │   │
│  └─────────────────┴─────────────────┴─────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### インセンティブ設計（フュージョン型固有）

```
【共通層の評価】
評価 = 自部門KPI(40%) + 事業層支援成果(40%) + VL1貢献(20%)

例: データ基盤チーム
・自部門KPI: データ品質、システム稼働率
・事業層支援: 「事業部からの満足度」「活用による成果」
・VL1貢献: 全社NPS、リピート率への寄与

【事業層の評価】
評価 = 事業KPI(50%) + VL1貢献(30%) + 共通層活用(20%)

例: サプリメント事業部
・事業KPI: 売上、利益、シェア
・VL1貢献: 「健康価値」認知向上への寄与
・共通層活用: データ活用度、品質基準遵守
```

### 推進メカニズム

| メカニズム | 内容 | 頻度 |
|-----------|------|------|
| **価値統合会議** | 共通層の貢献度レビュー、事業層からのフィードバック | 月次 |
| **共通層サービスレビュー** | 共通層サービスの品質・活用状況 | 四半期 |
| **VL1整合投資レビュー** | 共通層/事業層の投資配分調整 | 半期 |

### 共通層形骸化リスクへの対策

```
┌─────────────────────────────────────────────────────────────┐
│  リスク: 共通層が「お題目」になる                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  対策1: 共通層を「イネーブラー」として評価                   │
│  ・共通層の成果 = 事業層の能力向上度                        │
│  ・「使われる」ことがインセンティブに                       │
│                                                             │
│  対策2: 価値ストーリーの可視化                              │
│  ・共通層活動→事業層活動→顧客状態遷移→VL1の因果を常時可視  │
│  ・トレース不能な共通層活動は投資対象外                     │
│                                                             │
│  対策3: 価値起点の意思決定プロセス                          │
│  ・全投資をVL1トレーサビリティで検証                        │
│  ・価値ROI = VS遷移改善によるVL1貢献度向上 / 投資額         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 4. ケイパビリティ統合型パターンの価値貢献度ガバナンス

### 構造的特徴

```
┌─────────────────────────────────────────────────────────┐
│                  ケーパビリティ層                        │
│  研究開発    マーケティング    営業    製造    物流    │
└─────────────────────────────────────────────────────────┘
          │           │           │           │
          ▼           ▼           ▼           ▼
    ┌──────────────────────────────────────────────┐
    │              事業ポートフォリオ               │
    │    酒類    飲料    健康食品    外食    ...   │
    └──────────────────────────────────────────────┘
```

**特徴**: 機能別組織のため、**事業成果への責任が分散**するリスク。

### ガバナンス設計

| 観点 | 設計 |
|------|------|
| **価値定義** | VL1は全社共通、ケイパビリティはVS別に貢献 |
| **測定ポイント** | ケイパビリティの担当VS遷移率 |
| **課題** | ケイパビリティが事業成果から切り離される |
| **対策** | ケイパビリティを事業成果に連結評価 |

### 価値貢献度フレームワーク

```
┌─────────────────────────────────────────────────────────────┐
│  【Tier 1: VL1貢献指標】全社レベル                           │
│  価値指標（NPS、LTV等）                                      │
└─────────────────────────────────────────────────────────────┘
                    ↑ 集約
┌─────────────────────────────────────────────────────────────┐
│  【Tier 2: 事業別価値指標】                                  │
│  各事業のVS遷移率・価値指標                                  │
└─────────────────────────────────────────────────────────────┘
                    ↑ 分解
┌─────────────────────────────────────────────────────────────┐
│  【Tier 3: ケイパビリティ別貢献指標】                        │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐       │
│  │ R&D     │ マーケ  │ 営業    │ 製造    │ 物流    │       │
│  │ →VS2   │ →VS1,3 │ →VS3,4 │ →VS4   │ →VS4,5 │       │
│  │ 遷移貢献│ 遷移貢献│ 遷移貢献│ 遷移貢献│ 遷移貢献│       │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### インセンティブ設計

```
ケイパビリティ評価 = 機能効率(40%) + 事業貢献(40%) + VL1貢献(20%)

例: R&D部門
・機能効率: 開発サイクル、特許出願数、研究効率
・事業貢献: 各事業のVS2遷移率への寄与度
・VL1貢献: 全社の製品差別化による価値向上

例: 物流部門
・機能効率: 配送コスト、リードタイム
・事業貢献: 各事業のVS4-VS5遷移率への寄与度
・VL1貢献: 顧客体験（配送品質）による価値向上
```

### 推進メカニズム

| メカニズム | 内容 | 頻度 |
|-----------|------|------|
| **Cap-事業連携会議** | ケイパビリティと事業部の連携状況確認 | 月次 |
| **VS貢献レビュー** | 各ケイパビリティの担当VS遷移率評価 | 四半期 |
| **機能効率レビュー** | ケイパビリティ単体の効率改善 | 四半期 |

### 責任分散リスクへの対策

```
┌─────────────────────────────────────────────────────────────┐
│  リスク: 事業成果への責任が曖昧化                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  対策1: ケイパビリティを担当VSの遷移率で評価                │
│  ・R&D → VS2遷移率への寄与で評価                           │
│  ・物流 → VS4,5遷移率への寄与で評価                        │
│                                                             │
│  対策2: 事業パートナー制度                                  │
│  ・各事業にケイパビリティから「ビジネスパートナー」を配置  │
│  ・事業目線での要件・優先順位調整                          │
│                                                             │
│  対策3: 事業成果連動報酬                                    │
│  ・ケイパビリティ評価の40%を事業成果に連動                 │
│  ・機能効率だけでは評価されない仕組み                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 5. プラットフォーム軸パターンの価値貢献度ガバナンス

### 構造的特徴

```
┌──────────────────────────────────────────────────────────────┐
│  【需要者向け VStr】(Demand Side)                             │
│  DS0 → DS1 → DS2 → DS3 → DS4 → DS5 → DS6 → DS7              │
├──────────────────────────────────────────────────────────────┤
│  【プラットフォーム基盤】                                     │
│  マッチング │ 決済 │ 信頼構築 │ サポート │ データ            │
├──────────────────────────────────────────────────────────────┤
│  【供給者向け VStr】(Supply Side)                             │
│  SS0 → SS1 → SS2 → SS3 → SS4 → SS5 → SS6 → SS7              │
└──────────────────────────────────────────────────────────────┘
```

**特徴**: 需要者と供給者の**両面の価値を同時に追求**する必要がある。

### ガバナンス設計

| 観点 | 設計 |
|------|------|
| **価値定義** | VL1は「マッチング価値」、VL2は需要者/供給者別 |
| **測定ポイント** | 両VStrの遷移率 + ネットワーク効果指標 |
| **課題** | 両面のバランスが崩れるリスク |
| **対策** | 両面の遷移率を統合的にモニタリング |

### 価値貢献度フレームワーク

```
┌─────────────────────────────────────────────────────────────┐
│  【Tier 1: プラットフォーム価値指標】                        │
│  ・GMV（取引総額）                                           │
│  ・マッチング成功率                                          │
│  ・ネットワーク効果係数（両面成長率）                        │
└─────────────────────────────────────────────────────────────┘
                    ↑ 集約
┌─────────────────────────────────────────────────────────────┐
│  【Tier 2: 両面VStr遷移指標】                                │
│  ┌─────────────────────┬─────────────────────┐              │
│  │ 需要者VStr          │ 供給者VStr          │              │
│  │ DS遷移率            │ SS遷移率            │              │
│  │ ・DS0→DS7総合      │ ・SS0→SS7総合      │              │
│  │ ・リピート率        │ ・継続率            │              │
│  │ ・NPS               │ ・NPS               │              │
│  └─────────────────────┴─────────────────────┘              │
└─────────────────────────────────────────────────────────────┘
                    ↑ 支援
┌─────────────────────────────────────────────────────────────┐
│  【Tier 3: 基盤機能貢献指標】                                │
│  ┌─────────┬─────────┬─────────┬─────────┐                 │
│  │マッチング│ 決済    │ 信頼    │サポート │                 │
│  │精度     │成功率   │構築     │解決率   │                 │
│  │→両VStr │→DS4,SS4│→DS3,SS3│→DS5,SS5│                 │
│  └─────────┴─────────┴─────────┴─────────┘                 │
└─────────────────────────────────────────────────────────────┘
```

### インセンティブ設計

```
【両面バランス原則】
プラットフォーム評価 = 需要者価値(35%) + 供給者価値(35%) + 基盤効率(30%)

例: マッチングチーム
・需要者価値: DS2→DS3遷移率への寄与
・供給者価値: SS3→SS4遷移率への寄与
・基盤効率: マッチング精度、レスポンス速度

※ 片側だけの最適化を防ぐため、両面を均等にウェイト
```

### 推進メカニズム

| メカニズム | 内容 | 頻度 |
|-----------|------|------|
| **両面バランスレビュー** | 需要者/供給者VStrの遷移率比較 | 週次 |
| **ネットワーク効果分析** | 片側成長が他側に与える影響 | 月次 |
| **基盤機能レビュー** | 基盤機能の両VStrへの貢献度 | 四半期 |

### 両面バランスリスクへの対策

```
┌─────────────────────────────────────────────────────────────┐
│  リスク: 鶏と卵問題、片側偏重                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  対策1: 両面同時モニタリング                                │
│  ・片側の遷移率が急上昇/急落した場合にアラート             │
│  ・両面の成長バランスをKGIに設定                            │
│                                                             │
│  対策2: クロスサイドインセンティブ                          │
│  ・需要者チームの評価に供給者満足度を含める（逆も同様）    │
│  ・「両面の健全性」を全員の共通目標に                       │
│                                                             │
│  対策3: 基盤投資の両面効果評価                              │
│  ・基盤投資は「両VStrへの貢献」で評価                       │
│  ・片側にしか効かない投資は優先度を下げる                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 6. 多段VStrパターンの価値貢献度ガバナンス

### 構造的特徴

```
┌──────────────────────────────────────────────────────────────┐
│  【顧客VStr】Core Domain                                     │
│  VS0 → VS1 → VS2 → VS3 → VS4 → VS5 → VS6 → VS7              │
├──────────────────────────────────────────────────────────────┤
│  【従業員VStr】Supporting Domain                             │
│  ES0 → ES1 → ES2 → ES3 → ES4 → ES5 → ES6 → ES7              │
├──────────────────────────────────────────────────────────────┤
│  【パートナーVStr】Supporting Domain                         │
│  PS0 → PS1 → PS2 → PS3 → PS4 → PS5 → PS6 → PS7              │
├──────────────────────────────────────────────────────────────┤
│  【Generic Layer】全VStr共通基盤                             │
└──────────────────────────────────────────────────────────────┘
```

**特徴**: 複数ステークホルダーへの価値を**同時に設計・測定**する。

### ガバナンス設計

| 観点 | 設計 |
|------|------|
| **価値定義** | VL1は顧客価値、VL2として従業員/パートナー価値を包含 |
| **測定ポイント** | 3つのVStrの遷移率とVStr間の因果関係 |
| **課題** | VStr間の優先順位とリソース配分 |
| **対策** | サービス・プロフィット・チェーンでの因果分析 |

### 価値貢献度フレームワーク

```
┌─────────────────────────────────────────────────────────────┐
│  【Tier 1: 統合価値指標】                                    │
│  ・顧客NPS + 従業員eNPS + パートナー満足度の統合             │
│  ・サービス・プロフィット・チェーンの実現度                  │
└─────────────────────────────────────────────────────────────┘
                    ↑ 集約（因果関係で連結）
┌─────────────────────────────────────────────────────────────┐
│  【Tier 2: VStr別価値指標】                                  │
│  ┌───────────────┬───────────────┬───────────────┐          │
│  │ 顧客VStr      │ 従業員VStr    │ パートナーVStr│          │
│  │ VS遷移率      │ ES遷移率      │ PS遷移率      │          │
│  │ ・顧客LTV     │ ・定着率      │ ・継続率      │          │
│  │ ・NPS         │ ・eNPS        │ ・NPS         │          │
│  │               │               │               │          │
│  │ Core Domain   │ Supporting    │ Supporting    │          │
│  └───────────────┴───────────────┴───────────────┘          │
│                                                             │
│  【因果関係】                                                │
│  従業員VStr向上 → 顧客VStr向上                              │
│  パートナーVStr向上 → 顧客VStr能力補完                      │
└─────────────────────────────────────────────────────────────┘
                    ↑ 支援
┌─────────────────────────────────────────────────────────────┐
│  【Tier 3: Generic Layer貢献指標】                           │
│  全VStrへの共通基盤提供度                                    │
└─────────────────────────────────────────────────────────────┘
```

### インセンティブ設計

```
【VStr間因果を反映した評価】

顧客VStr担当 = 顧客VS遷移(60%) + 従業員VStr活用(25%) + パートナーVStr活用(15%)
従業員VStr担当 = 従業員ES遷移(50%) + 顧客VStr支援効果(40%) + 組織貢献(10%)
パートナーVStr担当 = パートナーPS遷移(50%) + 顧客VStr補完効果(40%) + エコシステム(10%)

※ サービス・プロフィット・チェーンの因果:
   従業員満足 → 顧客満足 → 収益
   を評価に反映
```

### 推進メカニズム

| メカニズム | 内容 | 頻度 |
|-----------|------|------|
| **VStr別レビュー** | 各VStrの遷移率と課題 | 月次 |
| **VStr間因果分析** | 従業員→顧客、パートナー→顧客の因果検証 | 四半期 |
| **統合価値レビュー** | 3VStr統合での価値創出状況 | 半期 |

### VStr間優先順位の設計

```
┌─────────────────────────────────────────────────────────────┐
│  原則: 顧客VStrを主軸に、他VStrは「イネーブラー」            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  優先順位ロジック:                                          │
│  1. 顧客VStrのボトルネックを特定                            │
│  2. ボトルネック解消に必要な従業員/パートナー能力を特定    │
│  3. 従業員VStr/パートナーVStrの該当ESステージに投資        │
│                                                             │
│  例: VS4（価値獲得）の遷移率が低い場合                      │
│  → 原因分析: 人材不足? パートナー能力不足?                 │
│  → ES4（活躍開始）の改善、またはPS4（協業開始）の改善      │
│                                                             │
│  ※ 従業員/パートナーVStrへの投資も                          │
│    「顧客VStrへの貢献」で最終評価                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
