# ケーパビリティ統合型パターン (Capability Integration Axis)

## 概要

**ケーパビリティ統合型**は、事業横断で共通する「能力（ケーパビリティ）」を中心に設計するアプローチです。機能別組織やシェアードサービス化を志向する企業に適しています。

## 適用条件

| 条件 | 説明 |
|------|------|
| **推奨** | 機能別組織への移行を検討している場合 |
| **推奨** | 重複投資の解消が急務の場合 |
| **推奨** | シェアードサービス化を推進中の場合 |
| **注意** | 事業理解が薄れるリスクがある |

## 基本構造

```
┌─────────────────────────────────────────────────────────┐
│                  ケーパビリティ層                        │
├─────────────────────────────────────────────────────────┤
│  研究開発    マーケティング    営業    製造    物流    │
│  ケーパビリティ ケーパビリティ   ケーパビリティ ...     │
└─────────────────────────────────────────────────────────┘
          │           │           │           │
          ▼           ▼           ▼           ▼
    ┌──────────────────────────────────────────────┐
    │              事業ポートフォリオ               │
    │    酒類    飲料    健康食品    外食    ...   │
    │   (ケーパビリティを組み合わせて価値創造)      │
    └──────────────────────────────────────────────┘
```

## ケーパビリティ分類

| カテゴリ | ケーパビリティ例 | バリューストリーム対応 |
|----------|-----------------|----------------------|
| **戦略系** | 経営企画、M&A | VS0 |
| **創造系** | R&D、商品開発 | VS2 |
| **市場系** | マーケティング、営業 | VS1, VS3, VS4 |
| **運営系** | 製造、物流、品質保証 | VS4, VS5 |
| **支援系** | IT、人事、財務 | VS6, VS7 |

## 強み

- **効率化**: 重複機能の統合でコスト削減
- **専門性**: 機能別に深い専門性を蓄積
- **柔軟性**: 事業間でケーパビリティを再利用
- **スケール**: 全社規模での最適化が可能

## 弱み

- **事業理解低下**: 現場から離れた機能組織化
- **調整コスト**: 事業とケーパビリティ間の調整増加
- **責任曖昧化**: 事業成果への責任が分散
- **官僚化**: 大規模化に伴う硬直化リスク

## CL2 = BC = チーム境界

ケーパビリティ統合型においても、CL2レベルがBC（Bounded Context）となります：

```
┌─────────────────────────────────────────────────────────────────────┐
│  CL2 = BC = チーム境界（Conway's Law準拠）                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ケーパビリティ（例：R&D）                                          │
│  │                                                                  │
│  └─ CL2: 発酵研究チーム = BC: fermentation-research-bc              │
│     │   （1チーム = 1 BC = Nサービス）                              │
│     │                                                               │
│     ├─ CL3: 酵母探索オペレーション                                  │
│     │   └─ Actor UC → View                                         │
│     │                                                               │
│     └─ CL3: 発酵条件最適化オペレーション                            │
│         └─ Actor UC → View                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Phase 4連携

CL2定義後、Phase 4でBC→マイクロサービスマッピングを実施：

| Phase | 活動 | 成果物 |
|-------|------|--------|
| Phase 3 | CL2（チーム境界）定義 | ケーパビリティ別チーム構成 |
| Phase 4 | BC→サービスマッピング | Context Map、サービス境界 |
| Phase 5 | API/DB設計 | ドメイン言語、スキーマ |

## 設計のポイント

1. **ケーパビリティの粒度設計**
   ```
   粗すぎる: 「マーケティング」→ 事業特性を無視
   細かすぎる: 「SNSマーケ」→ 統合メリット薄
   適切: 「ブランドマネジメント」→ 専門性と汎用性のバランス
   ```

2. **事業との接続方法**
   - 事業側に「ビジネスパートナー」を配置
   - ケーパビリティのサービスレベルを定義

3. **評価指標の設計**
   - ケーパビリティ単体の効率指標
   - 事業貢献の成果指標
   - 両方を組み合わせた評価

## 業種別の適用例

### 飲料・食品（例：サントリー ケーパビリティ型）

**統合ケーパビリティ**:
| ケーパビリティ | 内容 | 対象事業 |
|---------------|------|---------|
| 研究開発センター | 発酵・醸造・健康科学 | 全事業 |
| ブランドマネジメント | ブランド戦略・広告 | 全事業 |
| サプライチェーン | 調達・製造・物流 | 全事業 |
| デジタル基盤 | データ・IT・DX | 全事業 |

**事業固有**:
| 事業 | 固有要素 |
|------|---------|
| 酒類 | 酒類規制対応、成熟市場戦略 |
| 飲料 | 自販機網運営、季節商品開発 |
| 健康食品 | 機能性表示対応、通販運営 |

## 判断基準マトリクス

ケーパビリティ統合の判断:

```
                    事業間の類似度
                    低い        高い
                 ┌─────────┬─────────┐
    規模・投資  │ 事業固有 │ 統合検討 │
       大きい   │  維持   │   ◎    │
                ├─────────┼─────────┤
       小さい   │ 事業固有 │ 統合    │
                │  維持   │  推奨   │
                └─────────┴─────────┘
```

## 多段VStrとの関係

「支援系ケーパビリティ」は、多段VStrパターンでは独立したValue Streamとして再設計可能：

| 支援系ケーパビリティ | 多段VStrでの位置づけ | 再設計のメリット |
|-------------------|-------------------|----------------|
| **人事** | 従業員VStr（ES0-ES7） | 採用→活躍→定着の状態遷移を設計可能 |
| **パートナーハブ** | パートナーVStr（PS0-PS7） | 協業開拓→価値実現→拡大の設計可能 |
| **IT・財務** | Generic Layer | 全VStr共通基盤として維持 |

### 組み合わせパターン

```
ケイパビリティ統合型 + 多段VStr:
┌─────────────────────────────────────────────────┐
│  ケイパビリティ層（機能別組織）                    │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐           │
│  │ 顧客VStr │ │従業員VStr│ │パートナー│           │
│  │ 支援Cap │ │ 支援Cap │ │VStr支援 │           │
│  └─────────┘ └─────────┘ └─────────┘           │
├─────────────────────────────────────────────────┤
│  事業ポートフォリオ（VStrを組み合わせて価値創造）  │
```

- 機能別組織化を進めつつ、EX/PXも経営課題の場合
