# Value Stage (VS) as WHEN - 時間軸による価値設計

**バージョン**: V5.4
**最終更新**: 2025-01-06

---

## 概要

Value Stage（VS）は、Parasol V5における **「WHEN（いつ）」軸** を担う中核概念です。これは「顧客がどの状態からどの状態へ遷移するか」という **時間的・状態的なコンテキスト** を定義します。

---

## Value Space の3軸モデル

Parasol V5では価値空間を3つの軸で捉えます：

```
┌─────────────────────────────────────────────────────────────┐
│                      VALUE SPACE                            │
│                                                             │
│                         ▲ WHY                               │
│                         │ (なぜ・目的)                       │
│                         │ Value Level                       │
│                         │                                   │
│                         │                                   │
│              ───────────┼───────────▶ HOW                   │
│                         │            (どのように・手段)      │
│                         │            Value Stream           │
│                        ╱                                    │
│                       ╱ WHEN                                │
│                      ╱  (いつ・状態遷移)                     │
│                     ▼   Value Stage                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 軸 | 名称 | 定義 | 例 |
|----|------|------|-----|
| **WHY** | Value Level (VL) | なぜ価値があるのか（目的・存在意義） | 顧客満足向上、コスト削減 |
| **HOW** | Value Stream (VStr) | どのように価値を提供するか（手段・プロセス） | 受注→製造→配送→サポート |
| **WHEN** | Value Stage (VS) | いつ価値が発生するか（状態遷移・タイミング） | VS0認知→VS3購入→VS7継続 |

---

## WHEN軸（Value Stage）とは

### 定義

Value Stage は **「顧客ライフサイクルにおける状態遷移」** を表します。顧客がビジネスとの関わりの中でどのような状態（ステージ）にいるかを定義し、各ステージで必要なシステム能力を明確化します。

### VS0-VS7 の8段階

```
┌───────────────────────────────────────────────────────────────────────────┐
│                        Customer Lifecycle                                 │
│                                                                           │
│  VS0        VS1        VS2        VS3        VS4        VS5        VS6   VS7
│   │          │          │          │          │          │          │     │
│   ▼          ▼          ▼          ▼          ▼          ▼          ▼     ▼
│ ┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐  ┌────┐
│ │認知│───▶│関心│───▶│検討│───▶│購入│───▶│利用│───▶│満足│───▶│推奨│──▶│継続│
│ └────┘    └────┘    └────┘    └────┘    └────┘    └────┘    └────┘  └────┘
│  Aware    Interest  Consider  Purchase   Use    Satisfied  Advocate Loyalty
│                                                                           │
└───────────────────────────────────────────────────────────────────────────┘
```

| Stage | 名称 | 顧客の状態 | 代表的なシステム能力 |
|-------|------|-----------|-------------------|
| **VS0** | 認知 (Awareness) | 製品・サービスを知る | マーケティング、広告配信 |
| **VS1** | 関心 (Interest) | 興味を持ち情報収集 | コンテンツ管理、リード獲得 |
| **VS2** | 検討 (Consideration) | 比較検討、評価 | 見積もり、デモ、試用 |
| **VS3** | 購入 (Purchase) | 購入決定、契約 | 受注、決済、契約管理 |
| **VS4** | 利用 (Usage) | 製品・サービスを使用 | 配送、セットアップ、利用支援 |
| **VS5** | 満足 (Satisfaction) | 価値を実感、満足 | サポート、問題解決、改善 |
| **VS6** | 推奨 (Advocacy) | 他者に推薦 | レビュー、紹介プログラム |
| **VS7** | 継続 (Loyalty) | 継続利用、リピート | ロイヤルティ、アップセル |

## なぜ WHEN軸 が重要なのか

### 1. システム設計の時間的コンテキスト

従来のシステム設計は **「機能（WHAT）」** と **「手段（HOW）」** に焦点を当てがちでした。しかし、同じ機能でも **「いつ使われるか」** によって設計が大きく変わります。

```
例：「顧客情報表示」機能

VS2（検討段階）での表示:
  → 簡潔な概要情報
  → 比較しやすい形式
  → 匿名でも閲覧可能

VS5（満足段階）での表示:
  → 詳細な利用履歴
  → パーソナライズされた推奨
  → 認証必須
```

### 2. ビジネス価値との直接的な紐付け

WHEN軸を明確にすることで、**各システム機能がどのビジネスステージに貢献するか** が明確になります：

```
┌──────────────────────────────────────────────────────────────┐
│                     Value Chain Mapping                      │
│                                                              │
│  Business Goal: 顧客獲得コスト削減                            │
│                                                              │
│  VS0 認知 ──┬── マーケティング自動化 ──── 効果測定            │
│             │                                                │
│  VS1 関心 ──┼── リード育成システム ────── スコアリング        │
│             │                                                │
│  VS2 検討 ──┴── セルフサービス見積もり ── 営業工数削減        │
│                                                              │
│  → システム投資のROIが明確に測定可能                          │
└──────────────────────────────────────────────────────────────┘
```

### 3. 優先順位付けの明確化

すべてのVSを同時に実装する必要はありません。ビジネス戦略に応じて優先順位を付けられます：

```
戦略A: 新規顧客獲得優先
  → VS0, VS1, VS2, VS3 を優先実装

戦略B: 既存顧客維持優先
  → VS4, VS5, VS6, VS7 を優先実装

戦略C: 収益最大化
  → VS3（購入）とVS7（継続）を優先実装
```

## VS-Swimlane Model（時間軸のコンテキスト管理）

### 概念

VS-Swimlane Model は、**Value Stage を横軸（時間軸）として、ケイパビリティを配置する** 設計手法です。これにより「いつ・誰が・何を」が一目で把握できます。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           VS-Swimlane Model                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ Actors      │ VS0    │ VS1    │ VS2    │ VS3    │ VS4    │ VS5    │ VS6-7  │
│             │ 認知   │ 関心   │ 検討   │ 購入   │ 利用   │ 満足   │ 継続   │
├─────────────┼────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│             │        │        │        │        │        │        │        │
│ Customer    │ 広告   │ 資料   │ 見積   │ 注文   │ 受取   │ 問合せ │ 再購入 │
│ (顧客)      │ 閲覧   │ 請求   │ 依頼   │ 確定   │ 使用   │        │        │
│             │        │        │        │        │        │        │        │
├─────────────┼────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│             │        │        │        │        │        │        │        │
│ Sales       │ 施策   │ リード │ 提案   │ 契約   │ 引継ぎ │        │ アップ │
│ (営業)      │ 実行   │ 対応   │ 作成   │ 締結   │        │        │ セル   │
│             │        │        │        │        │        │        │        │
├─────────────┼────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│             │        │        │        │        │        │        │        │
│ Support     │        │        │        │        │ 導入   │ サポ   │ 定期   │
│ (サポート)  │        │        │        │        │ 支援   │ ート   │ レビュー│
│             │        │        │        │        │        │        │        │
├─────────────┼────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│             │        │        │        │        │        │        │        │
│ Operations  │        │        │ 在庫   │ 出荷   │ 配送   │ 返品   │        │
│ (オペ)      │        │        │ 確認   │ 準備   │ 追跡   │ 対応   │        │
│             │        │        │        │        │        │        │        │
└─────────────┴────────┴────────┴────────┴────────┴────────┴────────┴────────┘
```

### 活用方法

1. **カバレッジ分析**: どのVSにシステム支援が不足しているか特定
2. **依存関係の可視化**: VS間のデータ・プロセスの流れを明確化
3. **優先順位の合意形成**: ステークホルダー間でのスコープ合意
4. **段階的実装計画**: フェーズごとの実装範囲を明確化

## WHEN軸 と ケイパビリティ階層の関係

### CL1-CL3 と VS の対応

ケイパビリティ階層（CL1-CL3）は **「何ができるか」** を定義し、VSは **「いつそれが必要か」** を定義します：

```
┌────────────────────────────────────────────────────────────────────┐
│                   VS × CL Matrix Example                          │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│           │ VS2(検討)    │ VS3(購入)    │ VS4(利用)    │          │
├───────────┼──────────────┼──────────────┼──────────────┤          │
│ CL1       │ 見積提示     │ 注文確定     │ 配送追跡     │ Activity │
│ (活動)    │              │              │              │ Area     │
├───────────┼──────────────┼──────────────┼──────────────┤          │
│ CL2       │ 価格計算     │ 在庫引当     │ 配送手配     │Capability│
│ (能力)    │ 割引適用     │ 決済処理     │ 状態更新     │          │
├───────────┼──────────────┼──────────────┼──────────────┤          │
│ CL3       │ 見積作成     │ 注文登録     │ 出荷登録     │ Business │
│ (操作)    │ 見積送付     │ 請求発行     │ 配送完了     │Operation │
└───────────┴──────────────┴──────────────┴──────────────┴──────────┘
```

### 設計への適用

1. **VSごとにCL1を特定**: 各ステージで必要な活動領域を洗い出す
2. **CL2で能力を定義**: 活動を実現するケイパビリティを設計
3. **CL3で操作を具体化**: 実際のシステム操作として実装

### VS別のシステム要件

```yaml

    - チェックアウトフロー
```

## ZIGZAG プロセスでの WHEN軸 活用

公理的設計のZIGZAGプロセスでは、FR（機能要件）とDP（設計パラメータ）を往復しながら設計を具体化します。WHEN軸（VS）はこのプロセスで重要な制約条件となります：

```
FR1: VS3で注文を確定できる
  ↓
  DP1: 注文確定サービス
    ↓
    FR1.1: 在庫を引き当てられる（VS3時点で）
    FR1.2: 決済を処理できる（VS3時点で）
      ↓
      DP1.1: 在庫引当モジュール
      DP1.2: 決済連携モジュール
```

VSを制約として明示することで、**各機能がどの時点で必要か** が設計に組み込まれます。

## まとめ

### WHEN軸の価値

1. **ビジネスとシステムの橋渡し**: 顧客ライフサイクルをシステム設計に反映
2. **優先順位の明確化**: 戦略に基づいた実装順序の決定
3. **コンテキスト依存の設計**: 同じ機能でもVSによる要件の違いを考慮
4. **ROIの可視化**: どのVSへの投資がどの成果に繋がるかを追跡

### 他の軸との統合

```
完全な価値設計 = WHY × HOW × WHEN

WHY (VL): 顧客満足向上のため
HOW (VStr): 受注→配送→サポートのプロセスで
WHEN (VS): VS3購入〜VS5満足の段階で
```

3軸すべてを考慮することで、**ビジネス価値に直結したシステム設計** が実現できます。
