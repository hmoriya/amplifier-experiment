# Parasol V5 - 概要

## ☀️ 価値を守る日傘

Parasol（パラソル）は、価値駆動設計（Value-Driven Design）を中核に据えたビジネス設計フレームワークです。

### 日傘のメタファー

```
         【日傘】
      ╱────┼────╲
     ╱     ｜     ╲
    ╱      ｜      ╲
   ╱       ｜       ╲
    ☀️ 価値の涼しい日陰 ☀️
```

**遮るもの**：
- 🔥 想像の設計という「灼熱の日差し」を遮る
- ☀️ 根拠なき判断の「紫外線」を遮る
- 🌡️ 価値劣化の「熱波」を遮る

**提供するもの**：
- 🏖️ 構造的必然性に基づく「涼しい日陰」を提供
- 🌴 観測可能な価値という「快適な空間」を提供
- 🎨 カラフルな可能性と「明るい未来」を提供

詳細は [PARASOL-CONCEPT.md](./PARASOL-CONCEPT.md) を参照。

## 🏯 Parasol V5の革新性

### パラソルの構造

**🎯 中心軸：構造的必然性**
- 観測可能なデータに基づく判断
- 不可逆的な構造変化への適応
- 10年以上の継続的トレンド

**🏗️ 骨組み：九次元価値分析**
1. Impact（影響力）
2. Velocity（速度）
3. Messaging（伝達力）
4. Reach（到達力）
5. Sentiment（感情価値）
6. Metrics（測定可能性）
7. Balance（均衡）
8. Depth（深度）
9. Foundation（基盤）

**🎪 布地：価値トレーサビリティ**
- すべての判断を価値まで追跡
- フェーズ間の価値継承を保証
- 価値劣化を技術的に防止

**🌈 展開機構：価値マイルストーン（VMS1-VMS5）**

**重要**: VMSは「プロセス完了」ではなく「**顧客が得ている価値状態**」を定義します。

```
VMS1: 顧客が最初の価値を体験できる状態（3ヶ月後）
    ↓
VMS2: 顧客が価値を認識し選択できる状態（6ヶ月後）
    ↓
VMS3: 顧客が主要価値を日常的に体験できる状態（9ヶ月後）
    ↓
VMS4: 顧客が全面的に価値を実感している状態（12ヶ月後）
    ↓
VMS5: 顧客が期待を超える価値を日常的に享受している状態（18ヶ月後）
```

**設計フロー**: VL分解 → VMS導出（バックキャスト）→ VS設計 → VL×VSマッピング → 優先順位決定

詳細は [VALUE-ANALYSIS-FLOW-DETAILED.md](./VALUE-ANALYSIS-FLOW-DETAILED.md) を参照。

### 世界初の特徴

1. **想像の設計を技術的に不可能にする**
   - すべての設計判断に構造的必然性を要求
   - 観測可能なデータに基づかない判断を自動ブロック

2. **完全な価値トレーサビリティ**
   - すべての設計判断を価値まで追跡可能
   - フェーズ間の価値継承を自動検証

3. **段階的価値実現の保証**
   - VMS1-VMS5価値マイルストーンで価値実現を保証
   - 各段階で品質ゲートを自動実行

## 🆕 V5.6: 5W1H価値駆動アーキテクチャ

### WHATの二面性

V5.6では、BIZBOKのビジネスアーキテクチャ概念を統合し、**WHATの二面性**を明示化しました。

```
┌─────────────────────────────────────────────────────────┐
│                   5W1Hフレームワーク                      │
├─────────────────────────────────────────────────────────┤
│  WHO    誰のため        → Stakeholder                   │
│  WHY    なぜ価値か      → Value Level (VL1-VL3)         │
│  WHAT   何を提供か      → Value Component (外向き)       │
│         何ができるか    → Capability (内向き)            │
│  WHEN   いつ価値が      → Value Stage / Operational Stage│
│  WHERE  どこで提供か    → Channel (V5.7予定)            │
│  HOW    どう実現するか  → BC / API / Implementation      │
└─────────────────────────────────────────────────────────┘
```

### Value Component（VC）フレームワーク

顧客に提供する価値を5つの構成要素で分析：

| ID | 要素 | 顧客の問い | サブ要素 |
|----|------|-----------|----------|
| VC1 | 製品価値 | 「何を買うのか」 | 機能/品質/多様性 |
| VC2 | サービス価値 | 「どう手に入るか」 | アクセス/価格/安定性 |
| VC3 | 体験価値 | 「どう感じるか」 | 購買/利用/所有 |
| VC4 | 関係価値 | 「なぜ選び続けるか」 | 信頼/帰属/継続 |
| VC5 | 情報価値 | 「何を知れるか」 | 知識/発見/共有 |

詳細は [value-component-guide.md](./value-component-guide.md) を参照。

### Operational Stage（OS）

バリューストリーム内の運用段階を正式導入：

```
Value Stage (VS) - 顧客状態
  └── Value Stream (VStr) - 価値の流れ
      └── Operational Stage (OS) - 運用段階（組織視点）
          └── Capability - 1:1対応
```

**設計原則**：
- 各OSで価値が蓄積される
- 1ケイパビリティは1OSに対応（ステージを跨がない）
- 判断ポイント・測定ポイントを明確化

### ステージ依存TVDC

同一ケイパビリティでもValue Stageによって競争優位への貢献度が変化：

```yaml
capability:
  tvdc_default: "VCI"
  stage_overrides:
    VS1_VS2:
      tvdc: "Core"
      rationale: "選好形成において競争優位を生む"
```

### 3軸価値分解

WHY（価値）からWHAT（何を）への分解に3つの軸を適用：

| 軸 | 分解内容 |
|----|---------|
| ステークホルダー軸 | 顧客/株主/従業員/パートナー/社会 |
| 時間軸 | 現在価値/将来価値/持続価値 |
| 性質軸 | 機能的/情緒的/社会的 |

詳細は [V56-SPECIFICATION.md](../V56-SPECIFICATION.md) を参照。

## 🆕 V5.7: 価値継承構造駆動アーキテクチャ

### 核心原則: 価値継承構造 = VStream構成

V5.7では、**価値定義の継承構造を基盤として、VStreamも構造的整合性を持って構成される**という原則を確立します。

```
【V5.7 核心コンセプト】

価値は継承される → VStreamも継承される

  HD価値定義        →   HD VStream
      ↓ 継承
  OpCo価値定義      →   OpCo VStream
      ↓ 継承
  Brand価値定義     →   Brand VStream
      ↓ 継承
  Product価値定義   →   Product VStream
```

**重要**: VStreamの本数は固定ではなく、**組織の価値継承構造から導出**されます。

### V5.7の分析フロー

```
【従来（V5.6以前）】
対象の分析開始 → VL定義 → VStream（顧客セグメント別）

【V5.7】
全体組織構造の把握 → コンテキスト選択 → 継承関係分析 → VL定義 → VStream（継承構造と同じ構成）
```

### 価値継承の分析

| ステップ | 内容 | 成果物 |
|---------|------|--------|
| Step 0 | 全体組織構造の把握 | 組織マップ |
| Step 1 | ターゲットコンテキスト選択 | コンテキスト定義 |
| Step 2 | 価値継承関係の分析 | 継承チェーン |
| Step 3 | 価値定義（VL1-VL3） | 継承元を明示したVL |
| Step 4 | VStream設計 | 継承構造と同じ構成 |

### 継承タイプ

| タイプ | 説明 | 例 |
|-------|------|-----|
| 特殊化 | 上位価値を事業領域に適応 | 「おいしさ」→「飲料でのおいしさ」|
| セグメント化 | 上位価値を顧客層に特化 | 「飲料健康」→「乳酸菌による健康」|
| 精緻化 | 上位価値を具体化 | 「生活文化」→「家族の絆」|
| 拡張 | 上位価値に新要素を追加 | 「品質」→「品質＋利便性」|

### V5.7の価値

1. **「なぜこの価値か」が説明可能**
   - 継承チェーンにより価値の根拠を構造的に説明

2. **戦略的整合性の担保**
   - 下位の戦略が上位戦略と矛盾しないことを構造的に確認

3. **上位価値コンポーネントの活用**
   - 継承元の価値を再利用し、一貫性を維持

### エンタープライズ価値統制

各レベルで**同じアーキテクチャ（価値定義 + VStream）**を採用することで、コングロマリット/ホールディングの価値統制が可能に。

**VStream統合 → Capability統合の順序**:
```
価値定義の統合 → VStream統合 → Capability統合 → システム統合
     ↑              ↑              ↑
  「なぜ統合？」  「誰に何を？」  「何が必要？」
```

VStreamが先に統合されることで、Capability統合の目的・スコープ・優先順位が明確になり、統合が容易になります。

詳細は [V57-DESIGN-PRINCIPLES.md](../V57-DESIGN-PRINCIPLES.md) を参照。

## 📚 7つのフェーズ

**重要**: 各フェーズのマイルストーンは「プロセス完了」ではなく「顧客価値状態の達成」を表します。

### Phase 1: Context - コンテキスト確立
- **目的**: プロジェクトの文脈を理解
- **成果物**: ステークホルダー分析、制約条件、VL1-VL3価値分解
- **貢献マイルストーン**: VMS1への準備（価値体験の設計基盤）

### Phase 2: Value - 価値定義
- **目的**: ビジネス価値の明確化
- **成果物**: バリューストリーム設計、戦略ケイパビリティ
- **V5.6追加**: VC分析、3軸価値分解、VS-VC対応、Value Portfolio
- **達成マイルストーン**: VMS1 - 顧客が最初の価値を体験できる状態（3ヶ月後）

### Phase 3: Capabilities - ケイパビリティ分解
- **目的**: ビジネス能力の詳細化
- **成果物**: CL1活動領域識別（傾向的分類）、CL2ケイパビリティ設計（正式分類）、CL3詳細分解（≈BizOp）
- **V5.6追加**: OS定義、ステージ依存TVDC設定、VC-Cap対応マッピング
- **達成マイルストーン**: VMS2 - 顧客が価値を認識し選択できる状態（6ヶ月後）

### Phase 4: Application Design - アプリケーション設計
- **目的**: 技術的な境界定義
- **成果物**: サービス境界、API設計、データモデル
- **貢献マイルストーン**: VMS3への準備（主要価値の設計基盤）

### Phase 5: Software Design - ソフトウェア設計
- **目的**: 実装詳細の設計
- **成果物**: ドメイン言語、DB設計、ビジネスオペレーション
- **達成マイルストーン**: VMS3 - 顧客が主要価値を日常的に体験できる状態（9ヶ月後）

### Phase 6: Implementation - 実装
- **目的**: コードの生成と実装
- **成果物**: 実装ガイド、コード、テスト、ドキュメント
- **達成マイルストーン**: VMS4 - 顧客が全面的に価値を実感している状態（12ヶ月後）

### Phase 7: Platform - プラットフォーム
- **目的**: 運用基盤の構築
- **成果物**: インフラ設計、CI/CD、監視システム
- **達成マイルストーン**: VMS5 - 顧客が期待を超える価値を享受している状態（18ヶ月後）

## 🎯 Parasolの哲学

### 五本の支柱思想

1. **価値駆動** - すべては価値から始まる
2. **観測可能性** - 測定できないものは管理できない
3. **段階的実現** - 一歩ずつ確実に
4. **構造的思考** - 表層ではなく構造を見る
5. **継続的検証** - 常に価値を確認する

### 保守性と変更容易性

1. **シンプルで理解しやすいアーキテクチャ**
   - APIファーストの同期通信を基本
   - 複雑な分散トランザクションを避ける
   - 必要な場合のみバッチ処理を使用

2. **運用時の変更容易性**
   - サービスの停止・入れ替えが容易
   - ローカルでのテストが可能
   - デバッグとトラブルシューティングが簡単

3. **イベント駆動の落とし穴を回避**
   - キューシステムは停止できない → 修正が困難
   - トランザクション制御が複雑 → バグの温床
   - ローカルテストが困難 → リリースリスク増大

詳細は [PHILOSOPHY.md](./PHILOSOPHY.md) を参照。
