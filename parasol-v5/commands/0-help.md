# /parasol:0-help - Parasol V5 ヘルプガイド

## コマンド: `/parasol:0-help`

Parasol V5フレームワークの使い方とコマンド一覧を表示します。

## Parasol V5とは

Amplifier DDDフレームワークの構造を採用した、軽量版のParasol開発フレームワークです。
企業の価値創造プロセスを体系的に設計・実装するための段階的アプローチを提供します。

## 利用可能なコマンド

### 基本コマンド
```bash
/parasol:0-help        # このヘルプを表示
/parasol:status        # プロジェクトの現在ステータスを確認
```

### 実行コマンド（推奨順序）
```bash
/parasol:1-context     # プロジェクトコンテキストの確立
/parasol:2-value       # 価値定義と企業活動設計
/parasol:3-business    # バリューステージ設計
/parasol:4-architecture # アプリケーションアーキテクチャ設計
/parasol:5-software    # ソフトウェア設計
/parasol:6-implementation # ソフトウェア実装
/parasol:7-platform    # プラットフォーム設計
```

## 各フェーズの概要

### Phase 1: プロジェクトコンテキスト (`/parasol:1-context`)
- **目的**: 組織分析と市場評価
- **成果物**: 組織分析、市場評価、制約条件
- **実行時間**: 約1-2時間

### Phase 2: 価値定義 (`/parasol:2-value`)
- **目的**: 価値宣言と企業活動設計
- **成果物**: 価値宣言、価値分解、価値マイルストーン
- **実行時間**: 約2-3時間

### Phase 3: ビジネス設計 (`/parasol:3-business`)
- **目的**: バリューステージとケーパビリティ設計
- **成果物**: VS定義、ケーパビリティ階層、価値フロー
- **実行時間**: 約2-3時間

### Phase 4: アーキテクチャ (`/parasol:4-architecture`)
- **目的**: 技術アーキテクチャの設計
- **成果物**: 境界コンテキスト、マイクロサービス、API仕様
- **実行時間**: 約2-3時間

### Phase 5: ソフトウェア設計 (`/parasol:5-software`)
- **目的**: 詳細設計とドメインモデル
- **成果物**: ドメインモデル、ユースケース、DB設計
- **実行時間**: 約2-3時間

### Phase 6: 実装 (`/parasol:6-implementation`)
- **目的**: コード生成と実装
- **成果物**: 実装コード、テスト、ドキュメント
- **実行時間**: 約3-4時間

### Phase 7: プラットフォーム (`/parasol:7-platform`)
- **目的**: デプロイメントとインフラ設計
- **成果物**: K8s設定、CI/CD、監視設定
- **実行時間**: 約2時間

## クイックスタート

### 新規プロジェクトの開始
```bash
# 1. プロジェクトコンテキストから開始
/parasol:1-context

# 2. ステータスを確認
/parasol:status

# 3. 次のフェーズへ進む
/parasol:2-value
```

### 特定フェーズから再開
```bash
# フェーズ3から再開する場合
/parasol:3-business

# 前フェーズの成果物を確認
ls outputs/2-value/
```

## 成果物の場所

すべての成果物は`outputs/`ディレクトリに保存されます：

```
outputs/
├── 1-context/        # コンテキスト分析結果
├── 2-value/          # 価値定義ドキュメント
├── 3-business/       # ビジネス設計書
├── 4-architecture/   # アーキテクチャ設計書
├── 5-software/       # ソフトウェア設計書
├── 6-implementation/ # 実装コード
└── 7-platform/       # プラットフォーム設定
```

## ベストプラクティス

1. **順序通りの実行**: 各フェーズは前フェーズの成果物に依存
2. **レビューポイント**: 各フェーズ完了時に成果物をレビュー
3. **決定記録**: 重要な決定は決定記録として文書化
4. **インクリメンタル**: 必要に応じて前フェーズに戻って修正

## トラブルシューティング

### Q: 特定のフェーズをスキップできますか？
A: 可能ですが、前フェーズの成果物が必要な場合があります。

### Q: 成果物を修正したい場合は？
A: 該当するフェーズのコマンドを再実行するか、直接`outputs/`内のファイルを編集できます。

### Q: V4完全版との違いは？
A: Lite版は概念実証用の軽量版です。必要に応じて各コマンドを詳細化できます。
