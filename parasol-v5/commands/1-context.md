# /parasol:1-context - プロジェクトコンテキスト確立

## コマンド: `/parasol:1-context`

プロジェクトの基礎となるコンテキストを確立します。組織分析、市場評価、制約条件を明確化します。

## 実行時間
約1-2時間

## 前提条件
- プロジェクトの基本情報
- 組織の概要情報
- 市場/業界の基本知識

## 実行内容

### Task 1: 組織分析

#### 組織の現状評価
```yaml
組織プロファイル:
  名称: [組織名]
  業界: [業界分類]
  規模: [従業員数、売上規模]
  事業領域: [主要事業の説明]

組織構造:
  階層: [組織階層の数]
  部門: [主要部門のリスト]
  意思決定: [意思決定プロセス]

技術成熟度:
  現在のIT活用レベル: [1-5の5段階評価]
  デジタル化の進捗: [%表示]
  主要システム: [使用中の主要システム]

組織文化:
  変革への姿勢: [保守的/中立/積極的]
  リスク許容度: [低/中/高]
  学習志向: [1-5の5段階評価]
```

### Task 2: 市場評価

#### 市場環境分析
```yaml
市場規模:
  総市場規模: [金額]
  成長率: [年率%]
  主要トレンド: [3-5個のトレンド]

競合状況:
  主要競合: [上位3-5社]
  市場シェア: [自社のポジション]
  差別化要因: [独自の強み]

顧客セグメント:
  主要顧客層: [セグメント定義]
  顧客ニーズ: [主要なニーズ3-5個]
  顧客期待: [期待値の変化]

技術動向:
  業界標準技術: [主流技術]
  新興技術: [注目すべき新技術]
  技術的脅威: [ディスラプション要因]
```

### Task 3: 制約条件の特定

#### プロジェクト制約
```yaml
リソース制約:
  予算: [利用可能予算]
  人員: [投入可能人員]
  期間: [プロジェクト期限]

技術制約:
  既存システム: [統合が必要なシステム]
  技術スタック: [使用可能な技術]
  セキュリティ: [セキュリティ要件]

組織制約:
  承認プロセス: [必要な承認レベル]
  コンプライアンス: [準拠すべき規制]
  変更管理: [変更の許容範囲]

外部制約:
  規制要件: [業界規制]
  市場要因: [市場からの制約]
  パートナー: [外部依存関係]
```

### Task 4: ステークホルダー分析

#### ステークホルダーマップ
```yaml
内部ステークホルダー:
  経営層:
    期待: [経営層の期待]
    影響力: [高/中/低]
    関与度: [高/中/低]

  事業部門:
    期待: [事業部門の期待]
    影響力: [高/中/低]
    関与度: [高/中/低]

  IT部門:
    期待: [IT部門の期待]
    影響力: [高/中/低]
    関与度: [高/中/低]

外部ステークホルダー:
  顧客:
    期待: [顧客の期待]
    影響力: [高/中/低]

  パートナー:
    期待: [パートナーの期待]
    影響力: [高/中/低]
```

## 成果物

以下のファイルが`outputs/1-context/`に生成されます：

1. **organization-analysis.md**
   - 組織の現状分析
   - 強みと弱みの評価
   - 改善機会の特定

2. **market-assessment.md**
   - 市場環境の評価
   - 競合分析
   - 市場機会の特定

3. **constraints.md**
   - 制約条件の一覧
   - リスク評価
   - 対応策の提案

4. **stakeholder-map.md**
   - ステークホルダー分析
   - 影響力/関心度マトリクス
   - エンゲージメント戦略

## チェックリスト

- [ ] 組織の現状が明確に定義されているか
- [ ] 市場環境が適切に評価されているか
- [ ] すべての制約条件が識別されているか
- [ ] ステークホルダーが網羅的に分析されているか
- [ ] 次フェーズに必要な情報が揃っているか

## 次のステップ

コンテキスト分析が完了したら、価値定義フェーズへ：

```bash
/parasol:2-value
```

---

*このフェーズで確立したコンテキストが、後続のすべてのフェーズの基礎となります。*
