## 2. 軸パターン適合性評価

### 2.1 価値軸パターン

```
適合度: ★☆☆☆☆（1/5）
```

| 評価観点 | 判定 | 根拠 |
|---------|------|------|
| 単一価値の一貫性 | × | 5事業で顧客への価値提案が異なる |
| 顧客セグメントの統一性 | × | 6セグメント以上に分散 |
| ブランド体系 | × | 独立ブランド群 |

**結論**: 不適合。顧客セグメント分散が大きすぎる。

---

### 2.2 事業部軸パターン

```
適合度: ★★★☆☆（3/5）
```

| 評価観点 | 判定 | 根拠 |
|---------|------|------|
| 事業部独立性 | ○ | 各事業で異なる競合・チャネル |
| ブランド独立性 | ○ | ミンティア、アマノ、和光堂等が独立認知 |
| シナジー活用 | △ | 技術シナジーを見落とすリスク |
| 設計シンプルさ | ○ | 事業部単位で明確に分離可能 |

**結論**: 適合可能だが、技術シナジーの設計機会を逃す。

---

### 2.3 ケイパビリティ統合型パターン

```
適合度: ★★☆☆☆（2/5）
```

| 評価観点 | 判定 | 根拠 |
|---------|------|------|
| 共通ケイパビリティ | △ | 発酵・FD技術は一部事業のみ |
| 顧客接点の統合 | × | 各事業で異なるチャネル・営業体制 |
| R&D統合効果 | ○ | 研究開発の共通化は可能 |

**結論**: R&D以外の統合メリットが限定的。

---

### 2.4 フュージョン型パターン

```
適合度: ★★★★☆（4/5）
```

| 評価観点 | 判定 | 根拠 |
|---------|------|------|
| 共通層の定義可能性 | ○ | 「おいしさと健康」を上位価値として設定可能 |
| 事業固有性の尊重 | ○ | 各事業のVS1-VS5は独自設計 |
| 技術シナジー | ○ | 発酵・FD技術を共通層で設計 |
| 段階的適用 | ○ | 1事業からパイロット可能 |
| 親会社との整合 | ○ | アサヒグループのGeneric Layerと連携 |

**結論**: 最も適合。技術シナジーと事業独立性のバランスが取れる。

---

### 2.5 プラットフォーム軸パターン

```
適合度: ☆☆☆☆☆（0/5）
```

| 評価観点 | 判定 | 根拠 |
|---------|------|------|
| 両面市場構造 | × | マーケットプレイス型事業なし |
| ネットワーク効果 | × | 該当なし |

**結論**: 不適合。事業モデルが該当しない。

### 推奨理由

| 観点 | フュージョン型の優位性 |
|------|----------------------|
| **技術シナジー** | 発酵・FD技術を共通層で設計、投資効率化 |
| **事業独立性** | 各事業グループのVS1-VS5は独自設計可能 |
| **段階的適用** | 1グループからパイロット、リスク低減 |
| **親会社連携** | アサヒグループのGeneric Layerと自然に接続 |
| **スケーラビリティ** | 新事業追加時も共通層+事業層で対応 |

## 4. 代替案との比較

| 観点 | フュージョン型 | 事業部軸 |
|------|--------------|---------|
| 技術シナジー活用 | ○ 共通層で設計 | △ 見落としリスク |
| 設計複雑度 | △ やや複雑 | ○ シンプル |
| 初期導入コスト | △ やや高 | ○ 低 |
| 長期運用効率 | ○ 高 | △ 重複発生 |
| 組織変更インパクト | △ 共通層の体制必要 | ○ 現状維持可能 |

**判断**: 長期的な技術シナジー活用を重視し、フュージョン型を推奨。
ただし、組織の準備状況によっては事業部軸からスタートし、段階的にフュージョン型へ移行する選択肢もあり。

## 5. 次ステップ

### Phase 2 継続タスク

1. **共通層の詳細定義**
   - VS0「おいしさと健康」の具体化
   - VS6 データ基盤の範囲定義
   - VS7 品質保証・サステナビリティの設計

2. **事業グループのVStr設計**
   - 嗜好品グループ（ミンティア、アマノ）のVS1-VS5
   - 健康・ライフグループ（ディアナチュラ、和光堂）のVS1-VS5
   - BtoB原料事業のVS1-VS5

3. **Supporting/Generic Layerの定義**
   - 親会社システムとの連携範囲
   - 自社独自で持つべき機能
