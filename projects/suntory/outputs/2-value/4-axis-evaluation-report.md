```
│                                                                 │
│  対立軸②: スピード vs 一貫性                                   │
│  ┌─────────────────────────────────────────────────────────┐   │

│  │   → デメリット: 統合VStrと同期されない                   │   │
│  │                                                           │   │
│  │ Option B: 統合基盤の承認プロセス経由（6週間）            │   │

│  │   → デメリット: 市場投入遅延                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│  ★ 二者択一を迫られる                                          │
│                                                                 │
│  対立軸③: 標準化 vs カスタマイズ                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 「80%共通 + 20%カスタマイズ」理論は魅力的だが、          │   │
```

#### フュージョン型の未解決問題

```
┌─────────────────────────────────────────────────────────────────┐
│              フュージョン型でも解決されない課題                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ❶ ガバナンスの二重構造                                        │
│     ├─ 統合VStr責任者 vs 事業部VStr責任者の権限衝突            │
│     ├─ 統合基盤への投資決定者が不明確                          │

│     ├─ 1本VStr: 調整不要（0コスト）                            │
│     ├─ 4本VStr: 6組の組み合わせ                                │

│                                                                 │
│  ❸ 責任の曖昧さによる品質リスク                                │

│     ├─ 「どちらも一部担当」= 「どちらも全責任を負わない」      │
```
