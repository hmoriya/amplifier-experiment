## 1. Bounded Context設計方針

### 1.1 設計原則

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Bounded Context設計原則                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【ユビキタス言語の境界】                                            │
│  ├─ 同一用語が異なる意味を持つ箇所で境界を設定                      │
│  ├─ ドメインエキスパートの言葉を尊重                                │
│  └─ 用語集（Glossary）をBC単位で管理                                │
│                                                                     │
│  【チーム境界との整合】                                              │
│  ├─ 1BC = 1チーム（または1チームが複数BC）                          │
│  ├─ チーム間のコミュニケーションコストを最小化                      │
│  └─ 組織構造（コンウェイの法則）を考慮                              │
│                                                                     │
│  【技術境界の独立性】                                                │
│  ├─ BCごとに技術スタックを選択可能                                  │
│  ├─ 独立したデプロイが可能                                          │
│  └─ データストアの独立性を確保                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 BC命名規則

| 形式 | 例 |
|------|-----|
| `{ドメイン略称}-{機能}` | `RD-WaterScience`, `MKT-Beverage` |
