# サントリー Value Stream設計 - 5軸パターン分析レポート

**作成日**: 2025-12-09
**分析対象**: サントリーホールディングス株式会社
**フレームワーク**: Parasol V5 Lite

---

## エグゼクティブサマリー

本レポートは、サントリーを題材に5つの軸パターンでValue Stream（VStr）設計を並列分析した結果をまとめたものである。各パターンの特徴、適用可能性、推奨組み合わせを提示する。

### 5軸パターン比較一覧

| 軸パターン | 主な特徴 | 適用推奨度 | 最適ケース |
|-----------|---------|-----------|-----------|
| **価値軸** | 「水と生きる」を全VSに貫通 | ★★★★☆ | 事業横断シナジー重視 |
| **事業部軸** | 酒類・飲料・健康食品を独立設計 | ★★★★☆ | 事業部オーナーシップ重視 |
| **ケイパビリティ軸** | 研究開発・ブランドMgt等を統合 | ★★★★☆ | 機能効率化・重複排除重視 |
| **フュージョン型** | 共通層(VS0,6,7)+事業層(VS1-5) | ★★★★☆ | バランス・段階適用重視 |
| **多段VStr** | 顧客+従業員+パートナーの3層 | ★★★★☆ | ESG・ステークホルダー重視 |

---

## 1. 価値軸パターン

### 1.1 概要

「水と生きる」を企業理念としてVS0に据え、全事業を顧客への価値創造フローという単一視点で捉えるアプローチ。

### 1.2 VS0-VS7 定義

| ステージ | 名称 | 定義 | 「水と生きる」との接続 |
|---------|------|------|----------------------|
| **VS0** | ビジョン・戦略立案 | 「水と生きる」を核とした企業理念、長期経営戦略 | **根幹**: 水資源保全・自然との共生が全ての出発点 |
| **VS1** | 市場機会発見 | 消費者ニーズ調査、健康トレンド分析 | 水・自然由来の価値への消費者期待を起点に |
| **VS2** | 製品開発 | 発酵技術、抽出技術、原料研究 | 水の品質・自然素材の活用が競争優位の源泉 |
| **VS3** | ブランディング | ブランド構築、広告宣伝 | 「水と生きる」ストーリーによる情緒的価値 |
| **VS4** | 販売・流通 | 調達、製造、物流、販売チャネル | 水源地保全から消費者への一貫した品質管理 |
| **VS5** | 品質保証 | 品質管理、顧客対応 | 水の安全性・自然由来の信頼を担保 |
| **VS6** | データ活用・DX | 顧客データ統合、需要予測 | 水資源モニタリング、消費行動分析 |
| **VS7** | オペレーション | 製造効率化、持続可能運営 | 水使用量削減、循環型オペレーション |

### 1.4 強み・弱み評価

| 項目 | 評価 | 説明 |

| **技術シナジー** | ◎ | 発酵・抽出・水処理の基盤技術共有 |
| **ブランド統一** | ◎ | 「自然由来」で一貫したストーリー |
| **事業部独立性との摩擦** | △ | 商習慣の違いで調整コスト発生 |
| **抽象度の高さ** | △ | 現場オペレーションとの乖離リスク |

## 2. 事業部軸パターン

### 2.1 概要

既存の事業部構造を尊重し、各事業部ごとに独立したVS1-VS7を定義。全社共通はVS0のみ。

### 2.2 全体構造

```

│                       全社戦略層 (VS0)                           │
│  ・グループ経営戦略  ・ブランドポートフォリオ  ・サステナビリティ    │
└─────────────────────────────────────────────────────────────────┘
              │                    │                    │
              ▼                    ▼                    ▼
    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐

    │   VS1-VS7       │  │   VS1-VS7       │  │   VS1-VS7       │
    │   独自定義       │  │   独自定義       │  │   独自定義       │
```

## 3. ケイパビリティ軸パターン

### 3.1 概要

事業横断で共通する「能力（ケーパビリティ）」を中心に設計。機能別組織やシェアードサービス化を志向。

### 3.2 ケイパビリティ層構造

```
┌──────────────────────────────────────────────────────────────────────┐
│                     【戦略系ケイパビリティ層】                          │
│    経営企画・戦略  │  M&A・アライアンス  │  サステナビリティ             │
│                         (VS0 対応)                                   │
└──────────────────────────────────────────────────────────────────────┘
                                 │
┌──────────────────────────────────────────────────────────────────────┐
│                     【創造系ケイパビリティ層】                          │
│  研究開発センター  │  商品開発  │  パッケージ開発                       │
│                         (VS2 対応)                                   │
└──────────────────────────────────────────────────────────────────────┘
                                 │
┌──────────────────────────────────────────────────────────────────────┐
│                     【市場系ケイパビリティ層】                          │
│  ブランドマネジメント  │  営業・チャネル  │  デジタルマーケ              │
│                     (VS1, VS3, VS4 対応)                             │
└──────────────────────────────────────────────────────────────────────┘
                                 │
┌──────────────────────────────────────────────────────────────────────┐
│                     【運営系ケイパビリティ層】                          │
│  サプライチェーン  │  製造・品質管理  │  自販機運営                      │
│                      (VS4, VS5 対応)                                 │

│   デジタル基盤  │  人事・組織  │  財務・経理  │  法務・コンプライアンス   │
│                      (VS6, VS7 対応)                                 │
└──────────────────────────────────────────────────────────────────────┘
          │                  │                  │                  │

```

### 3.3 統合判断マトリクス

| ケイパビリティ | 類似度 | 規模 | 判断 | 理由 |
|--------------|-------|------|------|------|
| 研究開発センター | 高 | 大 | **統合◎** | 発酵・醸造技術は共通、健康科学は全事業展開可能 |
| ブランドマネジメント | 高 | 大 | **統合◎** | 「やってみなはれ」「水と生きる」は全社統一 |
| サプライチェーン | 高 | 大 | **統合◎** | 製造・物流の最適化は全事業共通課題 |
| デジタル基盤 | 高 | 中 | **統合○** | データ・IT基盤は全社共通化でスケールメリット |
| 営業・チャネル | 中 | 大 | **部分統合** | 量販店・CVSは統合、飲食店・自販機は事業固有 |
| 酒類規制対応 | 低 | 中 | **事業固有** | 酒類事業固有の法規制対応 |
| 自販機運営 | 低 | 大 | **事業固有** | 飲料事業固有のオペレーション |

### 3.4 強み・弱み評価

| 項目 | 評価 | 説明 |
|------|------|------|
| **重複投資削減** | ◎ | 研究開発・IT等の一元化 |
| **専門性向上** | ◎ | 機能別に深い専門知識蓄積 |
| **シナジー創出** | ◎ | ビール製造技術→ノンアルコール応用 |
| **事業理解低下** | △ | 機能組織が現場から乖離 |
| **調整コスト増大** | △ | 事業と機能間の調整複雑化 |

## 4. フュージョン型パターン

### 4.1 概要

価値軸と事業部軸のハイブリッド。共通層（VS0, VS6, VS7）で骨格を作り、事業層（VS1-VS5）で事業固有の拡張を許容。

### 4.2 層構造

```
┌────────────────────────────────────────────────────────────────────┐
│                        【共通層】                                  │
│  ┌──────────────┬──────────────────┬────────────────────┐         │
│  │ VS0: 戦略    │ VS6: データ基盤  │ VS7: オペレーション │         │
│  │ ・ブランド   │ ・顧客統合DB     │ ・品質管理基準     │         │
│  │  ポートフォリオ│ ・AI/ML基盤     │ ・サステナビリティ  │         │
│  │ ・投資配分   │ ・分析BIツール   │ ・調達プラットフォーム│        │
│  └──────────────┴──────────────────┴────────────────────┘         │
└────────────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│【酒類事業】    │     │【飲料事業】    │     │【健康食品事業】│
│ VS1:発酵技術  │     │ VS1:健康機能  │     │ VS1:機能性研究│
│ VS2:成熟市場  │     │ VS2:ファミリー │     │ VS2:エビデンス│
│ VS3:酒販店網  │     │ VS3:自販機    │     │ VS3:EC直販   │
│ VS4:蒸溜所ツアー│     │ VS4:工場見学  │     │ VS4:健康相談 │
│ VS5:熟成管理  │     │ VS5:リサイクル │     │ VS5:継続支援 │
└───────────────┘     └───────────────┘     └───────────────┘
                              │
┌────────────────────────────────────────────────────────────────────┐
│                     【連携層】インターフェース                      │
│   顧客ID統合  │  ブランド横断プロモ  │  原料・資材共同調達          │
└────────────────────────────────────────────────────────────────────┘
```

### 4.3 共通化判断

| 機能領域 | 判断 | 理由 |
|---------|------|------|
| ブランド管理 | **共通化** | 「水と生きる」統一メッセージ |
| 品質管理基準 | **共通化** | 食品安全は全社統一必須 |
| 顧客データ基盤 | **共通化** | 一元管理でシナジー創出 |
| 発酵・蒸溜技術 | **固有** | 競争優位の源泉 |
| 自販機オペレーション | **固有** | 事業特性が大きく異なる |
| 機能性エビデンス研究 | **固有** | 顧客接点が独自 |

### 4.4 導入ステップ

| Phase | 期間 | 内容 |
|-------|------|------|
| Phase 1 | 3-6ヶ月 | 共通層の定義、ガバナンス体制構築 |
| Phase 2 | 6-9ヶ月 | 飲料事業でパイロット検証 |
| Phase 3 | 9-18ヶ月 | 酒類・健康食品へ展開 |
| Phase 4 | 18-24ヶ月 | 連携層の強化、クロスセル仕組み化 |

### 4.5 強み・弱み評価

| 項目 | 評価 | 説明 |
|------|------|------|
| **バランス** | ◎ | 酒類の伝統と飲料の革新性を両立 |
| **現実的** | ◎ | 既存の事業会社体制を活かせる |
| **シナジー** | ◎ | 「水」という共通資源で横断価値創出 |
| **設計複雑** | △ | 境界設計が困難 |
| **ガバナンス** | △ | 持株会社と事業会社の二重管理 |

## 5. 多段VStrパターン

### 5.1 概要

顧客VStrを主軸としつつ、従業員VStr・パートナーVStrを階層的に構成。ESG/ステークホルダー資本主義への対応。

### 5.2 3層VStr定義

#### 顧客向けVStr（VS: Value Stage）- Core Domain

| Stage | 状態 | サントリー固有定義 |
|-------|------|-------------------|
| VS0 | 潜在ニーズ | 「喉が渇いた」「リフレッシュしたい」 |
| VS1 | 認知到達 | サントリー製品の存在を知る |
| VS2 | 選好形成 | 「サントリーを選びたい」と思う |
| VS3 | 意思決定 | 「この製品を買う」と決める |
| VS4 | 価値獲得 | 製品を購入し手に入れる |
| VS5 | 価値実感 | 「美味しい」を実感 |
| VS6 | 価値継続 | リピート購入、定番化 |
| VS7 | 価値共創 | SNSでシェア、友人に推奨 |

#### 従業員向けVStr（ES: Employee Stage）- Supporting Domain

| Stage | 状態 | サントリー固有定義 |
|-------|------|-------------------|
| ES0 | 就職潜在 | 飲料・酒類業界で働く可能性を意識していない |
| ES1 | 認知到達 | 「やってみなはれ」文化を知る |
| ES2 | 選考意向 | サントリーで働きたいと思う |
| ES3 | 入社決定 | 内定承諾、入社決定 |
| ES4 | 活躍開始 | 配属後、業務で価値を発揮 |
| ES5 | 成長実感 | スキル向上、やりがいを実感 |
| ES6 | 定着継続 | 長期的にキャリア構築 |
| ES7 | 組織共創 | 後輩育成、文化継承 |

#### パートナー向けVStr（PS: Partner Stage）- Supporting Domain

| Stage | 状態 | サントリー固有定義 |
|-------|------|-------------------|
| PS0 | 協業潜在 | 取引可能性を認識していない |
| PS1 | 認知到達 | サントリーとの取引可能性を知る |
| PS2 | 提携意向 | 取引・協業したいと思う |
| PS3 | 契約決定 | 取引契約を締結 |
| PS4 | 協業開始 | 実際の取引を開始 |
| PS5 | 価値実感 | 協業から利益・成長を実感 |
| PS6 | 拡大継続 | 取引規模拡大 |
| PS7 | 戦略共創 | 戦略的パートナーとして共創 |

### 7.2 アーキテクト視点（zen-architect）

#### 各パターンの技術評価

| パターン | 技術的特性 | BC整合性 | 実装難易度 |
|----------|-----------|---------|-----------|
| **価値軸** | 顧客接点で統合、バックエンドは事業別 | 顧客ドメインが肥大化しやすい | 中 |
| **事業部軸** | サイロ化しやすいが境界明確 | 自然なBC分割、統合コスト高 | 低 |
| **ケイパビリティ軸** | 共通基盤化で効率的だが硬直化リスク | 技術BC優先で業務BCが歪む | 高 |
| **フュージョン型** | 価値軸+事業部軸の組み合わせ | 二重構造でBC設計が複雑 | 中〜高 |
| **多段VStr** | 階層的な価値の流れを可視化 | 依存関係の明示に有効 | 中 |

#### 推奨案への見解

**賛成（条件付き）**

フュージョン型+多段VStrは理にかなう。理由：
- 顧客接点（CX層）の統合と事業独自性の両立が可能
- 多段構造により「全社→事業部→機能」の価値連鎖が追跡可能
- サントリーの事業多角性に適合

**前提条件：**
- CX統合層のBCを薄く保つ（オーケストレーションに徹する）
- 事業部BCは自律性を維持し、CX層への過度な依存を避ける
- 多段VStrは2〜3階層に制限（複雑性抑制）

#### 技術的懸念事項

1. **統合層の肥大化**: CX統合層に業務ロジックが漏れ出すとモノリス回帰
2. **データ整合性**: 顧客マスタの正規化戦略が不明確だと事業間で不整合
3. **変更波及**: 多段構造は上流変更が下流に波及しやすい
4. **チーム境界**: フュージョン型はConwayの法則と衝突しやすい

#### 提言

1. **CX層はAnti-Corruption Layerとして設計** — 事業BCを直接結合せず変換層を挟む
2. **顧客IDの統合戦略を先行定義** — 技術選定より先にデータモデル合意
3. **PoC対象を1事業に絞る** — 酒類など成熟事業で検証後、他事業へ展開
4. **Value Streamのオーナーシップを明確化** — 技術アーキテクチャだけでなく組織設計とセットで進める

---

### 7.3 イノベーション視点（insight-synthesizer）

#### 見落とされているシナジー

**パターン間の隠れた接続点：**

- **ケイパビリティ軸 × 多段VStr**: 研究開発力を顧客・従業員・パートナー全層に開放する「オープンイノベーション・プラットフォーム」が未活用。例：農家パートナーの水質データ × 顧客の健康データ × 社内研究を融合した新価値創造

- **事業部軸 × フュージョン型の逆転**: 事業部を「独立」ではなく「実験場」として捉え直す。酒類で成功したクラフト戦略を健康食品で試す等、事業間の学習ループが弱い

- **価値軸の「受動性」**: 「水と生きる」が全VSを「貫通」するだけでは不十分。水を起点にした**能動的な価値生成エンジン**として再設計すべき

#### 「水と生きる」の革新的活用

**現状の問題**: 理念が「精神的シンボル」に留まり、オペレーショナルな価値創造に変換されていない

**革新提案 - 「Water Intelligence Platform」構想：**

```
水データ収集（工場・農場・自然環境）
    ↓
AI解析による水質・生態系インサイト
    ↓
3事業への差別化価値として還元：
  - 酒類：テロワール証明
  - 飲料：環境負荷可視化
  - 健康食品：原料トレーサビリティ
```

これにより「水と生きる」がコストセンター（CSR）からプロフィットセンター（差別化資産）に転換する

#### 競争環境での差別化

**脅威シナリオ：**
- キリンが「CSV経営」を深化させ、社会価値とビジネス価値の統合でサントリーを逆転
- アサヒがグローバルM&Aで規模優位を確立し、日本市場でも価格攻勢

**サントリー固有の差別化ポイント：**
- 「水」は他社が真似できない資産。キリン・アサヒは「ビール」「発酵」が軸であり、水への深いコミットメントはサントリー独自
- この差を「ブランドストーリー」ではなく「事業構造」に埋め込む必要がある

#### メタパターンの発見 - 「生態系型Value Stream」

5つのパターンを統合する上位概念として、**水循環のメタファーを事業構造そのものに適用**する：

```
[蒸発] 顧客からのフィードバック・データ上昇
    ↓
[雲]  ケイパビリティ層での知識凝縮
    ↓
[雨]  3事業への価値還元
    ↓
[川]  パートナー・サプライチェーンへの流通
    ↓
[海]  社会・環境へのインパクト → 再び[蒸発]
```

**効果:**
1. 「水と生きる」が事業アーキテクチャそのものになる
2. 循環構造により持続的イノベーションが内蔵される
3. 競合が「コンセプト」だけでは模倣不能な構造優位を確立

### 7.4 批判的視点（ambiguity-guardian）

#### 内在する緊張関係

**「水と生きる」の価値軸 vs 事業部軸の矛盾**
- 酒類事業における「水と生きる」は健康食品事業と根本的に異なる意味を持つ
- 酒類は水を消費し健康を害する可能性があり、この緊張を「貫通する価値」で覆い隠すことは表面的な統合に過ぎない

**ケイパビリティ統合 vs 事業独立性のトレードオフ**
- 研究開発を統合すれば効率は上がるが、酒類と健康食品では必要な専門性が異なる
- 「共通化」という名の妥協が各事業の競争力を削ぐリスク

**多段VStrの複雑性**
- 顧客・従業員・パートナーの3層を同時最適化しようとすることは、結局どれも中途半端になる「八方美人」パターンに陥りやすい

#### 推奨案への反論

**フュージョン型の落とし穴**
- 「共通層+事業層」は理論的には美しいが、実際には「どこまでが共通でどこからが事業固有か」の境界線争いが永続化する
- この調整コストが設計の利点を相殺する

**失敗シナリオ**
- 健康食品事業が急成長し、酒類が規制強化で縮小した場合、「フュージョン型」の共通層が酒類最適化のまま残り、足枷になる
- 多段VStrは、ステークホルダー間で利害対立が起きた際の優先順位決定メカニズムが不明確

**楽観的見積もり**
- 「3事業の相乗効果」は実証されていない前提
- 実際にはカニバリゼーション（健康志向の消費者が酒類から離脱）が起きている可能性

#### 無視されている前提

**組織政治の現実**
- 各事業部長は自部門のP/Lに責任を持つ
- 「共通層への投資」は誰のコストになるのか？この権力闘争が設計を骨抜きにする

**外部環境の脆弱性**
- 酒類規制の世界的強化トレンド（WHO推進）を軽視していないか
- 「水と生きる」は水資源危機が深刻化すれば、企業責任の追及対象になりうる

**暗黙の仮定**
- 「Value Stream設計が戦略実行を改善する」という前提自体が未検証
- 組織構造ではなく、意思決定プロセスや人材が本質的課題である可能性

#### 代替案・異論

**シンプルな事業部軸の再評価**
- 複雑なフュージョン型より、明確な事業部独立+最小限の共通サービス（財務・法務のみ）の方が、責任の所在が明確で実行力が高い可能性

**Value Stream設計への根本的疑問**
- そもそも「設計」という静的アプローチは、変化の激しい市場に適さないのでは
- 継続的な実験と適応のメカニズム構築の方が重要

---

### 7.5 実行可能性分析（analysis-engine）

#### 組織インパクト評価

**変革抵抗の発生箇所**
- **事業部間**: 酒類・飲料・健康食品それぞれが独自のプロセス・文化を持つ。共通層導入時に「うちは違う」という抵抗が発生
- **ミドルマネジメント**: VStr導入で既存の権限・予算配分が変わる層が最も抵抗
- **現場オペレーション**: 新システム・新プロセスへの習熟コスト

**対策**
- 飲料事業をPhase 1に選んだのは正解。成功事例を作り、他事業への説得材料とする
- 事業部長クラスのスポンサーを各事業に設置。彼らが「自分ごと化」しないと動かない
- 現場には早期に小さな成功体験を提供

**必要なスキル・人材**
- VStr設計経験者（外部採用または育成）
- データエンジニア・アーキテクト
- チェンジマネジメント専門家（内部育成困難、外部支援必須）

#### ロードマップ評価

**27ヶ月は「やや楽観的」**
- 大企業の変革は通常36-48ヶ月が現実的
- ただし、Phase 1（飲料事業・顧客VStr）を9ヶ月で成功させれば、後続は加速する

**Phase間の依存関係リスク**
- 共通層の設計品質がPhase 2以降のボトルネックになる可能性
- 推奨: Phase 1終了時に共通層の設計レビューを実施し、必要なら2ヶ月のバッファを設ける

**クイックウィン設計**
- Phase 1の最初の3ヶ月で「顧客データ統合ダッシュボード」を1つの営業チームに提供
- 「見える化」だけでも現場の支持を獲得できる

#### 成功の条件

**必須の経営コミットメント**
- 役員レベルのVStr推進オーナー任命
- 予算の3年間コミット（年度ごとの再承認では遅延する）
- 事業部KPIにVStr関連指標を組み込む

**KPI設計（例）**

| VStr | KPI |
|------|-----|
| 顧客VStr | 顧客生涯価値（LTV）向上率、リピート率 |
| 従業員VStr | エンゲージメントスコア、離職率 |
| パートナーVStr | サプライヤー協働による新商品開発数 |

**早期警戒指標**
- Phase 1の3ヶ月時点で現場フィードバックが「役に立たない」が50%超 → 設計見直し
- 6ヶ月時点でデータ統合率が目標の50%未達 → 技術リソース増強

#### 具体的アクションプラン

**明日から始める最初の一歩**
1. 飲料事業の顧客接点プロセスを1週間で可視化（As-Is）
2. 3事業の部長クラスから「変革推進チーム」を任命
3. 外部のVStr/チェンジマネジメント専門家との初回ミーティング設定

**最初の90日で達成すべきこと**
- 飲料事業の顧客VStrのTo-Be設計完了
- 共通層（顧客データ基盤）の技術アーキテクチャ決定
- 経営会議で27ヶ月ロードマップを正式承認
- パイロット営業チーム（1チーム10-20名）の選定と巻き込み

---

### 7.6 討議総括

#### エージェント間の合意点

| 論点 | 合意内容 |
|------|---------|
| **推奨案の妥当性** | フュージョン型+多段VStrは「理論的には妥当」だが条件付き |
| **Phase 1の重要性** | 飲料事業・顧客VStrの成功がすべての前提 |
| **組織政治** | 技術設計より組織・人の巻き込みが成否を分ける |
| **「水と生きる」の活用** | 現状は精神的シンボルに留まり、事業構造への埋め込みが不十分 |

#### エージェント間の対立点

| 論点 | zen-architect | insight-synthesizer | ambiguity-guardian | analysis-engine |
|------|--------------|---------------------|-------------------|----------------|
| **フュージョン型の複雑性** | 管理可能（ACL設計で対応） | 革新の阻害要因にならない | 調整コストが利点を相殺 | 実行上のリスクだが対処可能 |
| **多段VStr** | 2-3層に制限すべき | 3層を超えて生態系へ拡張 | 八方美人リスク | 段階導入で管理 |
| **27ヶ月ロードマップ** | 技術的には可能 | イノベーションには短い | 楽観的すぎる | やや楽観的だが加速可能 |
| **事業部独立性** | BC自律性を維持 | 実験場として活用 | 最小共通化で十分 | 成功事例で巻き込み |

#### 統合提言

4エージェントの討議を踏まえ、以下の統合提言を提示する：

1. **Phase 1に全力集中**: 飲料事業・顧客VStrの成功を最優先。ここで失敗すると全体が頓挫

2. **共通層は薄く始める**: Anti-Corruption Layer思想で、最小限の統合から開始

3. **「水と生きる」の事業化**: Water Intelligence Platformなど、理念を事業構造に埋め込む具体策を検討

4. **組織政治への対処**: 事業部長スポンサー制、予算3年コミット、クイックウィンの設計

5. **早期警戒システム**: 3ヶ月・6ヶ月時点でのチェックポイントを設置し、失敗を早期検知

6. **代替シナリオの準備**: フュージョン型が機能しない場合、シンプルな事業部軸+最小共通サービスへのフォールバックを用意
