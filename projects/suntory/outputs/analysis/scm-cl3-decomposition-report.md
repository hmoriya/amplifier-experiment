# サントリー SCM (Supply Chain Management) CL3分解レポート

**作成日**: 2025-12-12
**分析対象**: サントリーグループ サプライチェーン管理
**フレームワーク**: Parasol V5 Lite - Capability Decomposition
**要求**: SCMをCL3まで分解し、BC（Bounded Context）とサービスを抽出

---

## エグゼクティブサマリー

本レポートは、サントリーグループのSCM（サプライチェーン管理）領域をCL3（Capability Level 3）まで詳細分解し、Bounded Context（BC）とサービスを抽出した結果をまとめたものである。

### SCM分解サマリー

| レベル | 分解数 | 説明 |
|--------|--------|------|
| **CL1** | 1 | サプライチェーン管理（全体ケイパビリティ） |
| **CL2** | 4 | 調達統括 / 需給計画 / 物流管理 / 在庫管理 |
| **CL3** | 16 | 詳細オペレーション（4×4構造） |
| **BC** | 6 | 境界づけられたコンテキスト |
| **Service** | 24 | マイクロサービス候補 |

### 1.2 Value Streamとの対応

SCMは主に以下のValue Stage（VS）に対応する：

| VS | 名称 | SCM関連活動 |
|----|------|------------|
| VS2 | 製品開発 | 原材料調達設計、サプライヤ開発 |
| VS4 | 販売・流通 | 物流、配送、在庫配置 |
| VS5 | 品質保証 | 入荷検査、トレーサビリティ |
| VS7 | オペレーション | 製造連携、効率化 |

### 1.3 フュージョン軸とSCMの関係

5軸分析で推奨された**フュージョン型パターン**（共通層＋事業層のハイブリッド構造）において、SCMは重要な位置を占める。

#### SCMのCL2-CL3とフュージョン層のマッピング

| CL2 | CL3 | フュージョン層 | 根拠 |
|-----|-----|---------------|------|

| | 在庫計画 | **連携層** | 共通ロジック＋事業別パラメータ |

| | 輸送管理 | **共通層** | 輸送業者管理、コスト管理は共通 |
| | ラストマイル | **事業固有層** | 66万台自販機は飲料固有 |

| | 在庫最適化 | **連携層** | 共通アルゴリズム＋事業別制約 |
| | 鮮度管理 | **事業固有層** | 日配(飲料) vs 10年熟成(酒類)で全く異なる |
| | 在庫分析 | **共通層** | KPI定義・分析基盤は共通 |

#### フュージョン型実装における設計原則

| 原則 | 説明 | SCMでの適用 |
|------|------|------------|
| **共通層は薄く** | 全社共通機能は最小限に抑制 | 在庫可視化・分析基盤・サプライヤ管理のみ |
| **事業固有層は自由度高く** | 事業特性に応じた拡張を許容 | 熟成管理、自販機IoT、定期配送は各事業で独自実装 |
| **連携層で調整** | 事業間の接点を明示的に管理 | S&OP会議、原材料融通、物流網共有 |
| **APIファースト** | 層間はAPI契約で疎結合 | 共通層APIを事業固有層が呼び出す構造 |

#### traevo/AIコラミング等のフュージョン配置

| 実SCM施策 | フュージョン層 | 配置理由 |
|-----------|--------------|---------|
| **traevo（車両追跡）** | 共通層 | 全事業の輸送を統合管理 |
| **日立Lumada** | 共通層 | 60工場・300倉庫の統合トレーサビリティ |
| **AIコラミング** | 飲料固有層 | 66万台自販機は飲料事業固有 |
| **Project44** | 共通層 | グローバル輸送可視化は全社共通 |
| **天然水の森** | 共通層 | 水源涵養は「水と生きる」の共通基盤 |
| **Sedex** | 共通層 | サステナブル調達は全社ESG共通 |


#### 発見2: 「予測-配送-補充」三位一体サービス

66万台自販機対応として、3サービスを統合した自律型補充システム：
```
統合サービス: Autonomous Replenishment Orchestrator
├─ 自販機IoT → リアルタイム在庫
├─ AI需要予測 → 補充量算出
├─ 車両追跡 → 動的ルート再計算
└─ 積載最適化 → 35→45ケース/パレット連動
```

#### メタパターン
サントリーSCM施策の共通原理：
1. **可視化→予測→自動化の3段階進化**
2. **エコシステム連携**（外部プラットフォーム活用）
3. **物理効率とデジタル効率の同時追求**

### 11.5 曖昧性・矛盾分析（ambiguity-guardian）

#### 生産的矛盾（保持推奨）

| 矛盾 | 保持理由 | 期待される洞察 |
|------|---------|---------------|
| スケール抽象化 vs 実規模 | 設計段階では抽象的BCが正しい | スケールアウト戦略の判断材料 |
| ラストマイル抽象度 vs 66万台IoT | IoTは別アーキテクチャ層の問題 | エッジ層とBC層の分離必要性 |
| 国内中心 vs グローバル展開 | 意図的スコープ限定の可能性 | マルチリージョン設計パターン選択 |

#### 解消すべき矛盾

| 矛盾 | 解消しないリスク | 推奨解消方法 |
|------|-----------------|-------------|
| 2024年問題対応 | 輸送能力23%減で設計破綻 | キャパシティ制約管理サブドメイン追加 |
| Project44との責任分界 | Build vs Buy判断迷走 | サービス別「代替可能性」マトリクス作成 |
| サステナビリティ深度 | ESG法的義務化に未対応 | 独立CL3またはエンティティ追加 |

#### パラドックスと両立方法

| パラドックス | 両立方法 |
|-------------|---------|
| スケール vs 俊敏性 | 計画系（週次バッチ可）と実行系（イベント駆動）の分離 |
| 標準化 vs 事業固有性 | Generic層とSpecific層のレイヤード設計 |
| 予測精度 vs 予測不可能な変動 | 「予測外れ時の高速リカバリ能力」を別KPIとして設計 |
